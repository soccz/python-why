01 — 이번 주 이야기: 전화번호부의 한계

1990년대 전화번호부. 홍길동의 번호를 찾으려면? — 가, 가, 가... ㅎ, 호, 홍. 수천 페이지를 넘기며 이진 탐색. O(log n).
그런데 홍길동의 번호를 1밀리초에 찾고 싶다면?

아이디어 — "이름을 숫자로 바꾸고, 그 숫자 칸에 저장하자"

"홍길동" → 숫자 137. 137번 페이지에 번호 저장. 찾을 때도 "홍길동" → 137 → 즉시 찾음. O(1).

이 "이름 → 숫자" 변환이 해시 함수(hash function). 그리고 그 숫자를 인덱스로 쓰는 배열이 해시 테이블(hash table).

문제 — "김길동"도 137이 되면?

두 이름이 같은 숫자로 매핑되는 걸 충돌(collision)이라 부릅니다. 해시 테이블 설계의 90%는 "충돌을 어떻게 우아하게 처리할 것인가"에 관한 것.

더 큰 문제 — "공격자가 일부러 모든 이름이 137이 되게 만든다면?"

2011년부터 알려진 알고리즘 공격: 공격자가 해시 함수를 예측해 모든 입력이 충돌하게 만듦. 해시 테이블이 O(n) 연결 리스트로 degradation, 웹 서버가 DoS. Python·Java·PHP 모두 취약했습니다.

해법은 2013년 Python 3.4의 SipHash 도입과 hash randomization. 매 프로세스마다 무작위 seed로 해시 — 공격자가 예측 불가.

WHY — Python dict는 이 모든 것을 숨기고 단순해 보인다

d = {}; d["홍길동"] = "010-...". 두 줄 뒤에는:

  • SipHash-1-3으로 키 해시 계산 (DoS 방어).
  • 해시값으로 테이블 인덱스 계산 (& (size − 1), power of 2 트릭).
  • 충돌하면 perturbation probing으로 다음 슬롯 탐색.
  • load factor가 2/3 초과하면 테이블 리사이징 (O(n) 한 번, 하지만 amortized O(1)).
  • Python 3.6+에서는 compact dict 구조로 삽입 순서 유지 + 메모리 30~95% 절약.

이 모든 디테일을 이해하고 나면 — dict의 평균 O(1)이 당연한 게 아니라 한 세대에 걸친 공학의 결과임을 알게 됩니다.

dict의 마법: 해시로 슬롯 직행 O(1). 충돌 시 perturbation으로 상위 비트를 점진 활용해 새 슬롯 탐색 (CPython 고유 기법, CLRS에 없음).
해시 테이블 = 우편함. hash가 슬롯 번호를 알려줌 → 직행 O(1). 우연히 같은 슬롯을 받으면(충돌) 다음 빈 자리로 이동 → 조회도 같은 순서로 따라감 → 정확히 발견.

02 — 해시 함수: 무엇이 좋은 해시인가

초급

해시 함수의 정의

해시 함수 h(x): 임의의 데이터 x를 고정 크기 정수로 변환하는 함수.

h: 임의 데이터 → ℤ ∩ [0, m)  (m = 테이블 크기)

예: "홍길동" → 137, "abc" → 42, 42 → 42. 같은 입력은 항상 같은 출력.

중급

좋은 해시 함수의 4가지 속성

  1. 결정론적 (Deterministic): 같은 입력 → 같은 출력. 무작위가 아님. (단, 다른 프로세스 간에는 다를 수 있음 — 해시 랜덤화.)
  2. 균등성 (Uniformity): 출력이 [0, m) 범위에 고르게 분포. 어떤 슬롯도 편향되지 않음.
  3. 빠름: O(key 크기). 키 하나 해싱에 O(1) 정도 바라야 dict 전체가 O(1).
  4. Avalanche effect (눈사태 효과): 입력 1비트 변화 → 출력의 약 절반 비트 변화. 연관된 입력들이 같은 슬롯에 뭉치지 않도록.
고급

Python의 hash() — 어떻게 작동하는가

python_hash_examples.py
hash(42)              # 42 (작은 int는 자기 자신)
hash(-1)              # -2 (Python 내부적으로 -1은 예약값)
hash(2**100)          # Python은 큰 int도 해시 가능 (정수 나머지 연산)
hash("hello")         # 프로세스마다 다름! (SipHash randomization)
hash((1, 2, 3))       # tuple — 원소들 해시 조합

hash([1, 2, 3])       # TypeError: unhashable type: 'list'
                      # list는 mutable → hashable 아님
  • int: 작은 값은 자기 자신, 큰 값은 x mod (2^61 − 1) (Mersenne prime).
  • str, bytes: SipHash (Python 3.4: SipHash-2-4 → 3.12: SipHash-1-3). 프로세스마다 seed 다름.
  • tuple, frozenset: 원소 해시들을 XOR·combine. 재귀적.
  • float: 정수와 일관되게 처리 — hash(3.0) == hash(3).
⚠️ hash(-1)은 왜 -2인가

CPython 내부에서 -1은 "해시 계산 오류"를 나타내는 sentinel 값. 따라서 실제 해시값이 -1이면 -2로 바꿈. 그래서 hash(-1) == hash(-2) == -2.

이게 사용자 레벨에서 뭔가 잘못될 일은 거의 없지만, 커스텀 __hash__를 만들 때 -1은 절대 반환하지 말 것.

03 — 충돌과 해결 전략: Chaining vs Open Addressing

중급

생일 역설 — 충돌은 필연이다

해시 테이블 크기 m, 원소 n개. 모두 다른 슬롯에 들어갈 확률은 생일 역설로 계산 가능:

P(충돌 없음) ≈ e^(−n²/2m)

n = √m만 돼도 P(충돌) > 50%. 예: m=365(1년), n=23(23명)이면 P(충돌) ≈ 50% (생일 역설). 해시 테이블에서 충돌은 드문 일이 아니라 일상. 대응이 필수.

고급

전략 1 — Chaining (CLRS §11.2)

각 슬롯이 linked list(또는 다른 자료구조). 충돌한 원소들이 같은 슬롯 리스트에 쌓임.

Slot 0: [A] → [B] → [C] (세 개가 slot 0에 충돌)
Slot 1: [D]
Slot 2: (비어있음)
Slot 3: [E] → [F]
  • 장점: 구현 단순. Load factor > 1도 가능.
  • 단점: linked list overhead (포인터, malloc, 캐시 미스).
  • 사용처: Java HashMap, C++ std::unordered_map.

평균 lookup: O(1 + α), α = n/m (load factor). α가 O(1)이면 O(1).

고급

전략 2 — Open Addressing (CLRS §11.4)

모든 원소를 해시 테이블 자체에 저장. 충돌 시 다른 슬롯을 probe(탐색).

Slot 0: A (hash → 0)
Slot 1: B (hash → 1)
Slot 2: C (hash → 0, probe 1 충돌, probe 2로)
Slot 3: (비어있음)
Slot 4: D (hash → 4)
  • 장점: 포인터 없음 → 캐시 친화적. Linked list 오버헤드 없음.
  • 단점: Load factor < 1이어야 (꽉 차면 무한 루프). 리사이징 비용.
  • 사용처: CPython dict, Go map, Rust HashMap.

평균 lookup: 1/(1 − α) (α = load factor). α=0.5면 평균 2회 probe, α=0.66이면 약 3회.

WHY — 왜 CPython은 open addressing을 선택했나
  1. 캐시 지역성: 모든 데이터가 연속 메모리. CPU 캐시가 효율적.
  2. 메모리 효율: linked list 포인터 overhead 없음.
  3. 예측 가능성: malloc 호출 없음 (chaining은 매 충돌마다 malloc).
  4. 단순성: 포인터 관리 코드 없음 → 버그 적음.

단점(리사이징 비용)은 amortized O(1)로 분산 (§07). 트레이드오프가 분명히 open addressing이 우위.

04 — Open Addressing 상세: 3가지 probing 기법

고급

Linear Probing — 가장 단순

충돌 시 "바로 다음 슬롯"으로 이동. 빈 슬롯 나올 때까지 선형 탐색.

probe(k, i) = (h(k) + i) mod m,   i = 0, 1, 2, ...
  • 장점: 캐시 친화 (인접 슬롯 연속 접근).
  • 단점: primary clustering — 한 곳에서 충돌이 시작되면 인접 슬롯들이 계속 채워져 긴 "군집" 형성. 긴 군집은 더 긴 probe를 유발.
고급

Quadratic Probing — 거리 증가

probe(k, i) = (h(k) + c₁·i + c₂·i²) mod m

거리가 1, 4, 9, 16, ... 점점 커짐. Primary clustering 줄임.

단점: secondary clustering — 같은 초기 해시값이면 같은 probe sequence. 일부 슬롯은 영원히 탐색 안 되기도 (m이 소수가 아니면).

고급

Double Hashing — 두 개의 해시 함수

probe(k, i) = (h₁(k) + i · h₂(k)) mod m

Probe sequence가 키마다 다름 (h₂(k)에 의존). Primary/secondary clustering 모두 해결.

단점: 두 해시 함수 필요. 캐시 불친화 (멀리 점프).

📖 원서 인용 — CLRS 4판, §11.4 "Open addressing"

"The probe sequence for a key k is the sequence of slots <h(k,0), h(k,1), ..., h(k,m-1)>. Every key must produce a permutation of the slot numbers, so that every slot is eventually considered."

Probe sequence가 모든 슬롯을 결국 방문해야 한다는 조건이 핵심. Linear/quadratic은 m이 특정 조건일 때만 이 성질 만족.

WHY — 이 3가지 중 CPython은 무엇을 쓰는가

셋 다 아니다. CPython은 교과서에 없는 영리한 4번째 기법을 씁니다.
Perturbation Probing — §05에서.

05 — CPython의 Perturbation Probing (CLRS에 없는 기법)

전공자

아이디어 — 해시의 상위 비트를 "섞어" 사용

표준 probing은 hash(k)의 하위 비트 (& (m-1))만 사용. 상위 비트 정보가 버려짐. CPython의 기법: probe마다 해시값 자체를 점진적으로 섞어 새로운 probe 위치 생성.

CPython dictobject.c — probe 루프 (개념 의사코드)
i = hash & mask              # 첫 probe
perturb = hash
while slot[i] != EMPTY and slot[i].hash != hash:
    perturb >>= PERTURB_SHIFT    # PERTURB_SHIFT = 5
    i = (i * 5 + perturb + 1) & mask

핵심: perturb가 매 iteration마다 해시의 상위 5비트를 흘려보냄. 처음 probe는 하위 비트만 반영, 그 다음부터 점점 상위 비트의 정보도 포함.

i * 5 + perturb + 1: linear probing처럼 보이지만 perturb 덕에 키마다 독특한 probe sequence. Double hashing의 효과 + 캐시 친화.

전공자

왜 i * 5인가 — Mersenne 추측의 영리함

테이블 크기 m은 항상 2의 거듭제곱. mask = m − 1이라 & mask가 mod m와 같음.

5는 2의 거듭제곱 modulo에서 특별한 속성을 가짐:

  • 모든 m에 대해 i → i * 5 + 1 mod m은 m의 모든 값을 순회.
  • 실제로 (5·i + c) mod 2^k의 순열 구조가 증명됨.
  • perturb의 non-zero 기여 덕에 clustering 최소화.

이게 경험적으로 매우 우수한 성능을 보여 CPython에서 채택 (Python 2.3 시대, ~2001–2002, Tim Peters 설계).

📖 원문 — dictobject.c 주석

"The probe sequence is computed by: i = (i*5 + perturb + 1) & mask. ... This gives excellent collision statistics. It's cheap (just a few operations per probe) and it guarantees that any probe can eventually reach every slot."

— CPython Objects/dictobject.c 주석 (Tim Peters 원 설계, Raymond Hettinger 유지).

전공자

시각화 — 어떻게 probe 순서가 퍼지나

m = 8 (mask = 7)일 때 hash = 0x12345678에 대한 probe 순서:

perturb = 0x12345678, i = hash & 7 = 0
probe 1: i=0
perturb >>= 5 → 0x0091A2B3, i = (0*5 + 0x91A2B3 + 1) & 7 = 4
probe 2: i=4
perturb >>= 5 → 0x0048D159, i = (4*5 + 0x48D159 + 1) & 7 = 2
probe 3: i=2
...

Linear probing처럼 순차적이지 않고, 전체 테이블을 예측 불가 패턴으로 순회. Clustering 효과적으로 깸.

06 — SipHash-1-3과 해시 랜덤화: DoS 방어

고급

2011년의 악몽 — Hash Collision DoS

2011년 말, Alexander Klink와 Julian Wälde가 발표: "Hash DoS".

  • Python, PHP, Java, Ruby의 해시 함수가 결정론적 + 공개.
  • 공격자가 같은 해시값을 갖는 문자열 수천 개를 만들어 POST 요청.
  • 서버 해시 테이블이 모두 한 슬롯에 충돌 → O(n) linked list로 전락.
  • Django 요청 처리가 n=1000에서 수 초 → DoS.

CVE-2012-1150 (Python), CVE-2011-4858 (Java) 등이 발급. 주요 언어들이 비상 대응.

고급

해법 1 — Hash Randomization

매 프로세스 시작 시 무작위 seed를 해시 함수에 주입. 공격자가 해시값을 예측 불가.

hash_randomization_demo.sh
# 프로세스마다 hash("hello")가 다르다
$ python3 -c "print(hash('hello'))"
-8183367247218815361

$ python3 -c "print(hash('hello'))"
-2847113481593718475

# PYTHONHASHSEED로 고정 가능 (재현 가능한 테스트용)
$ PYTHONHASHSEED=0 python3 -c "print(hash('hello'))"
4472863470232598073
$ PYTHONHASHSEED=0 python3 -c "print(hash('hello'))"
4472863470232598073

Python 3.3(2012)부터 기본 활성화 (3.2에서는 -R 옵션으로 opt-in), Python 3.4(2014)에서 SipHash-2-4로 해시 함수 자체도 강화. 정수는 randomization 영향 X (수학적 성질 유지). 문자열·bytes만.

전공자

해법 2 — SipHash 채택

Jean-Philippe Aumasson과 Daniel J. Bernstein이 2012년에 발표한 SipHash. PRF(Pseudo-Random Function) 기반 — 키 없이 역산 불가능.

  • SipHash-2-4 (원조, Python 3.4 도입): 2 round 압축 + 4 round finalization.
  • SipHash-1-3 (Python 3.12부터): 1 round + 3 round. 약간 빠르고 여전히 안전.
SipHash(key, data) = 128-bit key로 파라미터화된 64-bit PRF

Python은 프로세스 시작 시 256-bit 무작위 seed 생성 → SipHash key로 사용. str/bytes 해시에 적용. 공격자가 서버 메모리 읽지 않는 한 충돌 입력 생성 불가.

📖 원서 인용 — PEP 456 "Secure and interchangeable hash algorithm"

"This PEP proposes SipHash as default string and bytes hash algorithm to properly fix Python's hash vulnerability... SipHash is a cryptographic pseudorandom function with a 128-bit key and 64-bit output."

— PEP 456, Christian Heimes (2013). Python 3.4에서 구현.

⚠️ 성능과 보안의 트레이드오프

SipHash는 간단한 해시(FNV, DJB2 등)보다 2~3배 느림. 그래도 안전성을 위해 쓰는 것.

Python 3.12에서 SipHash-2-4 → SipHash-1-3으로 전환한 이유: "3회 finalization이면 여전히 공격자 저항 충분, 약 20% 빠름".

07 — Load Factor와 Rehashing

고급

Load Factor α = n / m

n: 저장된 원소 수. m: 테이블 크기. α가 너무 크면 충돌 폭증.

α평균 probe 수 (open addressing)
0.5~2
0.66 (CPython threshold)~3
0.75~4
0.9~10
0.99~100

공식: 평균 probe ≈ 1/(1 − α). α = 0.66에서 ~3회 — 매우 빠름. α = 0.9를 넘으면 성능 저하가 급격.

고급

CPython의 Resize 정책

  • Load factor 2/3 초과 시 resize.
  • 작은 dict (원소 < 50,000): 크기를 4배로.
  • 큰 dict: 2배로.
  • 모든 기존 원소를 새 테이블로 rehash (해시값 자체는 저장되어 있음, 마스크만 다시 적용).

Resize 자체는 O(n), 하지만 amortized O(1) per operation (Algo 08의 amortized 분석 동일 논리).

왜 작은 dict는 4배로?

작은 dict는 빠르게 커지는 경우가 많음. 2배로 자주 resize하기보다 4배로 한 번에 resize가 효율적. 큰 dict는 이미 메모리를 많이 쓰고 있어 4배는 과도 → 2배.

08 — Compact Dict: Raymond Hettinger의 혁신 (Python 3.6)

전공자

전통 dict 구조 (~Python 3.5)

전통적으로 dict 테이블의 각 슬롯은 (hash, key, value) 삼중체를 직접 저장:

전통 dict (Python 3.5 이하):
Slot 0: (hash=0x123, key="a", value=1)
Slot 1: EMPTY
Slot 2: EMPTY
Slot 3: (hash=0x456, key="b", value=2)
Slot 4: EMPTY
...

각 슬롯 = 24바이트 (8 hash + 8 key + 8 value)
Empty 슬롯도 24바이트 메모리 차지

Load factor 2/3이면 33% 메모리가 empty로 낭비됨. 큰 dict일수록 이 낭비가 큼.

문제 2: 삽입 순서 정보 없음. 순회 순서가 해시 순서 — 사용자 입장에서 예측 불가.

전공자

Raymond Hettinger의 혁신 (Python 3.6, 2016)

테이블을 두 부분으로 분리:

  1. Indices 배열: 작은 정수 (entries 배열로의 index). 슬롯이 많지만 각 슬롯이 1~8바이트.
  2. Entries 배열: (hash, key, value) 삼중체의 조밀한 배열. 삽입 순서대로 저장. n만큼만 공간 차지.
Compact dict (Python 3.6+):

Indices: [-1, -1, 0, -1, -1, 1, -1, -1]  (-1 = empty)
Entries: [
   (hash=0x123, key="a", value=1),
   (hash=0x456, key="b", value=2)
]

Lookup 과정:

  1. hash("a") 계산 → 2.
  2. Indices[2] = 0.
  3. Entries[0] 읽음 → ("a", 1). 키 일치 → 값 반환.
전공자

이점 — 메모리 절약 30-95%

  • 빈 슬롯의 크기: 전통 24바이트 → compact 1~8바이트. 33% 낭비가 1.4~11%로 줄어듦.
  • Index 크기 동적 조정: 테이블 크기 < 256이면 int8(1바이트). < 65536이면 int16(2바이트). 더 크면 int32 또는 int64.
  • 삽입 순서 유지: entries가 순서대로라 for k in d가 삽입 순서.
메모리 절약 = 30% (꽉 찬 dict) ~ 95% (거의 빈 dict)

특히 작은 dict를 많이 만드는 프로그램에서 효과 큼. CPython 자체의 object __dict__, 모듈 namespace 등이 모두 dict → 인터프리터 메모리 사용량 대폭 감소.

📖 원문 — Raymond Hettinger, python-dev 2012

"The current memory layout for dictionaries is unnecessarily inefficient. It has a sparse table of 24-byte entries containing the hash value, key pointer, and value pointer. Instead, the 24-byte entries should be stored in a dense table... with the hash table of indices being much smaller. This would reduce the memory footprint of dicts significantly."

— Raymond Hettinger, "More compact dictionaries" 제안서. 2012년 제안, 2016년 Python 3.6 구현.

WHY — 부작용: Python 3.7은 "dict는 삽입 순서 유지"를 언어 스펙에 추가

3.6의 compact dict는 구현 상세로 도입, 순서 유지는 "우연한 부수 효과". 커뮤니티 토론 후 3.7에서 언어 스펙에 보장하기로 결정.

이제 OrderedDict는 대부분의 용도에서 불필요 (move_to_end, 동등성 비교 시 순서 포함 같은 특수 기능만 남음). 다만 커뮤니티 합의 요지: dict가 insertion-ordered라고 해서 list를 대체하는 건 아니다 — 키 lookup 자료구조와 시퀀스는 다르다.

10 — Hashable 객체: __hash__와 __eq__

중급

Hashable의 세 조건

  1. __hash__ 정의 — 정수 반환.
  2. __eq__ 정의.
  3. 불변(immutable) — 해시값이 객체 생명 동안 변하지 않아야.
규칙: a == b ⟹ hash(a) == hash(b). (역은 아님.)

이게 깨지면 a == b인데 dict에서 찾을 때 다른 슬롯을 봐서 찾지 못하는 버그 발생.

고급

기본 hashable / 기본 unhashable

HashableUnhashable

str, bytesdict
tuple (원소들이 hashable일 때)set

None, Ellipsisnumpy array

원리: mutable 객체는 hashable 아님. 내용이 바뀌면 해시도 바뀌어야 하는데, dict는 해시가 고정이라 가정.

list를 dict key로 쓰고 싶다면 — tuple로 변환.

전공자

커스텀 클래스에 __hash__ 구현

custom_hash.py
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __eq__(self, other):
        if not isinstance(other, Point):
            return NotImplemented
        return self.x == other.x and self.y == other.y

    def __hash__(self):
        return hash((self.x, self.y))   # tuple 해시로 위임


p1 = Point(1, 2)
p2 = Point(1, 2)
d = {p1: "hello"}
print(d[p2])   # "hello" — p1과 p2가 == 이므로 같은 슬롯

편의 도구: @dataclass(frozen=True)는 자동으로 __hash__와 __eq__ 생성. 또한 frozen이라 수정 불가 (진짜 immutable 보장).

dataclass_hashable.py
from dataclasses import dataclass

@dataclass(frozen=True)
class Point:
    x: int
    y: int

# __eq__, __hash__ 자동 생성. 변경 시도 → FrozenInstanceError.
⚠️ __eq__만 override하면 __hash__가 None이 됨

Python의 규칙: __eq__ 구현 시 __hash__를 명시하지 않으면 자동으로 None. → 객체가 unhashable.

이유: "a == b but hash(a) != hash(b)"의 버그를 사전 방지. 직접 둘 다 구현하거나, @dataclass(frozen=True) 또는 @total_ordering 사용.

13 — 실습: 해시 테이블 직접 구현 + 충돌 시각화

고급

간단한 open addressing dict

mini_dict.py
class MiniDict:
    _EMPTY = object()
    _DELETED = object()

    def __init__(self, capacity=8):
        self._cap = capacity
        self._slots = [(self._EMPTY, None, None)] * capacity
        self._size = 0

    def _probe(self, key):
        h = hash(key)
        mask = self._cap - 1
        i = h & mask
        perturb = h
        while True:
            k, _, slot_hash = self._slots[i]
            if k is self._EMPTY or (k is not self._DELETED and slot_hash == h and k == key):
                return i
            perturb = perturb >> 5   # PERTURB_SHIFT = 5
            i = (i * 5 + perturb + 1) & mask

    def __setitem__(self, key, value):
        if self._size >= self._cap * 2 // 3:
            self._resize(self._cap * 4)
        i = self._probe(key)
        k, _, _ = self._slots[i]
        if k is self._EMPTY:
            self._size += 1
        self._slots[i] = (key, value, hash(key))

    def __getitem__(self, key):
        i = self._probe(key)
        k, v, _ = self._slots[i]
        if k is self._EMPTY:
            raise KeyError(key)
        return v

    def _resize(self, new_cap):
        old = self._slots
        self._cap = new_cap
        self._slots = [(self._EMPTY, None, None)] * new_cap
        self._size = 0
        for k, v, _ in old:
            if k is not self._EMPTY and k is not self._DELETED:
                self[k] = v


# 테스트
d = MiniDict()
for i in range(20):
    d[f"key{i}"] = i * 10

for i in range(20):
    assert d[f"key{i}"] == i * 10
print("MiniDict works ✓")

실습 2 — 충돌 분포 시각화

collision_histogram.py
import collections

def collision_stats(words, m):
    """n개 단어를 m 크기 테이블에 해시했을 때 슬롯별 충돌 수."""
    counts = collections.Counter()
    for w in words:
        counts[hash(w) & (m - 1)] += 1
    return counts


import string, random
random.seed(42)
words = [''.join(random.choices(string.ascii_lowercase, k=5)) for _ in range(1000)]

m = 2048   # load factor ~50%
stats = collision_stats(words, m)
print(f"슬롯당 평균: {sum(stats.values()) / len(stats):.2f}")
print(f"최대 충돌: {max(stats.values())}")
print(f"빈 슬롯 수: {m - len(stats)}")

실행하면 대부분의 슬롯이 0~2개, 최대 충돌 3~4개 정도. SipHash의 균등 분포 덕.

WHY 문제 5

def dedupe(items):

dict는 "key의 해시값이 생명 동안 고정"이라고 가정. mutable 객체는 내용 변경 → 해시 변경 → dict에서 다시 찾지 못하게 됨. 그래서 hashable 속성이 boolean 결정.

BUILD 문제 10

단어 빈도 카운터를 두 가지 방법으로 구현 (defaultdict, Counter). 결과 비교.

풀이 펼치기
word_freq.py
from collections import defaultdict, Counter

text = "the quick brown fox jumps over the lazy dog the"

# 방법 1: defaultdict
d = defaultdict(int)
for w in text.split():
    d[w] += 1

# 방법 2: Counter (훨씬 간결)
c = Counter(text.split())

print(dict(d))          # {'the': 3, 'quick': 1, ...}
print(dict(c))          # 같은 결과
print(c.most_common(3))   # [('the', 3), ...]
Counter의 장점

most_common, 산술 연산(+, -), elements() 등 빈도 관련 편의 메서드. 대부분의 경우 Counter가 우선.

AI 문제 11

AI가 "두 dict 병합"에 {**d1, **d2}와 d1 | d2(3.9+) 중 더 최신 방법을 제안했다. 차이는?

풀이 펼치기
답

기능적으로 동일 (둘 다 새 dict 생성, 중복 key는 오른쪽 우선). |가 더 읽기 쉽고 Python 3.9+ 표준.

In-place 병합은 d1.update(d2) 또는 d1 |= d2. 이건 새 dict 안 만들고 d1 수정.

WHY 문제 12

Python 3.7에서 "dict는 삽입 순서 유지"를 언어 스펙으로 보장한 영향은? (이전까지는 3.6의 구현 상세였음)

풀이 펼치기
영향
  1. OrderedDict 사용 감소: 대부분의 용도에서 그냥 dict로 충분.
  2. JSON 직렬화 순서 예측 가능: json.dumps(d)가 dict 순서 보존.

15 — 정리 + 용어 사전 + 다음 주 맛보기

핵심 정리

🎯

해시 함수의 4속성

결정론적·균등성·빠름·avalanche. Python은 SipHash-1-3으로 DoS 방어.

💥

충돌 해결

Chaining (Java) vs Open addressing (Python). 캐시 친화로 open addressing 우세.

🌀

Perturbation Probing

CPython 고유. i = (i·5 + perturb + 1) & mask. 상위 비트 점진 활용.

🛡️

SipHash 방어

2012년 hash DoS 공격 대응. 프로세스마다 무작위 seed. str/bytes만 영향.

📦

Compact Dict (3.6)

Raymond Hettinger의 혁신. Indices + Entries 분리. 30-95% 메모리 절약. 순서 유지는 부산물.

🔑

Hashable 규칙

a==b ⟹ hash(a)==hash(b). mutable은 unhashable. __eq__ override 시 __hash__도 필요.

이진 탐색 트리 (BST) → AVL → Red-Black

dict는 O(1) 평균이지만 정렬된 순회나 범위 조회가 불가능. 이때 필요한 자료구조 — 균형 이진 탐색 트리.

  • BST 기본 — 삽입·삭제·조회 + 불균형 문제
  • AVL 트리 — 엄격한 균형 (높이 차 ≤ 1)
  • Red-Black 트리 — 느슨한 균형 (실전 표준, Java TreeMap, C++ std::map)
  • 왜 Python은 내장 BST가 없는가 — sortedcontainers의 skip list 대안
  • CLRS §12 + §13 완전 정복
