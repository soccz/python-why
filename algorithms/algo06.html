ALGO 06

O(n log n) 정렬들
하한에 닿는 세 거인과, 그것을 우회하는 변종들

Algo 05에서 본 Ω(n log n) 하한 — 비교 기반 정렬의 근본적 한계. 이번 주는 이 하한에 정확히 닿는 세 알고리즘(병합·힙·퀵)과, 비교가 아닌 정보를 써서 하한을 우회하는 세 알고리즘(counting·radix·bucket)을 봅니다.

01 — 이번 주 이야기: 1조 원소와 세 가지 전략

데이터 엔지니어 A씨는 1조(10¹²)개의 로그 레코드를 정렬해야 합니다. O(n²) 정렬이면 10²⁴ 연산 — 지구의 모든 CPU를 다 써도 수조 년. 답은 O(n log n) 외엔 없습니다.
그런데 O(n log n) 정렬도 세 가지가 있습니다. 어느 것을 고를까요?

병합 정렬 — 안전하지만 메모리 2배

"항상 Θ(n log n) 보장, 안정 정렬." 하지만 보조 공간 O(n)이 필요합니다. 1조 레코드가 100GB라면 200GB 메모리를 써야 함. 디스크 기반 외부 정렬(external sort)에 최적 — 대규모 데이터베이스의 표준.

힙 정렬 — 보장되는데 아무도 안 씀

"Θ(n log n) 보장 + 제자리(O(1) 보조)." 완벽해 보이는데, 표준 라이브러리 어디도 힙 정렬을 기본으로 안 씁니다. 이유는 캐시 친화도 — 힙의 자식 노드 접근 패턴이 메모리를 건너뛰어 캐시 미스가 잦습니다. 실측에선 퀵소트보다 2~3배 느립니다.

퀵소트 — 최악 Θ(n²)인데 모두가 사랑함

"평균 Θ(n log n), 최악 Θ(n²), 제자리." 최악 케이스가 있는데도 — 상수가 가장 작고 캐시 친화적이라 실측에서 가장 빠릅니다. 무작위 피벗(Algo 04 §06) + introsort 트릭으로 최악 케이스를 무력화하면, 더할 나위 없는 선택.

WHY — 세 거인의 공존 이유

같은 Θ(n log n)이지만 세 정렬은 전혀 다른 트레이드오프 프로파일을 갖습니다.

속성병합힙퀵
최악 시간Θ(n log n)Θ(n log n)Θ(n²) ⚠️
보조 공간Θ(n) ⚠️Θ(1) ✓Θ(log n) ✓

실측 속도중느림 (캐시)가장 빠름 ✓

이번 주는 "왜 이 트레이드오프가 발생하는지"를 내부 구조에서 뜯어 봅니다.

02 — 세 거인 개요: 하한에 닿는 법

중급

같은 Θ(n log n)의 세 가지 기원

Algo 05 §07에서 본 결정 트리 논증: 비교 기반 정렬은 최소 Θ(n log n)번의 비교가 필요. 세 거인은 모두 이 하한에 정확히 닿지만, 그 방법이 서로 다릅니다.

알고리즘기원n log n이 나오는 이유
병합 정렬 재귀적 반분 재귀 깊이 log n × 각 깊이의 병합 O(n) (Algo 03)
힙 정렬 힙의 log n 깊이 n번의 extract-max × 각 O(log n)
퀵소트 기댓값 분석 무작위 피벗의 평균 분할 비율 × log n 깊이 (Algo 04)

이 세 가지는 "log n을 얻는 세 가지 방식"의 대표입니다 — 분할 정복, 트리 자료구조, 확률 분석. 다른 O(n log n) 알고리즘들(예: 이진 탐색 트리 정렬, FFT 기반 정렬)도 이 중 하나의 변종.

📖 원서 인용 — CLRS 4판, §6 도입부

"Like merge sort, but unlike insertion sort, heapsort's running time is O(n lg n). Like insertion sort, but unlike merge sort, heapsort sorts in place: only a constant number of array elements are stored outside the input array at any time."

— CLRS 4th ed., §6 "Heapsort"

03 — 병합 정렬 재방문: 안정성과 공간 트레이드오프

중급

Algo 03에서 본 것 요약

Algo 03 §04에서 병합 정렬의 정의·증명·복잡도를 자세히 다뤘습니다. 여기선 세 거인 비교 관점에서 요점만.

  • Θ(n log n) — 최선·평균·최악 모두 같음 (적응성 없음)
  • Θ(n) 보조 공간 — 병합에 임시 배열 필요
  • 안정 정렬 — <=로 비교하면 같은 키의 원소 순서 유지
  • 순차 접근 — 캐시 친화적
고급

Bottom-up 병합 정렬 — 재귀 없는 버전

보통 소개하는 top-down 병합 정렬은 재귀를 씁니다. bottom-up은 같은 결과를 반복문으로 얻습니다.

merge_sort_bottom_up.py
def merge_sort_bu(A):
    n = len(A)
    width = 1
    while width < n:
        for i in range(0, n, 2 * width):
            mid = min(i + width, n)
            right = min(i + 2 * width, n)
            A[i:right] = _merge(A[i:mid], A[mid:right])
        width *= 2
    return A


def _merge(L, R):
    out, i, j = [], 0, 0
    while i < len(L) and j < len(R):
        if L[i] <= R[j]:
            out.append(L[i]); i += 1
        else:
            out.append(R[j]); j += 1
    out.extend(L[i:]); out.extend(R[j:])
    return out

크기 1인 구간들부터 시작해 2, 4, 8, …로 확장하며 병합. 재귀 호출 스택 없음 — 매우 큰 배열에서도 RecursionError 걱정 없음. Timsort는 이 bottom-up 접근을 더 정교하게 발전시킨 것.

WHY — 외부 정렬(external sort)의 기본

데이터가 메모리에 안 들어가면 (예: 1TB 파일, 16GB RAM)? — 병합 정렬이 유일한 실용적 답입니다.

  1. 파일을 RAM에 들어갈 만한 청크로 쪼개 각각 메모리에서 정렬 (quicksort 등).
  2. 정렬된 청크들을 파일로 써둠.
  3. bottom-up k-way 병합으로 파일들을 합침 (k는 가용 메모리에 맞게).

이게 GNU sort 명령, 데이터베이스 ORDER BY, MapReduce shuffle 등이 내부에서 하는 일. 병합 정렬의 공간 비용이 디스크 I/O에서는 오히려 장점.

04 — 힙 정렬 (Heapsort, CLRS §6)

중급

아이디어 — "항상 최댓값을 꺼낼 수 있는 자료구조"

J.W.J. Williams, 1964 (CACM Algorithm 232, Vol. 7, June 1964). 같은 해 R.W. Floyd가 선형 시간 build-heap(sift-down bottom-up)을 추가로 제안.

최대 힙(max-heap): 부모가 자식보다 항상 크거나 같은 완전 이진 트리. 루트는 전체 최댓값. "루트를 빼서 마지막에 놓고, 힙을 다시 정돈" — 이걸 n번 반복하면 정렬 완료.

힙은 완전 이진 트리이자 배열. 인덱스 공식만으로 부모-자식 관계 O(1) 계산. 포인터 없이 메모리 효율 + 캐시 친화.
힙 정렬은 "최댓값을 뒤로 보내기" 반복. 각 단계마다 max-heap 루트(가장 큰 값)을 배열의 끝으로 옮김. 끝에서부터 정렬된 영역(녹색)이 자라남.
고급

배열을 이용한 힙 표현

완전 이진 트리는 배열로 아주 깔끔하게 표현됩니다. 루트를 A[0]이라 하면:

parent(i) = (i − 1) // 2,   left(i) = 2i + 1,   right(i) = 2i + 2

포인터 따라갈 필요 없이 인덱스 계산만으로 트리 순회 — O(1) 접근. 힙을 배열로 쓰는 게 자연스러운 이유.

(CLRS는 1-indexed 의사코드를 씁니다: parent(i) = i/2, left(i) = 2i, right(i) = 2i+1. 여기서는 Python 관례에 맞게 0-indexed.)

고급

핵심 3개 루틴 — MAX-HEAPIFY / BUILD-MAX-HEAP / HEAPSORT

heapsort.py (CLRS §6.1~§6.4 구현)
def max_heapify(A, i, size):
    """i번 노드 위치에서 '내려가며' 힙 속성 복구. O(log n)."""
    l = 2 * i + 1
    r = 2 * i + 2
    largest = i
    if l < size and A[l] > A[largest]:
        largest = l
    if r < size and A[r] > A[largest]:
        largest = r
    if largest != i:
        A[i], A[largest] = A[largest], A[i]
        max_heapify(A, largest, size)


def build_max_heap(A):
    """마지막 내부 노드부터 루트까지 heapify. O(n)."""
    n = len(A)
    for i in range(n // 2 - 1, -1, -1):
        max_heapify(A, i, n)


def heapsort(A):
    """O(n log n)."""
    build_max_heap(A)
    for i in range(len(A) - 1, 0, -1):
        A[0], A[i] = A[i], A[0]   # 루트(최댓값)을 끝으로
        max_heapify(A, 0, i)     # 줄어든 힙 복구
    return A


print(heapsort([5, 2, 4, 7, 1, 3]))
출력:
[1, 2, 3, 4, 5, 7]
⚠️ 힙 정렬의 두 가지 단점
  1. 불안정(unstable) — 힙 구조가 원래 순서를 기억하지 못함. 같은 키의 원소 순서 보장 없음.
  2. 캐시 불친화 — left(i) = 2i+1, right(i) = 2i+2 접근은 메모리를 크게 건너뜀. n = 10⁶일 때 자식 노드는 수 MB 떨어진 메모리를 읽음 → 캐시 미스 폭증. 실측에서 퀵소트보다 2~3배 느린 주요 원인.

05 — 힙 정렬의 복잡도: Build-Heap이 왜 Θ(n)인가

고급

순진한 분석 — "Θ(n log n)"은 왜 틀렸는가

BUILD-MAX-HEAP은 n/2개의 내부 노드에 대해 MAX-HEAPIFY를 호출. 각 호출은 O(log n). 그러니 (n/2) × O(log n) = O(n log n)? — 맞지만 너무 느슨한 상한.

진짜 놀라운 사실: BUILD-MAX-HEAP은 Θ(n)입니다. log 인자가 없습니다.

전공자

Tight 분석 (CLRS §6.3)

핵심 관찰: 노드의 높이가 낮을수록 MAX-HEAPIFY가 빠르고, 낮은 높이의 노드가 훨씬 많다.

높이 h해당 높이 노드 수 (상한)각 노드의 MAX-HEAPIFY 비용
0 (리프)⌈n/2⌉O(1) — heapify 안 함
1⌈n/4⌉O(1)
2⌈n/8⌉O(2)
………
h⌈n / 2h+1⌉O(h)
log n1 (루트)O(log n)
총 비용 = Σh=0log n (n / 2h+1) · O(h)
     = O(n) · Σh=0∞ h / 2h
     = O(n) · 2   // 미적분: Σ h/2^h = 2
     = O(n)

Σh=0∞ h/2h = 2는 유명한 수렴 급수. 미적분으로 증명 가능 (등비급수 Σ xh = 1/(1-x)를 미분).

📖 원서 인용 — CLRS 4판, §6.3 "Building a heap"

"Although the procedure runs in O(n lg n) time, which is correct but not tight, we can derive a tighter bound by observing that the time for MAX-HEAPIFY to run at a node varies with the height of the node in the tree, and the heights of most nodes are small. Our tighter analysis relies on the properties... We can build a max-heap from an unordered array in linear time."

전공자

HEAPSORT 전체 — Θ(n log n)

BUILD-MAX-HEAP = Θ(n), 그 뒤 n-1번의 MAX-HEAPIFY(루트에서) = (n−1) × Θ(log n) = Θ(n log n). 총합: Θ(n) + Θ(n log n) = Θ(n log n).

하지만 BUILD-MAX-HEAP의 Θ(n)은 의미 있는 사실 — 힙의 다른 용도(우선순위 큐 초기화, Algo 12 heapq)에서는 이게 주된 비용이 됩니다.

WHY — 왜 힙 정렬이 실무에서 밀리는가 심층 분석

점근 복잡도는 병합·힙·퀵 모두 Θ(n log n)이지만, 상수와 하드웨어 친화도가 다릅니다.

항목병합힙퀵
메모리 접근 패턴순차 ✓랜덤(부모-자식 점프) ⚠️대부분 순차 ✓
CPU 분기 예측잘 맞음잘 안 맞음잘 맞음 (파티션)
비교 수 (상수)~n log₂ n~2n log₂ n ⚠️~1.39 n log₂ n ✓
SIMD/병렬화쉬움어려움중간

힙 정렬은 비교 수 자체가 2배 많음(상수 인자). 거기에 캐시 미스 × 2~3. 결론: 같은 Θ(n log n)이라도 실측 속도는 퀵소트의 3~5배 느림.

그래서 힙 정렬은 직접 쓰기보단 "최악을 보장해야 할 때의 안전장치" — introsort에서 퀵소트가 깊어지면 힙 정렬로 전환하는 데 사용.

중급

Algo 04 §06에서 randomized quicksort의 기댓값 분석을 자세히 다뤘습니다. 여기선:

  • 정확성 증명 (루프 불변식 — Algo 02 도구)

  • 중복 키 처리의 3-way partition (§07)

고급

퀵소트의 핵심 구조 (CLRS §7.1)

quicksort_classic.py (CLRS §7.1 Lomuto)
def quicksort(A, p, r):
    if p < r:
        q = partition(A, p, r)     # pivot 위치
        quicksort(A, p, q - 1)     # 왼쪽: pivot보다 작은 부분
        quicksort(A, q + 1, r)     # 오른쪽: pivot보다 큰 부분


def partition(A, p, r):
    pivot = A[r]                # 마지막 원소를 pivot으로
    i = p - 1                 # "pivot 이하" 경계
    for j in range(p, r):
        if A[j] <= pivot:
            i += 1
            A[i], A[j] = A[j], A[i]
    A[i + 1], A[r] = A[r], A[i + 1]
    return i + 1

병합 정렬과 구조는 비슷합니다 — 쪼개고 재귀로 풀고. 차이:

  • 병합: Divide가 O(1)(인덱스 계산), Combine이 O(n)(실제 병합)
  • 퀵: Divide가 O(n)(partition), Combine이 O(1)(제자리 완성)

퀵소트의 Combine이 O(1)이라는 게 in-place가 되는 근본 이유 — 보조 배열이 필요 없습니다.

고급

partition의 루프 불변식

I(j): 매 for 반복 직전,
(a) A[p..i]의 모든 원소 ≤ pivot,  
(b) A[i+1..j-1]의 모든 원소 > pivot,  
(c) A[r] = pivot (고정).
  • 초기 (j = p): i = p−1이므로 A[p..p−1]과 A[p..p−1] 모두 빈 배열, 자명히 성립.
  • 유지: A[j] ≤ pivot이면 i++, swap — A[i](새 위치)가 이전 A[j]로 바뀌고 (a)에 들어감. A[j] > pivot이면 그대로 (b)에 남음. 둘 다 불변식 유지.
  • 종료 (j = r): A[p..i]은 ≤ pivot, A[i+1..r−1]은 > pivot, A[r] = pivot. swap(A[i+1], A[r])하면 A[p..i]≤pivot ≤ A[i+1] < A[i+2..r]이라 pivot이 제자리. ∎
📖 원서 인용 — CLRS 4판, §7.1 "Description of quicksort"

"Quicksort, like merge sort, applies the divide-and-conquer method. ... The PARTITION procedure always selects an element x = A[r] as a pivot element around which to partition the subarray A[p:r]. As the procedure runs, it partitions the array into (possibly empty) subarrays A[p:i], A[i+1:j-1], and A[j:r-1] such that each element of A[p:i] is less than or equal to x, each element of A[i+1:j-1] is greater than x, and x resides at A[r]."

07 — Partition 변종: Hoare · Lomuto · 3-way

고급

Hoare partition (원조, 1961)

Tony Hoare가 1961 Communications of the ACM에 처음 발표한 원조. 양 끝에서 가운데로 진행하는 두 포인터.

hoare_partition.py
def hoare_partition(A, p, r):
    pivot = A[p]
    i, j = p - 1, r + 1
    while True:
        i += 1
        while A[i] < pivot:
            i += 1
        j -= 1
        while A[j] > pivot:
            j -= 1
        if i >= j:
            return j
        A[i], A[j] = A[j], A[i]

특징: Hoare는 pivot을 원래 자리에 고정하지 않음. 반환값 j는 "왼쪽 부분배열의 끝" 인덱스. 재귀 호출은 qs(A, p, j), qs(A, j+1, r)로 호출.

  • 장점: Lomuto 대비 스왑 3배 적음. 실측 상수가 작음 — 실전 표준.
  • 단점: 이해 난이도 ↑. 경계 조건(pivot이 극단값일 때) 주의.
고급

Lomuto partition — CLRS 교과서 버전

Nico Lomuto가 1980년대에 제안. §06의 코드가 이것. Hoare보다 이해가 쉬워 교육용으로 자주 등장 (CLRS 본문도 이것).

  • 장점: 단일 포인터, 루프 불변식 증명 쉬움.
  • 단점: 스왑 더 많음. 중복 원소에 취약 — 모든 원소가 같으면 partition이 극단 분할(0 vs n−1)을 만들어 최악 Θ(n²).
전공자

3-way partition (Dijkstra의 Dutch National Flag)

중복이 많은 입력(예: 0/1/2만 있는 배열)에서 Lomuto도 Hoare도 Θ(n²)으로 망가집니다. Edsger Dijkstra의 해결책: 세 영역으로 쪼개기. < pivot, = pivot, > pivot.

three_way_partition.py
def quicksort_3way(A, lo, hi):
    if lo >= hi: return
    pivot = A[lo]
    lt, i, gt = lo, lo + 1, hi
    # 불변식: A[lo..lt-1] < pivot, A[lt..i-1] == pivot, A[gt+1..hi] > pivot
    while i <= gt:
        if A[i] < pivot:
            A[lt], A[i] = A[i], A[lt]
            lt += 1; i += 1
        elif A[i] > pivot:
            A[i], A[gt] = A[gt], A[i]
            gt -= 1
        else:
            i += 1
    quicksort_3way(A, lo, lt - 1)
    quicksort_3way(A, gt + 1, hi)

결과: 같은 값 여러 개는 A[lt..gt]에 모이고 재귀 X. 중복이 많은 입력에서 극도로 빨라짐 — 모두 같은 값이면 Θ(n). Java의 Arrays.sort가 쓰는 Dual-Pivot Quicksort가 이 아이디어의 변형입니다.

Sedgewick과 Bentley(1997)의 "Fast Three-Way Partition": 추가 최적화로 실전에서 거의 모든 입력에 안정적.

WHY — "어느 partition을 쓸 것인가"의 실무 기준
  • 입력 분포 미상 + 단순성 우선: CLRS Lomuto로 시작 (이해·증명 쉬움).
  • 성능 추구: Hoare. 스왑 수 1/3. 실측에서 30~50% 빠름.
  • 중복 많을 가능성: 3-way. 예: 범주형 데이터, 이미지 픽셀, 정수 ID.
  • 일반 라이브러리: Dual-pivot (Java) 또는 pattern-defeating quicksort (Go, Rust) — 여러 변종 결합.

08 — Introsort: C++ std::sort의 심장

전공자

Musser 1997 — 퀵소트의 최악 케이스를 봉인하는 법

David Musser가 1997년 "Introspective Sorting and Selection Algorithms" 논문에서 제안. 아이디어가 단순하면서 강력:

  1. 퀵소트를 시작한다. 재귀 깊이를 카운트한다.
  2. 깊이가 2 × log₂ n을 넘어가면 → 퀵소트가 나빠지고 있다는 신호.
  3. 그 시점에서 힙 정렬로 전환. 나머지는 힙 정렬이 Θ(n log n)으로 완료 보장.
  4. 부분배열이 충분히 작으면(보통 < 16) → 삽입 정렬.
Introsort(A) = Quicksort까지 깊이 2⌊log₂ n⌋ → 깊이 초과 시 Heapsort → 작은 구간 Insertion sort

복잡도 보장: 최악·평균·최선 모두 Θ(n log n). 동시에 평균 속도는 퀵소트 수준.
introsort.py (개념 구현)
import math


def introsort(A):
    depth_limit = 2 * math.floor(math.log2(len(A))) if len(A) > 1 else 0
    _introsort(A, 0, len(A) - 1, depth_limit)


def _introsort(A, lo, hi, depth):
    if hi - lo + 1 < 16:
        _insertion_sort(A, lo, hi)
        return
    if depth == 0:
        _heapsort_range(A, lo, hi)   # fallback
        return
    p = _partition(A, lo, hi)
    _introsort(A, lo, p - 1, depth - 1)
    _introsort(A, p + 1, hi, depth - 1)
📖 원서 인용 — Musser, "Introspective Sorting" (1997)

"Introsort behaves like quicksort in the average case but switches to heapsort to avoid quicksort's worst case. The overall effect is to guarantee O(n log n) running time in all cases while maintaining quicksort's performance on typical inputs."

전공자

Part B — 왜 2 × log₂ n인가?

재귀 깊이가 log₂ n 근처면 quicksort가 균형 있게 진행 중. 2 × log₂ n까지는 "피벗이 평균 1/4 이상을 차지하는" 꽤 균형 잡힌 분할. 그 이상 깊어지면 연속 불균형 분할 (최악 시나리오)에 가까워짐.

수학적으로: 한쪽이 항상 1/α 이상을 차지하면 깊이 ≤ logα/(α-1) n. α=4/3이면 log4 n ≈ log₂ n / 2. 2 log₂ n 제한은 α ≤ √2/(√2−1) = 2+√2 ≈ 3.41까지 허용 — 즉 "한쪽이 약 30% 이상만 차지하면 2 log₂ n 안에 들어감", 실전에서 충분히 넉넉.

WHY — Introsort가 "세 거인 모두 이기는" 이유

Introsort는 트레이드오프를 합친 알고리즘입니다.

  • 평균 속도: 퀵소트에서 상속 (가장 빠름).
  • 최악 보장: 힙 정렬에서 상속 (Θ(n log n)).
  • 작은 n 최적화: 삽입 정렬에서 상속 (Algo 05).

세 알고리즘 모두 각자의 장점이 있기에 — 하나로 합친 introsort가 C++ std::sort의 기본 구현이 됐습니다 (GCC libstdc++, Microsoft STL 등).

09 — 비비교 정렬: Ω(n log n) 우회 (CLRS §8.2~§8.4)

고급

하한을 "깨는" 것이 아니라 "벗어나는" 것

Algo 05 §07의 Ω(n log n) 하한은 비교 기반 정렬에만 해당. 원소의 다른 성질을 쓸 수 있다면:

  • 원소가 작은 정수 → Counting sort (§8.2). O(n + k).
  • 원소가 d자리 수 → Radix sort (§8.3). O(d(n + k)).
  • 원소가 [0, 1) 균등분포 → Bucket sort (§8.4). 기대 O(n).

이들의 공통점: 원소를 직접 배열의 인덱스로 사용. 비교가 없음.

고급

Counting Sort (CLRS §8.2)

"원소 값마다 카운터를 하나씩 두고, 카운터를 누적합으로 바꿔 각 원소의 최종 위치를 계산."

counting_sort.py
def counting_sort(A, k):
    """A의 원소는 0 이상 k 이하의 정수."""
    n = len(A)
    C = [0] * (k + 1)       # 카운터
    B = [0] * n             # 출력
    for x in A:
        C[x] += 1           # 각 값의 빈도
    for v in range(1, k + 1):
        C[v] += C[v - 1]     # 누적합 → "C[v] = v 이하 원소의 수"
    for j in range(n - 1, -1, -1):
        B[C[A[j]] - 1] = A[j]  # A[j]의 최종 위치
        C[A[j]] -= 1        # 같은 값 다음 원소는 한 칸 앞으로
    return B


print(counting_sort([2, 5, 3, 0, 2, 3, 0, 3], 5))
출력:
[0, 0, 2, 2, 3, 3, 3, 5]
  • 시간: O(n + k).
  • 공간: O(n + k).
  • 안정성: ✓ (뒤에서 앞으로 훑는 루프가 핵심).
  • 제약: 원소가 정수 + 값 범위 k가 n과 비슷하거나 작아야 유용.

k = n이면 Θ(n). k = n²이면 Θ(n²) — 하한은 못 넘음. "k가 작다"는 정보가 있어야 이득.

고급

Radix Sort (CLRS §8.3)

"d자리 숫자들을 1의 자리부터 d의 자리까지 차례로 안정 정렬." 각 자리는 counting sort로.

radix_sort.py (LSD, 10진법 가정)
def radix_sort(A):
    if not A: return A
    max_val = max(A)
    exp = 1
    while max_val // exp > 0:
        A = _counting_sort_by_digit(A, exp)
        exp *= 10
    return A


def _counting_sort_by_digit(A, exp):
    n = len(A)
    output = [0] * n
    count = [0] * 10
    for x in A:
        count[(x // exp) % 10] += 1
    for v in range(1, 10):
        count[v] += count[v - 1]
    for j in range(n - 1, -1, -1):
        d = (A[j] // exp) % 10
        output[count[d] - 1] = A[j]
        count[d] -= 1
    return output
  • 시간: O(d(n + k)) — 자릿수 d, 각 자리당 O(n + k) counting.
  • 안정성: ✓ (counting sort가 안정이므로).
  • 주의: LSD(least significant digit first) 방향이 올바름 — 일 자리부터 정렬해야 최종 안정성 보장. MSD는 다른 알고리즘.

32비트 정수 n개 정렬이면 d=8, k=256 (바이트 단위) → O(8·(n + 256)) ≈ O(n). 실제로 정수 배열에서 퀵소트보다 빠를 수 있음 — NumPy의 np.sort가 정수면 radix 변종을 쓰기도 함.

전공자

Bucket Sort (CLRS §8.4)

"입력이 [0, 1) 균등분포라 가정. n개의 버킷에 원소를 분배, 각 버킷을 정렬 후 연결."

1. n개의 빈 버킷 생성
2. 각 원소 x를 버킷 ⌊n·x⌋에 배치
3. 각 버킷을 삽입 정렬 (입력 균등이면 각 버킷 기대 원소 수 O(1))
4. 버킷들을 순서대로 연결

기댓값 분석: 각 버킷의 기대 원소 수 = 1. 삽입 정렬 비용은 O(k²), k=버킷 원소 수. E[Σ k²] = O(n) (확률론 기댓값 선형성). 따라서 기댓값 O(n).

⚠️ "균등분포" 가정이 깨지면 O(n²)

한 버킷에 다 몰리면 (예: 모든 원소가 비슷한 값) 삽입 정렬 O(n²) → 전체도 O(n²). bucket sort는 입력 분포를 잘 아는 특수 상황에서만 유용.

WHY — 하한 Ω(n log n)과 "선형 시간 정렬"의 공존

모순 같아 보이지만 — 둘 다 옳습니다.

  • Ω(n log n) 하한: "비교만으로" 정렬할 때. 결정 트리의 물리적 한계.
  • O(n) 선형 정렬: "원소가 작은 정수/균등분포" 등 추가 정보를 쓸 때. 비교 대신 인덱싱.

교훈: "이 입력에 대해 내가 아는 추가 정보가 있는가?"를 먼저 묻고, 있으면 비비교 정렬을 고려. 정보가 없으면 비교 정렬이 최선.

WHY — "모르면 sorted()"라는 경험칙

실무에서 Python의 sorted()·list.sort()는 Timsort입니다. Timsort는:

  • 안정 정렬 ✓
  • 적응성 있음 (거의 정렬된 입력에 O(n))
  • 최악 O(n log n) 보장
  • 작은 구간 binarysort로 자동 최적화

단 하나 약점: 추가 메모리 O(n) (병합 정렬 기반). 이게 문제가 아닌 한, sorted()가 거의 최선입니다. 직접 구현할 상황은 (1) 비교 함수가 비싸서 비교 수를 줄이고 싶음, (2) 외부 정렬, (3) 교육용 — 이 셋 외엔 드뭄.

Pattern-Defeating Quicksort (PDQSort)

2016년 Orson Peters가 제안. Go 1.19(2022)와 Rust의 sort_unstable이 채택. 기능 요약:

  • Introsort 기반: 깊이 초과 시 힙 정렬로 fallback.
  • Adaptive: 이미 정렬된 구간 감지 → O(n).
  • 중복 탐지: 같은 피벗이 반복되면 3-way partition으로 전환.
  • BlockQuicksort의 아이디어로 분기 예측 개선 — SIMD 없이도 기존 introsort 대비 30% 빠름.

2020년대 "현대 정렬"의 대표. 모든 입력 패턴에 안정적 성능 + 이론적 하한 도달.

12 — AI와 O(n log n) 정렬

AI — "quicksort 구현해줘"는 함정

"quicksort 구현해달라"고 하면 LLM은 Lomuto partition을 깔끔하게 씁니다. 거의 정확. 하지만 종종 다음을 누락:

  • 무작위 피벗 또는 median-of-three — 최악 Θ(n²) 방어 장치.
  • 작은 구간 삽입 정렬 fallback — 실전 속도의 핵심.
  • 깊이 제한 + 힙 정렬 fallback (introsort 스타일) — 최악 보장.
  • 3-way partition 중복 많은 입력 방어.

"교과서적 quicksort"는 이런 방어 없이 순진한 Lomuto. 실전 코드로 쓰면 적대적 입력에 Θ(n²)으로 죽습니다.

⚠️ "Accidentally Quadratic" — 실제로 발생한 사건들
  • Java 6까지의 Arrays.sort: 특정 패턴의 입력에 최악 O(n²). 2007년 CVE-2007-2440으로 DoS 취약점 보고 → Java 7 (2011)에서 Arrays.sort(Object[])가 Timsort로 교체.
  • libc++ std::sort (2010년대 초): 극단 입력에 Θ(n²). pattern-defeating 기법 도입으로 해결.
  • Python 2.x의 list.sort(): 초기 Timsort에 소수의 버그 (2015년 Stijn de Gouw가 발견). Merge stack invariant 오류로 일부 입력에 오동작. Python 3.4+ 수정.

교훈: "O(n log n)이 평균"과 "O(n log n)이 보장"은 다릅니다. 적대적 환경(사용자 입력 받는 서버)에서는 보장이 필요합니다.

WHY — AI에게 정렬 코드 받을 때 4가지 질문
  1. "이 코드의 최악 복잡도는? 최악 입력 예시를 보여봐."
  2. "피벗 선택 전략은? 무작위인가 결정론적인가?"
  3. "깊이 제한 또는 fallback 메커니즘이 있는가?"
  4. "안정 정렬인가? 증명은?"

sort_bench_advanced.py
import random, timeit, sys
sys.setrecursionlimit(10000)


def merge_sort(A):
    if len(A) <= 1: return A[:]
    mid = len(A) // 2
    L, R = merge_sort(A[:mid]), merge_sort(A[mid:])
    out, i, j = [], 0, 0
    while i < len(L) and j < len(R):
        if L[i] <= R[j]: out.append(L[i]); i += 1
        else: out.append(R[j]); j += 1
    out.extend(L[i:]); out.extend(R[j:])
    return out


def heapsort(A):
    A = A[:]
    def heapify(n, i):
        largest, l, r = i, 2*i+1, 2*i+2
        if l < n and A[l] > A[largest]: largest = l
        if r < n and A[r] > A[largest]: largest = r
        if largest != i:
            A[i], A[largest] = A[largest], A[i]
            heapify(n, largest)
    n = len(A)
    for i in range(n//2-1, -1, -1): heapify(n, i)
    for i in range(n-1, 0, -1):
        A[0], A[i] = A[i], A[0]
        heapify(i, 0)
    return A


def quicksort_naive(A):
    """고정 피벗(마지막) — 정렬된 입력에 Θ(n²) 함정."""
    A = A[:]
    def qs(lo, hi):
        if lo >= hi: return
        pivot = A[hi]
        i = lo - 1
        for j in range(lo, hi):
            if A[j] <= pivot:
                i += 1; A[i], A[j] = A[j], A[i]
        A[i+1], A[hi] = A[hi], A[i+1]
        qs(lo, i); qs(i+2, hi)
    qs(0, len(A)-1)
    return A


def bench(fn, data):
    start = timeit.default_timer()
    fn(data)
    return (timeit.default_timer() - start) * 1000


random.seed(42)

FIX 문제 3

def max_heapify(A, i, size):

        largest = l
    elif r < size and A[r] > A[i]:

BUILD 문제 4

iterative(재귀 없는) MAX-HEAPIFY를 구현하시오.

풀이 펼치기
iterative_heapify.py
def max_heapify_iter(A, i, size):
    while True:
        l, r = 2*i+1, 2*i+2
        largest = i
        if l < size and A[l] > A[largest]: largest = l
        if r < size and A[r] > A[largest]: largest = r
        if largest == i: break
        A[i], A[largest] = A[largest], A[i]
        i = largest
왜

재귀가 꼬리 위치(tail call)라 while로 직역 가능. Python은 tail-call optimization이 없어 iterative가 명시적 이득 — 스택 사용 O(1).

PREDICT 문제 5

Hoare partition이 Lomuto 대비 스왑을 몇 배 적게 하는가? 이유는?

풀이 펼치기
답

대략 3배 적음. Lomuto는 모든 "pivot 이하" 원소마다 스왑(평균 n/2번). Hoare는 "왼쪽에 pivot 초과 + 오른쪽에 pivot 이하"인 쌍만 스왑(평균 n/6번).

실측 차이는 입력에 따라 다르지만, Hoare가 꾸준히 Lomuto보다 20~50% 빠릅니다.

WHY 문제 6

3-way partition quicksort가 "모든 원소가 같은" 입력에서 Θ(n)인 이유를 설명하시오.

풀이 펼치기
답

모두 같은 값이면 partition 후 중앙 영역 A[lt..gt]에 전부 모이고 A[lo..lt-1]과 A[gt+1..hi]는 빈 구간. 재귀 호출이 빈 구간에만 일어나므로 O(1). 단일 패스 O(n)에 partition만 하고 끝.

Lomuto나 Hoare는 이 경우에도 불균형 분할(0 vs n-1)로 Θ(n²).

PREDICT 문제 7

Counting sort가 n = 10⁶, k = 10¹²에서 비교 정렬 대비 유리한가 불리한가? 왜?

풀이 펼치기
답

불리. Counting sort는 O(n + k) = O(10⁶ + 10¹²) = O(10¹²). 비교 정렬은 O(n log n) = O(10⁶ × 20) = O(2×10⁷). Counting이 5만 배 더 느림.

교훈: k가 n보다 훨씬 크면 counting sort는 의미 없음. k = O(n)일 때만 선형 시간.

BUILD 문제 8

32비트 부호 없는 정수 배열을 radix sort로 정렬하는 함수를 256진법(바이트 단위)으로 작성하시오.

풀이 펼치기
radix_byte.py
def radix_sort_byte(A):
    for shift in (0, 8, 16, 24):
        # 각 바이트별 counting sort (안정)
        count = [0] * 256
        for x in A:
            count[(x >> shift) & 0xFF] += 1
        for v in range(1, 256): count[v] += count[v-1]
        out = [0] * len(A)
        for x in reversed(A):
            d = (x >> shift) & 0xFF
            count[d] -= 1
            out[count[d]] = x
        A = out
    return A
분석

4 패스 (d=4) × 각 O(n + 256) ≈ O(n). Pure Python 구현은 C로 작성된 sorted()보다 느리지만, NumPy np.sort(kind='radix')처럼 C 레벨 구현이면 n = 10⁶ 이상 정수 데이터에서 Timsort보다 빠릅니다.

WHY 문제 9

Introsort가 깊이 한계를 2 log₂ n으로 잡는 이유를 피벗 분할 비율 관점에서 설명.

풀이 펼치기
답

한쪽 부분배열이 항상 1/α 이상을 차지하는 분할에서 재귀 깊이는 logα/(α-1) n. 2 log₂ n 한계는 α ≤ 2+√2 ≈ 3.41까지 허용 — 즉 "한쪽이 약 30% 이상 차지"만 지켜지면 OK. 피벗이 아주 작은 극단 영역에 걸리는 경우가 log n번 연속 발생해야 한계 도달. 확률적으로 매우 드물지만, 그래도 발생하면 heapsort로 안전 착륙.

PREDICT 문제 10

Python에서 sorted()가 "이미 정렬된 10⁶개" 입력을 처리하는 데 기대 복잡도는?

풀이 펼치기
답

Θ(n) = Θ(10⁶). Timsort는 natural run을 감지하므로 이미 정렬된 배열을 한 번 스캔하고 바로 종료. 10⁶ 원소 기준 약 0.05ms 예상.

안정 정렬을 이용해 "학생 배열을 성적순으로, 성적이 같으면 이름순으로" 정렬하는 2-key 정렬 방법을 두 가지 작성하시오.

풀이 펼치기
two_key_sort.py
# 방법 1: tuple key (한 번의 sorted 호출)
students.sort(key=lambda s: (-s.score, s.name))

# 방법 2: 안정 정렬을 2번 (2차 key 먼저, 1차 key 나중)

AI 문제 12

LLM이 C로 quicksort를 작성해줬는데 고정 피벗(첫 원소)을 썼다. 서버 코드에 배포하면 생길 수 있는 보안 문제는?

풀이 펼치기
답

알고리즘 복잡도 공격(DoS). 사용자가 악의적으로 "이미 정렬된" 또는 "역순" 입력을 보내면 quicksort가 Θ(n²)로 터져 CPU를 고갈. n = 10⁴만으로도 초 단위 지연 발생 가능.

수정: pivot = A[rand() % n] 또는 introsort로 전환.

핵심 정리

📚

병합 — 안정의 기둥

Θ(n log n) 보장, 안정, 순차 접근. O(n) 보조 공간이 약점이자 외부 정렬의 강점.

⛰️

힙 — 보장의 기둥

in-place, Θ(n log n) 보장. 하지만 캐시 불친화 + 비교 상수 2배로 실무에서 밀림.

⚡

퀵 — 속도의 기둥

평균 가장 빠름, 최악 Θ(n²). 무작위 피벗 + introsort로 최악 봉인 → 범용 표준.

🔀

3-way Partition

Dijkstra Dutch Flag. 중복 많은 입력에서 Θ(n). Java Dual-Pivot, Rust PDQ의 토대.

🛡️

Introsort — 세 거인의 합체

Quicksort 평균 속도 + Heapsort 최악 보장 + Insertion 작은 n. C++ std::sort.

🔢

비비교 정렬 — 하한 우회

Counting O(n+k), Radix O(d(n+k)), Bucket 기대 O(n). 원소 특성 알 때 비교 우회.
