최단 경로 I — Dijkstra
1956년 커피 브레이크에서 태어나 1959년에 발표된 알고리즘

01 — 이번 주 이야기: 커피 브레이크의 선물

"The shortest path algorithm was invented in about 1956... I designed it in about 20 minutes. One morning I was shopping in Amsterdam with my young fiancée, and tired, we sat down on the café terrace to drink a cup of coffee and I was just thinking about whether I could do this, and I then designed the algorithm for the shortest path."
— Edsger W. Dijkstra, Interview 2001

배경 — 네덜란드의 신생 컴퓨터 ARMAC

1956년, 네덜란드 수학센터(CWI)가 새 컴퓨터 ARMAC을 공개 시연. Dijkstra에게 "대중이 이해할 수 있는 문제"를 풀어달라 부탁. 그가 선택한 건 — "네덜란드 64개 도시 간 최단 경로".

커피 브레이크 20분 동안 떠올린 알고리즘. 3년 뒤인 1959년 "A Note on Two Problems in Connexion with Graphs" (Numerische Mathematik 1:269-271)에 겨우 3페이지로 발표.

60년 후의 오늘

Dijkstra 알고리즘이 지금 처리하는 일들:

  • 구글 지도 / 카카오맵: 길 안내. 도로망이 가중 그래프, 도시가 정점.
  • GPS 내비게이션: 실시간 교통 반영한 경로 재계산.
  • 인터넷 라우팅 (OSPF): 라우터 간 패킷 경로. 2022년 현재도 운영 중.
  • 게임 AI: NPC 경로 찾기 (보통 A*로 확장).
  • 3D 렌더링: 빛이 장면을 통과하는 최단 경로.
  • 의료 영상 (segmentation): 가중 그래프로 경로 트레이싱.
WHY — BFS로는 왜 안 되는가

Algo 14의 BFS는 "모든 간선이 같은 비용(1)" 가정. 가중 그래프에선 깨집니다:

  • A → B 직통 길(비용 10)
  • A → C → D → B (비용 1+1+1 = 3)

BFS는 "간선 수 적은 순"이라 A→B (1 간선)를 먼저 발견해 dist[B]=10으로 확정. 하지만 실제 최단은 3. BFS가 거짓말.

Dijkstra의 해결책: "간선 수" 대신 "누적 비용" 순서로 처리. 큐 대신 우선순위 큐 (heap).

02 — BFS가 가중 그래프에 실패하는 이유

중급

구체적 반례

그래프:
     A ─[10]─ B
     │
    [1]
     │
     C ─[1]─ D ─[1]─ B

BFS(A) 순서: A (dist=0) → B (dist=10, 1간선) → C (dist=1) → D (dist=2)
BFS는 B를 한 번만 발견하니 dist[B] = 10으로 고정. 잘못된 답.

실제 최단: A → C → D → B = 3.

BFS의 핵심 가정 — "큐가 거리 순서 보장" — 은 모든 간선 비용이 같을 때만 성립.

고급

Dijkstra의 수정

두 가지 수정이 필요:

  1. 큐 대신 heap — "현재 알려진 거리 최소" 노드를 꺼내기. FIFO → 최솟값 우선.
  2. dist 업데이트 허용 — 더 짧은 경로가 발견되면 갱신. BFS는 "한 번 방문=끝"이지만 Dijkstra는 "더 좋은 경로 찾으면 갱신".

결과: "현재 거리 최소 노드부터 처리"를 반복 → 비가중 BFS의 일반화.

03 — Dijkstra의 아이디어: 그리디

중급

"매 순간 현재까지 가장 가까운 미방문 노드 방문"

그리디 전략:

  1. 모든 노드의 거리를 ∞로 초기화. dist[start] = 0.
  2. 미방문 노드 중 dist가 최소인 u 선택.
  3. u의 모든 이웃 v에 대해: new_dist = dist[u] + weight(u,v). 기존보다 작으면 갱신.
  4. u를 "방문 완료"로 표시.
  5. 모든 노드가 방문될 때까지 2-4 반복.
Dijkstra = 그리디. BFS처럼 "한 번 방문=끝"이 아니라, 거리가 업데이트되며 점진적으로 확정. heap이 "현재 최소 거리 노드"를 빠르게 제공.

04 — 알고리즘 의사코드 (CLRS §22.3)

고급

CLRS 표준 의사코드

DIJKSTRA(G, w, s)
1. INITIALIZE-SINGLE-SOURCE(G, s)
2. S ← ∅ // 확정된 노드 집합
3. Q ← G.V // 우선순위 큐 (min-heap)
4. while Q ≠ ∅
5.     u ← EXTRACT-MIN(Q)
6.     S ← S ∪ {u}
7.     for each vertex v ∈ G.Adj[u]
8.        RELAX(u, v, w)

RELAX(u, v, w):

if v.d > u.d + w(u,v):
   v.d ← u.d + w(u,v)
   v.π ← u

"완화(relaxation)" — 새 경로가 기존보다 짧으면 업데이트.

📖 원서 인용 — CLRS 4판, §22.3 "Dijkstra's algorithm"

"Dijkstra's algorithm solves the single-source shortest-paths problem on a directed or undirected weighted graph G = (V, E) for the case in which all edge weights are nonnegative. ... Dijkstra's algorithm is a greedy algorithm: it grows a set S of vertices for which the final shortest-path weight from s to each vertex has already been determined."

05 — Python 구현: heapq + lazy deletion

전공자

실무 패턴 — decrease-key 대신 lazy deletion

교과서의 DIJKSTRA는 heap의 DECREASE-KEY 연산을 가정. Python heapq엔 이 연산이 없음 (Algo 12 §10). 해결: 같은 노드의 여러 entry를 heap에 두고, pop 시 "stale"이면 무시.

dijkstra.py
import heapq

def dijkstra(graph, src):
    """graph: {node: [(neighbor, weight), ...]}"""
    dist = {src: 0}
    parent = {src: None}
    pq = [(0, src)]    # (누적거리, 노드)

    while pq:
        d, u = heapq.heappop(pq)
        if d > dist.get(u, float('inf')):
            continue    # stale entry (이미 더 짧은 경로로 처리됨)
        for v, w in graph[u]:
            nd = d + w
            if nd < dist.get(v, float('inf')):
                dist[v] = nd
                parent[v] = u
                heapq.heappush(pq, (nd, v))   # 새 entry (옛 entry는 heap에 남아있지만 무시됨)

    return dist, parent


def reconstruct_path(parent, target):
    if target not in parent: return None
    path = []
    cur = target
    while cur is not None:
        path.append(cur)
        cur = parent[cur]
    return path[::-1]


# 테스트
graph = {
    'A': [('B', 10), ('C', 1)],
    'B': [],
    'C': [('D', 1)],
    'D': [('B', 1)],
}
dist, parent = dijkstra(graph, 'A')
print(dist)    # {'A': 0, 'B': 3, 'C': 1, 'D': 2}
print(reconstruct_path(parent, 'B'))    # ['A', 'C', 'D', 'B']
  • stale 체크: pop된 (d, u)에서 d가 현재 dist[u]보다 크면 skip. 이미 더 짧은 경로로 처리된 entry.
  • 공간: heap에 중복 entry가 있지만 최대 O(E) 크기. 각 간선당 최대 1번 push.
Dijkstra 한 주기. heap에서 거리 최소 노드를 꺼내 "확정" → 이웃 거리 갱신 → 반복. (10,B) 같은 옛 entry는 pop 시 stale 판정으로 skip.

06 — 정확성 증명: 왜 그리디가 통하는가

전공자

루프 불변식

불변식 I: 매 반복 직전, 확정 집합 S의 모든 u에 대해 dist[u] = δ(s, u) (진짜 최단 거리).

증명 (모순법): 처음으로 잘못된 u가 S에 들어간다고 가정. u ∈ S이지만 dist[u] > δ(s, u). 진짜 최단 경로 P: s → ... → x → y → ... → u에서 y가 첫 번째 미확정 노드(=V\S)라 하자.

  1. x ∈ S이고, x 이전까진 모두 확정. 귀납 가정에 의해 dist[x] = δ(s, x).
  2. RELAX(x, y)가 호출됐음 → dist[y] ≤ dist[x] + w(x, y) = δ(s, y).
  3. 음수 간선 없으므로 δ(s, y) ≤ δ(s, u). 따라서 dist[y] ≤ δ(s, u) < dist[u].
  4. Dijkstra가 u를 뽑았다는 건 u가 y보다 작거나 같은 dist라는 뜻 → dist[u] ≤ dist[y]. 모순!

핵심: 음수 간선 없음이 2→3 단계에서 필수. 음수 있으면 δ(s, y) > δ(s, u) 가능 → 증명 무너짐.

07 — 복잡도: O((V+E) log V)

고급

heapq 기반 분석

  • 각 노드는 heap에 최대 deg(v)+1번 push → 총 push = O(V + E).
  • 각 push/pop은 O(log V) (heap 크기 상한).
  • 총 시간 = O((V + E) log V).
자료구조시간공간
Array (min 선형 탐색)O(V²)O(V)
Binary heap (heapq)O((V+E) log V)O(V+E)
Fibonacci heapO(V log V + E)O(V)
Bucket (정수 가중치)O(V + E) (Dial)O(V+W)

실무에선 거의 항상 binary heap. Fibonacci heap은 이론적 개선이나 상수가 커 실측에서 느림.

08 — 음수 간선의 함정

고급

반례

그래프 (A가 시작점, 목표는 dist[C]):
A ─[4]→ B
A ─[2]→ C
B ─[−3]→ C

교과서 Dijkstra (visited 집합 버전):
1. A pop. visited={A}. Relax: dist[B]=4, dist[C]=2.
2. C pop (dist=2 — "확정"). visited={A,C}. C의 이웃 없음.
3. B pop (dist=4). visited={A,B,C}. Relax C: 4+(−3)=1 < 2 이지만, C는 이미 visited → 갱신 불가.
4. 결과: dist[C] = 2.  실제 최단 A→B→C = 1. ❌ 틀림.

문제의 본질: Dijkstra의 불변식 "pop된 노드의 dist는 확정" (greedy choice)이 음수 간선에서 깨진다. 음수 간선이 미래에 더 짧은 경로를 만들 수 있기 때문.
💡 lazy deletion과의 미묘한 차이

앞서 본 Python 구현(Section 05)은 visited 집합 대신 stale 엔트리 skip만 씀. 이 경우 C가 재삽입되어 정답(1)을 낼 수도 있다. 하지만 이는 보장이 아님:

  • 음수 간선이 많으면 노드별 재방문이 지수적으로 폭증 가능 (전형적인 "도미노형" 구성에서 Dijkstra-with-reinsertion은 최악 Ω(2n) — 음수 간선에는 Bellman-Ford를 써야 함).
  • 음수 사이클이 있으면 영원히 종료 못 함 (dist 무한 감소).
  • "운 좋게 맞는 경우"에 기대는 건 알고리즘이 아님.

원칙: 음수 간선이 있을 수 있으면 Dijkstra를 쓰지 말라.

해결: 음수 간선 허용 그래프는 Bellman-Ford (다음 주 Algo 16). O(VE)로 느리지만 정확하며 음수 사이클도 감지.

⚠️ 주의 — 음수 사이클

음수 사이클이 있으면 최단 경로가 −∞. Dijkstra, Bellman-Ford 모두 정답 없음. Bellman-Ford는 감지는 가능 (V-1번 반복 후에도 갱신되면 음수 사이클).

09 — 변종: Bidirectional, K-shortest

전공자

Bidirectional Dijkstra

시작점 s와 목표 t를 동시에 양방향에서 탐색 → 두 프런티어가 만나면 종료. 탐색 영역을 크게 줄임.

  • 단방향 Dijkstra: O(exploration of disk radius δ(s,t))
  • Bidirectional: O(2 × exploration of disk radius δ(s,t)/2) → 약 절반 노드만 방문

OSRM, GraphHopper 같은 오픈소스 라우팅 엔진의 표준. Contraction Hierarchies (CH) 같은 전처리 기법과 결합해 쓰이며, 상용 내비·지도 서비스들도 유사한 양방향 탐색 기법을 기반으로 한다고 공개 자료에서 언급됨.

전공자

K-Shortest Paths (Yen's algorithm, 1971)

"최단 경로 K개" 찾기. Yen's algorithm: K-1번의 Dijkstra 변형으로 K개 구함. O(KV(E + V log V)).

용도: "최단이 막혔을 때 두 번째 짧은 길" (실시간 교통 대응), 네트워크 fault tolerance 설계.

10 — A*: Dijkstra + 휴리스틱

고급

직관 (Hart, Nilsson, Raphael 1968)

Dijkstra는 모든 방향으로 퍼짐. A*는 "목표 방향으로" 편향된 탐색. 1968년 Peter Hart, Nils Nilsson, Bertram Raphael이 Stanford Research Institute에서 로봇 Shakey의 경로 계획을 위해 제안 — "A Formal Basis for the Heuristic Determination of Minimum Cost Paths" (IEEE Trans. SSC-4, pp. 100-107).

f(n) = g(n) + h(n)
  • g(n): s에서 n까지 실제 거리 (Dijkstra와 같음).
  • h(n): n에서 목표까지의 휴리스틱 추정 거리. 과대평가 금지 (admissible: h(n) ≤ 진짜 거리).
  • consistent(monotone): 모든 간선 (u,v,w)에 대해 h(u) ≤ w + h(v). admissible보다 강한 조건 — 만족하면 각 노드 한 번만 pop(재개방 불필요).

heap 우선순위로 f(n) 사용. 목표 근처 노드가 먼저 탐색됨 → 엄청난 속도 향상. admissible이면 최적 경로 보장 (Hart–Nilsson–Raphael 정리).

전공자

휴리스틱 예

상황휴리스틱admissible?
격자(grid) 맵맨해튼 거리 |dx|+|dy|✓ (대각선 못 가면)
격자(대각선 허용)체비쇼프 max(|dx|,|dy|)✓
지리 좌표유클리드 거리 √(dx²+dy²)✓ (지구 평면 가정)
네트워크 그래프0 (Dijkstra로 퇴화)✓ (trivially)

h(n) = 0이면 A* = Dijkstra. h(n) = 진짜 거리면 최적 탐색 (하지만 이것 자체가 답).

A*는 주로 게임 AI (경로 찾기), 로봇 공학, 퍼즐 풀기 (15-puzzle)에 사용. networkx.astar_path로 Python에서 바로.

11 — 실무: 구글 지도는 어떻게 쓰는가

중급

단순 Dijkstra는 충분한가?

미국 도로망: 노드 2천만 개, 간선 5천만 개. 한 쿼리 당 Dijkstra 실행하면 수십 초. 구글 지도는 0.1초 이내에 답. 뭔가 다른 것을 씁니다.

전공자

Contraction Hierarchies (CH) — 2008 Geisberger et al.

아이디어: 도로망 전처리 — 각 노드를 "중요도" 순서로 제거하며 shortcut 간선 추가. 쿼리 시엔 중요도 높은 노드 위주로 bidirectional Dijkstra.

  • 전처리: 도로망 규모에 따라 분~시간 단위 (한 번만; 실험적으로 근선형에 가깝게 보고됨).
  • 쿼리: 대륙 규모 네트워크에서 밀리초 단위.
  • OSRM(Open Source Routing Machine), GraphHopper가 오픈소스로 CH를 사용. Google Maps 같은 상용 엔진의 내부 알고리즘은 공개되지 않았으나, 공개된 논문·강연에서는 CH 계열 및 CRP(Customizable Route Planning) 같은 전처리 기법이 활용된다고 알려져 있음.

어떤 방식이든 근간은 Dijkstra (또는 양방향 Dijkstra)다. 1959년의 3페이지짜리 논문이 아직도 최단 경로 엔진의 뼈대.

📖 Sanders & Schultes 2008 — Engineering Fast Route Planning Algorithms

"With multilevel hierarchies and careful engineering, we can compute shortest paths in road networks of continental size in milliseconds on a PC. The underlying algorithm is a bidirectional variant of Dijkstra's."

AI — 4가지 흔한 실수
  1. queue.PriorityQueue 사용: 스레드 안전이라 10-100배 느림. heapq 권장.
  2. 음수 간선에 Dijkstra 적용: 조용히 잘못된 답. 음수 있으면 Bellman-Ford.
  3. lazy deletion 스킵 누락: if d > dist[u]: continue 없으면 같은 노드를 여러 번 처리 → O(VE) degradation.
  4. visited set 사용: 일부 구현은 "처음 pop된 건 확정"이라 visited set 쓰지만, lazy deletion과 함께 써야 정확. 섞어 쓰면 버그.
WHY — 5가지 검증 질문
  1. "이 그래프에 음수 간선 없나?"
  2. "heapq 썼나? lazy deletion 패턴 있나?"
  3. "부모 추적 (parent dict) 필요한가? 경로 복원해야 하면 필수."
  4. "목표 하나만 찾으면 목표 pop 시점에 조기 종료 가능."
  5. "거대한 그래프면 A* (휴리스틱) 또는 bidirectional 고려."

13 — 실습: Network Delay Time (LeetCode #743)

고급

문제

N개 노드와 유향 가중 간선 times[(u,v,w)]. 노드 k에서 신호를 보내면 모든 노드에 도달하는 데 걸리는 시간은? (도달 불가면 −1.)

network_delay.py
import heapq
from collections import defaultdict

def network_delay_time(times, n, k):
    graph = defaultdict(list)
    for u, v, w in times:
        graph[u].append((v, w))

    dist = {k: 0}
    pq = [(0, k)]
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist.get(u, float('inf')): continue
        for v, w in graph[u]:
            nd = d + w
            if nd < dist.get(v, float('inf')):
                dist[v] = nd
                heapq.heappush(pq, (nd, v))

    if len(dist) != n: return -1
    return max(dist.values())


print(network_delay_time([[2,1,1],[2,3,1],[3,4,1]], 4, 2))   # 2

"모든 노드 도달 시간" = dist의 최댓값. 모든 노드에 도달 불가능이면 len(dist) != n.

14 — 연습문제 (12문제)

WHY 문제 1

Dijkstra가 음수 간선에서 실패하는 이유를 한 문장으로.

풀이 펼치기
답

그리디 "확정된 노드는 다시 안 본다"는 "이미 확정된 노드보다 더 짧은 경로가 음수 간선으로 나중에 발견될 수 있다"는 가능성과 충돌.

PREDICT 문제 2

V=10⁵, E=5×10⁵ 그래프에서 binary heap Dijkstra 시간 복잡도 실측 추정.

풀이 펼치기
답

O((V+E) log V) = 6×10⁵ × 17 ≈ 10⁷. Python에서 약 2-5초. C++이면 0.1초 수준.

FIX 문제 3

다음 Dijkstra의 버그는?

buggy.py
def d(g, s):
    dist = {s: 0}
    pq = [(0, s)]
    while pq:
        d, u = heapq.heappop(pq)
        for v, w in g[u]:
            if v not in dist or d+w < dist[v]:
                dist[v] = d+w

풀이 펼치기
버그

stale entry skip 누락. (d, u) pop 후 if d > dist[u]: continue가 없으면 같은 u를 여러 번 처리해 O(VE+) degrade. 맨 위에 stale 체크 추가.

WHY 문제 4

Fibonacci heap이 이론적으로 빠른데 실무에서 안 쓰는 이유?

풀이 펼치기
답

구현 복잡도 ↑ + 상수 인자 큼. O(V log V + E)의 이론적 개선이 실제론 binary heap보다 느린 경우가 많음. 특히 sparse 그래프에서 binary heap이 캐시 친화적.

BUILD 문제 5

Dijkstra에 목표 t가 있을 때 조기 종료 최적화. 어떻게?

풀이 펼치기
답

heappop에서 u == t면 즉시 return. 그 시점의 dist[t]가 최단 거리 (이미 확정). 큰 그래프에서 쿼리 단축 핵심.

PREDICT 문제 6

A*의 h(n)이 진짜 거리보다 크다면 무엇이 잘못되나?

풀이 펼치기
답

최단 경로를 놓칠 수 있음 (최적성 깨짐). h가 과대평가하면 실제 최단 경로 위의 노드의 f 값이 부풀어 heap에서 먼저 처리 안 됨. admissible 조건 (h ≤ 진짜)이 최적성 보장의 전제.

BUILD 문제 7

격자에서 0은 길, 1은 벽. (0,0)에서 (n-1,n-1)까지 최단 경로 길이 (LeetCode #1091의 변형, 8방향).

풀이 펼치기
힌트

비가중 → BFS로 충분. 8방향 이웃. deque 사용, visited 체크. O(n²).

만약 칸마다 "비용"이 있으면 (가중) Dijkstra. heap으로.

WHY 문제 8

무방향 그래프에서 Dijkstra가 작동하는 이유 (유향 그래프용으로 설계됐는데).

풀이 펼치기
답

무방향 간선 (u,v,w)는 두 유향 간선 (u→v,w)와 (v→u,w)로 표현. Dijkstra는 그대로 작동. 음수 간선이 없다면 (대부분 무방향 그래프는 거리라 양수).

PREDICT 문제 9

Bidirectional Dijkstra가 단방향 대비 얼마나 빠른가 (경험적)?

풀이 펼치기
답

경험적으로 약 2배 (탐색 영역 절반). CH 같은 전처리와 결합하면 대륙 규모에서 수백~수천 배.

DESUGAR 문제 10

networkx.dijkstra_path(G, s, t)는 내부적으로 어떻게 동작하는가?

풀이 펼치기
답

heapq 기반 Dijkstra + parent 추적. 목표 노드 pop 시 조기 종료. parent 따라 역순 복원. 가중치 속성(weight) 이름 지정 가능.

대형 그래프엔 networkx.bidirectional_dijkstra가 더 빠름.

풀이 펼치기

WHY 문제 12

lazy deletion 패턴이 없으면 정확성은 유지되지만 복잡도만 degrade하는가?

풀이 펼치기

정확성은 유지 (RELAX가 멱등). 하지만 이미 확정된 노드를 다시 처리 → 각 노드 최대 O(deg(u))번 처리 → 최악 O(VE log V) 가능. stale check로 O((V+E) log V) 보장.

최단 경로 II — Bellman-Ford, Floyd-Warshall

Dijkstra가 못 하는 걸 할 수 있는 두 알고리즘.

  • Bellman-Ford (CLRS §22.1) — 음수 간선 허용, 음수 사이클 감지. O(VE).
  • Floyd-Warshall (CLRS §23.2) — 모든 쌍 최단 경로 (All-Pairs). O(V³) but 단순한 3중 루프.
  • SPFA — Bellman-Ford의 실전 최적화. 큐 기반.
  • Johnson 알고리즘 — sparse APSP. Bellman-Ford + Dijkstra 조합.
  • 언제 무엇을 — 음수·밀도·쿼리 수에 따른 선택 가이드.
