01 — 이번 주 이야기: 100점이 아니라 만점

수학 시험에서 100점을 받았습니다. 다음 시험도 100점을 받을 거라고 확신할 수 있나요?
없습니다. 운이 좋았을 수 있고, 시험 범위 밖 문제는 못 풀 수도 있습니다.
"100점 받았다"는 사실은 "앞으로 항상 맞다"를 보장하지 않습니다.

프로그래밍에서도 같습니다.

  • 친구가 보낸 정렬 함수 sort_clever()를 받았습니다.
  • 10개 입력으로 테스트 — 모두 통과.
  • 100개 무작위 입력 — 모두 통과.
  • 1만 개 입력 — 모두 통과.
  • 그런데 정말 모든 입력에 대해 맞나요?
WHY — 테스트는 "버그가 있다"를 찾을 뿐 "버그가 없다"를 증명하지 못한다

"Testing shows the presence, not the absence of bugs." — Edsger Dijkstra (1969)
통과한 테스트가 100만 개여도, 1만큼의 안 본 입력에서 틀릴 수 있습니다. 가능한 입력의 수는 무한입니다.

이번 주의 도구는 루프 불변식(loop invariant)입니다. "이 알고리즘이 모든 입력에 대해 옳다"를 수학적으로 증명하는 방법입니다.
이번 주의 핵심은 — "테스트는 보충재이고, 증명이 본진"이라는 시각의 전환입니다.

루프 불변식 3속성 = 수학적 귀납법. Init = base case, Maintain = inductive step, Terminate = 결론. 셋 모두 만족하면 루프의 모든 반복에 대해 P가 성립.
합계 함수의 루프가 옳다는 "증명"을 시각화. 매 반복 직전마다 s가 "지금까지 본 원소들의 합"임을 유지 → 루프 끝나면 자동으로 전체 합. 테스트 없이도 100% 확신.

출처: E. W. Dijkstra. "Notes on Structured Programming". EWD249, 1970, §3.

02 — 왜 증명하는가? 테스트의 한계

초급

"내가 지금 안 본 입력"의 무게

배열 정렬 함수는 길이 n의 배열, 각 원소가 int 범위인 모든 경우에 동작해야 합니다.
n=10, 각 원소가 0~99라면 — 가능한 입력은 100¹⁰ = 10²⁰ 개. 1초에 10억 개를 검사해도 3,000년.
"테스트로 모든 경우 확인"은 물리적으로 불가능합니다.

중급

유명한 사고 — 검증 안 된 정확성이 만든 재앙

  1. Ariane 5 (1996) — 발사 37초 후 폭발. 64-bit float를 16-bit int로 변환하는 코드가 오버플로우. 테스트는 통과했지만 새 발사체의 가속도 범위에서 처음 발생.
  2. Therac-25 (1985~87) — 방사선 치료기가 정상 용량의 100배를 환자에게 투여, 사고 6건·환자 사망 발생. race condition (입력 편집과 단일 스케줄러 사이의 타이밍 버그) — 테스트로는 절대 잡히지 않는 종류의 버그.
  3. Pentium FDIV bug (1994) — Intel CPU의 나눗셈 회로 오류. Intel 손실 4.75억 달러. 1024개 셀의 lookup table 중 5개가 잘못. 테스트로 발견될 확률은 90억 분의 1.

세 사고의 공통점: "이런 입력은 절대 안 들어올 거야"의 가정이 깨졌습니다.
증명은 "그런 가정 자체를 만들지 않는" 방법입니다.

중급

증명 vs 테스트 — 보완 관계

구분테스트증명
커버 범위유한한 입력모든 입력 (∀)
찾는 것버그 존재버그 부재
비용실행 시간사고 시간
실패 시"이 입력에서 틀림""가정이 틀림"
강점실제 환경 확인모든 경우 보장

현실의 프로젝트는 "핵심 로직은 증명, 통합은 테스트"로 둘을 섞습니다. 이번 주는 그 "증명" 쪽을 손에 익힙니다.

📖 원서 인용 — CLRS, §2.1 "Insertion sort"

"루프 불변식은 알고리즘이 왜 옳은지 이해하는 데 도움이 된다. 우리는 루프 불변식에 대해 세 가지를 보여야 한다: 초기성(initialization), 유지성(maintenance), 종료성(termination). 처음 둘이 성립하면 — 수학적 귀납법처럼 — 루프가 끝날 때까지 불변식이 유지된다."

출처: Cormen, Leiserson, Rivest, Stein. Introduction to Algorithms, 4th ed., MIT Press, §2.1, p. 19.

03 — Hoare triple — {P} S {Q}

중급

"코드의 정확성"이란 정확히 무엇인가

"코드가 맞다"는 막연한 표현입니다. 1969년 C. A. R. Hoare가 정확한 정의를 만들었습니다.

{P} S {Q}: "P가 참인 상태에서 S를 실행하면, 끝났을 때 Q가 참이다."
  • P — 사전조건(precondition). 실행 전에 만족해야 할 가정.
  • S — 명령문(statement). 실행할 코드.
  • Q — 사후조건(postcondition). 실행 후에 만족해야 할 결과.

이걸 한 줄로 쓴 것이 호어 트리플(Hoare triple)입니다.

중급

예제 — 절댓값 함수

abs_simple
def abs_simple(x):
    # P: x is an integer
    if x < 0:
        x = -x
    # Q: x >= 0 and (원래 x의 절댓값과 같다)
    return x

호어 트리플로 쓰면:

{ x ∈ ℤ }   if x<0: x = -x   { x ≥ 0 ∧ x = |x₀| }

여기서 x₀는 실행 전의 x를 가리키는 표기. 사후조건이 사전조건의 변수를 참조할 때 자주 씁니다.

고급

부분 정확성 vs 전체 정확성

  • 부분 정확성(partial correctness): S가 멈춘다면 Q가 성립.
  • 전체 정확성(total correctness): S가 반드시 멈추고 Q가 성립.

호어 트리플 {P} S {Q}는 보통 부분 정확성입니다. 종료성을 따로 증명해야 전체 정확성이 됩니다 (§09에서 다룸).

왜 둘로 나누나? — 종료성은 별도의 어려운 문제이기 때문입니다 (정지 문제는 일반적으로 결정 불가능). 두 단계로 나누면 각각을 따로 다룰 수 있습니다.

고급

호어 논리의 추론 규칙들

호어가 같은 1969년 논문에서 제시한 추론 규칙(일부):

대입 공리(Assignment axiom)

{ Q[x ← E] }   x := E   { Q }

"Q에서 x를 모두 E로 바꾼 식이 사전조건이면, 대입 후에 Q가 성립한다."
예: Q가 x > 5이면 사전조건은 E > 5. 대입 x := y+1 후 x > 5가 성립하려면 사전 y+1 > 5가 필요.

순차 합성(Sequential composition)

{P} S₁ {R},   {R} S₂ {Q}  ⟹  {P} S₁; S₂ {Q}

조건문(Conditional)

{P ∧ B} S₁ {Q},   {P ∧ ¬B} S₂ {Q}  ⟹  {P} if B then S₁ else S₂ {Q}

반복문(Loop)

{I ∧ B} S {I}  ⟹  {I} while B do S {I ∧ ¬B}

여기서 I가 바로 루프 불변식! 다음 절의 핵심.

출처: C. A. R. Hoare. "An Axiomatic Basis for Computer Programming". Communications of the ACM, 12(10), 1969, pp. 576–583.

전공자

가장 약한 사전조건 (weakest precondition, Dijkstra 1975)

Hoare 추론 규칙은 "사후조건 Q와 코드 S가 주어졌을 때, 가장 약한 사전조건 P가 무엇인가"를 묻는 함수형으로 다시 쓸 수 있습니다.

wp(S, Q) = "S 실행 후 Q가 성립하기 위한, S 실행 전 가장 약한 조건"
  • wp(skip, Q) = Q
  • wp(x := E, Q) = Q[x ← E]
  • wp(S₁; S₂, Q) = wp(S₁, wp(S₂, Q))
  • wp(if B then S₁ else S₂, Q) = (B ⇒ wp(S₁, Q)) ∧ (¬B ⇒ wp(S₂, Q))

wp는 역방향(backward) 분석을 줍니다. 결과(Q)에서 출발해 사전 조건을 계산. 이게 정형 검증 도구(Dafny, Why3, Spark/Ada)의 핵심 엔진.

출처: E. W. Dijkstra. "Guarded Commands, Nondeterminacy and Formal Derivation of Programs". CACM, 18(8), 1975.

04 — 루프 불변식의 3속성

중급

한 줄 요약

루프 불변식 I은 "루프의 매 반복 직전과 직후에 항상 참인 명제"입니다.
"이 루프가 무엇을 하고 있는지"를 한 문장으로 압축한 것이 좋은 불변식입니다.

중급

3속성 — Initialization · Maintenance · Termination

불변식 I가 정확성 증명에 쓰이려면, 다음 셋을 모두 보여야 합니다.

  1. 초기성 (Initialization) — 루프 첫 반복 직전에 I가 참이다.
  2. 유지성 (Maintenance) — I가 참인 상태에서 한 번 반복하면, 다음 반복 직전에도 I가 참이다.
  3. 종료성 (Termination) — 루프가 끝날 때 I가 우리가 원하는 결과 Q를 알려준다.

1과 2는 수학적 귀납법입니다. 기저(Initialization) + 귀납 단계(Maintenance) → I는 모든 반복에서 참.
그리고 종료 시점에 I + 종료 조건이 합쳐져 우리가 원하던 Q가 나옵니다.

WHY — 왜 "루프 직전"인가, "직후"가 아닌가?

불변식은 보통 "i번째 반복을 시작하기 직전의 상태"에 대해 씁니다. 루프 본문이 끝나면 변수가 갱신된 상태로 다음 반복 직전이 됩니다. 그래서 "첫 반복 직전(=초기성)"부터 "마지막 반복 직후(=루프 종료 직전)"까지 같은 명제가 늘 참이어야 합니다.
직후 기준으로 써도 본질은 같습니다 — 일관성만 지키면 됩니다.

고급

증명 양식 — 4단

앞으로의 모든 사례 증명은 다음 4단으로 씁니다.

  1. 불변식 I 명시: 매 반복 직전에 무엇이 참이어야 하는지 한 문장.
  2. 초기성 (Initialization): 첫 반복 직전(루프 진입 시) I 성립.
  3. 유지성 (Maintenance): I ∧ (루프 조건 B) 가정 → 본문 실행 후에도 I.
  4. 종료성 (Termination): 루프 종료 시 I ∧ ¬B → 사후조건 Q.

추가로 — 종료성의 "루프가 정말로 멈춤" 부분은 변형 함수(variant function)로 따로 보입니다 (§09).

전공자

좋은 불변식 vs 나쁜 불변식

불변식은 "참이기만 하면" 되는 게 아니라 — 충분히 강해서 사후조건 Q를 함의해야 합니다.

  • 너무 약한 불변식: I = True. 항상 참이지만 정보가 없음. 종료 시 Q를 끌어낼 수 없음.
  • 너무 강한 불변식: 첫 반복 직전(초기성)에 거짓일 수도 있음. 증명 시작부터 막힘.
  • 좋은 불변식: "지금까지 처리한 부분에 대해 우리가 원하는 결과가 부분적으로 만들어져 있다"의 형태. 알고리즘의 진행 상태 그 자체.

관용적 패턴: 불변식은 보통 "인덱스 i 시점에서, 부분 결과가 ___ 까지는 옳다"의 형태를 띱니다.

  • 합계: "s = sum(nums[0..i-1])"
  • 최댓값: "m = max(nums[0..i-1])"
  • 선형 탐색: "nums[0..i-1]에는 target이 없다"
  • 삽입 정렬: "nums[0..j-1]은 정렬된 상태로, 원래의 nums[0..j-1] 원소들의 순열"

05 — 사례 1: 합계 함수의 정확성

초급

증명할 코드

total
def total(nums):
    s = 0
    for i in range(len(nums)):
        s = s + nums[i]
    return s

호어 트리플:

{ nums는 숫자 리스트, len(nums) = n }   total(nums)   { 반환값 = nums[0] + nums[1] + … + nums[n-1] }
중급

불변식

I(i): 매 반복 직전에   s = nums[0] + nums[1] + … + nums[i-1]

특수 약속: i=0이면 합은 빈 합 = 0.

고급

증명 (4단)

1. 초기성 (Initialization)

첫 반복 직전: i=0, s=0. I(0): s = (빈 합) = 0. ✓

2. 유지성 (Maintenance)

I(i)가 성립하고 (s = nums[0] + … + nums[i-1]), 본문 s = s + nums[i]를 실행하면:
새로운 s = nums[0] + … + nums[i-1] + nums[i] = nums[0] + … + nums[i].
그리고 i가 i+1로 갱신됨. 따라서 다음 반복 직전 (i' = i+1, s' = nums[0] + … + nums[i' - 1]). I(i+1) ✓.

3. 종료성 (Termination)

range(n)은 정확히 n번 반복 후 종료. 종료 직후 i = n.
I(n): s = nums[0] + … + nums[n-1] = (전체 합).
함수가 s를 반환 → 사후조건 Q 성립. ∎

WHY — "for문이 정확히 n번 반복"의 근거

Python의 range(n)은 0, 1, …, n-1을 차례로 내는 이터레이터 — 정확히 n번 반복 후 StopIteration으로 종료.
이 자체가 또 하나의 작은 정확성 — Python range의 의미론에 우리가 의존하고 있습니다.

06 — 사례 2: 선형 탐색의 정확성

중급

증명할 코드

linear_search
def linear_search(nums, target):
    for i in range(len(nums)):
        if nums[i] == target:
            return i
    return -1

사후조건 Q: 반환값 r에 대해

  • r ≠ -1 ⇒ nums[r] = target
  • r = -1 ⇒ ∀j: nums[j] ≠ target
고급

불변식과 증명

I(i): 매 반복 직전에   nums[0..i-1] 안에는 target이 없다.

1. 초기성

i=0. nums[0..-1] = 빈 슬라이스. 빈 집합에는 어떤 원소도 없음 → I(0) ✓ (vacuously true).

2. 유지성

I(i) 성립 + 본문 진입. 두 분기:

  1. nums[i] == target → return i. 이 경우 사후조건 Q 즉시 성립 (반환값 = i, nums[i] = target). ✓
  2. nums[i] ≠ target → 다음 반복 진행. 다음 직전 i' = i+1, 그리고 I(i)에 의해 nums[0..i-1] 안에 target 없음. 게다가 nums[i] ≠ target도 알아냈음. 따라서 nums[0..i] = nums[0..i'-1] 안에 target 없음 → I(i') ✓.

3. 종료성

두 가지 종료 경로:

  • 중간에 return i — 이미 위에서 Q 성립.
  • for문이 모두 끝남 → i = n. I(n): nums[0..n-1] 안에 target 없음 → ∀j ∈ [0, n): nums[j] ≠ target. 함수가 -1 반환 → Q 성립. ∎

07 — 사례 3: 삽입 정렬 (CLRS §2.1)

고급

증명할 코드 (CLRS 의사코드를 Python으로)

insertion_sort
def insertion_sort(A):
    n = len(A)
    for j in range(1, n):           # 외부 루프 — j = 1, 2, ..., n-1
        key = A[j]
        i = j - 1
        while i >= 0 and A[i] > key:    # 내부 루프 — key를 자리 찾기
            A[i + 1] = A[i]
            i = i - 1
        A[i + 1] = key
    return A

사후조건 Q: 반환된 A는 (a) 원래 A의 순열이고 (b) 비내림차순 정렬되어 있다.

고급

외부 루프 불변식 (CLRS의 표현 그대로)

I_outer(j): j번째 반복 직전,   A[0..j-1]은 원래 A[0..j-1]의 원소들로만 구성된 정렬된 부분배열이다.

1. 초기성

j=1. A[0..0] = [A[0]] — 원소 한 개는 정렬됨, 원래 A[0]과 같음. I_outer(1) ✓.

2. 유지성

I_outer(j) 성립 → A[0..j-1] 정렬됨. 본문은 key = A[j]를 A[0..j-1]의 적절한 자리에 끼워 넣고 큰 원소들은 한 칸씩 밀어냄.
핵심 보조정리: while 루프가 끝났을 때 i+1은 key가 들어갈 자리이고, A[i+1+1..j]는 원래 A[i+1..j-1]을 한 칸씩 민 결과 = 정렬됨.
key를 A[i+1]에 대입 → A[0..j]가 정렬됨, 그리고 원래 A[0..j]의 순열. I_outer(j+1) ✓.

3. 종료성

외부 루프가 j = n에서 끝남. I_outer(n): A[0..n-1] = A 전체가 정렬됨, 원래 A의 순열. → 사후조건 Q. ∎

전공자

내부 while의 불변식 (보조정리 증명)

"유지성 단계"에서 한 줄로 넘긴 보조정리를 풀어보면 — 내부 while 자체에도 불변식이 필요합니다.

I_inner(i): 매 while 반복 직전,
(a) A[i+2..j]는 정렬된 비내림차순,
(b) A[i+2..j]의 모든 원소는 key보다 크다,
(c) A[0..i]는 원래 A[0..j-1]의 처음 i+1개 원소와 같다 (이 영역은 손대지 않았음).

초기성

i = j-1. A[i+2..j] = A[j+1..j] = 빈 슬라이스 → (a)(b) 자동. (c) i+1 = j, 처음 j개는 변경 없음 → ✓.

유지성

I_inner(i) ∧ (i ≥ 0 ∧ A[i] > key). A[i+1] = A[i] 실행 후, A[i+1] = (옛 A[i]) > key. A[i+2..j]는 그대로 (정렬+key보다 큼).
따라서 새 영역 A[i+1..j] = A[(i')+2..j] (i' = i-1)도 (a)(b) 만족. (c) 외부 영역 안 건드림. I_inner(i-1) ✓.

종료 + 삽입

while 종료 조건: i < 0 ∨ A[i] ≤ key.
A[i+1] = key 실행 후, A[0..i]는 ≤ key (모두 정렬되어 있고 마지막 원소가 key 이하), A[i+2..j]는 > key, A[i+1] = key. 따라서 A[0..j] 전체가 정렬됨.

(a) 원래 A[0..j-1]의 모든 원소가 보존됨 (한 칸 밀려있을 뿐) + key 추가 → A[0..j]는 원래 A[0..j]의 순열.

이로써 보조정리 성립 → 외부 유지성도 정식으로 증명됨. ∎

📖 원서 인용 — CLRS §2.1

"외부 for 루프의 시작 시점에서, A[1..j-1]은 원래 A[1..j-1]에 있던 원소들로 구성되며 정렬된 순서로 있다. 이 성질을 형식적인 명제로 만들면 루프 불변식이 된다."

출처: CLRS 4th ed., §2.1 "Insertion sort", pp. 19–20. (CLRS는 1-인덱스를 사용; 본 문서는 Python의 0-인덱스로 변환.)

08 — 사례 4: 이분 탐색의 함정

고급

증명할 코드

binary_search
def binary_search(A, target):
    # P: A는 비내림차순 정렬된 배열
    lo, hi = 0, len(A) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if A[mid] == target:
            return mid
        elif A[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1
고급

불변식

I: 매 반복 직전,   target이 A 안에 있다면 반드시 A[lo..hi] 안에 있다.
(즉: A[0..lo-1]에도 없고 A[hi+1..n-1]에도 없다.)

초기성

lo=0, hi=n-1. A[lo..hi] = A 전체. "있다면 A 안에 있다" — 자명히 ✓.

유지성

I 성립 + 루프 조건 (lo ≤ hi). mid = (lo+hi)//2 ∈ [lo, hi]. 세 분기:

  • A[mid] = target → return mid. 사후조건 Q 즉시 성립.
  • A[mid] < target → A는 정렬됨, 따라서 A[lo..mid] 모든 원소 ≤ A[mid] < target. 즉 target이 A[lo..mid]에는 없음. lo' = mid+1로 갱신 → I 유지 (A[lo'..hi] 안에만 있을 가능성).
  • A[mid] > target → 대칭적으로 A[mid..hi] 안에 없음. hi' = mid-1로 갱신 → I 유지.

종료성

종료 조건: lo > hi → A[lo..hi]가 빈 영역. I에 의해 "target이 있다면 빈 영역에 있다" — 없다는 의미. -1 반환 → 사후조건 ✓.

전공자

"왜 mid = (lo + hi) // 2가 위험한가" — Java/C에서는 버그

Java/C 스타일 (Python에서는 안전)
// Java
int mid = (lo + hi) / 2;  // ⚠️ lo + hi가 int 오버플로우

JDK의 Arrays.binarySearch가 9년간 이 버그를 지니고 있었습니다 (Joshua Bloch, 2006).
lo + hi가 Integer.MAX_VALUE를 넘으면 음수로 wrap-around → mid가 음수 → ArrayIndexOutOfBoundsException.
해결: mid = lo + (hi - lo) / 2. 이건 절대 오버플로우 안 남.

Python에서는 무관. int가 임의 정밀도라 오버플로우 자체가 없음. 그래도 코드 이식할 때 알아두면 좋음.

출처: J. Bloch. "Extra, Extra — Read All About It: Nearly All Binary Searches and Mergesorts are Broken". Google Research Blog, 2006-06-02.

전공자

off-by-one 함정 — 2가지 주요 변형 비교

이분 탐색은 인덱스 표기에 크게 2가지 주요 변형이 있습니다 (닫힘 vs 반열림). 각각 불변식이 다릅니다.

구간초기 (lo, hi)루프 조건이동
[lo, hi] (닫힘)(0, n-1)lo ≤ hilo=mid+1, hi=mid-1
[lo, hi) (반열림)(0, n)lo < hilo=mid+1, hi=mid

가장 자주 틀리는 곳: 루프 조건의 ≤ vs <, hi의 갱신값 mid vs mid-1. 불변식을 명시적으로 적으면 이 함정이 보입니다.

Python 표준 라이브러리 bisect는 [lo, hi) 변형을 사용합니다. bisect.bisect_left(a, x)의 사후조건: "a[i-1] < x ≤ a[i] for i = bisect_left(a, x)" — 즉, x를 끼워 넣어도 정렬이 유지되는 가장 왼쪽 위치.

참조: Python 3.14 docs — library/bisect.html.

09 — 종료성과 변형 함수

고급

"정말 멈추는가"는 별도의 문제다

불변식의 3속성 중 종료성은 사실 두 가지를 합친 것입니다.

  1. (a) 멈춘다 — 루프가 무한히 돌지 않는다.
  2. (b) 멈췄을 때 Q — 종료 시점의 상태가 사후조건을 함의한다.

앞 사례들에서 (a)는 "for문은 정확히 n번 반복" 같은 자명한 사실에 의존했습니다. 그런데 while문에서는 자명하지 않습니다.

고급

변형 함수 (Variant Function)

종료성의 (a)를 보이는 표준 도구가 변형 함수 V입니다.

변형 함수 V는 다음 두 조건을 만족하는 자연수 값 함수:
(1) 매 반복마다 V가 엄격하게 감소 (V_new < V_old)
(2) V는 아래로 유계 (보통 V ≥ 0)

자연수의 정렬된 집합에 무한 감소 수열은 없으므로(well-foundedness) — 루프는 유한 번 반복 후 종료. ∎

예제: 이분 탐색의 변형 함수

V = hi - lo + 1 (현재 탐색 영역 크기). 매 반복마다 lo가 ≥ mid+1이 되거나 hi가 ≤ mid-1이 됨 → 영역 크기가 최소 1만큼 감소. 그리고 V ≥ 0 (lo > hi가 되면 종료).
따라서 이분 탐색은 반드시 종료 (구체적으로 ≤ ⌈log₂(n+1)⌉ 번 반복).

예제: 삽입 정렬 내부 while의 변형 함수

V = i + 1. 매 반복마다 i가 1씩 감소 → V가 1씩 감소. 종료 조건 i < 0 → V = 0 도달. ✓

전공자

well-founded ordering — 자연수가 아니어도 된다

변형 함수의 치역이 꼭 ℕ일 필요는 없습니다. well-founded(정렬된, 무한 하강 사슬이 없는) 부분 순서면 충분.

  • ℕ — 표준
  • ℕ × ℕ (사전순) — 이중 루프의 종료 증명에 사용 (외부 i 감소가 우선, 같은 i면 내부 j 감소)
  • 유한 집합의 부분집합들의 ⊂ 순서
  • 유한 트리의 구조적 순서

Ackermann 함수의 종료 증명은 ℕ 만으로는 안 되고 — 다중 변형 (m, n) 사전순으로 증명합니다.

정지 문제와의 관계

정지 문제(Halting Problem)는 일반적으로 결정 불가능합니다 (Turing 1936) — 임의의 프로그램이 멈추는지 자동으로 판별하는 알고리즘은 존재하지 않음.
그러나 특정 알고리즘이 멈추는지를 사람이 변형 함수로 증명하는 것은 가능. 정형 검증 도구(Dafny, Coq)는 사용자에게 변형 함수를 명시할 것을 요구합니다.

10 — 재귀의 정확성: 강귀납법

고급

재귀에는 루프 불변식이 안 통한다 — 대신 귀납법

재귀 함수의 정확성은 입력 크기에 대한 수학적 귀납법으로 증명합니다.

  1. 기저(base case): 가장 작은 입력에서 옳다.
  2. 귀납 단계(inductive step): n보다 작은 모든 입력에 대해 옳다고 가정하고, n에서도 옳음을 보인다 (강귀납법 / strong induction).

예: factorial

factorial
def factorial(n):
    if n == 0:
        return 1
    return n * factorial(n - 1)

주장: factorial(n) = n! (모든 n ≥ 0에 대해).

기저: n=0. 코드 반환 1 = 0! ✓.
귀납: n > 0 가정하에 factorial(n-1) = (n-1)! (귀납 가설). 코드 반환 n × factorial(n-1) = n × (n-1)! = n!. ✓ ∎

고급

예: 병합 정렬

merge_sort
def merge_sort(A):
    if len(A) <= 1:
        return A
    mid = len(A) // 2
    left = merge_sort(A[:mid])
    right = merge_sort(A[mid:])
    return merge(left, right)

주장: merge_sort(A) = sorted(A) (정렬된 결과).

기저: len(A) ≤ 1 → A 그대로 반환. 길이 0 또는 1은 자명히 정렬됨. ✓
귀납: len(A) = n > 1. 부분 호출의 입력 크기 n/2 < n. 강귀납 가설로 left와 right는 정렬됨. merge가 두 정렬 배열을 정렬된 하나로 합친다는 별도 보조정리가 있다면 → 결과는 정렬됨. ✓ ∎

merge 자체의 정확성은 따로 루프 불변식으로 증명 — 이게 "큰 증명을 작은 증명으로 분해"하는 표준 양식.

전공자

구조적 귀납법 (Structural Induction)

트리, 리스트 같은 재귀적 자료구조에는 자연수 귀납법 대신 구조적 귀납법을 씁니다.

이진 트리에 대한 명제 P: 모든 이진 트리 T에 대해 P(T) 성립함을 보이려면:

  1. P(빈 트리) 성립
  2. P(left), P(right) 성립을 가정하고 P(Node(v, left, right)) 성립

예: 이진 트리의 노드 수와 잎 수의 관계 — "내부 노드 i개를 가진 이진 트리는 잎 i+1개를 가진다"는 구조적 귀납으로 증명.

이는 "well-founded 집합 위의 귀납법"의 한 형태이며, 자연수 귀납과 본질이 같음.

11 — AI는 정확성을 어떻게 다루는가

AI의 일반적 한계

LLM에게 "이 함수가 옳은가?"를 물으면 — 보통 "코드가 합리적으로 보입니다" 류의 답을 합니다. 이는 증명이 아니라 "비슷해 보임"입니다.

  • AI는 학습 데이터에서 본 비슷한 코드 패턴과 매칭해 "이건 정렬 코드 같다"고 답합니다.
  • 그러나 off-by-one, 경계값, 빈 입력, 중복 키 같은 함정에 약합니다.
  • "이 코드의 루프 불변식을 명시해줘"라고 요청하면 — 그럴듯한 명제를 내지만 증명까지 검증된 것은 아님.
WHY — 우리가 직접 불변식을 적어야 하는 이유

AI에게 코드만 주면 그럴듯한 답이 옵니다. 그러나 AI에게 "이 불변식이 맞는지 검증해줘"로 묻는 순간, 우리가 제안한 불변식의 사고 흐름이 들어갑니다.
좋은 사용 패턴: 사람이 불변식을 적고 → AI에게 "이 불변식이 초기/유지/종료 3속성을 만족하는지 한 단계씩 검토" 요청. 이러면 AI가 정형 증명의 보조 도구가 됩니다.

증명 보조 도구 — AI 시대의 새 풍경

  • Coq, Lean 4 — 수학 정리의 기계 검증. 페르마 소정리부터 연속체 가설까지.
  • Dafny, Why3, Spark/Ada — 실용 코드의 정확성 검증. 사용자가 사전/사후조건과 불변식을 명시하면 SMT solver가 자동 검증.
  • seL4 — 운영체제 마이크로커널 전체를 Isabelle로 정형 증명한 사례. 약 1만 줄 코드, 20만 줄 증명.

AI가 "Lean 증명을 자동 생성"하는 연구도 활발합니다 (DeepMind FunSearch, OpenAI o1 등). 그러나 사람이 무엇을 증명할 것인가 — 명제와 불변식 — 를 명확히 정의해야 합니다. 이 부분은 외주 불가.

12 — 실습: 깨진 코드 찾기

중급

실습 1: 이 함수는 어디서 틀리는가?

find_max_buggy
def find_max_buggy(nums):
    m = 0
    for x in nums:
        if x > m:
            m = x
    return m

불변식 후보: "매 반복 직전, m = max(nums[0..i-1] ∪ {0})". 그런데 사후조건 Q는 "m = max(nums)"입니다.
입력 [-3, -1, -2]에서 코드 반환 0, 정답 -1. 실패! 불변식과 사후조건이 어긋남 — m의 초기값 0이 max에 안 들어가는 입력에서 깨짐.

수정: m = nums[0]로 초기화하고 빈 리스트는 별도 처리. 새 불변식: "m = max(nums[0..i-1])" (i ≥ 1).

고급

실습 2: 무한 루프를 만들어보기

binary_search_buggy
def binary_search_buggy(A, target):

    while lo <= hi:
        mid = (lo + hi) // 2
        if A[mid] == target:
            return mid

            lo = mid              # ⚠️ +1 빠짐

실습 3: 불변식이 너무 약한 사례

remove_duplicates
# 정렬된 배열에서 중복을 제거 (in-place, 새 길이 반환)
def remove_dup(A):
    if not A: return 0
    k = 1
    for i in range(1, len(A)):
        if A[i] != A[i-1]:    # 직전 원소와 비교
            A[k] = A[i]
            k += 1
    return k

약한 불변식: "k ≤ i". 이건 사실이지만 사후조건을 함의 못 함.
좋은 불변식: "A[0..k-1]은 원래 A[0..i-1]의 모든 서로 다른 원소를 정렬된 순서로 담고 있다." 이게 종료 시 (i = n) "A[0..k-1] = unique elements of A"를 줌.
"맞는 코드 + 약한 불변식"이면 — 코드는 옳지만 우리는 그걸 모르는 상태. 강한 불변식을 찾는 게 증명의 본질.

변형 함수 고급 문제 7

다음 코드에 대한 변형 함수를 제시하고 종료성을 증명하시오.

def f(n):

        else:

    return n

불변식 + 증명 고급 문제 8

반전 함수(reverse)에 대해 in-place 구현의 불변식을 적고 정확성을 증명하시오.

reverse_inplace
def reverse(A):
    lo, hi = 0, len(A) - 1
    while lo < hi:
        A[lo], A[hi] = A[hi], A[lo]
        lo += 1
        hi -= 1
    return A
정답

I: A[0..lo-1]은 원래 A[n-1..n-lo]의 역순 ∧ A[hi+1..n-1]은 원래 A[n-1-hi..0]의 역순 ∧ A[lo..hi]는 원래 A[lo..hi] 그대로.

변형 함수: V = hi - lo. 매 반복 V가 2 감소.

초기: lo=0, hi=n-1 → 양 끝이 빈 영역, 가운데가 원본 = ✓. 유지: 한 쌍 교환 + lo,hi 이동 → 새 양 끝 영역에 한 쌍씩 추가됨. 종료: lo ≥ hi → 가운데 영역이 비거나 1원소 → 전체가 역순. ∎

재귀 증명 고급 문제 9

power(x, n) = xⁿ 의 정확성을 강귀납법으로 증명하시오.

power
def power(x, n):
    if n == 0:
        return 1
    half = power(x, n // 2)
    if n % 2 == 0:
        return half * half
    else:
        return half * half * x
정답

주장: power(x, n) = xⁿ (모든 n ≥ 0).

기저: n=0 → 1 = x⁰ ✓.

귀납: n > 0. n//2 < n이므로 강귀납 가설로 power(x, n//2) = x^(n//2). 짝수 n: half² = x^(2(n/2)) = xⁿ ✓. 홀수 n: half²·x = x^(2((n-1)/2)) · x = x^(n-1) · x = xⁿ ✓. ∎

호어 논리 전공자 문제 10

대입 공리 {Q[x ← E]} x := E {Q}를 사용해, {y > 5} x := y + 1 {x > 5}가 성립함을 보이시오.

정답

Q = (x > 5). Q[x ← y+1] = (y+1 > 5) = (y > 4).
대입 공리에 의해 {y > 4} x := y+1 {x > 5} 성립.
우리에게 주어진 사전조건은 (y > 5)인데, y > 5 ⇒ y > 4 이므로 강화 규칙(rule of consequence)으로 {y > 5} x := y+1 {x > 5} 성립. ∎

wp 계산 전공자 문제 11

wp("if x > 0 then y := x else y := -x", y ≥ 0)을 계산하시오.

정답

wp(if-then-else) = (B ⇒ wp(S₁, Q)) ∧ (¬B ⇒ wp(S₂, Q)).
wp(y := x, y ≥ 0) = (x ≥ 0).
wp(y := -x, y ≥ 0) = (-x ≥ 0) = (x ≤ 0).
∴ wp = (x > 0 ⇒ x ≥ 0) ∧ (x ≤ 0 ⇒ x ≤ 0) = True.
즉, 어떤 x에 대해서도 사후조건이 성립 — 그래서 이 함수가 절댓값 함수의 정확한 구현. ∎

실제 사례 분석 전공자 문제 12

Joshua Bloch (2006)가 발견한 Java Arrays.binarySearch 버그 — (low + high) / 2 — 가 Python에서는 왜 발생하지 않는지, 그리고 발생하지 않는다고 해서 Python 코드가 "정확성이 보장된" 코드인지 답하시오.

정답

왜 안 일어나는가: Python int는 임의 정밀도(arbitrary precision). low + high가 어떤 값이든 정확한 정수로 계산되므로 wrap-around가 없음. CPython 내부적으로는 PyLongObject가 자릿수에 비례한 메모리를 자동 할당.

"정확성이 보장된" 코드인가: 부분 정확성과 종료성은 별개. Python 자체의 오버플로우는 없지만, 알고리즘 로직(루프 조건, 인덱스 갱신 등)에 다른 종류의 off-by-one 버그가 있을 수 있음. "이 언어는 안전하다"는 명제는 "이 코드가 옳다"를 함의하지 않음. 정확성 증명은 여전히 별도로 필요.

14 — 이번 주 정리

  1. 테스트는 버그 존재를 찾고, 증명은 버그 부재를 보장한다 (Dijkstra).
  2. 코드의 정확성은 호어 트리플 {P} S {Q}로 표현된다.
  3. 루프의 정확성은 루프 불변식 I의 3속성으로 보인다: 초기성, 유지성, 종료성.
  4. 3속성 중 1, 2는 수학적 귀납법의 기저/귀납. 3은 종료 조건 + I → 사후조건.
  5. "멈춤" 자체는 변형 함수 V로 별도 증명 — 매 반복 엄격히 감소 + 아래로 유계.
  6. 재귀의 정확성은 입력 크기에 대한 강귀납법으로.
  7. 좋은 불변식은 "지금까지 처리한 부분에 대해 옳다"의 형태. 너무 약하면 사후조건을 함의하지 못한다.
  8. AI는 패턴 매칭으로 그럴듯한 답을 주지만, 증명의 검증은 사람이 불변식을 명시해야 한다.
