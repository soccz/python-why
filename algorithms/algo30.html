종합 — 알고리즘 설계 전략
30주 여정의 통합

29주간 200개 이상의 알고리즘과 자료구조를 배웠습니다. 1810년 Gauss의 소거법부터 2020년 Karlin-Klein-Oveis Gharan의 TSP 1.5-ε까지 — 약 210년의 알고리즘 역사를 한 권의 책으로.
하지만 알고리즘 학습의 진짜 목표는 "특정 알고리즘 외우기"가 아니라 "새 문제에서 적절한 도구를 고르는 안목". 이번 주는 그 안목을 기르기 위한 종합 — 문제 분류 → 패러다임 선택 → 복잡도 직관 → 실무 적용의 사고 흐름.
마지막 질문: "30주 뒤 여러분은 어떤 알고리즘 사용자 · 설계자 · 연구자가 되어 있는가?"

01 — 이번 주 이야기: 30주의 여정

"Computer science is no more about computers than astronomy is about telescopes."
— Edsger W. Dijkstra (attributed)

우리가 걸어온 길

30주 동안 1810년 Gauss(소거법)에서 2020년 Karlin-Klein-Oveis Gharan(TSP 1.5-ε)까지 약 210년의 알고리즘 역사를 훑었습니다. 핵심 인물들 — Dijkstra, Knuth, Bellman, Floyd, Warshall, Ford, Fulkerson, Cook, Karp, Hopcroft, Tarjan, Hoare, Huffman, Bayer, Fenwick, Pugh, Pagh, Bloom, Karger, Bentley, Fortune, Christofides — 한 세기의 CS를 만든 사람들.

단순한 알고리즘의 놀라운 깊이

처음 이진 탐색을 배웠을 때는 "당연한 로직"처럼 보였지만, Jon Bentley는 "이진 탐색을 버그 없이 구현할 수 있는 프로그래머는 10%도 안 된다"고 썼습니다. Knuth는 "이 문제는 1946년 처음 출판됐지만, 첫 번째 올바른 구현은 1962년에 나타났다"고 지적합니다. 16년. 이 단순한 알고리즘의 엣지 케이스(mid의 오버플로우, 경계 조건)가 그토록 까다롭습니다.

같은 단순함의 깊이를 Dijkstra, Floyd-Warshall, B-Tree, DP 점화식, NP-completeness 증명, Christofides 1.5 모두 공유합니다. 간단해 보이는 통찰이 수십 년의 연구 이후에야 정제되었음을 우리는 30주 동안 배웠습니다.

알고리즘이 바꾼 세상

  • Dijkstra 1959 → Google 지도, GPS 내비게이션, 인터넷 라우팅.
  • Bellman 1957 → 강화학습(Q-learning, AlphaGo), 최적 제어 이론.
  • Cook-Karp 1971-72 → 현대 암호학의 안전성 가정.
  • Huffman 1952 → ZIP, JPEG, MP3, PNG.
  • Bayer-McCreight 1972 → PostgreSQL, MySQL, SQLite 인덱스.
  • Bloom 1970 → Chrome Safe Browsing, 블록체인 SPV.
  • Karger 1997 → Akamai, CDN, 분산 캐시.
  • KMP/BM 1977 → grep, 텍스트 에디터, BLAST.
  • Ford-Fulkerson 1956 → 항공 승무원 스케줄링, 이미지 segmentation, 스포츠 순위.

교과서에만 있는 수학이 아니라 — 여러분이 매일 쓰는 서비스의 뼈대가 알고리즘입니다.

02 — 새 문제를 만났을 때 — 사고 흐름

중급

6단계 사고 프로세스

  1. 문제 이해: 입력/출력 정확히 명시. 제약 조건(크기, 정수/실수, 음수 허용, 시간/메모리 한계). "무엇을 해야 하는가"를 한 문장으로.
  2. 분류: 익숙한 문제 유형 중 어디에 속하는가? (다음 섹션 참고)
  3. 순진한 해법: 우선 brute force로 풀어보기. O(n!), O(2n), O(n³) 등. 정확성 확인이 우선.
  4. 최적화: 순진 해법이 너무 느리면 — DP(중복 제거)? 그리디(탐욕 선택)? 자료구조(효율 질의)? 수학적 통찰?
  5. 구현: 엣지 케이스(빈 입력, 단일 원소, 중복, 음수, 오버플로우). 작은 예제로 검증.
  6. 복잡도 증명/분석: 실제로 요구되는 복잡도를 달성했는지. 최악·평균·공간.
고급

실수하지 않기 위한 체크리스트

  • 🎯 문제 제약: n 크기는? 시간 한계는? (n=10⁵면 O(n²) 가능, n=10⁹면 O(n log n) 필수.)
  • 🎯 특수 구조: 그래프가 DAG? 가중치 비음수? 이분 그래프? 문자열 short alphabet?
  • 🎯 질의 패턴: 한 번 계산? 반복 질의? 업데이트? (prefix sum vs segment tree.)
  • 🎯 근사 허용?: 정확해 필수인가 (1±ε)-approx로 충분한가?
  • 🎯 메모리: 모든 데이터 RAM 가능? 디스크 스트리밍? 외부 메모리 알고리즘?
  • 🎯 병렬: 단일 CPU? GPU? 분산 클러스터?

03 — 문제 분류 트리

구조 인식: "이 문제에 특수 구조가 있는가?" (DAG, 이분, 비음, 작은 알파벳) • 변환: "기존 유명 문제로 환원할 수 있는가?" (매칭 → max-flow, 배낭 → DP) • 복잡도: 입력 크기에서 "가능한 방법" 역추론. (n=10⁸면 O(n log n) 이하만.) • trade-off: 정확도 vs 속도 vs 공간 vs 구현 복잡도.
문제 분류 트리: 새 문제를 다섯 주요 영역 중 하나로. 각 영역 내에서 특수 구조·제약·질의 패턴으로 적절 도구 선정. 이 트리는 가이드라인 — 실제론 영역 간 교차(예: 그래프 + DP)가 많음.

고급

어느 패러다임을 쓸 것인가 — 결정 플로우

  1. 문제가 "최적해 존재 증명 필요" or "유일해 확정"인가?
    • Yes → 정확 알고리즘: D&C, Greedy(증명), DP, 정확한 탐색(Backtracking/B&B).
    • 근사로 충분 → 근사 알고리즘 (Vertex Cover 2-approx, Christofides 1.5).
  2. 부분 문제가 서로 독립?
    • Yes → D&C (merge sort, closest pair).
    • 중복 → DP.
  3. 지역 최선이 전역 최선? (증명 가능)
    • Yes → Greedy. Exchange argument로 증명.
    • No → DP 또는 Backtracking.
  4. 모든 가능성 탐색 필요 + 가지치기 가능?
    • 존재/개수 → Backtracking.
    • 최적 + 하한 계산 가능 → B&B.
  5. 최적해가 NP-hard?
    • 실제 입력이 특수 구조 → FPT/휴리스틱/SAT solver.
    • 근사로 충분 → 근사 알고리즘.
    • 작은 n → Brute force 또는 DP.

05 — 복잡도 직관 — n에서 알고리즘 예측

중급

경험 법칙 — n에서 "가능한 알고리즘" 역추론

CPU가 초당 ~108 연산 가정 (Python은 107, C/C++은 109). 1초 제한에서:

n에서 복잡도 역추론. 경쟁 프로그래밍·실무 모두 이 표가 출발점. "n=10⁵, 1초" 제한이면 O(n²)=10¹⁰은 불가능, O(n log n)=10⁶·17≈2·10⁷은 여유.

06 — Part I 복습 — 기초 (Algo 01-06)

중급

핵심 개념

  • Algo 01-02: 복잡도 표기(Big-O, Big-Θ, Big-Ω). Master theorem.
  • Algo 03-05: 비교 기반 정렬의 하한 Ω(n log n). Merge sort, Quick sort, Heap sort, Timsort(하이브리드).
  • Algo 06: 이진 탐색과 변형(lower_bound, upper_bound, 답 이진 탐색).

Part I의 교훈: 정렬은 알고리즘의 기초 — 대부분 문제가 "먼저 정렬하면" 훨씬 쉬워짐. Timsort(Python 내장)의 현명함을 배웠습니다.

07 — Part II 복습 — 자료구조 (07-13)

중급

핵심 개념

  • Algo 07: Timsort, 하이브리드의 세련됨.
  • Algo 08-09: 동적 배열(CPython list), 연결 리스트, deque(blocks).
  • Algo 10: 해시 테이블. Collision 해결, load factor.
  • Algo 11: BST, AVL, Red-Black. 균형 유지의 회전.
  • Algo 12: Heap, Priority Queue, Python heapq.
  • Algo 13: 트리 순회(preorder, inorder, postorder, level-order), Morris.

교훈: 올바른 자료구조 선택은 알고리즘 절반의 완성. 해시는 평균 O(1), 트리는 정렬 유지의 O(log n), heap은 최솟값 O(log n) — 각각의 강점과 trade-off.

08 — Part III 복습 — 그래프 (14-18)

중급

핵심 개념

  • Algo 14: BFS, DFS, 위상 정렬, SCC.
  • Algo 15: Dijkstra (비음 가중치 SSSP, heap 기반).
  • Algo 16: Bellman-Ford (음수 허용, V-1 relax), Floyd-Warshall (APSP O(V³), 5줄), SPFA, Johnson.
  • Algo 17: MST — Kruskal(Union-Find), Prim(heap), Borůvka(병렬). Cut Property가 3개 알고리즘의 공통 기반.
  • Algo 18: Max-Flow — Ford-Fulkerson, Edmonds-Karp, Dinitz. Max-Flow Min-Cut 쌍대성. 이분 매칭 환원.

교훈: 많은 문제가 그래프로 환원 가능. "이 문제를 그래프로 볼 수 있나?"가 첫 질문이 될 만큼 그래프는 범용 모델.

09 — Part IV 복습 — 설계 패러다임 (19-21)

중급

핵심 개념

  • Algo 19: Greedy의 일반 원리. Activity Selection, Huffman, Fractional Knapsack. Matroid(Rado-Edmonds). 0/1 배낭은 왜 그리디로 안 되나.
  • Algo 20: DP I — Bellman 1957, memoization vs tabulation, 피보나치 O(2ⁿ)→O(n), 막대 자르기, 동전 교환, 집 털기.
  • Algo 21: DP II 고전 — 0/1 배낭, LCS, 편집 거리(Wagner-Fischer 1974), LIS (patience O(n log n)), 행렬 체인.

교훈: "그리디가 되는가?" → Exchange argument로 검증. 안 되면 DP. DP는 "점화식 먼저 펜으로, 메모이제이션은 한 줄 @cache".

10 — Part V 복습 — 탐색/해시 (22-25)

중급

핵심 개념

  • Algo 23: Branch & Bound. Land-Doig 1960, Little 1963 TSP. LP relaxation. CPLEX/Gurobi/HiGHS.

11 — Part VI 복습 — 고급 (26-29)

중급

핵심 개념

교훈: 30주 마지막에 도달한 주제들은 "단일 알고리즘"이 아니라 "자료구조 생태계"와 "복잡도의 지도". 실제 큰 시스템은 이들의 조합.

12 — 메타 교훈 — 알고리즘이 가르친 것

고급

6가지 근본 원리

  1. 분할과 정복 — 큰 문제를 작은 문제로. Merge sort부터 재귀의 철학.
  2. 기억과 재사용 — 같은 계산을 두 번 하지 말라. DP, 해시, 캐시.
  3. 구조의 힘 — 올바른 자료구조가 알고리즘의 절반. 정렬된 배열 vs hash vs tree.
  4. 증명과 직관의 춤 — 그리디는 exchange argument로, DP는 귀납법으로, NP-complete는 환원으로. "왜 맞는가"가 "어떻게 하는가"만큼 중요.
  5. Trade-off — 시간 vs 공간, 정확도 vs 속도, 최악 vs 평균. 절대 답은 없다.
  6. 추상화와 환원 — 새 문제는 기존 문제로 환원. "이걸 그래프 문제로 볼 수 있나?" "max-flow로?" "DP로?"
전공자

철학적 유산 — 알고리즘이 지식 전반에 남긴 것

  • 복잡도 이론: "풀 수 있는 문제"와 "어려운 문제"의 경계. 인간 인지의 한계와도 연결.
  • 증명의 기계화: SAT solver, 정리 증명기. 수학 증명도 알고리즘화.
  • 최적화의 일반성: Bellman 방정식 → 강화학습 → AI 전반.
  • 랜덤성의 활용: Rabin-Karp, Skip List, Monte Carlo, Miller-Rabin. "결정적이 아니어도 효율적".
  • 근사의 철학: 정확도를 포기해서 얻는 효율. ML과 본질적으로 같은 trade-off.
전공자

인물들의 공통점

  • 단순함에 매달림: Dijkstra 20분, Huffman 기말과제, Floyd 반 페이지 코드 — 우아한 해법은 거의 항상 짧다.
  • 분야 횡단: Bellman(OR → DP), Ford(군사 → BF), Bloom(spell check → CDN), Bentley(기하 → 구간).
  • 동시 발견: Cook-Levin, Christofides-Serdyukov, Jarník-Prim-Dijkstra. "때가 무르익으면" 여러 곳에서 동시에.
  • 긴 잠복기: Borůvka 1926 → 1965까지 서방에 안 알려짐. Ryabko 1989 → Fenwick 1994가 이름 차지.

  • 데이터베이스 내부: PostgreSQL 소스 읽기, RocksDB 기여, 트랜잭션/MVCC.

  5. Skiena "The Algorithm Design Manual": 경쟁/실무 도구로 접근.

14 — 종합 연습문제 (12문제)

WHY 문제 1

"n=10⁶ 정점, 10⁷ 간선 그래프의 최단 경로." 어떤 알고리즘?

풀이 펼치기
답

음수 간선 없으면 Dijkstra + heap: O((V+E) log V) = 1.1·10⁷ · 20 ≈ 2·10⁸ — 몇 초 내. 음수 있으면 BF O(VE) = 10⁶·10⁷ = 10¹³ — 불가능. Johnson은 APSP용(O(VE+V²log V))이라 SSSP엔 과잉. 현실적 대안은 SPFA(평균 괜찮음, 최악 BF와 같음) 또는 음수 사이클 여부만 BF로 먼저 확인 후 재가중치(reweighting).

PREDICT 문제 2

"n=10개 도시 TSP" vs "n=1000개 도시 TSP" — 각각 어떤 접근?

풀이 펼치기
답

n=10: Held-Karp DP O(n²·2ⁿ) = 10⁵ — 즉시. 또는 순진 순열 10! = 3.6M — 몇 초. n=1000: Metric이면 Christofides 1.5-approx (몇 초). 정확해 원하면 Concorde 같은 전용 solver. 휴리스틱(Lin-Kernighan) 실무 표준.

BUILD 문제 3

"수십 GB 로그 파일에서 가장 빈번한 IP 100개"를 찾으려면?

풀이 펼치기
답

외부 메모리 알고리즘 필요. Count-Min Sketch(streaming, 근사)로 top-k 추정. 또는 MapReduce로 hash-partition → 각 파티션에서 heap top-100 → merge. 정확한 top-k은 space Ω(n) 필요하지만, 근사는 O(log n) 공간으로 가능.

DESUGAR 문제 4

"쇼핑몰 상품 10억 개 중 키워드 자동완성" 설계.

풀이 펼치기
답

Trie + 각 노드에 상위 k개 후보(빈도/가중치) 저장. prefix 입력 시 O(|prefix|)에 후보 리스트. 상품 10억이면 메모리 부담 — FST(Elasticsearch suggester) 또는 Radix tree로 압축. 실시간 업데이트엔 batch 재빌드.

WHY 문제 5

"부품 재고 최적화"에서 DP로 풀리는지 판단하는 기준.

풀이 펼치기
답

조건: (1) 상태 공간이 이산적이고 작음. (2) Optimal substructure — 전체 최적이 부분 최적의 결합. (3) Overlapping subproblems — 같은 상태 재방문. 만약 연속 변수거나 제약이 많으면 LP/ILP(Algo 23). 확률적이면 MDP + Value Iteration(Bellman 방정식 재등장).

BUILD 문제 6

"친구 관계 그래프에서 2촌 친구 추천" 설계.

풀이 펼치기
답

사용자 u에서 BFS 깊이 2 → 2촌 후보. 순위: 공통 친구 수(Jaccard-like) 또는 관심사 유사도. 대규모엔 pre-compute — 각 사용자의 "2촌 후보 top-k"를 배치 작업으로. LinkedIn의 초기 "People You May Know" 알고리즘.

PREDICT 문제 7

"10만 개 문서에서 plagiarism 검사" 시스템.

풀이 펼치기
답

각 문서를 shingles(k-그램)으로 분해 → MinHash 또는 Winnowing(Rabin-Karp 기반)로 fingerprint 추출. LSH(Locality-Sensitive Hashing)로 유사 문서 후보 빠르게. O(n²) 비교 회피. MOSS(Stanford)의 원리.

DESUGAR 문제 8

"실시간 지도 앱에서 교통 반영 경로 재계산" 아키텍처.

풀이 펼치기
답

도로망을 Contraction Hierarchies(CH)로 전처리 (도로망 고정). 교통 가중치는 실시간 업데이트. CH + Bidirectional Dijkstra로 대륙 규모 밀리초 쿼리 가능. 거시적 라우팅은 CH, 국지적 lane-level은 A*. Google Maps, OSRM 구조.

WHY 문제 9

같은 문제가 "LeetCode Easy"와 "ICPC World Finals" 사이의 차이.

풀이 펼치기
답

Easy는 알고리즘 식별이 핵심 (DP? BFS?). ICPC는 관찰·창의성·조합이 핵심 — 여러 알고리즘을 엮거나 영리한 관찰로 복잡도를 줄이기. 예: "DP를 segment tree로 전이 최적화"(Convex Hull Trick), "mo's algorithm으로 offline 처리". 기본기가 같아도 추상화 층이 다름.

BUILD 문제 10

"n=50 도시에 가족 관광 여행 스케줄링. 모든 도시 방문 + 최저 비용".

풀이 펼치기
답

전형적 TSP. n=50은 정확 해법 중간 — Held-Karp O(n²·2ⁿ) = 2500·10¹⁵ 불가능. B&B(Algo 23) 또는 ILP solver(CPLEX, Gurobi) 가능. 근사로 Christofides 1.5 + 2-opt 개선. 관광이라 정확 최적 불필요 — Lin-Kernighan 휴리스틱이 실무 최선. Python networkx.traveling_salesman_problem.

AI 문제 11

AI 코딩 보조의 시대에 알고리즘 공부의 의미.

풀이 펼치기
답

AI는 이미 알려진 패턴을 빠르게 적용. 하지만:

  • 새 문제를 분류할 수 있어야 AI에게 적절한 질문.
  • AI의 답이 맞는지 검증하려면 알고리즘 이해 필수.
  • AI가 잘못된 알고리즘을 제안할 때(NP-hard에 순진 brute force 등) 교정 가능.
  • "왜 이 알고리즘이 작동하는가"의 이해는 AI가 대체 못 함.

오히려 AI 시대에 메타 이해의 가치 상승. 알고리즘 이해가 AI와 대화하는 공통 언어.

WHY 문제 12

30주 중 가장 아름답다고 느낀 알고리즘과 그 이유.

풀이 펼치기
답 — 정답 없음, 개인적 선택

후보들:

  • Dijkstra: 20분 커피에서 태어난 우아함. 그리디가 전역 최적을 보장하는 기적.
  • Floyd-Warshall: 5줄의 3중 for-loop로 모든 쌍 최단경로.
  • KMP failure function: "실패에서 배움"이라는 철학적 깊이.
  • Max-Flow Min-Cut: 두 전혀 다른 문제가 동치인 쌍대성의 아름다움.
  • Fenwick Tree의 x&-x: 이진 표현의 트릭 하나로 O(log n) 달성.
  • Bloom Filter: "공간을 위해 약간 틀려도 OK"라는 도발.
  • Cook-Levin: 임의의 NP 문제를 SAT으로 환원하는 메타 증명.
  • FFT: 다항식 곱셈이 주파수 영역에서 단순 곱셈이 되는 기적.

각자의 답이 다를 것 — 자신이 "깊이 와닿은" 알고리즘이 자신의 스타일을 말해줌.
