01 — 이번 주 이야기: 도서관 책 찾기

상상해보세요. 당신은 도서관 사서입니다.
어떤 학생이 와서 "홍길동전 좀 찾아주세요"라고 합니다.
같은 도서관, 같은 책, 같은 학생. 그런데 두 가지 일하는 방식이 있습니다.

방식 A — 처음부터 한 권씩

책장 맨 앞에서 시작해서 한 권 한 권 제목을 확인합니다. 운이 좋으면 첫 번째 책, 운이 없으면 마지막 책.

  • 책이 100권이면? 평균 50번 확인.
  • 책이 1만 권이면? 평균 5,000번 확인.
  • 책이 100만 권이면? 평균 50만 번 확인.

방식 B — 가나다순 정렬을 활용 (이분 탐색)

책이 가나다순으로 꽂혀 있습니다. 책장의 정중앙을 펴서 "ㅎ"이 어느 쪽인지 본 다음, 절반을 버립니다. 다시 남은 절반의 정중앙. 또 절반을 버립니다.

  • 책이 100권이면? 약 7번 확인. (log₂100 ≈ 6.6)
  • 책이 1만 권이면? 약 14번 확인.
  • 책이 100만 권이면? 약 20번 확인.
WHY — 같은 문제, 25,000배 차이

100만 권 도서관에서 방식 A는 50만 번, 방식 B는 20번.
이 차이는 "사서가 더 빠르다"가 아닙니다. 방법이 본질적으로 다릅니다.
알고리즘을 공부한다는 것은 — 같은 문제를 푸는 본질적으로 다른 방법들을 알고, 입력이 커질 때 어떻게 갈라지는지를 예측하는 일입니다.

이번 주의 도구는 빅오 표기법(Big-O Notation)입니다.
"방식 A는 O(n), 방식 B는 O(log n)" — 이 한 줄로 위의 표 전체를 요약할 수 있습니다.
빅오는 시계가 아닙니다. "입력이 두 배가 되면 시간이 몇 배가 되는가"를 보는 안경입니다.

입력 n이 커질 때 각 복잡도가 자라는 속도. O(1)·O(log n)은 거의 평평, O(n²)·O(2ⁿ)은 폭발적. 알고리즘의 "본질적 차이"가 곡선의 모양 차이.
👀 마커가 어떤 카드를 보고 있는지 따라가보세요. 위(빨강): 한 칸씩 천천히. 아래(초록): 가운데서 절반으로 점프. 위는 11번 머물고 아래는 4번 머문 뒤 답을 찾습니다.

02 — 왜 빅오가 필요한가? 시계의 한계

초급

"내 컴퓨터에서는 0.3초인데?"

두 친구가 같은 문제를 풀었다고 합시다. 친구 A는 자기 컴퓨터에서 0.3초, 친구 B는 0.5초가 나왔습니다.
A의 코드가 더 빠른 걸까요? 알 수 없습니다.

  • A가 더 빠른 컴퓨터를 쓰고 있을 수 있습니다.
  • A가 입력 크기를 더 작게 했을 수 있습니다.
  • A의 컴퓨터가 마침 다른 일을 안 하고 있었을 수 있습니다.

시간(시계)으로는 알고리즘 자체를 비교할 수 없습니다. 우리는 컴퓨터·언어·운영체제·온도까지 무관한 척도가 필요합니다.

중급

측정의 4가지 함정

  1. 하드웨어 의존: CPU 클럭, 캐시 크기, 메모리 대역폭이 다르면 같은 코드도 다른 속도가 납니다.
  2. 입력 의존: 정렬된 입력 vs 무작위 vs 역순 — 같은 알고리즘이 10배씩 차이 납니다.
  3. 측정 잡음: GC, OS 컨텍스트 스위치, JIT 워밍업이 들어가면 ±20% 흔들립니다.
  4. 외삽 불가: n=100에서 빠르다고 n=10⁶에서도 빠르리란 보장이 없습니다.

그래서 우리는 두 단계의 측정을 합니다:

  • 이론 분석 (이번 주): 입력 크기 n에 대해 연산 횟수가 어떤 함수로 자라는가? — 이게 빅오.
  • 실측 (보조): timeit으로 실제 환경에서 검증.

이론이 먼저고 실측이 나중입니다. 이론 없이 측정만 하면 "내 컴퓨터에서는…"의 늪에 빠집니다.

📖 원서 인용 — CLRS, Introduction to Algorithms, 3.1절

"우리는 알고리즘의 점근적 효율(asymptotic efficiency)에 관심이 있다. 입력 크기가 한도 없이 커질 때 알고리즘의 실행 시간이 어떻게 증가하는가를 따지는 것이다. 보통, 점근적으로 더 효율적인 알고리즘은 — 작은 입력에서는 예외이지만 — 큰 입력에서 최선의 선택이다."

출처: Cormen, Leiserson, Rivest, Stein. Introduction to Algorithms, 4th ed., MIT Press, Ch. 3.1.

03 — 알고리즘이란 무엇인가?

초급

요리 레시피와 같다

라면 끓이는 법도 알고리즘입니다.

  1. 물 550ml를 냄비에 붓는다.
  2. 물이 끓으면 면과 분말스프, 건더기를 넣는다.
  3. 4분 30초 더 끓인다.
  4. 불을 끄고 그릇에 옮긴다.

입력(물, 라면) → 유한한 단계(1~4) → 출력(끓인 라면). 이것이 알고리즘입니다.
"알아서 잘 끓여라"는 알고리즘이 아닙니다. 모호함이 없어야 합니다.

중급

알고리즘의 5가지 조건 (Knuth)

Donald Knuth가 The Art of Computer Programming, Vol.1 §1.1에서 정한 다섯 가지:

  1. 유한성(Finiteness) — 반드시 끝나야 한다. 무한 루프는 알고리즘이 아니다.
  2. 명확성(Definiteness) — 각 단계가 모호함 없이 정의되어야 한다.
  3. 입력(Input) — 0개 이상의 입력을 받는다.
  4. 출력(Output) — 1개 이상의 출력을 만든다.
  5. 실효성(Effectiveness) — 사람이 종이와 연필로도 (원리상) 수행 가능해야 한다.
고급

계산 모델 — 무엇을 "1번"으로 셀 것인가?

"이 알고리즘은 100번 연산한다"라고 말하려면, "1번"의 단위가 정해져야 합니다.
알고리즘 분석의 표준 모델은 RAM 모델(Random Access Machine)입니다:

  • 기본 산술(+, -, ×, ÷, %), 비교(<, ==), 대입, 배열 임의 접근(a[i]) — 모두 1단위 시간.
  • 분기(if), 함수 호출/반환 — 1단위.
  • 메모리는 무한, 모든 정수는 한 번에 처리.

현실과의 차이: 캐시 미스, 가상 메모리 페이지폴트, 큰 정수의 자릿수 등은 모두 무시합니다.
이 단순화 덕분에 알고리즘을 수학적으로 분석할 수 있습니다.

WHY — Python에서 RAM 모델이 어긋나는 순간

Python의 int는 임의 정밀도(arbitrary precision)입니다. 2 ** 1000의 곱셈은 1단위 시간이 아닙니다 — 자릿수 d에 대해 O(d) 또는 O(d log d).
문자열 비교 s1 == s2도 O(min(|s1|, |s2|))입니다.
이번 주 분석에서 우리는 "비교/연산은 O(1)"이라고 가정합니다. 이 가정이 깨지는 경우는 별도 표기합니다.

04 — 입력 크기 n과 기본 연산

초급

n은 무엇인가?

알고리즘 책에서 n은 "입력의 크기"를 가리키는 약속된 글자입니다.
문제마다 n이 의미하는 것이 다릅니다.

  • 리스트 정렬: n = 리스트의 길이
  • 문자열 검색: n = 문자열의 길이
  • 그래프 탐색: n = 정점(vertex) 수, m = 간선(edge) 수
  • 행렬 곱셈: n = 한 변의 길이 (총 원소는 n²)
중급

"기본 연산"을 세는 법

다음 코드를 봅시다.

합계 구하기 
def total(nums):
    s = 0                  # ← 연산 1번
    for x in nums:         # ← n번 반복
        s = s + x          # ← 매 반복마다 1번
    return s               # ← 1번

총 연산 횟수: 1 + n + 1 = n + 2.
n이 100이면 102번, n이 1,000,000이면 1,000,002번. n에 비례해서 자랍니다.

이렇게 "연산 횟수를 n의 식으로 쓴 것"을 시간 복잡도 함수 T(n)이라고 합니다.
방금 코드는 T(n) = n + 2 입니다.

고급

최선·최악·평균

같은 알고리즘도 입력에 따라 시간이 다릅니다. 세 가지를 구분합니다.

  • 최선(best-case): 가장 운 좋은 입력. 거의 의미 없음.
  • 최악(worst-case): 가장 운 나쁜 입력. 알고리즘 분석의 표준.
  • 평균(average-case): 입력 분포를 가정한 기댓값. 분석이 어려움.

왜 최악을 보는가?

  1. "이 시간 안에 끝난다"는 보장이 된다 (상한).
  2. 실제로 자주 일어난다 — 입력은 무작위가 아니다 (예: 정렬된 입력에 quicksort).
  3. 평균은 입력 분포 가정이 필요한데, 그 가정이 틀리면 무용지물이다.
📖 원서 인용 — CLRS 2.2절

"이 책의 나머지에서는 보통 알고리즘의 최악의 경우 실행 시간에 집중한다. 이는 임의의 입력에 대한 실행 시간의 상한(upper bound)을 제공하기 때문이다."

출처: CLRS 4th ed., §2.2 "Analyzing algorithms".

고급

예: 선형 탐색 (linear search)

linear_search
def linear_search(nums, target):
    for i, x in enumerate(nums):
        if x == target:
            return i
    return -1
  • 최선: target이 0번 인덱스 — 비교 1번 → T_best(n) = 1.
  • 최악: target이 마지막 또는 없음 — 비교 n번 → T_worst(n) = n.
  • 평균: target이 균등하게 분포 → 평균 (n+1)/2 ≈ n/2 → T_avg(n) = Θ(n).

세 경우 모두 "n에 비례"는 같습니다 (상수만 다름). 그래서 점근적으로는 같은 클래스 — O(n).
이제 우리는 "T(n) = n+2", "T(n) = n", "T(n) = n/2 + 0.5"의 차이를 무시하고 O(n) 하나로 묶을 수 있습니다. 다음 절에서 그 정의를 봅니다.

05 — 빅오 표기법 (O) — 상한

초급

한 줄 요약

O(f(n))은 "이 알고리즘은 n이 충분히 클 때 f(n)에 비례하는 시간보다 더 오래 걸리지 않는다"는 약속입니다.

  • O(1) — n에 상관없이 늘 같은 시간
  • O(n) — n이 두 배면 시간도 두 배
  • O(n²) — n이 두 배면 시간은 네 배
  • O(log n) — n이 두 배여도 시간은 거의 그대로 (살짝만 늘어남)
중급

왜 "상한"인가?

위에서 본 합계 함수는 T(n) = n + 2 였습니다. 우리는 이걸 "O(n)"이라고 씁니다.
더 구체적으로: "n이 충분히 클 때, T(n) ≤ 어떤 상수 × n". 예를 들어 c=2를 잡으면, n ≥ 2부터 n + 2 ≤ 2n 이 성립합니다. ✓

O는 부등식입니다. "T(n) = O(n)"은 사실 "T(n)은 어떤 상수 c에 대해 cn보다 작거나 같다 (n이 충분히 큰 곳부터)"의 줄임말입니다.
그래서 T(n) = n + 2 는 O(n)이기도 하고, O(n²)이기도 하고, O(n³)이기도 합니다 — O는 상한이지 정확한 값이 아닙니다.

WHY — "O(n²)인데 왜 O(n)이라고 안 쓰면 안 돼?"

됩니다. 다만 최대한 타이트하게(tight) 잡는 것이 관례입니다. "이 차는 시속 1,000km 이하로 달린다"는 사실이지만 정보량이 0입니다. 정확한 클래스를 잡아야 의미가 있습니다.

고급

형식적 정의

f(n)과 g(n)이 양의 함수일 때:

f(n) = O(g(n)) ⟺ ∃ c > 0, ∃ n₀ > 0 가 존재하여 모든 n ≥ n₀에 대해 0 ≤ f(n) ≤ c·g(n).

한국어로 풀면: "n이 어떤 문턱(n₀)을 넘은 후로는, f(n)이 항상 c배 이내의 g(n)을 따라간다."
두 가지 자유 매개변수 — 상수 c(배수)와 n₀(시작점) — 를 자기에게 유리하게 골라도 됩니다.

예제 1: 3n + 5 = O(n) 증명

c = 4, n₀ = 5 를 잡습니다. n ≥ 5 일 때, 5 ≤ n 이므로:
3n + 5 ≤ 3n + n = 4n = c·n. ✓   따라서 3n + 5 = O(n).

예제 2: n² + 100n = O(n²) 증명

c = 2, n₀ = 100 를 잡습니다. n ≥ 100 일 때, 100n ≤ n·n = n² 이므로:
n² + 100n ≤ n² + n² = 2n² = c·n². ✓

예제 3: 2ⁿ ≠ O(n^k) — 모든 다항식보다 빠르다

어떤 c, n₀, k를 잡아도 결국 2ⁿ > c·n^k 인 n이 나타납니다.
증명 스케치: limn→∞ 2ⁿ / n^k = ∞. 따라서 2ⁿ ∉ O(n^k).

전공자

Part A — Python에서의 빅오 계산: dis로 확인하기

Python의 시간 분석은 "바이트코드 한 줄 = O(1)"이 아닙니다. 중요한 예외들을 짚습니다.

"같아 보이는" 두 합계의 바이트코드
import dis

def sum_loop(nums):
    s = 0
    for x in nums:
        s += x
    return s

def sum_builtin(nums):
    return sum(nums)

dis.dis(sum_loop)
print('---')
dis.dis(sum_builtin)

sum_loop의 본문은 FOR_ITER + LOAD_FAST + BINARY_OP + STORE_FAST + JUMP_BACKWARD 가 매 반복.
sum_builtin은 LOAD_GLOBAL sum + CALL — 그러나 그 CALL 안에서 C 레벨 루프가 같은 n번을 돕니다.
둘 다 O(n). 다만 상수가 다릅니다 (보통 5~10배). 빅오는 같지만 "현실 시간"은 다릅니다.

list.append vs list 앞에 insert

  • lst.append(x) — 평균 O(1) (amortized). 내부 동적 배열의 capacity가 부족할 때만 재할당.
  • lst.insert(0, x) — O(n). 모든 원소를 한 칸씩 뒤로 미뤄야 함.

참조: wiki.python.org/moin/TimeComplexity — 표준 컬렉션의 연산별 빅오.

전공자

Part B — 점근의 위상학(topology): 등급(rank) 구조

함수들 사이에 "점근적으로 작다(≺)" 관계가 있습니다:

f ≺ g   ⟺   limn→∞ f(n) / g(n) = 0

이 관계로 정렬하면:
1 ≺ log log n ≺ log n ≺ √n ≺ n ≺ n log n ≺ n² ≺ n³ ≺ 2ⁿ ≺ n! ≺ nⁿ

"다항식 ≺ 지수" 가 핵심 분기점입니다 — 다항식 시간은 "현실적", 지수 시간은 "이론적으로만 가능"으로 취급됩니다 (이게 P vs NP의 출발선).

로그의 밑은 무관: log₂ n 과 log₁₀ n 은 상수배 차이 (log₂ n = log n / log 2). 그래서 그냥 "log n"이라고만 씁니다.

형식적 정의

f(n) = Ω(g(n)) ⟺ ∃ c > 0, n₀ > 0 ∀ n ≥ n₀   0 ≤ c·g(n) ≤ f(n)

f(n) = Θ(g(n)) ⟺ ∃ c₁, c₂ > 0, n₀ > 0 ∀ n ≥ n₀   0 ≤ c₁·g(n) ≤ f(n) ≤ c₂·g(n)

핵심 정리: f = Θ(g) ⟺ f = O(g) 그리고 f = Ω(g).

작은 o (little-o) — 엄격한 상한

f = o(g) ⟺ limn→∞ f(n)/g(n) = 0.
"f는 g보다 진짜로 느리게 자란다"는 뜻. 예: n = o(n²), log n = o(n), n^99 = o(2ⁿ).

작은 ω (little-omega) — 엄격한 하한

f = ω(g) ⟺ limn→∞ f(n)/g(n) = ∞. o의 반대.

전공자

"= O(g)"의 표기 남용

엄밀히 말하면 O(g)는 함수의 집합입니다. "f = O(g)"는 사실 "f ∈ O(g)"의 줄임말 (Knuth가 채택한 관행).
그래서 "O(n) = O(n²)"는 한 방향 포함 (좌변 ⊂ 우변)만 의미하고 등호의 대칭성이 깨집니다.
Knuth는 TAOCP Vol. 1 §1.2.11에서 이를 "one-way equality"로 부릅니다. 글을 쓸 때는 ∈ 표기가 더 안전합니다.

출처: D. E. Knuth, The Art of Computer Programming Vol. 1, §1.2.11 "Asymptotic Representations".

중급

코드로 보는 각 클래스

O(1) — 상수

O(1)
def first(nums):
    return nums[0] if nums else None

입력 크기와 무관하게 항상 같은 작업.

O(log n) — 로그

O(log n)
def binary_search(sorted_nums, target):
    lo, hi = 0, len(sorted_nums) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if sorted_nums[mid] == target:
            return mid
        elif sorted_nums[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1

매 반복마다 탐색 범위가 절반으로 줄어들기 때문. n → n/2 → n/4 → … → 1. 횟수는 log₂ n.

O(n) — 선형

O(n)
def maximum(nums):
    m = nums[0]
    for x in nums[1:]:
        if x > m:
            m = x
    return m

O(n log n) — 선형로그

O(n log n)
def sort_then_unique(nums):
    nums.sort()              # O(n log n) — Timsort
    result = []
    for x in nums:
        if not result or result[-1] != x:
            result.append(x)
    return result            # 전체 O(n log n)

O(n²) — 이차

O(n²)
def has_duplicate_pair(nums):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] == nums[j]:
                return True
    return False

이중 for문 → 비교 횟수 n(n-1)/2 = Θ(n²).

O(2ⁿ) — 지수

O(2ⁿ)
def fib_naive(n):
    if n < 2:
        return n
    return fib_naive(n-1) + fib_naive(n-2)

매 호출마다 두 번씩 분기 → 호출 트리 크기가 ≈ 2ⁿ. n=40만 되어도 매우 느림.

O(n!) — 팩토리얼

O(n!)
from itertools import permutations

def all_orderings(items):
    return list(permutations(items))   # n! 개

n=12 → 약 5억. n=15 → 약 1.3조. n=20 → 우리 우주에서 처리 불가.

08 — 점근 표기 계산 규칙

중급

실전 5가지 규칙

규칙 1 — 상수는 무시한다

O(5n) = O(n). 상수 5는 컴퓨터 속도, 명령어 한 줄의 비용에 흡수됩니다.
"이건 그냥 더 빠른 컴퓨터에서 돌리는 것과 같다"고 보면 됩니다.

규칙 2 — 가장 큰 항만 남긴다 (지배항)

O(n² + n + 100) = O(n²). n이 커질수록 n² 외의 항은 비율이 0으로 수렴.

두 부분으로 나뉜 코드
def f(nums):
    s = 0
    for x in nums:           # O(n)
        s += x
    for i in range(len(nums)):
        for j in range(len(nums)):   # O(n²)
            ...
    # 전체: O(n + n²) = O(n²)

규칙 3 — 곱셈은 곱셈으로 (중첩 루프)

바깥 루프 O(a), 안쪽 루프 O(b) → 전체 O(a × b).

행렬 순회
for i in range(n):       # O(n)
    for j in range(m):   # O(m)
        do_O1()
# 전체: O(n × m)

규칙 4 — 덧셈은 덧셈으로 (순차)

두 단계가 차례로 실행되면 O(a) + O(b) = O(max(a, b)).

규칙 5 — 함수 호출 비용은 그대로 곱한다

함정 — in 연산자
def slow(nums, queries):
    result = []
    for q in queries:        # O(|queries|)
        if q in nums:        # ⚠️ list의 in은 O(n)
            result.append(q)
    return result
# 전체: O(|queries| × |nums|), 보통 O(n²)

q in nums가 O(1)이 아니라 O(n)이라는 걸 모르면 빅오를 한 단계 잘못 잡습니다.
set으로 바꾸면 O(1) → 전체 O(|queries| + |nums|) = O(n).

고급

주의 — "겉모습 ≠ 빅오"

예 1: 중첩 루프인데 O(n)인 경우

투 포인터
def two_pointer(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo < hi:                  # 보기엔 O(n)
        s = nums[lo] + nums[hi]
        if s == target: return (lo, hi)
        elif s < target: lo += 1
        else: hi -= 1
    return None

while 루프이고 안에 보이는 비용은 O(1)이지만, 각 반복마다 lo+1 또는 hi-1 — 즉 lo와 hi가 만나기까지 정확히 n번 반복.
따라서 O(n). "while = O(n²)"이라는 미신은 틀렸습니다.

예 2: 단일 루프인데 O(n²)인 경우

감춰진 비용
def collect(nums):
    result = ""
    for x in nums:           # n번
        result += str(x)     # ⚠️ Python 문자열은 immutable
    return result

result += str(x)는 매번 새 문자열을 만듭니다. i번째 반복에서 길이 i 문자열을 복사 → 1+2+…+n = O(n²).
해결: "".join(...)를 쓰면 O(n).

WHY — CPython 한정 최적화

실제로는 s += t가 참조 카운트 == 1일 때 in-place로 동작하는 최적화가 있어 O(n)이 되기도 합니다. 그러나 PyPy/Jython/IronPython에는 이 최적화가 없습니다. "문자열 누적은 join"이 안전한 규칙.

예 3: 재귀 호출의 비용

재귀
def f(n):
    if n == 0: return 1
    return f(n-1) + f(n-1)   # ⚠️ 같은 부분 두 번 계산

매 단계마다 호출 수가 2배 → T(n) = 2·T(n-1) + O(1) → T(n) = Θ(2ⁿ).
v = f(n-1); return v + v로 바꾸면 T(n) = T(n-1) + O(1) → Θ(n).

고급

덧셈 vs 곱셈 — 실전 판별

  • 덧셈: 두 단계가 순차로 한 번씩 실행 → max만 남김.
  • 곱셈: 한 단계 안에서 다른 단계가 반복 → 곱한다.

이 둘을 헷갈리면 O(n+m)을 O(nm)으로 부풀리거나 그 반대. 코드를 읽을 때 "이 루프는 저 루프 안에 들어 있나?"를 항상 묻습니다.

09 — 점화식과 마스터 정리

고급

분할 정복 알고리즘은 점화식으로 표현된다

병합 정렬은 입력을 절반으로 나눠 각각 정렬하고 합칩니다.

merge sort 골격
def merge_sort(a):
    if len(a) <= 1:
        return a
    mid = len(a) // 2
    left = merge_sort(a[:mid])      # T(n/2)
    right = merge_sort(a[mid:])     # T(n/2)
    return merge(left, right)       # O(n)

이걸 시간 점화식으로 쓰면:

T(n) = 2·T(n/2) + Θ(n),   T(1) = Θ(1)

이 점화식의 닫힌 형태(closed form)가 바로 Θ(n log n)입니다.

3가지 풀이법

  1. 치환법(substitution) — 답을 추측하고 귀납으로 증명.
  2. 재귀 트리(recursion tree) — 호출 트리를 그려 레벨별 비용을 합산.
  3. 마스터 정리(master theorem) — 정형 패턴이면 공식으로.

방법 2: 재귀 트리로 T(n) = 2T(n/2) + n 풀기

각 레벨의 총 비용을 봅니다.

레벨 수 = log₂ n, 각 레벨 비용 = n → 총 n log n. ∎

전공자

Part A — 마스터 정리(Master Theorem) — CLRS 4.5

형태가 다음과 같은 점화식:

T(n) = a·T(n/b) + f(n),   a ≥ 1,   b > 1

여기서 a는 부분 문제의 개수, b는 분할 비율, f(n)은 분할/결합 비용. 이 점화식의 답은 n^(log_b a)와 f(n) 중 어느 쪽이 큰가에 따라 결정됩니다. 임계 함수를 n^c (c = log_b a)로 둡니다.

경우조건해해석
1 f(n) = O(n^(c-ε)), ε > 0 T(n) = Θ(n^c) 분할이 비싼 경우 — 잎(leaf)이 지배
2 f(n) = Θ(n^c · log^k n), k ≥ 0 T(n) = Θ(n^c · log^(k+1) n) 분할과 잎이 균형 — 모든 레벨이 같은 무게
3 f(n) = Ω(n^(c+ε)), ε > 0
(+ 정규성 조건)		 T(n) = Θ(f(n)) 분할이 싼 경우 — 뿌리(root)가 지배

적용 예제

  1. 병합 정렬: T(n) = 2T(n/2) + n. a=2, b=2, c=log₂ 2 = 1, f(n)=n=Θ(n¹). 경우 2 (k=0). → Θ(n log n).
  2. 이분 탐색: T(n) = T(n/2) + 1. a=1, b=2, c=0, f(n)=1=Θ(n⁰·log⁰). 경우 2 (k=0). → Θ(log n).
  3. Strassen 행렬 곱: T(n) = 7T(n/2) + n². c = log₂ 7 ≈ 2.807. f(n)=n² = O(n^(2.807-ε)). 경우 1. → Θ(n^2.807).
  4. 이진 트리 순회: T(n) = 2T(n/2) + 1. c=1, f(n)=1 = O(n^(1-ε)). 경우 1. → Θ(n).
📖 원서 인용 — CLRS §4.5

"마스터 정리는 a ≥ 1과 b > 1이 상수이고 f(n)이 양의 함수일 때, T(n) = aT(n/b) + f(n) 형태의 점화식에 대해 점근적 해를 제공한다. n/b는 ⌊n/b⌋ 또는 ⌈n/b⌉로 해석할 수 있다."

출처: CLRS 4th ed., §4.5 "The master method for solving recurrences", Theorem 4.1.

전공자

Part B — 마스터 정리가 안 통하는 경우

마스터 정리는 만능이 아닙니다. 다음 같은 점화식은 적용 불가:

  • T(n) = T(n-1) + n — n/b 형태가 아님 (감산형 점화식). → 합 공식으로 직접: T(n) = Θ(n²).
  • T(n) = 2T(n/2) + n log n — 경우 2의 정확한 형태(n^c · log^k n)에 살짝 맞으나 (k=1) Θ(n log² n). 실수로 n log n으로 답하면 안 됨.
  • T(n) = 2T(n/2) + n / log n — 경우 1, 2, 3 어디에도 안 맞음. Akra-Bazzi 정리 필요.
  • T(n) = T(n/3) + T(2n/3) + n — 부분 문제 크기가 다름. Akra-Bazzi 또는 재귀 트리.

Akra-Bazzi 정리는 마스터의 일반화로, 부분 문제 크기가 다르고 비율이 달라도 적용됩니다 (Akra & Bazzi, 1998).

참고: Akra, M.; Bazzi, L. (1998). "On the solution of linear recurrence equations". Computational Optimization and Applications. 10 (2): 195–210.

10 — AI는 빅오를 어떻게 분석하는가

LLM(대규모 언어 모델)에게 코드를 보여주면 빅오를 곧잘 답합니다. 어떻게 가능한 걸까요?

AI가 잘하는 것 — 패턴 매칭

  • 이중 for문 + 안쪽이 O(1) → "O(n²)일 가능성이 높다"
  • while lo <= hi: mid = (lo+hi)//2 → "이분 탐색, O(log n)"
  • def f(n): return f(n-1) + f(n-1) → "지수 재귀, O(2ⁿ)"

AI는 학습 데이터에서 본 무수한 코드와 그 분석을 패턴으로 압축하고 있습니다. 거의 모든 표준 코드의 빅오는 정확히 답합니다.

AI가 자주 틀리는 것 — 감춰진 비용

  1. 언어별 내장 연산의 진짜 비용 — Python list.insert(0, x)가 O(n)인 줄 모를 때가 있음.
  2. "하나처럼 보이는데 두 개" — s = s + t가 새 문자열을 만든다는 점.
  3. amortized vs worst-case — list.append는 평균 O(1)이지만 최악 한 번은 O(n) (재할당). 평균만 답하기 쉬움.
  4. 해시 충돌 — dict 조회는 평균 O(1)이지만 최악 O(n) (적대적 입력).
  5. 입력 의존 — quicksort는 평균 O(n log n) / 최악 O(n²)인데 "O(n log n)"으로만 답하기 쉬움.
WHY — 우리가 빅오를 직접 익혀야 하는 이유

AI에게 "이 코드의 빅오는?"이라 물으면 80% 정확한 답이 옵니다. 하지만 나머지 20%가 시스템을 죽입니다. "평균 O(1)"이라 믿었던 dict가 적대적 입력에서 O(n)이 되어 서버가 다운된 사례 (28C3에서 공개된 hash-collision DoS, CVE-2012-1150 → Python 3.4의 PEP 456 SipHash 도입)가 실제 있었습니다.
AI의 답을 검증하려면 빅오의 정의와 계산 규칙을 직접 알아야 합니다. 빅오는 AI 시대에도 외주 불가능한 사고력입니다.

좋은 질문 패턴

AI에게 빅오를 물을 때는 다음 형태가 더 정확한 답을 끌어냅니다.

  • ❌ "이 코드의 시간복잡도는?"
  • ✅ "이 코드의 최악 시간 복잡도를 입력 크기 n에 대해 분석해줘. 사용된 모든 표준 라이브러리 호출의 비용도 명시해줘."
  • ✅ "여기서 amortized와 worst-case가 다르다면 둘 다 답해줘."
  • ✅ "공간 복잡도도 함께."

11 — 실습: timeit으로 이론과 측정 비교

중급

실험 1: O(n) vs O(n²) — 입력 두 배 실험

"n을 2배로 했을 때 시간이 2배가 되는가, 4배가 되는가" 를 직접 측정합니다.

timeit_compare.py
import timeit, random

def linear(nums, target):
    for x in nums:
        if x == target: return True
    return False

def quadratic(nums):
    n = len(nums)
    cnt = 0
    for i in range(n):
        for j in range(n):
            cnt += 1
    return cnt

for n in [1000, 2000, 4000, 8000]:
    nums = [random.randint(0, 10**9) for _ in range(n)]
    target = -1                                  # 항상 못 찾도록 (worst-case)
    t1 = timeit.timeit(lambda: linear(nums, target), number=100)
    t2 = timeit.timeit(lambda: quadratic(nums), number=1)
    print(f"n={n:5d}  linear={t1*1000:7.2f} ms   quadratic={t2*1000:8.2f} ms")
예상 출력 (값은 환경마다 다름):
n= 1000 linear= 3.10 ms quadratic= 25.40 ms n= 2000 linear= 6.20 ms quadratic= 101.60 ms n= 4000 linear= 12.40 ms quadratic= 406.00 ms n= 8000 linear= 24.70 ms quadratic= 1623.00 ms

n이 2배가 될 때:

  • linear: 시간이 ≈ 2배 → O(n) 확인
  • quadratic: 시간이 ≈ 4배 → O(n²) 확인

이론과 측정이 일치합니다. 이게 "이론을 측정으로 검증하는" 표준 절차입니다.

실험 2: O(log n) — 두 배가 되어도 거의 안 늘어남

log_growth.py
import timeit, bisect

for n in [10**4, 10**5, 10**6, 10**7]:
    sorted_nums = list(range(n))

12 — 연습문제 (12문제)

빅오 판별 초급 문제 1

다음 코드의 빅오는?

코드
def f(nums):
    return nums[0] + nums[-1]
정답 및 해설

O(1)

왜?

인덱싱 두 번과 덧셈 한 번 — n에 무관하게 늘 3번 연산.

빅오 판별 초급 문제 2

다음 코드의 빅오는?

코드
def f(n):
    s = 0
    for i in range(n):
        for j in range(n):
            s += 1
    return s
정답 및 해설

O(n²)

왜?

두 루프 모두 n번 → n × n = n² 회 반복.

한계 추정 초급 문제 3

1초에 10⁸ 연산하는 컴퓨터에서, O(n²) 알고리즘으로 1초 안에 처리 가능한 n의 최대치는?

정답 및 해설

약 n = 10⁴ ((10⁴)² = 10⁸).

왜?

n² ≤ 10⁸ → n ≤ 10⁴. 안전하게 n = 10,000 정도가 한계.

빅오 판별 — 함정 중급 문제 4

다음 코드의 빅오는? (Python의 list 동작을 고려할 것)

코드
def f(nums):
    out = []
    for x in nums:
        out.insert(0, x)        # 앞에 삽입
    return out
정답 및 해설

O(n²)

왜?

list.insert(0, x)는 기존 원소를 한 칸씩 뒤로 미는 O(n) 연산. 이걸 n번 → O(n²).
해결: out.append(x) 후 out.reverse() → O(n). 또는 collections.deque의 appendleft O(1).

빅오 판별 중급 문제 5

다음 코드의 빅오는?

코드
def f(n):
    i = 1
    cnt = 0
    while i < n:
        i *= 2
        cnt += 1
    return cnt
정답 및 해설

O(log n)

왜?

i가 매 반복마다 2배 → 1, 2, 4, 8, …, 2^k. 2^k ≥ n이 되는 순간 멈춤 → k = ⌈log₂ n⌉.

빅오 단순화 중급 문제 6

다음을 가장 타이트한 빅오로 단순화: T(n) = 5n³ + 100n² + 99n + 12345

정답 및 해설

O(n³)

왜?

지배항만 남기고(5n³ → n³) 상수는 무시. n=10에서도 5n³가 다른 항을 압도.

빅오 증명 고급 문제 7

7n + 50 = O(n) 임을 정의에 따라 증명하시오. (c와 n₀를 명시)

정답 및 해설
증명

c = 8, n₀ = 50 으로 잡는다. n ≥ 50 일 때 50 ≤ n 이므로:
7n + 50 ≤ 7n + n = 8n = c·n. ✓   따라서 7n + 50 = O(n). ∎

투 포인터 분석 고급 문제 8

"보기엔 O(n²)지만 실제로 O(n)"인 코드를 직접 만들어 빅오 분석을 해보시오 (투 포인터, 슬라이딩 윈도우 등).

예시 답안
슬라이딩 윈도우
# 합이 정확히 target인 연속 부분배열 개수 (모두 양수)
def count_subarrays(nums, target):
    left, s, cnt = 0, 0, 0
    for right in range(len(nums)):
        s += nums[right]
        while s > target and left <= right:
            s -= nums[left]
            left += 1
        if s == target:
            cnt += 1
    return cnt
분석

안쪽 while가 있어 보이지만, left는 절대 감소하지 않고 right도 증가만 함. left와 right는 각각 최대 n번만 움직임 → 총 ≤ 2n 연산 → O(n).

점화식 풀이 고급 문제 9

T(n) = 4T(n/2) + n 의 점근적 해를 마스터 정리로 구하시오.

정답 및 해설

T(n) = Θ(n²)

풀이

a=4, b=2 → c = log₂ 4 = 2. f(n) = n = O(n^(2-1)) = O(n^(c-ε)) (ε=1).
마스터 정리 경우 1 적용 → T(n) = Θ(n^c) = Θ(n²). ∎

엄밀 증명 전공자 문제 10

2ⁿ ≠ O(n^k) (모든 양의 상수 k에 대해) 임을 증명하시오.

정답 및 해설
증명 (대우)

2ⁿ = O(n^k) 가정 → ∃ c, n₀: n ≥ n₀ ⇒ 2ⁿ ≤ c·n^k.
양변 로그: n·ln 2 ≤ ln c + k·ln n.
n / ln n → ∞ (n → ∞) 이므로, 좌변(n·ln 2)은 우변(상수 + k·ln n)보다 결국 커짐 — 모순.
따라서 2ⁿ ≠ O(n^k). ∎

점화식 풀이 (마스터 외) 전공자 문제 11

T(n) = 2T(n/2) + n log n 의 해를 구하시오. (마스터 정리의 경우 2 확장형 사용)

정답 및 해설

T(n) = Θ(n log² n)

풀이

a=2, b=2, c=log₂ 2 = 1. f(n) = n log n = Θ(n^c · log¹ n), k=1.
경우 2의 확장 → T(n) = Θ(n^c · log^(k+1) n) = Θ(n · log² n). ∎

참고: CLRS §4.5 Theorem 4.1 case 2.

CPython 해석 전공자 문제 12

Python의 list.append가 "amortized O(1)"이라고 합니다. 이 의미를 설명하고, 단일 호출의 최악 시간이 무엇인지 답하시오. CPython의 list 재할당 전략(growth factor)을 함께 설명하시오.

정답 및 해설
amortized 설명

"n번의 연속된 append의 총 비용이 O(n)이다 → 평균 O(1)/회" 라는 의미. 어떤 호출은 O(n) (재할당이 필요한 시점), 다른 호출은 O(1).

CPython 구현 (cpython/Objects/listobject.c — list_resize)

새 capacity = ((size_t)newsize + (newsize >> 3) + 6) & ~(size_t)3 공식으로 약 1.125× 성장 (4바이트 정렬).
"두 배" 성장이 아니라 ~12.5% 추가. 이래도 amortized 분석에서는 여전히 O(1)/append.
증명 스케치: n번 append 시 재할당 횟수는 O(log n)이고, k번째 재할당 비용은 약 1.125^k. 합 = 등비급수 ≈ O(n). 평균 O(1). ∎

참조: cpython/Objects/listobject.c, 함수 list_resize — 공식 문서로는 docs.python.org의 wiki TimeComplexity.

13 — 이번 주 정리

  1. 알고리즘은 유한·명확·실효한 단계의 모음. 같은 문제도 알고리즘에 따라 시간이 갈라진다.
  2. 실측 시간은 환경에 흔들린다 → 입력 크기 n에 대한 점근 분석이 표준.
  3. O는 상한, Ω는 하한, Θ는 정확. 일상에선 O가 통용.
  4. 계산 규칙: 상수 무시 / 지배항만 / 중첩=곱 / 순차=덧셈(=max).
  5. 분할정복은 점화식 T(n) = aT(n/b) + f(n)로 표현, 마스터 정리로 풀 수 있음.
  6. "보기엔 O(n²) → 실제 O(n)"(투 포인터)도, 그 반대(insert(0), 문자열 누적)도 있다. 코드의 속까지 봐야 한다.
  7. AI는 빅오를 잘 답하지만, amortized·worst-case·언어별 비용은 자주 틀린다 — 우리가 검증해야 한다.
