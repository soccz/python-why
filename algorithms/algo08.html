01 — 이번 주 이야기: 공책에서 1억 항목까지

학생 A는 작은 공책을 씁니다. 단어를 한 페이지에 하나씩 적어가다, 공책이 꽉 차면 더 큰 공책으로 옮깁니다. 어떻게 크기를 결정할까요?

전략 A — "한 페이지씩 늘리자"

공책이 꽉 차면 한 페이지만 더 큰 공책으로. 매번 복사해야 하는 페이지 수: 1, 2, 3, ..., n. 총 복사 = n(n+1)/2 = Θ(n²).
n=10,000이면 5천만 번 복사. 너무 느림.

전략 B — "두 배로 늘리자"

꽉 차면 두 배 크기의 공책으로. 복사는 덜 자주, 하지만 일어나면 큰 복사. 복사량: 1, 2, 4, 8, ..., n/2. 총 = n − 1 ≈ Θ(n).
n=10,000이면 약 20,000번 복사. 한 번의 append 평균 비용 O(1).

전략 C — "1/8만 더 늘리자" (Python의 선택)

꽉 차면 1.125배(= 1 + 1/8)만큼만. 복사 총량은 여전히 Θ(n) — 상수 배 차이일 뿐. 메모리 낭비는 전략 B의 1/8 수준.
Python 개발자들의 선택: "CPU 몇 초 덜 쓰는 것보다 메모리 2배 아끼는 게 낫다."

WHY — "가끔의 큰 비용"을 평균에 녹인다

이 세 전략을 정확하게 분석하려면 — amortized analysis라는 도구가 필요합니다.

  • "최악은 O(n)이지만 평균은 O(1)"을 수학적으로 증명하는 방법.
  • "한 작업의 비용"이 아니라 "n개 작업의 총 비용"을 계산 → n으로 나눔.
  • Python의 list.append뿐 아니라, Java ArrayList.add, C++ vector::push_back, Go append() — 모두 같은 분석.

CLRS §17이 통째로 이 주제를 다룹니다. 세 가지 방법(aggregate / accounting / potential)을 알면 "amortized O(1)"을 자유자재로 증명할 수 있습니다.

append는 대부분 O(1) (빈 슬롯), 가끔 O(n) (resize). 평균 비용이 O(1)인 이유는 비싼 resize가 기하급수적으로 드물게 발생하기 때문.
가끔의 큰 비용 (resize)을 평균에 녹인다. 8번 중 1번만 비싸고 나머지는 빠름 → 평균 O(1). 이게 "amortized" 분석의 직관.

02 — 정적 배열 vs 동적 배열

초급

정적 배열 — C의 int arr[100]

"처음에 크기를 정하고 영원히 고정." 크기를 바꾸려면 새로 만들어야 함. C, Fortran 같은 전통 언어의 기본.

  • 인덱싱 O(1): 주소 계산만 (base + i * sizeof).
  • 크기 변경 불가능.
  • append라는 개념 자체가 없음.
중급

동적 배열 — Python의 list

"크기가 필요에 따라 자동으로 늘어남." 사용자 입장에선 "무한히 큰 배열". 내부적으로는:

  • 용량(capacity): 실제 할당된 공간. 예: 64칸.
  • 크기(size): 실제 쓰인 원소 수. 예: 50칸.
  • append 호출 시 size++. size == capacity면 → 새 배열 할당 + 복사.

Python의 list는 본질적으로 동적 배열. C의 array + 자동 resize 로직.

연산정적 배열 (C)동적 배열 (Python list)
인덱싱 A[i]O(1)O(1)
수정 A[i] = xO(1)O(1)
끝에 추가 append❌ 불가amortized O(1)
앞에 추가 insert(0)❌ 불가O(n) ⚠️
중간 삭제 del A[i]❌ 불가O(n)
크기 확장❌ 불가 (새 배열)자동
WHY — Python list는 "배열"일까 "연결 리스트"일까?

초보자 질문: "Python list는 C의 array 같은 건가요, 아니면 linked list인가요?"

답: 동적 배열. 인덱싱은 O(1) (linked list였다면 O(n)). 이름은 "list"지만 내부는 array. Java의 ArrayList, C++의 std::vector와 본질적으로 같음.

C의 int a[100] 같은 고정 배열과의 차이는 "자동 resize 로직" 하나뿐. 이 로직을 이해하는 것이 이번 주의 핵심.

03 — 동적 배열의 구조: append가 하는 일

중급

append의 2가지 시나리오

Case 1: size < capacity (여유 있음)
   A[size] = new_value
   size += 1
   → O(1)

Case 2: size == capacity (꽉 참)
   new_cap = capacity * growth_factor  // Python: 1.125×
   new_array = allocate(new_cap)
   copy A[0..size-1] to new_array
   free old array
   A = new_array; capacity = new_cap
   A[size] = new_value; size += 1
   → O(size) = O(n)

Case 1이 대부분(여유가 있을 때), Case 2는 드물게(꽉 찼을 때). 이 두 경우의 평균적 비용이 우리의 질문.

고급

구조체 레벨 — 실제 메모리 레이아웃

동적 배열 의사구조 (C로)
typedef struct {
    void **items;     // 실제 데이터 포인터 배열
    size_t size;       // 현재 원소 개수
    size_t capacity;   // 할당된 공간
} DynamicArray;


void append(DynamicArray *a, void *x) {
    if (a->size == a->capacity) {
        a->capacity = a->capacity * 9 / 8 + 6;  // 1.125× + 6 (Python 스타일)
        a->items = realloc(a->items, a->capacity * sizeof(void*));
    }
    a->items[a->size++] = x;
}

Python의 list는 이것과 거의 동일. 다만 items가 PyObject*의 배열이라 각 원소가 포인터 — 실제 데이터는 힙에 따로 존재.

04 — Amortized의 직관: "평균 O(1)"의 의미

고급

"평균"의 두 가지 의미 — 혼동 금지

컴퓨터 과학에서 "평균"은 두 가지 맥락이 있습니다.

구분평균 대상예보장
Average-case 입력 분포의 평균 Quicksort Θ(n log n) 평균 확률적 (적대적 입력에는 X)
Amortized 연산 수열의 평균 append Θ(1) amortized 결정적 — 입력 무관, 최악 수열에서도 성립

Amortized는 더 강한 보장. "어떤 입력이든, n번의 append에 드는 총 시간은 Θ(n)"이 결정론적으로 참.

고급

실제 비용 시계열 — 2× growth에서

초기 용량 1에서 2배씩 늘리며 n번 append:

연산 i123456789
size 직전 012345678
용량 꽉? ✓✓✓✓
실제 비용 121311151

첫 4번까지 총 비용 = 1+2+1+3 = 7. 평균 1.75. 첫 9번까지 = 16, 평균 1.78. n이 커져도 평균은 3 이하로 수렴.

n번 append 총 비용 = n + (1 + 2 + 4 + ... + n/2) ≈ n + n = 2n  ⟹  평균 ≤ 2 = O(1)

05 — Aggregate 분석 (CLRS §17.1)

고급

가장 단순한 방법 — "다 더하고 n으로 나누기"

정의: n번의 연산 전체 최악 비용을 n으로 나눈 것을 amortized 비용으로 정의.

amortized_cost = total_cost(n operations) / n

적용이 쉽다는 장점, 모든 연산의 amortized 비용이 같게 나온다는 단점(실제로는 연산별로 다를 수 있음).

Python list append의 aggregate 분석 (2× growth 가정)

n번의 append. 매 append는 비용 1의 "기본 삽입" + 가끔의 "전체 복사". 전체 복사는 용량이 1, 2, 4, 8, ..., 2^k까지 확장될 때 각각 1, 2, 4, 8, ..., 2^k 비용.

기본 삽입: n
전체 복사: 1 + 2 + 4 + ... + 2^⌊log₂ n⌋ ≤ 2n − 1
총 비용: n + 2n − 1 = 3n − 1 = O(n)

amortized_cost = (3n − 1) / n ≈ 3 = O(1)

각 append에 대해 "실제로는 1 또는 n의 비용이 들지만, 평균적으로 3의 비용을 지불하는 것과 같다"고 말할 수 있습니다.

📖 원서 인용 — CLRS 4판, §17.1 "Aggregate analysis"

"In aggregate analysis, we show that for all n, a sequence of n operations takes worst-case time T(n) in total. In the worst case, the average cost, or amortized cost, per operation is therefore T(n)/n. Note that this amortized cost applies to each operation, even when there are several types of operations in the sequence."

06 — Accounting 분석 (CLRS §17.2)

고급

비유 — "저금통에 동전 쌓기"

아이디어: 각 연산에 실제 비용 ≥ 지불하는 amortized 비용이면 문제 없음. 반대로 실제 비용 < 지불이면 남는 금액을 "저금통"에 저장해뒀다가 나중에 비싼 연산 때 꺼내 씀.

조건: 저금통이 절대 음수가 되면 안 됨 (미리 지불 원칙).

고급

Python list append의 accounting (2× growth)

전략: 모든 append에 amortized 비용 $3을 "지불".

  • $1: 이번 append의 실제 비용.
  • $2: 저금통에 저장. 이 $2는 "나중에 이 원소가 복사될 때" 쓸 비용.

증명: 용량이 k에서 2k로 확장되는 순간, 배열에는 k개의 원소. 이 k개를 복사하는 비용은 k.
이 확장 직전에 저금통에 얼마가 있나? 지난 k/2번의 append(= 용량이 k/2 → k로 확장된 이후)에서 각 $2씩 저금. 총 $k.
저금 $k로 실제 복사 비용 $k를 정확히 지불. 저금통 다시 0.

각 연산 amortized = $3,   총 n번의 비용 ≤ $3n = O(n)

Aggregate 분석과 같은 결론. 접근 방식이 다를 뿐.

WHY — 왜 $3인가? (2× growth 기준)
  • $1: 이번 삽입 자체.
  • $1: 자기 자신이 나중에 복사될 때의 비용.
  • $1: 이전 사본의 복사될 때의 비용을 분담 (기하급수라서 유한 상수 안에서 해결).

1.125× growth면 amortized는 $10 (더 큼 — 공식 1 + g/(g−1)로 g=1.125 대입). g가 커질수록 amortized 비용은 단조감소하므로 "2×가 가장 싸다"는 건 아니고, 메모리 여유(공간 낭비) vs 시간(amortized)의 균형점으로 2× 전후를 흔히 씁니다.
그럼 왜 Python은 2×가 아닌 1.125×를 쓸까? → 다음 §08에서.

07 — Potential 분석 (CLRS §17.3)

전공자

가장 형식적인 방법 — Potential Function

자료구조의 상태에 포텐셜 함수 Φ를 정의. 실제 비용이 아닌 "포텐셜 변화"를 고려한 amortized 비용:

amortizedi = actuali + Φ(Di) − Φ(Di-1)

- Di: i번째 연산 후 자료구조 상태.
- Φ는 항상 Φ(D0) = 0이고 Φ(Di) ≥ 0이도록 설계.

총합을 취하면 텔레스코핑:

Σ amortizedi = Σ actuali + Φ(Dn) − Φ(D0)
                                     = total_actual + Φ(Dn)
                                     ≥ total_actual

따라서 총 amortized 비용 ≥ 총 실제 비용. 각 연산에 amortized 상수를 지정하면 전체 최악 상한을 얻을 수 있음.

전공자

Python list append의 potential 분석

포텐셜 정의(2× growth의 경우):

Φ(D) = 2 · size − capacity

- 배열이 가득 차면 capacity = size → Φ = size.
- 확장 직후 capacity = 2·size → Φ = 0.
- Φ ≥ 0 조건: size > capacity/2일 때만 양수. 일반적으로 성립.

경우 1: 확장 없는 append (size < capacity)

  • actual = 1
  • ΔΦ = 2·(size+1) − capacity − (2·size − capacity) = 2
  • amortized = 1 + 2 = 3

경우 2: 확장 있는 append (size = capacity)

  • actual = size + 1 (복사 size + 삽입 1)
  • Φ 이전 = 2·size − capacity = size (꽉 찬 상태)
  • Φ 이후 = 2·(size+1) − 2·capacity = 2 (새 용량 2·old)
  • ΔΦ = 2 − size
  • amortized = (size + 1) + (2 − size) = 3

두 경우 모두 amortized = 3 = O(1). 지불 금액이 실제 비용을 항상 덮음.

📖 원서 인용 — CLRS 4판, §17.3 "The potential method"

"The potential method represents the prepaid work as 'potential energy' that can be released to pay for future operations. We associate the potential with the data structure as a whole rather than with specific objects within the data structure."

WHY — 세 방법 중 어느 것을 쓸까?
방법장점단점
Aggregate 가장 단순, 계산 쉬움 모든 연산에 같은 amortized 부여 — 섬세한 분석 불가
Accounting 직관적 (저금통), 연산별 다른 비용 가능 "저금통 음수 금지"가 어디서나 성립하는지 증명 필요
Potential 가장 형식적, 복잡한 자료구조에도 적용 Φ 찾기가 creative — 전문가 수준 기법

실무에서는 accounting이 가장 많이 쓰입니다 — 직관적이면서도 대부분의 자료구조에 적용 가능. Potential은 splay tree, Fibonacci heap 등 고급 자료구조에 필수.

08 — 성장 인자: Python 1.125 vs Java 2 vs C++ 1.5

고급

각 언어의 선택

언어 / 컨테이너성장 인자amortized 비용메모리 낭비 최대
Python list~1.125 (+ 6)~1012.5%
Go slice (n > 1024)1.25625%
C++ std::vector (GCC libstdc++)23100%
C++ std::vector (Clang libc++)23100%
C++ std::vector (MSVC)1.5450%
Java ArrayList1.5450%
Go slice (n ≤ 1024)23100%

amortized 비용 공식: growth_factor가 g일 때, amortized = 1 + g/(g−1). 예: g=2면 3, g=1.5면 4, g=1.125면 10.

전공자

왜 Python은 1.125인가 — 메모리 우선 철학

Python list는 일반 용도 컨테이너이고 Python 프로그램에선 list가 매우 흔함. 모든 list가 2× 성장하면 메모리 사용이 2배로 부풀 수 있음.

  • 1.125배: 최대 12.5% 낭비. 극단적으론 8 단위 배열이 9로.
  • CPU 시간은 1.125 (~10 amortized) vs 2.0 (~3 amortized) — 약 3배 차이.
  • 결론: CPU는 싸고 메모리는 비싸다는 Python의 입장. 메모리 효율 우선.
전공자

왜 C++/Clang libc++는 2인가 — 재활용 가능성 포기

"Geometric mean" 논증: g > 2면 old와 new를 합쳐도 메모리가 다시 쓰일 수 없음 (옆에 끼어들 수 있는 크기가 아니니까).

old_total = 1 + g + g² + ... + g^k = (g^(k+1) − 1) / (g − 1)
새 할당 g^(k+1) < old_total ⟺ g < φ = (1+√5)/2 ≈ 1.618 (황금비)

이론적으로 g < φ ≈ 1.618이면 과거 해제된 메모리를 재사용 가능. 1.5도 이 경계 안 (φ 미만), 1.125도 안.
2.0은 경계 밖이라 메모리 재사용 불가. 그러나 C++ 표준 라이브러리는 이 이론적 효율보다 amortized 비용을 우선.

Stack Overflow에 아주 유명한 글이 있습니다: "Why do vectors grow by a factor of 1.5?" — Facebook의 Folly 프로젝트(Alexandrescu)도 1.5를 택.

WHY — "올바른" 성장 인자는 없다

성장 인자는 철학적 선택:

  • CPU 우선: 2.0 (GCC libstdc++, MSVC vector는 1.5 절충).
  • 메모리 우선: 1.125 (Python), 1.25 (Go 큰 크기).
  • 재사용 가능: < 1.618 (황금비).

어떤 선택도 "최적"이 아닙니다. 용도와 환경에 따라 다른 트레이드오프.

09 — CPython 내부: PyListObject와 list_resize

전공자

Part A — PyListObject 구조체

Include/cpython/listobject.h
typedef struct {
    PyObject_VAR_HEAD          // ob_refcnt, ob_type, ob_size
    /* Vector of pointers to list elements. list[0] is ob_item[0], etc. */
    PyObject **ob_item;

    /* ob_item 배열의 할당된 크기. ob_size <= allocated.
       할당되지 않았다면 NULL이 아니라 포인터를 NULL로 둠. */
    Py_ssize_t allocated;
} PyListObject;
  • ob_refcnt: reference count (GC).
  • ob_type: 타입 포인터 (모든 PyObject 공통).
  • ob_size: 현재 원소 수 (= len(list)).
  • ob_item: PyObject 포인터 배열의 포인터. 원소가 8바이트 포인터 × allocated개.
  • allocated: 실제 할당된 슬롯 수. ob_size ≤ allocated.

sys.getsizeof([]) ≈ 56바이트 (Python 3.12, 64-bit): PyListObject 헤더 자체. 원소가 추가되면 ob_item에 할당된 만큼 추가 바이트 필요 — sys.getsizeof([1,2,3]) ≈ 88바이트.

전공자

Part B — list_resize 소스 (Objects/listobject.c)

list_resize — 발췌 (Python 3.12)
static int
list_resize(PyListObject *self, Py_ssize_t newsize)
{
    PyObject **items;
    size_t new_allocated, num_allocated_bytes;
    Py_ssize_t allocated = self->allocated;

    /* Bypass realloc() when a previous overallocation is large enough
       to accommodate the newsize.  If the newsize falls lower than half
       the allocated size, then proceed with the realloc() to shrink the list. */
    if (allocated >= newsize && newsize >= (allocated >> 1)) {
        Py_SET_SIZE(self, newsize);
        return 0;
    }

    /* This over-allocates proportional to the list size, making room
       for additional growth.  The over-allocation is mild, but is
       enough to give linear-time amortized behavior over a long
       sequence of appends() in the presence of a poorly-performing
       system realloc().
       Add padding to make the allocated size a multiple of 4.
       The growth pattern is:  0, 4, 8, 16, 24, 32, 40, 52, 64, 76, ... */
    new_allocated = ((size_t)newsize + (newsize >> 3) + 6) & ~(size_t)3;
    /* Do not overallocate if the new size is closer to overallocated size
       than to the old size. */
    if (newsize - Py_SIZE(self) > (Py_ssize_t)(new_allocated - newsize))
        new_allocated = ((size_t)newsize + 3) & ~(size_t)3;

    ...
    items = (PyObject **)PyMem_Realloc(self->ob_item, num_allocated_bytes);
    ...
    self->ob_item = items;
    Py_SET_SIZE(self, newsize);
    self->allocated = new_allocated;
    return 0;
}

핵심 공식:

new_allocated = ((newsize + (newsize >> 3) + 6) & ~3)
= ((newsize · 9/8) + 6) 을 4의 배수로 내림

성장 시퀀스: 0, 4, 8, 16, 24, 32, 40, 52, 64, 76, 92, 108, 128, ...

검증: newsize=17일 때 new_allocated = (17 + 2 + 6) & ~3 = 25 & ~3 = 24. 17보다 큰 4의 배수 중 적절.

Shrinking — 메모리 반환

pop()이나 del로 크기가 줄면, allocated의 절반 미만이 되면 realloc으로 축소. 메모리 낭비 방지. 이 "크기 반환" 로직이 다른 언어 대비 Python list의 특징.

10 — list.insert(0)의 함정, 그리고 deque

중급

왜 O(n)인가

list.insert(0, x)를 호출하면 기존 모든 원소가 한 칸씩 뒤로 밀립니다. n개 원소라면 n번의 메모리 이동.

A = [1, 2, 3, 4, 5]
A.insert(0, 99):
   A = [_, 1, 2, 3, 4, 5]  // 한 칸씩 shift
   A = [99, 1, 2, 3, 4, 5]

list.pop(0)도 같은 이유로 O(n). 그리고 이게 Python 초보자들이 자주 빠지는 함정:

accidentally_quadratic.py
# 큐를 구현한다고 리스트를 쓰면...
queue = []
for i in range(100000):
    queue.append(i)        # O(1)
while queue:
    x = queue.pop(0)       # O(n)! 100000 × 50000 = 5×10⁹ 연산
    process(x)

n=100,000이면 총 연산 수 약 5×10⁹. Python 1초에 약 10⁷ 순수 연산이니 수 분 걸림. 실무에서 실제로 마주치는 "왜 이렇게 느리지?" 버그.

중급

해결 — collections.deque

Python 표준 라이브러리의 deque(doubly-ended queue)는 양 끝 모두 O(1).

use_deque.py
from collections import deque

queue = deque()
for i in range(100000):
    queue.append(i)        # O(1)
while queue:
    x = queue.popleft()    # O(1) — 양 끝 모두 빠름
    process(x)

n=100,000이면 2×10⁵ 연산 → 수십 ms. 25,000배 빠름.

deque의 내부 구조는 다음 주(Algo 09)에서 — 사실 linked list of blocks라는 하이브리드 구조.

⚠️ "리스트를 스택/큐로 쓸 때" 결정 기준
  • 스택 (push/pop 끝에서만) → list OK. append/pop(-1) 둘 다 O(1).
  • 큐 (push 끝, pop 앞) → deque 필수. list는 O(n²) 지옥.
  • 양방향 → deque.
  • 임의 위치 삽입 → 어느 쪽이든 O(n). 진짜 필요하면 자료구조 재설계 (segment tree, balanced BST 등).

11 — Slicing의 메모리 비밀

중급

A[i:j]는 새 리스트를 만든다

이 사실이 list.insert(0)만큼 놓치기 쉬운 함정.

slice_copies.py
A = [1, 2, 3, 4, 5]
B = A[1:4]           # [2, 3, 4] — 새 리스트 객체, O(k)
print(B is A)          # False (독립 객체)
B[0] = 99
print(A)              # [1, 2, 3, 4, 5] — A는 변화 없음
  • A[i:j]는 원소 포인터를 복사 (PyObject 자체는 아님). 복사 비용 O(j − i).
  • A가 100만 원소인데 A[:100]을 쓰면 100개만 복사 — 빠름.
  • A[:-1]은 n-1개 복사 — O(n).
고급

A[i:j] = [...] 는 in-place 수정

할당문의 왼쪽에 오면 A 자체가 수정됩니다. 새 리스트가 만들어지지 않음.

slice_assignment.py
A = [1, 2, 3, 4, 5]
A[1:4] = [99, 88]   # A[1], A[2], A[3] 대체
print(A)                # [1, 99, 88, 5]

# 크기도 바꿀 수 있음 (자동으로 shrink/grow)
A[0:2] = [100, 200, 300]
print(A)                # [100, 200, 300, 88, 5]

right-hand side가 길면 list가 자동으로 커짐 (memmove로 나머지 밀기). 짧으면 줄어듦.

WHY — "[::] = new_data" 관용구

A[:] = B는 "A의 내용을 B로 교체" (같은 객체 유지). 이게 왜 중요한가?

  • A = B: A가 새 이름이 됨. 다른 곳에서 A를 참조하고 있다면 그 참조는 old A를 가리킴.
  • A[:] = B: A 객체 자체 내용이 바뀜. 다른 참조들도 새 데이터를 봄.

이 구분이 shared state 디버깅의 핵심. 다음 코드:

shared_state.py
original = [1, 2, 3]
backup = original

# 방법 A
original = [4, 5, 6]
print(backup)       # [1, 2, 3] — 옛 값

# 방법 B
original[:] = [4, 5, 6]
print(backup)       # [4, 5, 6] — 같이 바뀜!

AI — 3대 누락 패턴

LLM에게 Python 코드를 작성시킬 때 list 관련 가장 흔한 실수들:

  1. 큐 구현에 list + pop(0): 초보자 수준 답. 모던 LLM은 보통 deque로 간다. 하지만 "간단한 예제"라고 하면 list로 쓰기도.

실습 2 — list vs deque 벤치마크

list_vs_deque.py
from collections import deque
import timeit


def queue_with_list(n):
    q = []
    for i in range(n): q.append(i)
    while q: q.pop(0)


def queue_with_deque(n):
    q = deque()
    for i in range(n): q.append(i)
    while q: q.popleft()


for n in [1000, 10_000, 100_000]:
    t1 = timeit.timeit(lambda: queue_with_list(n), number=1)
    t2 = timeit.timeit(lambda: queue_with_deque(n), number=1)
    print(f"n={n}: list {t1*1000:6.1f}ms, deque {t2*1000:6.1f}ms, ratio {t1/t2:.0f}×")
예상 출력:

n=10000: list  200ms, deque    3ms, ratio  67×
n=100000: list   20s, deque   30ms, ratio 666×

n이 10배 커지면 list는 100배 느려짐 (Θ(n²)), deque는 10배 (Θ(n)). 차이가 정확히 O(n)배로 벌어짐.

PREDICT 문제 1

Python list의 성장 인자가 g라면, amortized 비용 공식은? g = 1.125 대입 값은?

풀이

PREDICT 문제 4

Potential method로 2× growth list append 분석 시 Φ = 2·size − capacity. size=5, capacity=8 상태에서 append 시 ΔΦ는?

BUILD 문제 5

1.5× growth를 쓰는 간단한 동적 배열을 Python으로 구현. list처럼 동작하는 최소 기능(append, __getitem__, __len__).

풀이 펼치기
dynamic_array.py
class DynamicArray:
    def __init__(self):
        self._capacity = 4
        self._size = 0
        self._buf = [None] * self._capacity

    def append(self, x):
        if self._size == self._capacity:
            new_cap = max(self._capacity * 3 // 2, self._capacity + 1)
            new_buf = [None] * new_cap
            for i in range(self._size):
                new_buf[i] = self._buf[i]
            self._buf = new_buf
            self._capacity = new_cap
        self._buf[self._size] = x
        self._size += 1

    def __getitem__(self, i): return self._buf[i]
    def __len__(self): return self._size
분석

amortized = 1 + 1.5/0.5 = 4. 각 append 평균 4 연산. Java ArrayList와 같은 성장 전략.

WHY 문제 6

성장 인자 g가 황금비 φ = 1.618 미만일 때 해제된 메모리를 재사용 가능한 이유를 수식으로.

풀이 펼치기
증명

과거에 해제된 블록 합: 1 + g + g² + ... + g^(k−1) = (g^k − 1)/(g − 1).
다음 확장에 필요한 크기: g^(k+1).
재사용 조건: g^(k+1) ≤ (g^k − 1)/(g − 1). 양변에 (g−1)를 곱하면 g^(k+1)(g−1) ≤ g^k − 1 ⟺ g^(k+2) − g^(k+1) ≤ g^k − 1. 양변을 g^k로 나누고 k → ∞ 극한을 취하면 g² − g ≤ 1, 즉 g² − g − 1 ≤ 0.
이차부등식 풀면 g ≤ (1 + √5)/2 = φ ≈ 1.618.

A += [x]와 A.append(x)와 A = A + [x]의 차이는?

풀이 펼치기
분석

BUILD 문제 9

Accounting 방법으로 "n번의 push + n번의 pop(끝에서)"이 총 O(n)임을 증명하시오.

풀이 펼치기

  3. "collections.deque로 다시 작성해달라."
  4. "list와 deque의 내부 구조 차이는?"

AI가 "pop(0) O(n)이라서 전체 O(n²)"을 스스로 인정하게 만드는 것이 핵심 학습 이벤트.

WHY 문제 12

CPython이 2×가 아닌 1.125× 성장 인자를 선택한 이유. 현재 하드웨어(메모리 풍부)에서도 여전히 적절한가?

풀이 펼치기
답

역사적 이유 — 2002년대 메모리가 상대적으로 비쌌음. 또한 Python 프로그램은 list를 매우 많이 만들어서 총 메모리 사용이 2× growth 시 실제로 문제.

현재 하드웨어에서도 적절: list는 여전히 흔함, 그리고 1.125× vs 2×의 CPU 차이(~3배 amortized)는 대부분 user code에서 감지 불가. 메모리 효율이 여전히 이득. 게다가 Python 코드가 C로 처리되므로 amortized 10 vs 3 차이가 실제 시간에는 미미.

15 — 정리 + 용어 사전 + 다음 주 맛보기

핵심 정리

📊

Amortized ≠ Average

Amortized는 결정적 (적대 입력에도), Average는 확률적 (분포 가정). list.append는 amortized.

🎯

3가지 분석 방법

Aggregate (단순) / Accounting (직관 저금통) / Potential (형식 Φ). 모두 같은 결론.

📈

Growth factor 트레이드오프

큰 g → CPU 이득, 메모리 낭비. 작은 g → 메모리 절약, CPU 느려짐. Python은 1.125 (메모리 우선).

🐍

PyListObject

C 구조체로 ob_item(포인터 배열) + size + allocated. list_resize가 amortized 핵심.

⚠️

insert(0) / pop(0) 함정

O(n). 큐 구현에 list 쓰면 O(n²) 지옥. deque로 해결 (다음 주).

✂️

Slicing 메모리 비밀

A[i:j]는 새 객체. A[i:j]=...는 in-place. A[:] = B와 A = B는 다름.

용어 사전 (25개)

Dynamic Array
크기가 필요 시 자동 확장되는 배열. Python list, Java ArrayList, C++ vector.
Static Array
고정 크기 배열. C의 int a[100].
Capacity
할당된 공간 크기 (allocated).
Size
실제 사용 중인 원소 수 (len). size ≤ capacity.
Growth Factor
확장 시 새 용량/옛 용량 비율. Python 1.125, Java 1.5, C++ 2.0.
Amortized Analysis
n번 연산의 평균 비용. 입력 무관 결정적 보장. (CLRS §17)
Aggregate Method
n번 연산 총 비용 / n. 가장 단순. (§17.1)
Accounting Method
각 연산에 amortized 부여 + 저금통. 직관적. (§17.2)
Potential Method
상태의 포텐셜 Φ 정의. amortized = actual + ΔΦ. (§17.3)
Potential Function Φ
자료구조 상태를 수치로. Φ(D₀) = 0, Φ(D_i) ≥ 0.
PyListObject
CPython list 구조체. ob_item (포인터 배열), allocated, ob_size.
list_resize
listobject.c 함수. 새 용량 계산 + realloc.
ob_item
PyObject 포인터의 배열. Python list의 "본체".
allocated
ob_item에 할당된 슬롯 수 (capacity).
Overallocation
필요 이상 할당해 미래 append 대비. amortized O(1)의 비밀.
Golden Ratio φ
(1+√5)/2 ≈ 1.618. growth factor < φ이면 해제 메모리 재사용 가능.
insert(0) / pop(0)
list의 앞 원소 조작. O(n) (모든 원소 shift).
deque
collections.deque. 양 끝 O(1). linked list of blocks 구조.
Slicing Copy
A[i:j]는 새 list. 원소 포인터만 복사 (얕은 복사).
Slice Assignment
A[i:j] = [...] in-place 수정. 크기도 자동 조정.
Shallow Copy
얕은 복사. 최상위 레벨만 복사 (하위 객체는 공유). list.copy(), A[:].
Deep Copy
copy.deepcopy. 재귀적으로 모든 중첩 객체 복사.
realloc
C 라이브러리 함수. 기존 메모리 블록 크기 조정. 가능하면 in-place, 아니면 새 위치에 복사.
Telescoping
Σ (a_i − a_{i-1}) = a_n − a_0. Potential 분석의 수학적 트릭.
Accidentally Quadratic
실수로 O(n²)이 된 코드. list + pop(0)의 흔한 결과.
