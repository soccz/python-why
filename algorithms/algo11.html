01 — 이번 주 이야기: 사전과 전화번호부

dict의 한계

Algo 10의 Python dict. d["hello"] 조회 O(1) — 빠릅니다. 그런데:

  • "h로 시작하는 모든 단어 찾아줘" — dict는 못 함 (해시는 순서 정보 없음).
  • "사전순으로 단어 순회해줘" — sorted(d)로 매번 O(n log n) 정렬.
  • "K번째로 큰 값" — 전체 정렬 필요 O(n log n).
  • "값 100 이상 200 이하인 키들" — 전수 스캔 O(n).

문제: dict는 "정렬 정보"를 버립니다. 해시 테이블의 본질. 순서가 필요하면 다른 자료구조.

BST — 정렬된 dict

이진 탐색 트리(Binary Search Tree, BST)는 키가 정렬된 상태로 저장됩니다. 각 노드에서:

  • 왼쪽 자식 < 노드 < 오른쪽 자식.
  • 중위 순회(in-order traversal)하면 자동으로 정렬된 순서.
  • 범위 조회 O(log n + k), k는 결과 개수.

하지만 — 균형이 잡혀 있어야만 O(log n). 순진한 BST는 최악 O(n). 균형 잡는 법이 이번 주의 주제: AVL과 Red-Black.

WHY — dict와 BST는 보완재다
요구사항적합한 자료구조복잡도
특정 key 조회dict (hash)O(1) 평균
key 존재 확인dict (hash)O(1) 평균
정렬 순서 순회BST (균형)O(n)
최소/최대 keyBST (균형)O(log n)
k번째 작은 keyBST (augmented)O(log n)
범위 조회 [a, b]BST (균형)O(log n + 결과 수)

Java는 HashMap(dict)과 TreeMap(Red-Black BST) 둘 다 제공. C++도 unordered_map(dict)과 map(Red-Black BST) 둘 다.
Python은 dict만 내장 제공. BST는 외부 라이브러리(sortedcontainers) — 이유는 §09에서.

Rotation은 균형 BST의 기본 연산. 포인터 3개만 갱신하는 O(1) 변형으로 트리 모양을 자유롭게 바꿈. AVL/Red-Black 모두 이 연산 위에 구축.
BST의 비밀: 입력 순서가 운명. 정렬된 데이터가 들어오면 한쪽으로 기울어 O(n)이 됨 (linked list 수준). 무작위면 평균 O(log n). 그래서 AVL·Red-Black 같은 자동 균형 BST가 필요.

02 — BST의 정의와 성질 (CLRS §12.1)

초급

BST 속성

이진 트리(binary tree)이면서 다음 속성 만족:

모든 노드 x에 대해: key(left subtree of x) ≤ x.key ≤ key(right subtree of x)

예:

          [8]
        /      \
     [3]     [10]
    /  \          \
 [1]  [6]     [14]
     /  \       /
 [4]  [7]  [13]

in-order 순회: 1, 3, 4, 6, 7, 8, 10, 13, 14 — 자동으로 정렬. BST의 핵심 성질.

중급

노드 구조

bst_node.py
class Node:
    def __init__(self, key, value=None):
        self.key = key
        self.value = value
        self.left = None
        self.right = None
        self.parent = None       # 선택적 — 일부 연산에 필요

CLRS는 parent 포인터를 포함하는 형태로 제시. successor/predecessor 연산에 유용하지만 필수는 아님.

고급

BST 속성 증명 — 왜 in-order가 정렬인가

귀납법으로 증명. n=1: 한 노드 순회 → 단일 원소 정렬. 참.
n>1: 노드 x에서 in-order = [in-order of left] + [x] + [in-order of right].

  • 귀납 가정: 왼쪽 서브트리 in-order는 정렬, 오른쪽도 정렬.
  • BST 속성: 왼쪽 서브트리 모든 키 ≤ x ≤ 오른쪽 서브트리 모든 키.
  • 따라서 전체 in-order = 왼쪽(정렬) → x → 오른쪽(정렬). 전체 정렬. ∎
📖 원서 인용 — CLRS 4판, §12.1 "What is a binary search tree?"

"Let x be a node in a binary search tree. If y is a node in the left subtree of x, then y.key ≤ x.key. If y is a node in the right subtree of x, then y.key ≥ x.key."

03 — BST 연산: search, insert, delete

중급

search: 재귀 탐색 O(h)

bst_search.py
def search(root, key):
    if root is None or root.key == key:
        return root
    if key < root.key:
        return search(root.left, key)
    return search(root.right, key)

한 번의 비교로 반쪽을 제거. 트리 높이 h만큼 진행 → O(h). 균형 잡힌 트리: h = O(log n). 불균형: h = O(n).

중급

insert: search와 같은 경로 + 추가

bst_insert.py
def insert(root, key, value=None):
    if root is None:
        return Node(key, value)
    if key < root.key:
        root.left = insert(root.left, key, value)
    elif key > root.key:
        root.right = insert(root.right, key, value)
    else:
        root.value = value         # 중복 key는 value만 업데이트
    return root

leaf까지 내려가 새 노드 추가. O(h).

고급

delete: 3가지 경우

삭제할 노드 x의 자식 개수에 따라:

  1. 자식 0개 (leaf): 그냥 제거.
  2. 자식 1개: 그 자식으로 x 대체.
  3. 자식 2개: successor(= 오른쪽 서브트리의 최솟값)을 찾아 x와 교체.
bst_delete.py
def find_min(node):
    while node.left:
        node = node.left
    return node


def delete(root, key):
    if root is None:
        return None
    if key < root.key:
        root.left = delete(root.left, key)
    elif key > root.key:
        root.right = delete(root.right, key)
    else:
        # 삭제할 노드
        if root.left is None:
            return root.right
        if root.right is None:
            return root.left
        # 두 자식 모두: successor로 교체
        succ = find_min(root.right)
        root.key, root.value = succ.key, succ.value
        root.right = delete(root.right, succ.key)
    return root

Successor가 오른쪽 서브트리의 leftmost이므로 자식이 최대 하나 (위 case 1 또는 2로 해결 가능). 재귀로 깔끔하게 처리됨.

WHY — successor 대신 predecessor도 OK

"왼쪽 서브트리의 최댓값(predecessor)"으로 교체해도 BST 속성 유지. CLRS는 successor를 씀 (관례). 대칭적이어서 어느 쪽이든 정확함. 일관성을 위해 한 가지만 선택해 사용.

구현 팁: 대부분 구현이 한쪽만 씀 → tree가 한쪽으로 기울 수 있음. 고급 구현은 무작위로 선택해 균형 유지 (randomized BST).

04 — BST의 악몽: 정렬 입력이 linked list 되는 순간

고급

1, 2, 3, 4, 5를 순서대로 삽입하면

insert(1): [1]
insert(2): [1]
              \
            [2]
insert(3): [1]
             \
            [2]
               \
               [3]
...
결과: 완전히 오른쪽으로 기운 linked list. 높이 = n − 1.

이때 search/insert/delete가 모두 O(n). BST의 장점(O(log n)) 완전 상실.

고급

언제 발생하나 — 실세계의 함정

  • 로그 타임스탬프 저장: 시간순 삽입 → 매번 오른쪽으로 기움.
  • 오름차순 ID 입력: 사용자 가입 ID 1, 2, 3... 순서대로 트리 삽입.
  • 적대적 공격: Algo 06 §12의 "accidentally quadratic"처럼, 악의적 입력으로 O(n²) 유발.

"평균 O(log n)"은 무작위 입력 가정. 실세계 데이터는 무작위가 아닙니다. 그래서 순진한 BST는 교육용 이상으로 쓰지 않음. 실무는 항상 균형 BST.

WHY — 균형 BST의 두 가지 접근
  1. 엄격한 균형 (AVL): 삽입/삭제 후 높이 차가 1 이하가 되도록 즉시 조정. 탐색 빠름, 수정 느림.
  2. 느슨한 균형 (Red-Black): 약간의 불균형 허용. 삽입/삭제 시 덜 수정. 탐색 약간 느림, 수정 빠름.

두 접근 모두 rotation이라는 기본 연산에 의존. 다음 섹션에서.

05 — Rotation: 균형의 기본 연산 (CLRS §13.2)

고급

Left Rotation

노드 x와 그 오른쪽 자식 y를 교환. BST 속성 유지하면서 구조 변경.

      x
   /  \
  α     y        Left-Rotate(x)
      / \         ─────────→
     β   γ

       y
    /  \
  x     γ
  / \
α   β

속성 보존: α < x < β < y < γ라는 순서가 유지됨. BST 여전히 유효.

left_rotate.py
def left_rotate(x):
    assert x.right is not None
    y = x.right
    x.right = y.left        # β가 x의 오른쪽 자식이 됨
    y.left = x              # x가 y의 왼쪽 자식이 됨
    return y                # y가 새 루트

O(1) 시간. 포인터 3개만 갱신. (parent 포인터 쓴다면 6개.)

고급

Right Rotation (Left의 대칭)

       y
    /  \
  x     γ        Right-Rotate(y)
  / \          ─────────→
α   β

      x
   /  \
  α     y
      / \
     β   γ

Left와 대칭. 좌/우 rotation만으로 어떤 BST든 다른 BST로 변환 가능 (같은 키 집합 내에서). 이게 균형 잡기의 토대.

📖 원서 인용 — CLRS 4판, §13.2 "Rotations"

"When we do a left rotation on a node x, we assume that its right child y is not T.nil; x may be any node in the tree whose right child is not T.nil. The left rotation 'pivots' around the link from x to y."

06 — AVL 트리: Adelson-Velsky & Landis 1962

고급

역사 — 최초의 자체 균형 BST

Georgy Adelson-Velsky와 Evgenii Landis (소련 수학자) — 1962년 논문 "An algorithm for the organization of information"에서 발표. 자체 균형 BST의 최초 형태. 이름은 두 창시자 이니셜 A.V.L.

고급

Balance Factor — AVL의 핵심

각 노드의 balance factor (bf):

bf(node) = height(left subtree) − height(right subtree)

AVL 조건: 모든 노드에서 |bf| ≤ 1.

  • bf = -1, 0, +1만 허용. 그 외 (-2, +2 등)이면 불균형 상태.
  • 불균형 발생 시 즉시 rotation으로 균형 복구.
전공자

AVL 삽입의 4가지 불균형 케이스

삽입 후 ancestor 중 어떤 노드의 bf가 ±2 되면 rotation 필요. 4가지 패턴:

케이스불균형 위치해결 rotation
LL왼쪽 자식의 왼쪽에 삽입Right rotate
RR오른쪽 자식의 오른쪽에 삽입Left rotate
LR왼쪽 자식의 오른쪽에 삽입Left rotate 후 Right rotate
RL오른쪽 자식의 왼쪽에 삽입Right rotate 후 Left rotate

LL/RR은 단일 rotation, LR/RL은 더블 rotation. 4 케이스 모두 O(1) 재구조.

전공자

AVL 트리의 높이 bound

N(h) = 높이 h인 AVL 트리의 최소 노드 수. 피보나치-like 점화식:

N(h) = N(h−1) + N(h−2) + 1,   N(0) = 1, N(1) = 2

풀면 N(h) ≥ φ^h / √5, φ = 황금비. 따라서:

h ≤ logφ(n · √5) ≈ 1.44 log₂ n

최악의 AVL 트리 높이가 1.44 log₂ n. 즉 O(log n) 엄격히 보장. 모든 탐색/삽입/삭제 O(log n).

WHY — AVL이 왜 CLRS 주요 챕터가 아닌가

CLRS 4판에서 AVL 트리는 Chapter 13 Red-Black 챕터의 문제(Problem 13-3)로만 제시. 본문은 Red-Black 중심.

이유: "AVL은 엄격하나 복잡도 증명이 어렵고, Red-Black이 실무에서 더 자주 쓰임". 교과서는 실용성 우선. AVL은 단독 전공 과목에서 다루거나, 경쟁 프로그래밍에서 Segment Tree/Treap 같은 변형으로 더 자주 등장.

07 — Red-Black 트리: 5가지 속성 (CLRS §13.1)

고급

역사 — Red-Black의 탄생

Rudolf Bayer가 1972년 "symmetric binary B-trees"로 원형 제시.
Leonidas Guibas와 Robert Sedgewick이 1978년 "A Dichromatic Framework for Balanced Trees"에서 "red-black tree"라는 이름과 현대적 형태로 정리.

고급

Red-Black의 5가지 속성 (CLRS Thm 13.1)

RB Property 1: 모든 노드는 red 또는 black.
RB Property 2: 루트는 black.
RB Property 3: 모든 leaf (NIL)는 black.
RB Property 4: red 노드의 자식은 모두 black (= red-red 연속 금지).
RB Property 5: 모든 노드에서 descendants leaf까지의 모든 경로의 black 노드 수가 같다 (= black-height 일정).

NIL 노드(sentinel)는 실제 저장 안 되는 "빈 자식" 자리를 가리킴. 구현상 하나의 sentinel 객체 공유.

전공자

높이 bound — 왜 O(log n) 보장되는가

Red-Black 트리의 높이 h ≤ 2 log₂(n+1)

증명 스케치:

  1. Property 5: 루트에서 leaf까지 black-height 같음. 이 값을 bh(root)라 하자.
  2. Property 4: red 연속 금지 → 경로에서 red:black 비율 최대 1:1 → 전체 경로 길이 ≤ 2·bh(root).
  3. bh=k인 트리의 최소 노드 수 ≥ 2^k − 1 (완전 이진 트리 근사).
  4. 따라서 n ≥ 2^bh − 1 ⟹ bh ≤ log₂(n+1) ⟹ h ≤ 2 log₂(n+1). ∎

AVL의 1.44 log₂ n보다 느슨하지만 여전히 O(log n). 모든 연산 O(log n) 보장.

📖 원서 인용 — CLRS 4판, §13.1 "Properties of red-black trees"

"A red-black tree is a binary search tree with one extra bit of storage per node: its color, which can be either RED or BLACK. By constraining the node colors on any simple path from the root to a leaf, red-black trees ensure that no such path is more than twice as long as any other, so that the tree is approximately balanced."

전공자

삽입 후 재균형 — 3가지 케이스

새로 삽입된 노드는 항상 red로 시작 (Property 5 안 깨기 위해). 부모가 red이면 Property 4 위반 → 재균형.

부모의 형제(uncle)가 누구냐에 따라 3가지 케이스:

  • Case 1: uncle도 red → 부모·uncle을 black으로, 조부모를 red로. 조부모에서 재귀.
  • Case 2: uncle이 black, 삽입이 inner child → rotation 한 번으로 Case 3로 환원.
  • Case 3: uncle이 black, 삽입이 outer child → rotation + 색상 교환. 끝.

최악의 경우 Case 1이 루트까지 전파 — 그래도 O(log n) 재균형. Rotation은 최대 2번 (Cases 2+3).

08 — AVL vs Red-Black: 왜 실무는 Red-Black인가

핵심 차이: AVL은 탐색 조금 빠름, Red-Black은 수정(insert/delete)이 훨씬 빠름.

실제 범용 라이브러리는 모두 Red-Black을 선택:

  • Java: TreeMap, TreeSet — Red-Black.
  • C++: std::map, std::set (대부분 구현) — Red-Black.
  • Linux 커널: CFS 스케줄러의 task queue — Red-Black.
  • PostgreSQL: 일부 인덱스 구조 — Red-Black 변형.

이유: 실무 데이터는 읽기보다 쓰기가 많은 경우가 많고, rotation 적은 쪽의 현저한 이득이 있음.

📖 Sedgewick의 Red-Black 개량 — Left-Leaning Red-Black (2008)

Robert Sedgewick은 2008년 "Left-leaning red-black trees"로 구현을 대폭 단순화한 변형을 제시. 모든 red 링크가 왼쪽으로만 기울게 강제. 원조 Red-Black의 3·4 케이스가 단순화됨.

이론적 성질은 동일. 교육용으로 LLRB가 자주 등장.

Python만 유일하게 "sorted" 자료구조가 없음. 이상해 보이지만 이유가 있음.

고급

이유 3가지

  1. 대안이 있음: bisect 모듈(정렬된 list에 이진 탐색) + heapq(우선순위 큐) + sorted()(매번 O(n log n) 정렬)로 많은 경우 해결.
  2. 구현 복잡도: Red-Black 트리는 C로 정확히 구현하기 까다로움. CPython 코어가 그만큼 커짐. 유지 보수 부담.
  3. 범용 대안이 존재: sortedcontainers가 pure Python + 매우 빠름. "stdlib에 넣을 만큼의 가치"가 있는지에 대한 Guido의 의문.

PEP 3128 (Daniel Stutzbach 2007): "BList"(B+-tree 기반 list) 제안되었으나 채택되지 못함. 이유: 기존 list가 대부분의 경우 충분히 빠르고, 유지보수 비용 대비 이득이 작음.

WHY — Python 철학: "There should be one obvious way"

Zen of Python: "단 하나의, 그리고 가능하면 오직 하나의 명백한 방법."

여러 자료구조(dict, BST, skip list, B-tree)를 모두 제공하면 초보자 혼란. Python은 dict + list + deque + heapq만으로 대부분 문제 해결 가능하다고 판단. 정말 필요하면 sortedcontainers 설치 (pip 한 줄).

실제로 많은 Python 알고리즘 경진 참가자들이 sortedcontainers를 쓰지 않고 bisect.insort(정렬된 list 유지)로 풀어냅니다. n < 10⁶ 정도면 충분.

10 — sortedcontainers: Grant Jenks의 list-of-lists

고급

놀라운 사실 — 트리가 아니다

많은 사람이 sortedcontainers를 "Python의 TreeMap"이라 생각하지만 — 이진 탐색 트리가 아닙니다. 또한 skip list도 아닙니다 (흔한 오해).

Grant Jenks의 영리한 설계: "list of sorted lists".

고급

구조 — 조각난 정렬 배열

SortedList의 내부 (load=1000 가정):

_lists = [
   [1, 5, 8, 12, ..., 999],   # 1000개 정렬됨
   [1001, 1003, ..., 1999],   # 1000개 정렬됨
   [2001, ..., 2500]   # 500개 정렬됨 (마지막은 덜 참)

  • 전체 원소 n개를 약 √n 개의 서브리스트로 분할. 각 서브리스트 크기 약 √n.
  • 삽입: 이진 탐색으로 서브리스트 찾음 → 서브리스트 내 이진 탐색 → bisect.insort. O(√n) (!).

왜 이게 BST를 이기는가

  • Pure Python인데 C BST만큼 빠름: list와 bisect가 C로 구현되어 매우 빠름. 이들을 조합하면 tree pointer 추적 없이도 경쟁력.
  • 캐시 친화: 서브리스트가 연속 메모리. BST는 포인터로 메모리 도처에 흩어짐.
  • 메모리 효율: tree node overhead 없음. 순수 list.
  • 구현 단순: ~1500 줄 pure Python. BST는 C로 수천 줄.

Grant Jenks의 벤치마크: SortedList가 C로 구현된 blist(B+-tree) + 많은 Python BST 구현을 압도.

11 — 언제 BST가 필요한가

고급

5가지 대표 시나리오

  1. 범위 조회 (Range query): "[a, b] 사이의 모든 key를 순서대로." BST로 O(log n + k).
  2. k번째 원소 (Order statistics): "k번째로 작은/큰 값." Augmented BST로 O(log n). (각 노드에 subtree size 저장.)
  3. Predecessor / Successor: "주어진 key보다 작은 가장 큰 key." bisect.bisect_left와 같은 역할. BST로 O(log n).
  4. 정렬 순회: "전체 데이터를 정렬 순서로 순회." in-order 순회 O(n).
  5. Stream에서 중앙값: "데이터 스트림의 running median." 두 개의 heap으로도 가능하지만 BST로도.

WHY — 실무에서 "진짜 BST가 필요한 경우"는 드물다

대부분의 "BST가 필요해"라는 느낌은 실제로는 dict + 정렬로 해결되는 경우. "언제 진짜 BST 필요?"의 체크리스트:

  1. 데이터가 동적으로 삽입·삭제된다.
  2. 정렬된 순서 접근 또는 범위 조회가 빈번하다.
  3. n이 커서 매번 sorted()하면 O(n² log n) 지옥.

세 조건 모두 충족 → sortedcontainers 사용. 하나라도 빠지면 dict/list/heap으로 충분.

13 — 실습: BST 구현 + k번째 원소 + 범위 조회

고급

실습 1 — 기본 BST 구현

simple_bst.py
class BST:
    class Node:
        __slots__ = ('key', 'left', 'right', 'size')
        def __init__(self, key):
            self.key = key
            self.left = self.right = None
            self.size = 1   # subtree 크기 — k번째 원소 빠르게

    def __init__(self):
        self.root = None

    def insert(self, key):
        self.root = self._insert(self.root, key)

    def _insert(self, node, key):
        if node is None:
            return self.Node(key)
        if key < node.key:
            node.left = self._insert(node.left, key)
        elif key > node.key:
            node.right = self._insert(node.right, key)
        node.size = 1 + self._sz(node.left) + self._sz(node.right)
        return node

    def _sz(self, node):
        return 0 if node is None else node.size

    def kth_smallest(self, k):
        """k번째 작은 원소 (1-indexed). Augmented BST로 O(h)."""
        node = self.root
        while node:
            left_size = self._sz(node.left)
            if k == left_size + 1:
                return node.key
            elif k <= left_size:
                node = node.left
            else:
                k -= left_size + 1
                node = node.right
        raise IndexError("k out of range")

    def range_query(self, lo, hi):
        """[lo, hi] 범위의 모든 key를 정렬 순서로 yield."""
        yield from self._range(self.root, lo, hi)

    def _range(self, node, lo, hi):
        if node is None: return
        if lo < node.key:
            yield from self._range(node.left, lo, hi)
        if lo <= node.key <= hi:
            yield node.key
        if node.key < hi:
            yield from self._range(node.right, lo, hi)


# 테스트
bst = BST()
for x in [5, 3, 8, 1, 4, 7, 9, 2, 6]:
    bst.insert(x)

print(bst.kth_smallest(1))              # 1
print(bst.kth_smallest(5))              # 5
print(list(bst.range_query(3, 7)))    # [3, 4, 5, 6, 7]

고급

실습 2 — 균형 확인 함수

bst_height.py
def height(node):
    if node is None: return -1
    return 1 + max(height(node.left), height(node.right))


def is_balanced(node):
    """AVL 의미에서 균형인지 확인: 모든 노드의 bf가 |.| ≤ 1."""
    if node is None: return True, -1
    lb, lh = is_balanced(node.left)
    rb, rh = is_balanced(node.right)
    balanced = lb and rb and abs(lh - rh) <= 1
    return balanced, 1 + max(lh, rh)


# 두 입력의 높이 비교
import random
for desc, keys in [("sorted", list(range(100))),
                   ("random", random.sample(range(100), 100))]:
    bst = BST()
    for k in keys: bst.insert(k)
    print(f"{desc}: height = {height(bst.root)}, balanced = {is_balanced(bst.root)[0]}")

def inorder_iter(root):
    stack = []
    cur = root
    while cur or stack:

            stack.append(cur)

def successor(root, target):
    """target보다 큰 가장 작은 key. O(h)."""
    best = None
    cur = root
    while cur:
        if cur.key > target:
            best = cur
            cur = cur.left      # 더 작은 candidate 찾기
        else:
            cur = cur.right
    return best
원리

답

평균 O(log n)이라고 공식 문서가 명시. 내부 구조상 worst-case 분석으로는 O(√n) (sublist 찾기 O(log(n/load)) + sublist 내부 삽입 O(load))이지만, load=1000으로 고정되어 일반적인 n에서는 실측 O(log n)에 가까움.

BST의 O(log n)과 같은 수준 실측. C 구현 BST보다도 대부분 빠름.

핵심 정리

🌳

BST 기본

left < node < right. in-order 순회 = 정렬. search/insert/delete 모두 O(h).

⚠️

순진한 BST의 함정

정렬 입력에서 linked list 되어 O(n). 실무 사용 불가.

🔄

Rotation

Left/Right rotation이 균형의 기본 연산. BST 속성 유지하며 구조 변경. O(1).

⚖️

AVL 트리

Adelson-Velsky & Landis 1962. |bf| ≤ 1. 엄격 균형. 높이 ≤ 1.44 log n.

🔴

Red-Black 트리

Guibas & Sedgewick 1978. 5가지 속성. 높이 ≤ 2 log(n+1). 실무 표준.

📚

sortedcontainers

Grant Jenks. List-of-lists (트리 아님). Pure Python으로 C BST 수준 성능.
