ALGO 26

구간 자료구조 — Segment Tree, Fenwick
"구간 합 + 업데이트"를 O(log n)에

1977년 Jon Bentley(프로그래밍 펄 시리즈의 그 Bentley)가 계산 기하 문제 — "점이 몇 개 선분에 포함되는가" — 를 위해 Segment Tree를 고안. 17년 뒤인 1994년 Peter Fenwick이 arithmetic coding의 빈도표를 위해 더 간결한 Binary Indexed Tree(BIT/Fenwick Tree)를 발표. 사실 1989년 러시아 Boris Ryabko가 먼저 제시했지만 Fenwick의 이름이 굳어짐.
naive O(n) 업데이트를 O(log n)으로, 구간 질의도 O(log n)으로. 경쟁 프로그래밍 단골, PostgreSQL/MySQL 인덱스, 게임 공간 질의의 기반.
핵심 질문: "왜 prefix sum만으론 부족한가? Fenwick이 세그먼트 트리보다 작은데 왜 둘 다 쓰이는가?"

01 — 이번 주 이야기: Bentley 1977 & Fenwick 1994

Segment tree는 Jon Bentley가 1977년 미발표 원고 "Solutions to Klee's rectangle problems"에서 처음 도입한 자료구조로, 이후 Bentley & Wood의 1980년 논문 "An optimal worst case algorithm for reporting intersections of rectangles" (IEEE Trans. Computers C-29(7):571-577)에서 공식 출판되었습니다.

1977 — Jon Bentley, CMU

Jon Louis Bentley(1953-), Programming Pearls의 저자. 1976년 UNC Chapel Hill에서 박사 학위를 받고 1977년 CMU 교수로 임용된 해에 "Solutions to Klee's rectangle problems"의 계산 기하 문제를 풀면서 segment tree를 고안. "점 p가 주어진 n개 구간 중 몇 개에 포함되는가?"를 O(log n)에.

원래 기하 자료구조였지만, 1990-2000년대 경쟁 프로그래밍에서 "배열의 구간 합/최솟값 질의 + 업데이트"의 만능 해답으로 재발견. 지금은 대회 문제의 단골.

1989 — Boris Ryabko, 러시아

Ryabko가 정보 이론 맥락에서 오늘날의 Fenwick tree와 동등한 자료구조를 발표. 산술 부호화(arithmetic coding)에서 누적 빈도표를 업데이트/조회하기 위함. 러시아어로 발표돼 서방에 늦게 알려짐.

1994 — Peter Fenwick, University of Auckland

"A new data structure for cumulative frequency tables"(Software: Practice & Experience 24(3):327-336, 1994). Fenwick도 산술 부호화 맥락. 이름이 붙으며 서방에 전파. Ryabko 1989가 먼저였지만 Fenwick tree / BIT(Binary Indexed Tree)로 굳어짐.

Fenwick의 천재성: 세그먼트 트리의 4n 대비 약 1/4 공간(n+1), 더 짧은 코드 — 이진 표현의 최하위 1비트 추출(x & -x) 트릭. 경쟁 프로그래밍 필수 기법.

1995 이후 — 변종의 시대

  • Lazy propagation: 구간 업데이트까지 O(log n)으로. 1990년대 말 ICPC 대회에서 유행.
  • Persistent segment tree: 버전 관리. k-th smallest in range query 등.
  • 2D segment tree: 평면 영역 질의.
  • Segment tree beats(Jiry 2016): chmin/chmax 같은 복잡한 연산.
구간 자료구조의 진화: Bentley 1977 기하용 → 1989-94 Ryabko/Fenwick의 간결 버전 → 1990s 경쟁 프로그래밍 표준 → 2016 Jiry의 Segment Tree Beats.

02 — 문제 — 구간 질의 + 업데이트

초급

상황: "매일 주식 거래량" 분석

배열 A[0..n-1]에 각 날 거래량 저장. 자주 다음 두 연산 필요:

  • Range Query: 구간 [l, r]의 합 / 최솟값 / 최댓값.
  • Point Update: A[i]를 새 값으로.

예: n = 100만, 질의와 업데이트 각각 100만 번. 어떤 자료구조가 적합?

중급

naive 전략들의 한계

전략QueryUpdate전체 시간 (10⁶ · 10⁶)
배열 + 반복 합산O(n)O(1)10¹² — 불가능
Prefix sumO(1)O(n)10¹² — 불가능
Segment TreeO(log n)O(log n)4·10⁷ — 충분

둘 다 O(log n) — 업데이트와 질의가 모두 빈번할 때의 균형.

03 — Prefix Sum의 한계

중급

복습: 구간 합 O(1)

prefix[0] = 0
prefix[i+1] = prefix[i] + A[i]

sum(l, r) = prefix[r+1] − prefix[l] // O(1)
  • 구축: O(n).
  • 질의: O(1) — 빼기 한 번.
  • 업데이트 문제: A[i] 변경 시 prefix[i+1..n]을 모두 갱신 → O(n). 업데이트 많으면 치명적.

읽기만 하는 데이터엔 prefix sum이 최고. 하지만 실시간 업데이트가 있으면 segment tree/BIT가 필요.

04 — Segment Tree 아이디어

중급

"반씩 쪼개어 미리 계산"

배열 [0..n-1]을 이진 트리로 표현:

  • 루트: 전체 구간 [0, n-1]의 집약값(합/최솟값/...).
  • 자식: 부모 구간을 정확히 반으로 나눈 두 부분 구간.
  • 리프: 단일 원소.

총 노드 수 ≈ 2n (정확히는 < 4n 안전 상한). 높이 O(log n).

구간 [l, r] 질의: 트리를 DFS하며 "완전히 포함" 노드의 값을 직접 사용, "일부 겹침"이면 자식으로 내려감. 최대 O(log n) 노드 방문.

업데이트 A[i] = v: i를 포함하는 노드들의 경로만 업데이트. 리프→루트 O(log n).

Segment Tree 구조: 각 내부 노드가 자식 구간의 합. 구간 질의는 최대 O(log n)개 노드만 결합. 업데이트도 루트 경로 O(log n). 공간 O(4n) — 완전 이진 트리 상한.

05 — 구축 O(n)

고급

재귀 구축

segment_tree_build.py
class SegmentTree:
    def __init__(self, arr):
        n = len(arr)
        self.n = n
        self.tree = [0] * (4 * n)    # 안전 상한 4n
        if n > 0:
            self._build(arr, 1, 0, n - 1)

    def _build(self, arr, node, start, end):
        if start == end:
            self.tree[node] = arr[start]
            return
        mid = (start + end) // 2
        self._build(arr, 2*node,     start,   mid)
        self._build(arr, 2*node + 1, mid+1, end)
        self.tree[node] = self.tree[2*node] + self.tree[2*node + 1]
  • 인덱싱 컨벤션: 1-indexed. 노드 node의 자식은 2*node, 2*node+1. 부모는 node // 2. 힙과 같은 구조.
  • 공간 4n: n이 2의 거듭제곱이면 정확히 2n-1. 아니면 최대 4n-1. 4n이 안전.
  • 시간 O(n): 각 노드 O(1) 계산, 총 노드 < 4n.

06 — 구간 질의 O(log n)

고급

재귀 질의

현재 노드의 범위 [start, end] vs 질의 범위 [l, r]:

  1. 완전 분리 ([end < l] 또는 [r < start]): 0 반환 (합의 항등원).
  2. 완전 포함 ([l ≤ start ≤ end ≤ r]): tree[node] 반환.
  3. 부분 겹침: 자식 두 개 재귀 호출 후 합산.
segment_tree_query.py
    def query(self, l, r):
        return self._query(1, 0, self.n - 1, l, r)

    def _query(self, node, start, end, l, r):
        if r < start or end < l: return 0    # 분리
        if l <= start and end <= r:        # 완전 포함
            return self.tree[node]
        mid = (start + end) // 2
        return (self._query(2*node,     start,   mid, l, r) +
                self._query(2*node + 1, mid+1, end, l, r))

왜 O(log n)? 깊이마다 최대 4개 노드만 방문 (다른 연구에선 2개 부분 겹침 노드만 양측 재귀 + 2개 완전 포함 노드). 총 O(log n).

07 — 점 업데이트 O(log n)

고급

리프 → 루트 경로

segment_tree_update.py
    def update(self, idx, value):
        self._update(1, 0, self.n - 1, idx, value)

    def _update(self, node, start, end, idx, value):
        if start == end:
            self.tree[node] = value
            return
        mid = (start + end) // 2
        if idx <= mid:
            self._update(2*node, start, mid, idx, value)
        else:
            self._update(2*node + 1, mid+1, end, idx, value)
        self.tree[node] = self.tree[2*node] + self.tree[2*node + 1]


# 테스트
st = SegmentTree([1, 3, 5, 7, 9, 11, 13, 15])
print(st.query(1, 5))    # 35 (3+5+7+9+11)
st.update(3, 100)         # A[3]을 100으로
print(st.query(1, 5))    # 128 (35 - 7 + 100)
  • 변형: sum → min/max/gcd/xor로 바꾸면 구간 최솟값/최댓값/gcd/xor 질의. 결합 법칙이 성립하는 모든 연산에 적용 가능 (semigroup).
  • Idempotent 연산(min, max, gcd)에는 Sparse Table(Bender-Farach-Colton 2000)이 O(n log n) 전처리 + O(1) 질의 — 업데이트 없을 때 더 빠름.

08 — Lazy Propagation — 구간 업데이트

전공자

"나중에 처리하자"

구간 [l, r]에 모두 v를 더하라면? naive하면 O((r-l+1) · log n) — 크면 느림. Lazy: 각 노드에 "아직 자식에게 전파하지 않은 업데이트"를 저장.

  1. 노드에 내려오면 먼저 저장된 lazy가 있으면 자식으로 전파.
  2. 업데이트 [l, r, v]가 현재 노드 범위에 완전 포함이면: lazy에 저장만 (자식 내려가지 않음).
  3. 부분 겹침이면: lazy 전파 + 자식 재귀.
segment_tree_lazy.py
class SegTreeLazy:
    def __init__(self, arr):
        n = len(arr)
        self.n = n
        self.tree = [0] * (4 * n)
        self.lazy = [0] * (4 * n)
        if n > 0:
            self._build(arr, 1, 0, n - 1)

    def _push(self, node, start, end):
        """lazy를 자식으로 전파."""
        if self.lazy[node] != 0:
            mid = (start + end) // 2
            self.tree[2*node]     += self.lazy[node] * (mid - start + 1)
            self.tree[2*node + 1] += self.lazy[node] * (end - mid)
            self.lazy[2*node]     += self.lazy[node]
            self.lazy[2*node + 1] += self.lazy[node]
            self.lazy[node] = 0

    def update_range(self, l, r, v):
        self._update(1, 0, self.n - 1, l, r, v)

    def _update(self, node, start, end, l, r, v):
        if r < start or end < l: return
        if l <= start and end <= r:
            self.tree[node] += v * (end - start + 1)
            self.lazy[node] += v
            return
        self._push(node, start, end)
        mid = (start + end) // 2
        self._update(2*node,     start,   mid, l, r, v)
        self._update(2*node + 1, mid+1, end, l, r, v)
        self.tree[node] = self.tree[2*node] + self.tree[2*node + 1]

결과: 구간 업데이트 O(log n), 구간 질의 O(log n). 경쟁 프로그래밍 고수 기법.

09 — Fenwick Tree 아이디어

고급

이진 표현의 마법

Fenwick의 핵심: 1-indexed 배열 BIT[1..n]에서 BIT[i]는 i의 이진 표현에서 최하위 1비트만큼의 길이에 해당하는 구간 합 저장.

lowbit(i) = i & (-i) // 최하위 1비트 (2's complement 트릭)

BIT[i] = sum(A[i − lowbit(i) + 1 .. i])

예: i = 12 = 1100₂, lowbit = 100₂ = 4. BIT[12] = A[9]+A[10]+A[11]+A[12].

prefix 합 질의 sum(1..i): i에서 lowbit만큼 빼며 거슬러 올라감. O(log n).

업데이트: A[i]에 delta 더할 때, i에서 lowbit만큼 더하며 올라감. O(log n).

Fenwick Tree query(7): i = 7 → 6 → 4 → 0 으로 lowbit씩 감소. 각 단계 BIT[i] 누적. 3회 접근 = O(log n). 배열만큼 작은 공간(n+1)으로 구간 질의·업데이트 모두 O(log n).

10 — Fenwick Python 구현

고급

10줄의 마법

fenwick_tree.py
class FenwickTree:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)    # 1-indexed

    def update(self, i, delta):
        """A[i]에 delta 더하기. 1-indexed."""
        while i <= self.n:
            self.bit[i] += delta
            i += i & -i    # lowbit 더하기

    def query(self, i):
        """A[1..i] prefix sum."""
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i    # lowbit 빼기
        return s

    def range_query(self, l, r):
        return self.query(r) - self.query(l - 1)


# 테스트
ft = FenwickTree(8)
for i, v in enumerate([1, 3, 5, 7, 9, 11, 13, 15], 1):
    ft.update(i, v)

print(ft.query(7))    # 49 (1+3+5+7+9+11+13)
print(ft.range_query(3, 6))    # 32 (5+7+9+11)
ft.update(3, 10)    # A[3] += 10 (5 → 15)
print(ft.range_query(3, 6))    # 42
  • 시간: query, update 모두 O(log n).
  • 공간: n+1개 원소. Segment tree의 4n 대비 25%.
  • lowbit 트릭: i & -i는 2's complement에서 최하위 1비트만 남김. x=12=1100₂ → -x=...0100₂ → AND = 0100₂ = 4.
  • 한계: 합 외에 min/max 등 역원 없는 연산엔 잘 안 맞음. XOR은 됨 (자기 자신이 역원).

  • 경쟁 프로그래밍에서 둘 다 필수 — 문제에 맞는 것 선택.

12 — 변종 — 2D, Persistent, Merge Sort Tree

전공자

2D Segment Tree / BIT

2차원 격자의 사각형 영역 합. "segment tree of segment trees". 시간 O(log² n), 공간 O(n²). 게임 공간 질의, 이미지 처리에 사용.

전공자

Persistent Segment Tree

각 업데이트마다 새 버전을 만듦. O(log n) 추가 노드만 생성 (공유 가능한 노드는 재사용). 응용: "k-th smallest in range [l, r]" 질의를 O(log n)에 — Wavelet tree의 대안.

전공자

Merge Sort Tree

각 노드에 해당 구간의 정렬된 배열 저장. "[l, r]에서 x보다 작은 원소 개수" 같은 질의 O(log² n). 공간 O(n log n).

전공자

Segment Tree Beats (Jiry 2016)

"각 원소에 min(x, v) 적용" 같은 복잡한 범위 연산. 각 노드에 최댓값, 차순 최댓값, 개수를 저장. 특정 조건 만족 시 "break" 또는 "tag" — amortized O(n log² n).

13 — 응용 — DB, 게임, 주식

고급

데이터베이스 인덱스

Range query가 DB 핵심 연산. "가격 1000-5000 사이 상품 수?" 같은. 실무 DB의 주 인덱스는 B-tree/B+tree(다음 주 Algo 27)지만, 특수 aggregation 인덱스나 인메모리 OLAP 레이어에서 segment tree/BIT 아이디어가 사용됨.

  • PostgreSQL GIN/GiST: 트리 인덱스의 일반화 (segment tree의 직접 사용은 아님).
  • Time-series DB (InfluxDB, TimescaleDB): 시간 구간 aggregation을 위한 bucket tree — segment tree의 변형 아이디어.
  • OLAP 데이터 큐브: 차원별 prefix sum / 구간 축약.
고급

게임 / 시뮬레이션

  • 공간 분할: 2D segment tree로 "영역 내 적 수", "사각형 내 플레이어 경험치 합" 등.
  • 경로 탐색: Grid map에서 구간 장애물 업데이트 + 구간 조회.
  • 이벤트 관리: 시간 축의 segment tree로 "이 시점에 활성 이벤트들".
고급

금융 / 시계열

  • 고빈도 거래(HFT): "최근 N틱 내 최고가" 같은 슬라이딩 max (monotonic deque가 주로 쓰이지만 segment tree도).
  • 기술적 지표: 이동 평균, 변동성 — 구간 합/분산 계산.
  • 백테스트: 과거 기간의 통계 빠르게.
전공자

AI가 구간 자료구조를 잘못 쓰는 패턴

  • Prefix sum만 사용: 업데이트 빈번하면 O(n) 업데이트. BIT가 적절.
  • Segment tree 과잉: 합만 필요하면 BIT 10줄이 충분. 과잉 엔지니어링.
  • Lazy 구현 버그: push 순서, 타이밍을 틀리기 쉬움. 꼭 완전 포함 전에 push, 업데이트 후 tree 갱신.
  • 1-indexed vs 0-indexed 혼동: Fenwick은 1-indexed가 표준. 섞어 쓰면 off-by-one 지옥.
  • Semigroup 아닌 연산을 segment tree로: 예: 평균(결합 법칙 없음). 합/개수 두 개를 따로 저장.

교정 질문: "합/min/max 어느 것?", "구간 업데이트 필요?", "lazy에 push 넣었나?"

14 — 연습문제 (12문제)

WHY 문제 1

왜 segment tree 공간이 4n인가? 정확한 상한?

풀이 펼치기
답

n이 2의 거듭제곱이면 2n-1 노드 (리프 n + 내부 n-1). n이 2^k와 2^(k+1) 사이면 최대 2·2^(k+1) - 1 ≈ 4n. 정확히는 ⌈2·2^⌈log₂n⌉ − 1⌉. 4n이 안전한 상한. n=5 → 2·8-1=15, n=8 → 15, n=9 → 31.

PREDICT 문제 2

Fenwick tree에서 x & -x가 왜 최하위 1비트인지 2's complement 관점에서.

풀이 펼치기
답

−x = ~x + 1 (2's complement). x=12=01100₂, ~x=10011₂, −x=10100₂. x & −x = 00100₂ = 4. 원리: ~x는 모든 비트 반전 → +1이면 최하위 0 비트들이 1로 뒤집히며 carry 발생, 최하위 1 비트까지 carry 전파되어 그 비트만 1로 남고 위 비트는 원 x와 반전된 상태. AND 결과는 그 1 비트만 살아남음.

BUILD 문제 3

"배열 역전(inversion) 수 세기" Fenwick으로 풀기.

풀이 펼치기
답

값 좌표 압축 후 뒤에서 앞으로 순회. BIT에 각 값의 "이후 등장 횟수" 저장. a[i]에 대해 query(a[i] − 1) = "뒤에 더 작은 값 수" = inversion 기여. 업데이트 → 다음 i로. 총 O(n log n). merge sort로도 O(n log n), 둘 다 표준.

DESUGAR 문제 4

Segment tree에서 "완전 포함"과 "분리"의 경계 조건이 왜 O(log n)?

풀이 펼치기
답

질의 범위의 경계 l, r에 대해 각 깊이에서 최대 2개 "부분 겹침" 노드가 생김. 부분 겹침은 자식에게 전파되지만 각 자식 중 최대 하나만 또 부분 겹침. 다른 자식은 완전 포함 또는 분리 → 즉시 종료. 깊이당 O(1) → 총 O(log n). CLRS 같은 상세 증명.

WHY 문제 5

왜 Fenwick이 min/max에 맞지 않나?

풀이 펼치기
답

Fenwick의 query는 prefix에서 구간을 빼기로 구함 (sum(r) − sum(l−1)). min은 역원 없음: min(a,b,c,d) − min(a,b)를 계산 불가. 따라서 업데이트 시 해당 원소만 감소하는 경우 BIT로 복원 불가. 구간 min/max는 segment tree나 sparse table(업데이트 없을 때)이 표준.

BUILD 문제 6

Lazy propagation으로 "구간 대입(set) + 구간 합 질의" 구현 아이디어.

풀이 펼치기
답

lazy[node]: "이 노드의 모든 원소가 이 값으로 설정됨" 또는 None(미설정). 업데이트 [l,r,v] 완전 포함 시 lazy[node]=v, tree[node]=v·(end-start+1). push 시 자식 lazy 덮어쓰기 + tree 재계산. 구간 덧셈과 구간 대입이 동시에 있으면 lazy 두 개 (add, set) 관리 — 순서 주의.

PREDICT 문제 7

n=10⁶, 구간 질의+업데이트 10⁶번. Segment tree vs Fenwick vs Prefix sum 실측.

풀이 펼치기
답

Segment tree: ~2초 (C++), ~20-30초 (Python). Fenwick: 더 빠름 (~1초 C++, 코드 짧고 캐시 좋음). Prefix sum: 업데이트가 O(n)이라 총 10¹² — 불가능. 경쟁에서 합만 필요하면 무조건 Fenwick.

DESUGAR 문제 8

"구간 덧셈 + 점 질의"를 Fenwick으로.

풀이 펼치기
답

트릭: BIT에 차분 배열(difference array) 저장. 구간 [l, r]에 v 더하기 = BIT.update(l, v); BIT.update(r+1, −v). 점 질의 A[i] = BIT.query(i). 모두 O(log n). "구간 덧셈 + 구간 합"은 Fenwick 두 개로 가능 (더 복잡).

WHY 문제 9

Persistent segment tree가 "k-th smallest in range" 문제를 어떻게 푸나?

풀이 펼치기
답

각 prefix [1..i]에 대한 segment tree (값 도메인). 위치 i의 트리 = 위치 i-1의 트리 + a[i] 추가 (log n 새 노드). 질의 [l, r]: root[r] − root[l-1]이 개념상 [l, r] 트리. k-th smallest는 이 합 트리에서 좌/우 자식 count 비교하며 내려감 — O(log V) 시간.

BUILD 문제 10

Segment tree를 이용한 "가장 긴 증가 부분수열 길이" 풀이 (LIS).

풀이 펼치기
답

값 도메인 [1, max]에 segment tree. tree[v] = "값 v로 끝나는 LIS 길이". 각 a[i]마다 query(1, a[i]-1)로 이전 최대 LIS 얻고 +1, update(a[i], 값). O(n log n). Algo 21의 patience sort와 복잡도 같음, 다른 관점.

AI 문제 11

AI가 "LeetCode Range Sum Query 문제에 Fenwick"을 제안. 문제가 update 없으면?

풀이 펼치기
답

업데이트 없으면 prefix sum만으로 O(1) 질의 + O(n) 전처리. Fenwick 은 O(log n) 질의라 오히려 느림. "Range Sum Query — Immutable"(LC 303) = prefix sum. "Mutable"(LC 307) = Fenwick/Segment tree. 문제 읽기가 핵심.

WHY 문제 12

Bentley가 segment tree를 계산 기하용으로 만들었는데, 왜 "배열 구간 합"에도 완벽히 맞나?

풀이 펼치기
답

공통 구조: "구간들을 쪼개어 관리". 기하에선 실수 구간, 배열에선 정수 인덱스 구간 — 추상 수준에서 같은 문제. Bentley의 자료구조가 이 추상화를 포착. 좋은 자료구조는 예상 못한 문제에도 맞아들어감 — Dijkstra가 최단경로를 넘어 MST·A*에 스며들듯.
