ALGO 25

해시 심화 — Bloom, Universal, Consistent, Cuckoo
단순 dict를 넘어서 분산 시스템으로

1970년 Burton Bloom이 "공간 절약을 위해 약간의 오류를 허용하면?"이라는 도발적 질문으로 Bloom filter를 발명. 9년 뒤 Carter & Wegman이 악의적 입력 방어를 위한 Universal hashing 이론 정립. 1997년 Karger et al.이 분산 캐시용 Consistent hashing을 MIT에서 발표 — 이 논문이 Akamai의 뿌리. 2001년 Pagh & Rodler가 O(1) 최악 조회를 보장하는 Cuckoo hashing을 발표.
Algo 10의 해시 테이블은 "작은 규모에서 잘 작동". 이 주는 "대규모, 분산, 악의적 입력, 공간 극단 절약"을 다루는 해시의 정수.
핵심 질문: "왜 N개 노드에 1조 개 키를 균등 분배하는 것이 비자명한가? 왜 DoS 공격이 해시 테이블을 무력화하는가?"

01 — 이번 주 이야기: 해시의 진화 55년

"In this paper trade-offs among certain computational factors in hash coding are analyzed. The new methods are intended to reduce the amount of space required to contain the hash-coded information from that associated with conventional methods."
— Burton H. Bloom, CACM 13(7):422-426, July 1970

1953 — 해시 테이블의 탄생

IBM 701에서 Hans Peter Luhn이 linked chaining hash table을 최초 제안(IBM Research Memo, 1953년 1월). 아직 "hash"라는 용어가 정착되기 전이었음. 이후 1956 Arnold Dumey, 1963 Werner Buchholz가 "hashing" 용어 확산.

1970 — Burton Bloom, MITRE Corporation

"Space/time trade-offs in hash coding with allowable errors"(CACM 13(7):422-426). Bloom은 MITRE에서 맞춤법 검사기를 개발 중. "정확히 답하지 말고, '확실히 없다' 또는 '있을 수도 있다' 두 가지만 답하라"는 도발적 아이디어 → Bloom filter. 공간 효율 극단 추구.

이후 50여 년간 CDN, 블록체인, 데이터베이스, 웹 브라우저 안전 검색 등에 광범위 응용.

1979 — Larry Carter & Mark Wegman, IBM

"Universal Classes of Hash Functions"(J. Comput. Syst. Sci. 18:143-154). 악의적 입력도 평균적으로 안전한 랜덤 해시 함수 집합이라는 이론 제시. 기대 시간이 입력과 무관 → DoS 공격 방어의 이론적 기반.

1984 — FKS Perfect Hashing

Fredman, Komlós, Szemerédi (모두 헝가리 조합론 전설). "Storing a Sparse Table with O(1) Worst Case Access Time"(J. ACM 31:538-544, 1984). 정적 집합에 대해 O(1) 최악 조회를 보장하는 2단계 해싱.

1997 — Karger et al., MIT / Akamai

"Consistent Hashing and Random Trees: Distributed Caching Protocols for Relieving Hot Spots on the World Wide Web"(STOC 1997). 저자: David Karger, Eric Lehman, Tom Leighton, Matthew Levine, Daniel Lewin, Rina Panigrahy.

핵심: 노드 N개에 키 K개를 분배. 노드가 추가/제거될 때 K/N개의 키만 재배치. 이 논문이 Akamai Technologies(Leighton + Lewin 공동 설립, 1998)의 근간. Lewin은 2001년 9·11 테러 AA11편 탑승 희생자. Akamai는 오늘날 세계 최대 CDN 중 하나.

2001 — Cuckoo Hashing

Rasmus Pagh & Flemming F. Rodler(덴마크 Aarhus), "Cuckoo Hashing"(ESA 2001, LNCS 2161:121-133). 뻐꾸기(cuckoo)가 남의 둥지에 알을 낳는 습성에서 이름. O(1) 최악 조회를 동적 해시로 달성.

해시의 55년: 1953 Luhn이 IBM 701에서 첫 해시 테이블. 1970 Bloom, 1979 Carter-Wegman이 이론적 기반. 1997 Karger가 분산 시스템의 문을 엶 → Akamai → CDN 산업. 2001 Cuckoo로 최악 O(1) 달성.

02 — 복습 — 충돌, load factor, 기본 해시

중급

Algo 10 빠른 복습

  • 해시 함수 h: U → [0, m). 이상적으로 uniform random.
  • 충돌(collision): h(x) = h(y)이고 x ≠ y.
  • Load factor α = n/m (원소 수 / 테이블 크기).
  • Birthday paradox: m=365일 때 √m ≈ 19명으로 50% 확률 충돌. 해시 테이블에도 같은 원리.
고급

충돌 해결의 두 전략

  1. Separate Chaining: 각 버킷이 연결 리스트. 충돌 시 리스트에 추가.
  2. Open Addressing: 충돌 시 다른 버킷을 찾아 저장. Linear/quadratic/double hashing.

Python dict, Go map → open addressing. Java HashMap, C++ unordered_map → chaining (Java 8+부터 chain이 8개 넘으면 red-black tree로 전환).

03 — Separate Chaining

중급

구조와 성능

각 버킷이 연결 리스트(또는 더 세련된 구조). 충돌된 모든 원소가 한 리스트에.

  • 삽입: O(1) (리스트 head에).
  • 조회: 평균 O(1 + α), 최악 O(n) (모두 충돌).
  • 삭제: O(1 + α).
  • 장점: α > 1도 가능. 삭제 쉬움. 구현 단순.
  • 단점: 포인터 오버헤드, 캐시 비효율.
chaining_hashmap.py
class ChainingHashMap:
    def __init__(self, capacity=16):
        self.capacity = capacity
        self.size = 0
        self.buckets = [[] for _ in range(capacity)]

    def _hash(self, key):
        return hash(key) % self.capacity

    def put(self, key, value):
        bucket = self.buckets[self._hash(key)]
        for i, (k, v) in enumerate(bucket):
            if k == key:
                bucket[i] = (key, value)
                return
        bucket.append((key, value))
        self.size += 1
        if self.size > self.capacity * 0.75:
            self._resize()

    def get(self, key):
        bucket = self.buckets[self._hash(key)]
        for k, v in bucket:
            if k == key: return v
        raise KeyError(key)

    def _resize(self):
        old = self.buckets
        self.capacity *= 2
        self.buckets = [[] for _ in range(self.capacity)]
        self.size = 0
        for bucket in old:
            for k, v in bucket: self.put(k, v)

04 — Open Addressing — Linear / Quadratic

고급

Probing 전략 3종

  • Linear Probing: 충돌 시 h(k), h(k)+1, h(k)+2, ... 탐사. 캐시 효율 좋음, 클러스터링 심함.
  • Quadratic Probing: h(k), h(k)+1, h(k)+4, h(k)+9, ... (c1·i + c2·i²). 1차 클러스터링 완화.
  • Double Hashing: h1(k), h1(k)+h2(k), h1(k)+2h2(k), ... 가장 랜덤한 분포, 캐시 효율은 떨어짐.

중요 제약: α < 1 (테이블이 꽉 차면 탐사 실패). 보통 α=0.5-0.75에서 resize.

삭제 문제: 단순히 지우면 탐사 체인이 끊어짐. Tombstone(삭제 마커)으로 표시하고 rehash 시 정리. 또는 체인의 뒤쪽 원소를 앞으로 shift(Robin Hood는 이를 자연스럽게 함).

같은 충돌, 다른 해결: Chaining은 버킷 밖으로 리스트 확장. Open Addressing은 같은 배열 안에서 다른 자리 찾기. Python dict은 open addressing + perturbation, Java HashMap은 chaining.

05 — Universal Hashing — Carter-Wegman 1979 (CLRS §11.3.3)

전공자

왜 단일 해시 함수는 위험한가

고정된 해시 함수 h를 쓰면, 공격자가 h를 분석해 모두 같은 버킷에 해시되는 입력 집합을 만들 수 있음. 모든 삽입이 O(n) 충돌 → 서버 DoS (HashDoS 공격, 2003 Crosby & Wallach).

해결: 매 프로그램 실행마다 해시 함수를 무작위 선택. 공격자가 미리 입력을 준비할 수 없음.

전공자

Universal family 정의

해시 함수의 유한집합 H가 universal이면:

∀ x, y ∈ U, x ≠ y:   Prh ∈ H[h(x) = h(y)] ≤ 1/m

"무작위로 고른 h에 대해 충돌 확률이 ≤ 1/m" — 이상적 균등 해시와 동등.

구체 예: U = {0, 1, ..., p−1}(p는 큰 소수 > max key). 함수 ha,b(x) = ((ax + b) mod p) mod m. a ∈ {1,...,p−1}, b ∈ {0,...,p−1} 중 랜덤 선택.

Hp,m = {ha,b} 집합이 universal — Carter-Wegman 1979 증명. 오늘날 Python, Go, Java 해시가 이런 무작위화를 사용.

06 — Perfect Hashing — FKS 1984

전공자

정적 집합 O(1) 최악 조회

Fredman, Komlós, Szemerédi 1984: "정적 n개 키를 O(n) 공간에 저장하면서 최악 O(1) 조회" 가능. 2단계 해싱:

  1. 1단계: 크기 m=n인 주 테이블. 해시 h1로 각 키를 버킷에 분배. 예상 충돌 수 O(n).
  2. 2단계: 각 버킷 j에 대해, 그 버킷의 원소 수가 nj라면, 크기 nj²의 내부 테이블 할당. 내부 해시 h2,j를 무작위로 선택 — 충돌 없을 때까지.
  3. 전체 공간 O(∑nj²) = O(n) (기대값).

응용: 컴파일러의 keyword lookup, CPU명령어 디코딩 등 미리 정해진 집합에 대한 O(1) 보장 조회. 동적 삽입 필요하면 Dynamic Perfect Hashing(Dietzfelbinger et al. 1994)로 확장.

07 — Consistent Hashing — Karger 1997

고급

문제 — 분산 캐시의 재배치 지옥

N개 서버에 K개 키를 hash(key) mod N으로 분배. 서버가 하나 추가(N → N+1)되면? 거의 모든 키가 다른 서버로 이동해야 함 (mod N+1은 mod N과 완전히 다름). 재분배 트래픽 폭발.

Karger의 해결책: 키와 서버 모두를 원형 해시 공간(hash ring)에 배치. 키는 시계 방향으로 가장 가까운 서버에.

Consistent hashing 링: 노드와 키 모두 원형 해시 공간에 배치. 키는 시계 방향으로 가장 가까운 노드에. 노드 추가/제거 시 K/N개 키만 재배치 — DHT(Chord, Dynamo), CDN(Akamai), 분산 캐시(Memcached), Cassandra/DynamoDB.
전공자

Virtual Nodes

문제: 노드 수 적으면 링에 불균등 분포 → 일부 노드 과부하. 해결: 각 물리 노드를 여러 가상 노드(replica, 보통 100-200개)로. 해시 링에서 골고루 분포.

  • Amazon Dynamo (2007 논문) → DynamoDB (2012 AWS 서비스): 가상 노드 기반.
  • Apache Cassandra: 토큰 링(virtual nodes).
  • Redis Cluster: 16384 슬롯으로 확장 (consistent hashing 변종).

08 — Bloom Filter — Bloom 1970

고급

공간을 위한 거래

Bloom filter: 크기 m 비트 배열 + k개 해시 함수. 집합 S의 확률적 표현.

  • 삽입(x): h1(x), h2(x), ..., hk(x) 위치의 비트를 1로.
  • 조회(x): 모든 hi(x) 비트가 1이면 "있을 수도", 하나라도 0이면 "확실히 없음".
  • False positive: "있다"고 답했는데 실제로 없을 수 있음.
  • False negative 없음: "없다"고 답했으면 확실히 없음.
  • 삭제 불가: 비트를 0으로 되돌리면 다른 원소의 정보도 손실. (Counting Bloom filter로 해결.)

False positive rate: n개 원소 삽입 후 p ≈ (1 − e−kn/m)k. 최적 k = (m/n) · ln 2.

4프레임 Bloom filter: 빈 필터 → apple 삽입 → banana 삽입 (비트 5 공유) → cherry 조회 (비트 10=0 → 확실히 없음). 공간 m=12비트로 2원소 집합 표현 — 극단 공간 절약 + 확률적 false positive.
전공자

응용

  • Chrome Safe Browsing: 악성 URL 목록(수천만 개) → Bloom filter (수 MB). 로컬에서 빠르게 "안전하지 않을 수도" 판별 후 서버 확인.
  • Bitcoin SPV client: 관심 주소만 풀 노드에서 받기 (privacy 논란 있음).
  • Cassandra, LevelDB, RocksDB: 디스크 SSTable 읽기 전 "이 키 있을지" Bloom filter로 체크 → 디스크 IO 절약.
  • CDN 캐시: "이 URL 캐싱된 적 있나?" 빠른 체크.
  • Spell check: Bloom의 원래 응용.

09 — Cuckoo Hashing — Pagh-Rodler 2001

전공자

O(1) 최악 조회의 동적 해시

아이디어: 각 키 x는 두 개의 해시 함수 h1(x), h2(x) 중 정확히 한 위치에 저장.

  • 조회(x): T1[h1(x)] 또는 T2[h2(x)] 둘 중 하나만 확인 → 최악 O(1).
  • 삽입(x): h1(x)에 놓기. 자리가 비어있지 않으면 거기 있던 y를 꺼내고 x를 놓음. 꺼낸 y는 자신의 다른 해시 위치로 이동. 이게 자리가 차있으면 또 쫓아냄 — 뻐꾸기(cuckoo)가 둥지에서 알을 밀어내는 것처럼.
  • 사이클: 무한 루프 가능 → 일정 횟수 이상 실패하면 rehash(새 해시 함수 + 테이블 확장).

삽입 평균 O(1) (load factor < 0.5 보장 시), rehash 확률 낮음. 하드웨어 친화적(조회 상수 시간 guarantees) — 네트워크 장비, 데이터베이스 인덱스에 인기.

4프레임 Cuckoo 삽입: Z→1 → X 쫓겨남 → T2[2]로 → Y 쫓겨남 → T1[0]에 안착. 조회는 항상 T1 또는 T2의 한 위치만 확인해 O(1) 최악. 사이클 발생 시 rehash.

10 — Robin Hood Hashing

전공자

Celis-Larson-Munro 1985

Open addressing의 변종. "부자에게 빼앗아 가난한 자에게" 철학:

  • 각 키는 probe distance를 가짐 — 본래 위치에서 얼마나 멀리 탐사됐는가.
  • 삽입 시 현재 위치의 키와 probe distance 비교 → 내가 더 오래 probe했으면 위치 양보.
  • 결과: probe distance의 분산이 작아짐. 최악·평균 모두 개선.

현대 고성능 해시 테이블(Rust의 HashMap, Emilib, Google SwissTable의 일부 변종)에 사용. 캐시 효율이 중요한 이유.

전공자

Hopscotch Hashing (Herlihy-Shavit-Tzafrir 2008)

각 키는 자기 홈 버킷 + 작은 근접 범위(hop range, 보통 H=32) 내에 저장. 삽입/이동이 이 범위 내에서만 일어남 → 캐시 친화적. 락 프리 버전 설계 용이 → 멀티스레드 해시에 인기.

11 — CPython dict 내부

고급

Python의 선택들

  • Open addressing: chaining 아님. 포인터 오버헤드 회피.
  • Perturbation probing: i = (5*i + perturb + 1) mod m; perturb >>= 5. 해시 비트를 점진적으로 투입 — linear probing의 클러스터링 완화.
  • Load factor ≤ 2/3: 넘으면 테이블 크기 2-4배로 resize.
  • Compact dict (Python 3.6+, Raymond Hettinger): 삽입 순서 보존 + 공간 20-25% 절약. 인덱스 배열 + 엔트리 배열 분리.
  • SipHash(2012 Aumasson-Bernstein): str 해시에 랜덤 시드 추가 → HashDoS 방어. Python 3.4-3.11은 SipHash-2-4, 3.12+부터 SipHash-1-3(더 빠름). PYTHONHASHSEED=0로 비활성화 가능(재현성).
  • int 해시는 identity: hash(5) == 5. 단순하지만 패턴 있는 int(예: 2의 배수) 입력에 약할 수 있음.
CPython_dict_probe.c
// CPython dictobject.c의 핵심 probing (의사코드)
for (perturb = hash; ; perturb >>= 5) {
    i = ((5 * i) + perturb + 1) & mask;
    if (entry[i].key == NULL) break;    // 빈 슬롯 → 삽입
    if (entry[i].hash == hash && keys_equal(...))
        return entry[i];    // 매치
}

12 — 현대 해시 함수 — MurmurHash, SipHash

고급

비암호 해시 — 속도 우선

  • MurmurHash(Austin Appleby 2008): 비암호. 매우 빠름. MurmurHash3가 현재 표준. Cassandra, Redis, Elasticsearch 등.
  • CityHash(Google 2011): MurmurHash 후속. SSE4.2 CRC 명령어 활용.
  • xxHash(Yann Collet 2012): LZ4 저자. 현재 가장 빠른 비암호 해시. Zstandard에 번들.
  • FarmHash(Google 2014): CityHash 후속.
  • wyhash(2019): 단순하고 극도로 빠름.

이들은 속도 우선 — 암호학적 안전성 없음. MD5처럼 해시 충돌 공격 가능 → 악의적 입력엔 취약.

고급

SipHash — 적대적 안전성

SipHash(Aumasson & Bernstein 2012): key를 쓰는 pseudorandom function. 속도는 MurmurHash보다 약간 느리지만 HashDoS 방어. 2012년 발표 후 Python 3.4(2014, PEP 456), Ruby, Rust, Haskell, Perl 등이 수년 내 기본 해시로 채택.

  • SipHash-2-4: 2라운드 + 4라운드 출력.
  • Python 3.4-3.11 기본은 SipHash-2-4 (PEP 456). Python 3.12+부터 SipHash-1-3 (더 빠른 변종).
  • Key는 프로세스 시작 시 무작위 생성. 재시작마다 다른 해시.
고급

암호 해시

  • MD5(1991), SHA-1(1995): 현재 충돌 공격 가능. 해시 테이블에만 쓰고 보안 목적엔 금지.
  • SHA-2(2001, SHA-256/512): 현재 안전. 블록체인, TLS.
  • SHA-3(2015, Keccak): NIST 표준화. SHA-2 대안.
  • BLAKE3(2020): 가장 빠른 안전 해시. SIMD·SSE·AVX 활용.

13 — 응용 — Akamai, Redis, 블록체인

고급

Akamai와 CDN

Daniel Lewin(MIT 박사생)과 Tom Leighton(MIT 교수)은 Karger 1997 논문을 기반으로 1998년 Akamai Technologies 창립. 웹 콘텐츠를 세계 곳곳의 엣지 서버에 분산해 "핫스팟" 제거. 1999년 IPO, 현재 세계 최대 CDN 중 하나.

같은 원리로 Cloudflare, Fastly, AWS CloudFront 등 현대 CDN 인프라가 구축됨. MIT 박사 논문 → 수십조 원 산업.

고급

분산 데이터베이스

  • Amazon DynamoDB(2007 논문, 2012 서비스): virtual nodes + consistent hashing. 토큰 링.
  • Apache Cassandra: DynamoDB 논문에서 영감. 오픈소스.
  • Redis Cluster: 16384 슬롯 + consistent hashing 변종. 노드 추가 시 일부 슬롯만 마이그레이션.
  • Memcached: ketama consistent hashing 라이브러리 (2007).
전공자

블록체인

  • SHA-256 Merkle tree: 블록 내 거래 집약 — 효율적 증명.
  • Bitcoin SPV: Bloom filter로 관심 주소만 필터링 (프라이버시 문제로 2018 이후 비권장).
  • Ethereum Merkle Patricia Trie: trie + 해시. 상태 저장.
  • Content-Addressed Storage (IPFS): 데이터의 해시가 곧 주소.
전공자

AI가 해시를 잘못 쓰는 패턴

  • mod N로 분산 해싱: N 바뀌면 거의 모든 키 재배치. Consistent hashing 권장.
  • MD5로 해시 테이블: 너무 느림(128비트). MurmurHash/xxHash가 적합.
  • Bloom filter에 삭제: 불가능(일반 Bloom). Counting Bloom 필요.
  • Cuckoo 사이클 무시: 무한 루프 가능. rehash trigger 필수.
  • 암호 해시를 non-crypto 목적에: 너무 느림.
  • Non-crypto 해시를 보안 목적에: 충돌 공격 가능. MurmurHash는 HashDoS 취약.

교정 질문: "분산 시스템이면 consistent hashing인가?", "악의적 입력 가능한가? SipHash인가?", "삭제 필요하면 Counting Bloom인가?"

14 — 연습문제 (12문제)

WHY 문제 1

Bloom filter에서 "false negative가 절대 없다"의 의미와 증명.

풀이 펼치기
답

x를 삽입했다면 h1(x), ..., hk(x) 비트가 모두 1. 이후 조회 시 이 비트들은 0으로 돌아갈 수 없다 (비트는 1로만 설정, 0으로 내리지 않음). 따라서 조회하면 항상 "있을 수도"라고 답함. "없다"라는 답은 비트 중 하나가 0이어야만 가능 — 그 경우 x는 삽입된 적이 없음이 수학적으로 보장.

PREDICT 문제 2

1000만 URL을 Bloom filter에 넣고 FP rate 1%를 원한다면 크기 m과 k?

풀이 펼치기
답

n=107, p=0.01.
m = −n·ln(p) / (ln 2)² ≈ 107 × 4.6 / 0.48 ≈ 9.6·107 비트 ≈ 12 MB.
k = (m/n) · ln 2 ≈ 9.6 × 0.693 ≈ 6.6 → 7 해시 함수.
참고: 평균 URL 60바이트면 원본 저장 600MB, Bloom은 12MB — 50배 절약.

BUILD 문제 3

Consistent hashing에서 N=3에 노드 하나 추가 시 이동하는 키 비율.

풀이 펼치기
답

새 노드가 차지하는 호(arc) 위의 키만 이동. 균등 분포 기대값 = 1/(N+1) = 1/4 = 25%. 단순 mod N → mod (N+1) 방식은 ~100% 이동. 분산 시스템에서 25% vs 100%는 엄청난 차이.

DESUGAR 문제 4

Universal hashing이 HashDoS를 방어하는 원리.

풀이 펼치기
답

공격자는 해시 함수를 알아야 악의적 충돌 입력을 만들 수 있음. Universal family에서 매 실행마다 무작위 h 선택 → 공격자가 h를 모르면 어떤 n개 입력에 대해서도 평균 충돌 확률 ≤ 1/m. 입력이 무엇이든 기대 시간 O(n). SipHash + 프로세스 시작 시 랜덤 시드 = 실무 구현.

WHY 문제 5

Cuckoo hashing의 load factor가 왜 0.5 이하여야 하나?

풀이 펼치기
답

확률 분석: 각 키는 2개 위치 후보. 무작위 그래프 모델로 "2-core"가 생길 확률이 α=0.5에서 급격히 커짐 — cycle 생기면 무한 루프 가능성 증가 → rehash 필요. 실용적으로 0.4-0.45에서 resize. 변종 d-ary cuckoo (d>2 해시 함수)는 더 높은 load factor 허용 (d=3이면 0.91).

BUILD 문제 6

Counting Bloom filter 구현 아이디어.

풀이 펼치기
답

비트 대신 4비트 카운터 배열. 삽입 시 각 hi(x) 위치 카운터 ++. 삭제 시 --. 0이 아니면 "있을 수도". 공간 4배 증가, 삭제 가능. 카운터 오버플로우 시 saturate. Cisco의 네트워크 상태 추적에 사용.

PREDICT 문제 7

Python hash(5)가 5를 반환하는 이유와 함정.

풀이 펼치기
답

작은 int의 해시는 값 자체 (성능 최적화). 함정: 모든 키가 m의 배수(예: {0, m, 2m, ...})면 모두 같은 슬롯 → 최악 충돌. CPython은 perturbation으로 완화 — 해시 상위 비트를 섞어 linear probing 클러스터링 방지. hash(str)은 SipHash, hash(int)는 단순.

WHY 문제 8

Virtual nodes가 consistent hashing에 왜 필수?

풀이 펼치기
답

N=3 실제 노드만 링에 놓으면 운 나쁘게 두 노드가 근처에 배치 → 한 노드가 거의 모든 키 받음 (hot spot). Virtual nodes(각 물리 노드가 200개 가상)면 링의 고르게 분포 → 부하 균등. 노드 하나가 전체의 1/N에 ±O(log N / √V) 수렴 (Karger 정리).

BUILD 문제 9

Bloom filter를 이용한 "중복 URL 크롤링 방지" 시스템 설계.

풀이 펼치기
답
  1. 방문 예정 URL을 Bloom filter에 삽입.
  2. 새 URL 발견 시 조회: "있을 수도" → 실제 DB 확인. "없음" → 확정 미방문.
  3. False positive = 가끔 DB 조회 (비용 수용 가능).
  4. False negative 없음 = 미방문 URL을 놓치지 않음.

Google, Bing 크롤러의 표준 패턴. 메모리 효율로 수십억 URL 추적 가능.

DESUGAR 문제 10

Linear probing과 double hashing 중 캐시 효율이 더 좋은 쪽.

풀이 펼치기
답

Linear probing. 연속된 메모리에 접근 → CPU 캐시 라인 로드 한 번이면 여러 슬롯 포함. Double hashing은 랜덤 위치 → cache miss 자주. 단, linear은 클러스터링 심해 α↑ 때 탐사 길이 증가. 실무에선 load factor 낮게 유지 + linear probing 조합이 캐시 시대 이상.

AI 문제 11

AI가 "웹 스케일 중복 제거에 Python set" 제안 — 교정?

풀이 펼치기
질문
  1. "규모? 10억 URL이면 set은 수십 GB 메모리. 서버 터짐."
  2. "정확성 필수인가? '거의 확실히 없음'이면 충분하면 Bloom filter (수백 MB로 충분)."
  3. "정확 필요하면 외부 DB + Bloom filter 2단계 — 대부분 Bloom에서 필터 후 DB 쿼리 드물게."
  4. "Counting Bloom으로 삭제도 가능."
WHY 문제 12

SHA-256을 해시 테이블 키로 쓰면 안 되는 이유.

풀이 펼치기
답

너무 느림. SHA-256은 블록당 수백 사이클 — MurmurHash/xxHash는 수 사이클. 해시 테이블은 초당 수백만 연산 → 병목. SHA-256은 보안 목적(서명, 블록체인)에만. 해시 테이블엔 비암호 해시 + SipHash(DoS 방어 필요 시). 빠른 해시는 충돌이 가끔 있어도 amortized 문제 없음.
