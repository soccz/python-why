01 — 이번 주 이야기: 퀵소트의 역설

1959년, 25세의 토니 호어(Tony Hoare)는 소련 모스크바 국립대학교에 방문학생으로 가 영어-러시아어 기계 번역 프로젝트에 참여했습니다. 러시아어 단어를 사전에서 찾아야 했는데, 사전은 정렬되어 있지 않았습니다. 호어는 호텔 방에서 새로운 정렬 알고리즘을 발명합니다 — Quicksort (CACM 1961, Computer Journal 1962에 발표).

퀵소트의 아이디어 — 한 문장

배열에서 원소 하나(피벗)를 고르고, 그것보다 작은 것은 왼쪽, 큰 것은 오른쪽으로 옮긴 뒤, 양쪽을 각각 재귀로 정렬합니다. 병합 단계가 없습니다 — "제자리에서 정렬됨"이 핵심.

가장 작은 n에서 시연:
[3, 1, 4, 1, 5, 9, 2, 6] → 피벗 4 선택 → [3, 1, 1, 2, | 4 | 5, 9, 6] (작은 것 왼쪽, 큰 것 오른쪽). 이제 왼쪽, 오른쪽을 각각 재귀.

피벗 선택이 운명을 가른다. 좋은 피벗은 균형 분할 → 깊이 log n → Θ(n log n). 나쁜 피벗(이미 정렬된 입력에서 첫 원소)은 한쪽 0개 → 깊이 n → Θ(n²). 무작위 피벗이 적대 입력을 무력화.
퀵소트 = 피벗으로 갈라치고 양쪽 재귀. 무작위 피벗이 적대 입력을 막아 평균 Θ(n log n) 보장. 5단계로 압축한 버전 — 실제로는 더 많은 재귀가 일어나지만 본질적 흐름은 같음.

역설 — 최악은 Θ(n²)인데도 표준 라이브러리가 퀵소트를 쓰는 이유

이미 정렬된 배열 [1, 2, 3, …, n]에 퀵소트를 쓰고 피벗을 항상 맨 앞으로 고르면: 첫 분할은 [] | 1 | [2,3,…,n], 다음은 [] | 2 | [3,…,n]… 분할이 (0개, n−1개)로 완전히 한쪽으로 치우쳐 재귀 깊이가 n.

그래서 T(n) = T(n−1) + Θ(n) = Θ(n²). 병합 정렬(Θ(n log n))보다 훨씬 나쁩니다.

그런데 C++의 std::sort(Introsort = Quicksort + Heapsort), Java의 Arrays.sort, Go의 sort.Slice는 모두 퀵소트 기반. Python의 Timsort는 예외지만 — 여전히 C 표준라이브러리 qsort는 퀵소트. 왜 이 "최악 n²" 알고리즘이 세계를 지배할까요?

WHY — 피벗을 무작위로 고르면 최악이 사라진다

적대자가 입력 배열을 고를 수 있어도, 알고리즘이 피벗을 무작위로 고르면 적대자가 어떤 배열을 고르든 상관없이 기댓값 Θ(n log n)이 보장됩니다.

이게 이번 주의 핵심 아이디어 — "알고리즘이 동전을 던지면 입력의 적대성이 사라진다." 수학적으로 말하면, 무작위화는 입력에 대한 평균을 알고리즘의 동전 던지기에 대한 평균으로 바꿉니다. 전자는 "운이 나쁘면 항상 느림", 후자는 "운이 나쁜 경우는 지수적으로 드묾".

📖 역사 인용 — Hoare, "Quicksort" (Computer Journal, 1962)

"The algorithm is based on a simple idea: the given set is partitioned in place by selecting one of its elements and moving all smaller elements to one side and all larger ones to the other. The two resulting parts are then sorted recursively by the same method."

— C. A. R. Hoare. "Quicksort." The Computer Journal, Vol. 5, Issue 1, 1962.

02 — 왜 무작위인가? 적대적 입력을 이기는 법

초급

가위바위보 비유

친구와 가위바위보를 100번 하는데, 친구는 여러분의 손버릇을 압니다 — 여러분이 "주먹을 자주 낸다"는 걸 눈치채고 보를 냅니다. 어떻게 이길까요? 답은 "예측 불가능해지는 것" — 주사위를 굴려서 선택하는 것.

퀵소트도 똑같습니다. 적대자가 "퀵소트는 정렬된 입력에 약하다"는 걸 알면 정렬된 배열을 줘서 n² 시간을 만듭니다. 하지만 피벗을 주사위로 고르면, 적대자가 아무리 영리해도 어떤 배열로도 여러분을 붙잡을 수 없습니다.

중급

두 가지 "평균" — 입력 평균 vs 코인 평균

"평균 시간 복잡도"라는 말은 두 가지 의미가 있습니다. 중요한 구분입니다.

구분Average-case (입력 평균)Expected time (무작위 평균)
기준 입력의 분포에 대한 평균 알고리즘 내부 무작위 선택에 대한 평균
가정 입력이 균등 분포라 가정 (비현실적) 알고리즘이 동전을 던짐 — 입력은 고정
적대자 입력 분포를 깰 수 있음 어떤 입력이든 무력화
예 "결정론적 퀵소트는 평균 Θ(n log n)" — 입력이 균등할 때만 "Randomized 퀵소트는 기댓값 Θ(n log n)" — 모든 입력에서

Randomized 알고리즘이 강력한 건 입력에 대한 가정을 안 한다는 점입니다. 적대자가 최악의 입력을 골라도 소용없죠.

고급

무작위화의 세 가지 의미

  1. 대칭성 깨기: 분산 시스템에서 여러 프로세스가 동시에 같은 결정을 내릴 때 무작위로 하나를 뽑아 교착 상태 해결. 예: 이더넷의 CSMA/CD backoff.
  2. 적대적 입력 무력화: 퀵소트·해시 테이블의 universal hashing처럼, 입력이 적대적이어도 평균 보장.
  3. 결정론으로는 비싼 것을 싸게: 소수성 검사(Miller-Rabin), 큰 행렬 곱셈 검증(Freivalds) — 결정론은 다항식이지만 높은 차수, 무작위는 훨씬 낮은 차수.
결정론적 최악 하한   ≥   randomized 기댓값 하한

즉, 무작위화로 하한 자체가 내려갈 수도 있습니다. 이것이 randomized 알고리즘이 이론적으로 흥미로운 이유.

📖 원서 인용 — CLRS 4판, §5 도입부

"The behavior of a randomized algorithm is determined not only by its input but also by values produced by a random-number generator. ... We often analyze a randomized algorithm by measuring its expected running time, where the expectation is taken over the values returned by the random-number generator."

— Cormen, Leiserson, Rivest, Stein. Introduction to Algorithms, 4th ed., §5 "Probabilistic Analysis and Randomized Algorithms".

03 — 확률 복습: 표본공간부터 기댓값까지

중급

기본 용어 — 한눈에

용어뜻예 (주사위 한 번)
표본공간 Ω가능한 결과 전체 집합{1, 2, 3, 4, 5, 6}
사건 AΩ의 부분집합"짝수" = {2, 4, 6}
확률 P(A)A가 일어날 가능도 (0 ≤ P ≤ 1)P("짝수") = 3/6 = 0.5
확률 변수 X결과를 수로 매핑하는 함수X = 주사위 눈금
기댓값 E[X]X의 "평균값" — Σ x·P(X=x)E[주사위] = 1/6·(1+2+…+6) = 3.5
고급

기댓값의 선형성 — 모든 것의 열쇠

이번 주에 가장 많이 쓸 한 줄의 성질 — 기댓값은 선형입니다.

E[X + Y] = E[X] + E[Y]   (X와 Y가 독립이 아니어도 항상 성립!)

"더해서 평균 내는 것과, 평균 낸 것을 더하는 것이 같다." 직관적으로 자명하지만 증명은 정의에서 바로 나옵니다:

E[X + Y] = Σ (x + y)·P(X=x, Y=y)  
     = Σ x·P(X=x, Y=y) + Σ y·P(X=x, Y=y)
     = Σ x·P(X=x) + Σ y·P(Y=y) = E[X] + E[Y]. ∎

핵심은 "X와 Y의 독립이 필요 없다"는 것. 서로 얽혀 있어도 성립합니다. 퀵소트 분석에서 여러 사건이 얽혀 있을 텐데, 독립성 확인 없이 바로 더할 수 있으니 증명이 깔끔해집니다.

고급

조건부 확률 & 독립

P(A | B) = P(A ∩ B) / P(B),   A ⊥ B ⟺ P(A ∩ B) = P(A)·P(B)

조건부 확률은 "B가 일어났다는 걸 알고 A의 확률". 독립이면 B의 정보가 A에 영향 없음.

이번 주 분석에서는 기댓값의 선형성을 쓰는 한 조건부와 독립은 별로 필요하지 않습니다 — 지표 확률 변수 기법(§04)이 이들을 피해가는 우아한 도구이기 때문.

⚠️ "평균"이라는 말의 함정

"Randomized quicksort의 평균 복잡도는 Θ(n log n)" — 이 "평균"은 코인 던지기의 평균이지 "운이 좋으면"이 아닙니다. 기댓값이 Θ(n log n)이라는 건 모든 입력에 대해 코인 분포에서 평균 Θ(n log n)이라는 뜻.

또 집중 부등식(§11 Chernoff)에 의해 꼬리 확률도 매우 작습니다 — Θ(n log n)에서 크게 벗어날 확률이 지수적으로 작음.

04 — 지표 확률 변수 (Indicator Random Variable, CLRS §5.2)

고급

정의 — "사건 A가 일어났는지"를 0/1로

사건 A에 대한 지표 확률 변수 I_A는 다음과 같이 정의됩니다.

I_A = 1   if A가 일어나면    otherwise   I_A = 0

이때 기댓값은 정확히 사건의 확률.

E[I_A] = 1·P(A) + 0·P(¬A) = P(A)
고급

왜 이게 강력한가 — "세는 문제"를 "더하는 문제"로

"어떤 일이 몇 번 일어났는가?"라는 문제를 풀 때, 각 가능한 사건에 지표 변수를 하나씩 만들고 다 더하면 횟수가 됩니다.

X = I₁ + I₂ + … + I_n   (I_k = k번째 시행에서 "성공"이면 1, 아니면 0)
E[X] = E[I₁] + E[I₂] + … + E[I_n] = P₁ + P₂ + … + P_n

독립성이 필요 없는 것이 결정적입니다. I_k들이 서로 얽혀 있어도 그냥 더할 수 있습니다.

예제 — 동전 n번 던질 때 앞면 개수

I_k = "k번째 던지기에서 앞면이면 1"이라 하면 X = 총 앞면 수 = Σ I_k. E[I_k] = 1/2 (공정 동전). 따라서 E[X] = n/2. 직관과 일치 ✓

CLRS의 고용 문제(§5.1) 예고편

n명의 지원자를 무작위 순서로 면접하며 "지금까지 최고면 고용, 아니면 무시". 몇 명 고용할까? I_k = "k번째 지원자를 고용하면 1"이라 하면 E[I_k] = 1/k (k번째가 1~k 중 최고일 확률). 총 고용 수 기댓값 = Σ 1/k ≈ ln n — n번 면접해도 평균 log n명만 고용. §05에서 자세히.

📖 원서 인용 — CLRS 4판, §5.2 "Indicator random variables"

"Indicator random variables provide a convenient method for converting between probabilities and expectations. Suppose we are given a sample space S and an event A. Then the indicator random variable I{A} associated with event A is defined as... The expected value of I{A} equals the probability P(A) that A occurs."

WHY — 지표 변수가 "복잡한 확률"을 "간단한 산수"로 바꾸는 이유

"세는 문제"를 직접 풀면 경우의 수가 지수적으로 많아 손을 못 댑니다. 하지만 지표 변수 + 기댓값 선형성은:

  1. 개별 사건의 확률 P_k만 구하면 됩니다 (보통 간단).
  2. 다 더하면 답이 나옵니다 (독립성 확인 없이).

"세는 것 → 확률로 바꾸는 것 → 다 더하는 것" — 이 3단계가 이번 주 모든 분석의 뼈대입니다.

05 — 고용 문제 — 기대 채용 횟수 Θ(log n) (CLRS §5.1)

고급

문제 정의

당신은 비서 한 명을 뽑으려 합니다. 인력소개소가 n명의 지원자를 한 명씩 보냅니다. 각 지원자를 면접(비용 c_i, 저렴)하고, 지금까지 본 지원자 중 가장 뛰어나면 즉시 이전 비서를 해고하고 채용합니다(비용 c_h, 비쌈). 지원자 순서는 무작위 순열.

총 비용은 (모든 면접비 = n·c_i) + (채용 횟수 × c_h). 면접비는 고정이니 "채용 횟수의 기댓값"이 관건.

hire_assistant.py (CLRS §5.1 의사코드)
def hire_assistant(candidates):
    best = -1  # 가상의 "누구보다 낮은" 더미
    hired = 0
    for i, score in enumerate(candidates):
        if score > best:
            best = score
            hired += 1   # 비싼 채용 비용 발생
    return hired
고급

지표 변수로 풀기

I_k = "k번째 지원자가 채용되면 1, 아니면 0" (k = 1, 2, …, n). 총 채용 수 X = I₁ + I₂ + … + I_n.

핵심 확률 계산: k번째 지원자가 처음 k명 중 가장 뛰어날 확률은? 무작위 순열에서 k명 중 누구도 특별하지 않으니, 각자 가장 높을 확률은 1/k. 따라서 E[I_k] = 1/k.

E[X] = Σk=1n E[I_k] = Σk=1n 1/k = H_n   (n번째 조화수)
H_n = 1 + 1/2 + 1/3 + … + 1/n = ln n + γ + O(1/n)

여기서 γ ≈ 0.5772 (오일러-마스케로니 상수). 따라서 채용 횟수 기댓값 ≈ ln n.

총 비용 기댓값 = n·c_i + ln n · c_h = O(c_h · ln n + n·c_i). 최악의 경우(오름차순 입력)는 n번 채용해 c_h·n이지만, 무작위 순서면 지수적으로 적음.

전공자

왜 1/k인가 — 꼼꼼히

처음 k명(지원자 번호 1, 2, …, k)의 점수를 s₁, s₂, …, s_k라 합시다. 무작위 순열이라면 이들의 상대적 순위는 균등 분포. 즉 "가장 높은 점수가 k번째 자리에 있을 확률"은 정확히 k개 중 하나를 고를 확률 = 1/k.

중요한 건 이 확률이 뒤쪽 지원자(k+1 ~ n)와 독립이라는 것. "앞의 k명만 보고 판단"이라 뒤는 상관없음.

WHY — "무작위 순서"가 "무작위 알고리즘"으로 바뀌는 순간

원래 이 문제는 "입력이 무작위 순서라는 가정" 하에 평균을 본 것 — average-case 분석. 하지만 입력이 적대적이면(오름차순) 어쩔 건가요?

해법: 받은 배열을 우리가 먼저 무작위로 섞는다. 그러면 적대자가 어떤 배열을 줘도 결과는 무작위 순서 → 기댓값 O(ln n) 보장. 이것이 randomized 고용 문제.

Quicksort도 같은 변환을 합니다 (§06).

06 — 사례 1: Randomized Quicksort (CLRS §7)

초급

한눈에 보는 퀵소트

[3, 1, 4, 1, 5, 9, 2, 6]을 정렬합니다.

[3, 1, 4, 1, 5, 9, 2, 6]
  ↓ 피벗 = 4 (무작위 선택)
  ↓ partition: 4보다 작은 것 왼쪽, 큰 것 오른쪽
[3, 1, 1, 2]  |  4  |  [5, 9, 6]
  ↓ 왼쪽 재귀 (피벗 = 1)
[1, 1] | 2 | [3]  |  4  |  [5, 9, 6]
  ↓ 왼쪽도 더, 오른쪽도 재귀 …
[1, 1, 2, 3, 4, 5, 6, 9] ✅
중급

Python 구현 (간단형)

quicksort_simple.py
import random

def quicksort(A):
    if len(A) <= 1:
        return A[:]
    pivot = random.choice(A)                 # 무작위 피벗!
    less    = [x for x in A if x < pivot]
    equal   = [x for x in A if x == pivot]
    greater = [x for x in A if x > pivot]
    return quicksort(less) + equal + quicksort(greater)


print(quicksort([3, 1, 4, 1, 5, 9, 2, 6]))
출력:
[1, 1, 2, 3, 4, 5, 6, 9]

짧고 명확하지만 O(n) 추가 공간 — 리스트 컴프리헨션이 새 리스트를 만듭니다. 실제 구현은 in-place.

고급

In-place 버전 (CLRS §7.1 Lomuto partition)

quicksort_inplace.py
import random

def quicksort(A, p=0, r=None):
    if r is None:
        r = len(A) - 1
    if p < r:
        q = randomized_partition(A, p, r)
        quicksort(A, p, q - 1)
        quicksort(A, q + 1, r)


def randomized_partition(A, p, r):
    # 무작위 인덱스를 r로 이동 → 마지막 원소를 피벗으로
    i = random.randint(p, r)
    A[i], A[r] = A[r], A[i]
    return partition(A, p, r)


def partition(A, p, r):
    pivot = A[r]
    i = p - 1                     # "pivot 이하" 경계
    for j in range(p, r):
        if A[j] <= pivot:
            i += 1
            A[i], A[j] = A[j], A[i]
    A[i + 1], A[r] = A[r], A[i + 1]  # pivot을 제자리에
    return i + 1

루프 불변식(CLRS §7.1): 매 반복 직전, A[p..i]는 모두 pivot 이하, A[i+1..j-1]은 모두 pivot 초과. 증명은 Algo 02에서 본 3속성(초기/유지/종료) 방식.

⚠️ 왜 "무작위 인덱스 → 마지막으로 스왑"을 먼저 하는가

partition 함수는 "마지막 원소를 피벗으로" 가정해 작성됐습니다. "무작위 원소를 피벗으로"로 쓰려면 매번 인덱싱을 고쳐야 하니, 간단한 해법: 무작위로 고른 원소를 먼저 마지막 자리로 스왑. 이 두 줄이 partition 본체를 전혀 안 고치게 해줍니다.

전공자

더 발전된 partition들

  • Hoare partition(원조 1961): 양쪽에서 들어오는 두 포인터. 스왑 수 절반 — 보통 Lomuto보다 빠름. CLRS는 단순성 때문에 Lomuto 채택.
  • 3-way partition (Dutch National Flag, Dijkstra): 피벗과 같은 원소 여러 개일 때 O(n²) 회피. A[p..lt-1] < pivot, A[lt..gt] = pivot, A[gt+1..r] > pivot 세 구간. Python sort 구현의 fallback에도 쓰임.
  • median-of-3: 첫·중간·끝 3개 중 중앙값을 피벗으로. 이미 정렬된 입력에 강함. 무작위화보다 결정론적이고 빠르지만 적대적 입력에서는 여전히 "median-of-3 킬러" 입력이 존재(McIlroy 1999).
📖 원서 인용 — CLRS 4판, §7.3 "A randomized version of quicksort"

"By randomly choosing the pivot element, we expect the split to be reasonably well balanced on average, and thus the expected running time of randomized-quicksort is O(n lg n). ... We add randomization to quicksort to obtain an algorithm whose performance is good on every input."

07 — Quicksort 기댓값 분석: Θ(n log n) 증명 (CLRS §7.4)

고급

핵심 질문 — "무엇을 세어야 하는가"

퀵소트의 실행 시간은 비교 횟수에 비례합니다 (partition이 비교만 함, 스왑과 비교는 같은 차수). 그러니 E[비교 횟수]를 구하면 됩니다.

정렬 후 배열을 z₁ < z₂ < … < z_n이라 합시다. (같은 값은 없다고 가정.) 핵심 관찰: 퀵소트의 재귀 트리 안에서 z_i와 z_j는 비교하거나, 비교하지 않거나 둘 중 하나 — 그리고 비교한다면 정확히 한 번만 합니다.

왜 정확히 한 번 또는 0번인가

두 원소가 비교되는 유일한 경우는 둘 중 하나가 피벗일 때. 한번 피벗이 되면 그 자리에 고정되고 재귀는 양쪽으로 나뉘어 서로 다시 만날 수 없습니다. 따라서 최대 한 번 비교.

고급

지표 변수로 세기

X_{ij} = "z_i와 z_j가 비교되면 1, 아니면 0" 이라 합시다. 총 비교 횟수 X = Σi<j X_{ij}.

E[X] = Σi<j E[X_{ij}] = Σi<j P(z_i와 z_j가 비교됨)

핵심 확률 — P(z_i와 z_j가 비교될 확률)

정렬된 z_i, z_{i+1}, …, z_j (총 j − i + 1개)를 생각합니다. 이 구간에서 처음으로 피벗으로 뽑힌 원소가:

  • z_i 또는 z_j라면 → z_i와 z_j는 비교됩니다 (피벗과 나머지들이 비교되니).
  • 가운데 중 하나(z_k, i < k < j)라면 → z_i는 왼쪽, z_j는 오른쪽으로 나뉘어 영원히 안 만남.

구간 안 j − i + 1개 중 어떤 것이 먼저 피벗이 될지는 균등이므로:

P(z_i와 z_j 비교됨) = 2 / (j − i + 1)
전공자

합 계산 — H_n으로 귀결

E[X] = Σi=1n-1 Σj=i+1n 2 / (j − i + 1)
     = Σi=1n-1 Σk=2n-i+1 2/k   (k = j − i + 1 치환)
     ≤ Σi=1n-1 2·H_n
     = 2(n − 1)·H_n   ≈   2n·ln n = O(n log n)

즉 randomized quicksort의 기대 비교 횟수 ≈ 2n ln n ≈ 1.39·n log₂ n. 병합 정렬의 최악 n log₂ n보다 약 1.39배 많지만 in-place에서 오는 캐시 이점으로 실무에서는 더 빠릅니다.

E[비교 횟수] = Θ(n log n),   따라서 기대 실행 시간 = Θ(n log n)
📖 원서 인용 — CLRS 4판, §7.4.2 "Expected running time"

"Theorem 7.2: The expected running time of randomized-quicksort on an input of n distinct elements is O(n lg n). Proof: ... We compute the total expected number of comparisons by summing over all pairs."

WHY — 이 증명이 아름다운 이유

퀵소트의 실제 실행은 재귀 트리가 복잡하게 얽혀있고, 한 비교가 다른 비교에 영향을 줍니다 — 독립성이 없습니다. 그런데 지표 변수 + 기댓값 선형성으로 독립성 걱정 없이 바로 더했습니다.

그리고 "P(z_i와 z_j 비교됨) = 2/(j−i+1)"이라는 명확한 공식을 뽑아낸 것 — 복잡한 과정을 "정렬 후 거리"라는 기하학적 양으로 환원한 것. 이게 randomized 알고리즘 분석의 전형적 패턴입니다.

08 — Quicksort 최악 분석: 언제 Θ(n²)이 되는가

고급

최악 시나리오

피벗이 매번 가장 작거나 가장 큰 원소면 분할이 (0, n−1)로 치우칩니다. 그러면:

T(n) = T(n-1) + Θ(n)  ⟹  T(n) = Θ(n²)

결정론적 "첫 원소 피벗" 퀵소트는 오름차순·내림차순 입력에서 이 최악에 빠집니다 — 거의 정렬된 데이터에서 재앙.

고급

Randomized 퀵소트의 최악 확률

Randomized 버전에서 최악(Θ(n²))이 발생할 확률은? 모든 재귀 단계에서 "가장 극단의 피벗"이 뽑혀야 — 각 단계의 확률 2/n, 총 재귀 n-1번이면 2^(n-1) / n!. n=50이면 대략 10^(-50) 수준 — 사실상 안 일어납니다.

더 강력한 보장은 Chernoff 부등식(§11): Pr(실행 시간 > c·n log n) ≤ n^(-c) 정도. 즉 평균에서 벗어날 확률이 지수적으로 작음.

Worst-case randomized quicksort

최악 기대 시간(어떤 입력이든)은 Θ(n log n)이지만, 최악 실제 시간(운이 정말 나쁜 코인 던지기)은 여전히 Θ(n²). 확률이 지수적으로 작아 실용에서는 무시 가능.

"진짜 최악도 Θ(n log n)이게 하려면" → Introsort(Musser 1997): quicksort 재귀 깊이가 2 log n을 넘으면 heapsort로 전환. std::sort의 기본 구현. 평균은 quicksort처럼 빠르고 최악은 heapsort처럼 보장.

전공자

공간 복잡도와 꼬리 재귀 최적화

재귀 호출 스택 깊이는 평균 O(log n), 최악 O(n). 최악에서 RecursionError 위험. Sedgewick의 팁: 두 부분문제 중 큰 쪽을 루프로, 작은 쪽을 재귀로 — 스택 깊이가 O(log n)으로 보장됩니다.

quicksort_tail.py
def quicksort(A, p=0, r=None):
    if r is None:
        r = len(A) - 1
    while p < r:
        q = randomized_partition(A, p, r)
        if q - p < r - q:
            quicksort(A, p, q - 1)   # 작은 쪽 재귀
            p = q + 1                # 큰 쪽은 루프로
        else:
            quicksort(A, q + 1, r)
            r = q - 1

이 패턴을 쓰면 스택 깊이 O(log n) 보장 — 큰 배열에도 안전.

09 — 사례 2: Reservoir Sampling (스트림 균등 샘플링)

중급

문제 — 크기를 모르는 스트림에서 균등 샘플링

실시간 로그 스트림에서 하나의 로그를 균등하게 뽑고 싶습니다. 문제는:

  • 스트림이 언제 끝날지 모릅니다 (길이 n을 모름).
  • 모든 로그를 저장할 메모리가 없습니다 (수 TB일 수도).

단순한 방법: 다 저장했다가 끝나면 random.choice. → 메모리 O(n). 불가.

고급

Reservoir Sampling (Algorithm R; Waterman, Knuth TAOCP §3.4.2; 분석은 Vitter 1985)

reservoir.py
import random

def reservoir_sample(stream):
    """스트림에서 원소 하나를 균등 확률로 뽑음. O(1) 메모리."""
    sample = None
    for i, x in enumerate(stream, start=1):
        if random.randint(1, i) == 1:
            sample = x     # 확률 1/i로 교체
    return sample

메모리 단 하나 — sample. i번째 원소를 만날 때마다 1/i 확률로 교체.

정확성 증명 — 모든 원소가 1/n 확률

n개짜리 스트림에서 k번째 원소(xₖ)가 마지막에 남아 있을 확률을 계산합니다.

  1. k번째 시점에 xₖ가 sample로 선택될 확률: 1/k.
  2. 이후 i = k+1, k+2, …, n 각 단계에서 xₖ가 교체되지 않을 확률: (i-1)/i.
P(최종 sample = xₖ) = (1/k) · (k/(k+1)) · ((k+1)/(k+2)) · … · ((n-1)/n)
     = (1/k) · (k/n)   (망원합)
     = 1/n ✓

k가 무엇이든 정확히 1/n — 균등 분포. 스트림 길이 n을 몰라도 균등 샘플링이 되는 것이 마법.

전공자

k개 뽑기로 일반화 (Algorithm R)

reservoir_k.py
import random

def reservoir_sample_k(stream, k):
    reservoir = []
    for i, x in enumerate(stream):
        if i < k:
            reservoir.append(x)   # 처음 k개는 그대로
        else:
            j = random.randint(0, i)   # 0 ~ i 균등
            if j < k:
                reservoir[j] = x   # 확률 k/(i+1)로 교체
    return reservoir

메모리 O(k), 시간 O(n). 모든 길이-k 부분집합이 정확히 같은 확률 C(n, k)-1.

WHY — 한 번의 pass, 일정 메모리

Reservoir sampling은 "한 번만 읽는" one-pass 알고리즘의 대표. 빅데이터·스트리밍 DB(Kafka, Flink)의 기초 기술. Python 표준라이브러리의 random.sample은 sequence만 받으며 내부적으로 selection-tracking / pool-swap 방식을 씁니다 (reservoir 아님). 스트림에서 샘플링하려면 reservoir를 직접 구현해야 합니다.

📖 원서 인용 — Vitter, "Random Sampling with a Reservoir" (TOMS 1985)

"Algorithm R is the simplest reservoir algorithm: as each new record is encountered, it replaces a randomly chosen record in the reservoir with probability n/t, where t is the total number of records seen so far."

고급

이름의 유래: Las Vegas는 도박의 도시 — 시간(돈) 걸고 확실한 답을 얻음. Monte Carlo는 모나코의 카지노 — 결과(답)가 도박.

전공자

예 — Miller-Rabin 소수성 검사 (Monte Carlo)

큰 수 n이 소수인지 판정. 결정론적 AKS는 Õ(log⁶ n)이지만 상수가 큼. Miller-Rabin은 틀릴 확률이 아주 작은 Monte Carlo.

miller_rabin.py (요약)
import random

def miller_rabin(n, k=20):
    """확률 ≥ 1 - (1/4)^k 로 정확."""
    if n < 2: return False
    for p in [2, 3, 5, 7, 11, 13]:
        if n % p == 0: return n == p

    d, r = n - 1, 0
    while d % 2 == 0:
        d //= 2; r += 1

    for _ in range(k):
        a = random.randrange(2, n - 1)
        x = pow(a, d, n)
        if x == 1 or x == n - 1:
            continue
        for _ in range(r - 1):
            x = x * x % n
            if x == n - 1:
                break
        else:
            return False  # 확실히 합성수
    return True      # 확률적으로 소수

k = 20으로 시행하면 오차 확률 ≤ (1/4)²⁰ ≈ 10-12. RSA 키 생성 등 실용 암호에 쓰입니다. Python의 sympy.isprime은 내부적으로 Miller-Rabin + BPSW 조합.

Las Vegas ↔ Monte Carlo 변환

Las Vegas 알고리즘이 있으면 Monte Carlo로 바꿀 수 있습니다: "시간 T 내에 답이 나오면 반환, 아니면 임의 답 반환." 역방향(Monte Carlo → Las Vegas)은 "답을 효율적으로 검증 가능할 때만" 가능 — 반환된 답을 확인, 틀리면 재시도.

📖 역사 — Las Vegas 용어의 유래

"Las Vegas algorithm"이라는 용어는 László Babai가 1979년 논문 "Monte-Carlo algorithms in graph isomorphism testing"에서 Monte Carlo의 대비 개념으로 도입. "정답이 확실한 무작위 알고리즘"에 도박 도시의 이름을 붙인 게 인상적.

11 — Markov, Chebyshev, Chernoff 부등식 (전공자)

전공자

왜 꼬리 확률이 중요한가

"기댓값 = Θ(n log n)"은 평균값일 뿐 — 실제 실행이 평균에서 얼마나 벗어날까도 중요합니다. 이 "벗어남"을 제한하는 도구가 집중 부등식(concentration inequalities).

전공자

Markov 부등식 — 가장 기본

X가 음이 아닌 확률 변수라면:

P(X ≥ a) ≤ E[X] / a

증명: E[X] = ∫ x f(x) dx ≥ ∫[a, ∞) x f(x) dx ≥ a·P(X ≥ a). ∎

매우 약한 경계지만 조건이 느슨함 (분산 필요 없음, 음이 아니면 됨). 예: 평균 연봉 5000만 원이면 연봉 ≥ 1억 받는 사람은 최대 절반.

전공자

Chebyshev 부등식 — 분산을 쓰면 더 정확

P(|X − E[X]| ≥ a) ≤ Var(X) / a²

증명: Markov를 (X − E[X])²에 적용. 대칭적이고 a²이 있어 꼬리 감소가 Markov보다 빠름.

예: E[X] = 100, Var(X) = 25 → "|X − 100| ≥ 10"의 확률 ≤ 25/100 = 0.25. Markov는 1까지도 가능.

전공자

Chernoff 부등식 — 지수적 꼬리

X₁, …, X_n이 독립인 [0,1] 값 확률 변수, S = ΣX_i, μ = E[S]이면:

P(S ≥ (1 + δ)μ) ≤ (eδ / (1+δ)1+δ)μ   (상측)
P(S ≤ (1 − δ)μ) ≤ e-μδ²/2   (하측, 간단형)

꼬리가 지수적으로 감소 — Markov·Chebyshev보다 훨씬 강함. 대신 독립성 요구.

활용 — Randomized quicksort의 "거의 항상" 보장

퀵소트 재귀 깊이 D를 분석하면 Chernoff로 다음이 나옵니다:

P(D > c·log n) ≤ n-(c-1)²/2

c = 10이면 P(D > 10 log n) ≤ n-40 — 천문학적으로 작음. 그래서 "평균 log n 깊이"뿐만 아니라 "거의 항상 log n 수준"이 보장.

Chernoff가 통계적 학습 이론, 분산 시스템 분석, 근사 알고리즘 등 곳곳에 등장합니다.

📖 원서 인용 — Motwani & Raghavan, Randomized Algorithms §4

"The Chernoff bound provides a sharp estimate for the deviation from the expectation of a sum of independent random variables. It is the workhorse of randomized algorithm analysis, yielding tight bounds where Markov and Chebyshev give only loose ones."

14 — 실습: Fisher-Yates 무작위 순열

고급

문제 — 배열을 균등 무작위 순열로

배열 A의 원소들을 "n! 가지 순열 중 하나"가 정확히 같은 확률로 나오도록 섞습니다. 잘못 짜면 편향이 생기기 쉬운 문제 — 분석을 꼼꼼히.

고급

잘못된 풀이 — "모든 i에 대해 임의 j와 스왑"

bad_shuffle.py — 편향된 구현
import random

def bad_shuffle(A):
    n = len(A)
    for i in range(n):
        j = random.randint(0, n - 1)   # 전체 범위!
        A[i], A[j] = A[j], A[i]

전체 가능 조합 수 = nn. 하지만 가능한 순열 수 = n!. nn이 n!로 나누어떨어지지 않으므로 어떤 순열은 더 자주, 어떤 건 덜 나옵니다 — 편향.

예: n = 3 → 27가지 실행 경로 vs 6가지 순열. 27/6 = 4.5 — 정수가 아님! 어떤 순열은 5번, 어떤 건 4번 나옴.

고급

올바른 풀이 — Fisher-Yates (Durstenfeld 1964, Knuth 변형)

fisher_yates.py
import random

def shuffle(A):
    """In-place, 균등 확률 n! 순열 생성."""
    n = len(A)
    for i in range(n - 1, 0, -1):
        j = random.randint(0, i)    # 0 ~ i (i 포함)
        A[i], A[j] = A[j], A[i]

핵심 차이 — j의 범위가 [0, i]로 i번째 반복마다 축소됩니다. 가능한 경로 수 = n × (n−1) × … × 1 = n! — 정확히 순열 수와 일치.

정확성 증명 — 지표 변수 + 귀납

주장: 각 위치 k에 임의의 특정 원소 x가 올 확률 = 1/n.

귀납법으로. 첫 반복(i = n−1)에서 x가 위치 n−1에 올 확률: x가 0~n−1 중 하나이고 random.randint(0, n−1)이 그 인덱스를 뽑을 확률 = 1/n. 그 반복 후 나머지 n−1개에 대해 크기 n−1의 부분 배열에서 Fisher-Yates를 재귀적으로 수행 → 귀납 가정으로 각 위치에 각 원소가 1/(n−1) 확률. 전체적으로 각 위치-원소 쌍이 1/n 확률. ∎

⚠️ Python의 random.shuffle은 정확히 Fisher-Yates

소스: Lib/random.py의 Random.shuffle. 내부적으로 _randbelow(i+1)로 올바른 범위 생성. 직접 구현할 필요 없이 random.shuffle(A) 쓰면 됩니다 — 테스트된 구현.

WHY — 순열 생성이 randomized 알고리즘의 도우미

"입력이 적대적이면 먼저 무작위로 섞는다"가 randomized 알고리즘의 기본 트릭. 정확히 Fisher-Yates로.

  • 퀵소트: 입력을 먼저 섞으면 피벗을 "첫 원소"로 고정해도 기댓값 Θ(n log n).
  • 고용 문제: 입력을 먼저 섞으면 오름차순 같은 적대적 순서가 무작위가 됨.
  • Skip list: 레벨 결정에 무작위 사용 — 균형 BST 대안.

"섞기"가 이렇게 단순한 한 줄로 구현되고, 그게 최악을 평균으로 바꿉니다 — 이것이 이번 주의 가장 실용적 교훈.

📖 역사 — Fisher-Yates의 변천

원조 Fisher-Yates(1938)는 종이와 연필용 — O(n²)였습니다. Durstenfeld(1964)가 컴퓨터용 O(n) 변형을 발표. Knuth가 TAOCP Vol. 2 §3.4.2에 "Algorithm P"로 수록 — 그래서 "Knuth shuffle"이라고도 불립니다.

def randomized_select(A, p, r, k):
    if p == r:
        return A[p]
    q = randomized_partition(A, p, r)   # 무작위 피벗
    rank = q - p + 1                      # 피벗의 랭크

AI 문제 12

LLM에게 "Monte Carlo와 Las Vegas의 차이"를 물었더니 "Las Vegas는 빠르고 Monte Carlo는 정확하다"고 답했다. 이 답이 왜 틀렸는지 한 문장으로.

풀이 펼치기
답

정반대입니다. Las Vegas는 답이 항상 정확하고 시간이 확률 변수, Monte Carlo는 시간이 결정적이고 답이 확률적으로 정확. "빠름"은 둘 다 할 수 있고 차이의 본질이 아님 — 본질은 "무엇이 무작위인가(답 or 시간)".

핵심 정리

🎲

두 가지 평균

Average-case(입력에 대한 평균)와 Expected time(코인에 대한 평균). 후자는 적대적 입력 무력화.

🪙

지표 확률 변수

복잡한 "세는 문제"를 I_A들로 쪼갠 뒤 기댓값 선형성으로 합. 독립성 불필요.

⚡

Randomized Quicksort

피벗 무작위 → 기댓값 2n ln n, 모든 입력에서. 최악 n²의 확률은 지수적으로 작음.

🪣

Reservoir Sampling

스트림 크기를 몰라도 1/n 확률로 균등 샘플링. O(1) 메모리, one-pass.

🎰

Las Vegas vs Monte Carlo

LV: 답은 확실, 시간은 랜덤. MC: 시간은 확실, 답은 랜덤. 상호 변환 가능 (대개).

📏

집중 부등식

Markov(음이 아닌) → Chebyshev(분산) → Chernoff(지수적 꼬리, 독립 합).

용어 사전 (25개)

Randomized Algorithm
내부에서 무작위 선택을 이용하는 알고리즘. 평균 분석과 구분됨.
Expected Running Time
알고리즘 내부의 무작위 수에 대한 실행시간의 기댓값. 입력은 고정.
Average-case Analysis
입력이 특정 분포를 따른다고 가정한 평균 분석. 결정론적 알고리즘용.
Sample Space (Ω)
확률 실험의 모든 가능한 결과의 집합.
Random Variable
표본공간에서 실수로의 함수. 이산형/연속형.
Expectation (E[X])
확률 변수의 평균값. Σ x·P(X=x) 또는 ∫ x f(x) dx.
Linearity of Expectation
E[X+Y] = E[X]+E[Y]. 독립성 불필요. 분석의 핵심 도구.
Indicator Random Variable
사건 A에 대해 I_A = 1 if A, else 0. E[I_A] = P(A).
Independence
P(A ∩ B) = P(A)·P(B). 사건이 서로 정보를 주지 않음.
Harmonic Number (H_n)
1 + 1/2 + … + 1/n ≈ ln n + γ. 고용 문제·퀵소트에 등장.
Hiring Problem
CLRS §5.1의 대표 예제. 무작위 순서에서 채용 횟수 기댓값 Θ(log n).
Randomized Quicksort
피벗을 무작위 선택하는 quicksort. 기댓값 Θ(n log n), 입력 독립.
Partition
배열을 피벗 이하와 초과로 나누는 O(n) 연산. Lomuto/Hoare 두 변형.
Introsort
Quicksort + Heapsort 조합. 재귀 깊이가 커지면 heapsort로 전환, 최악 Θ(n log n).
Reservoir Sampling
스트림 크기 미지 상태에서 균등 샘플링. O(1)/O(k) 메모리.
Fisher-Yates Shuffle
O(n)에 균등 순열 생성. Python random.shuffle의 구현.
Las Vegas Algorithm
답은 항상 옳고 시간이 확률 변수인 randomized 알고리즘.
Monte Carlo Algorithm
시간은 결정적이고 답이 확률적으로 옳은 randomized 알고리즘.
Miller-Rabin Primality Test
Monte Carlo 소수성 검사. k회 반복으로 오차 ≤ (1/4)^k.
Markov's Inequality
음이 아닌 X에 대해 P(X ≥ a) ≤ E[X]/a. 가장 약한 꼬리 경계.
Chebyshev's Inequality
P(|X − μ| ≥ a) ≤ Var(X)/a². 분산 필요, Markov보다 강함.
Chernoff Bound
독립 합의 꼬리 확률이 지수적으로 감소. 집중도 분석의 표준 도구.
Mersenne Twister (MT19937)
주기 219937−1의 의사난수 생성기. Python random의 기본 RNG.
Cryptographically Secure RNG
공격자가 이전 출력으로 미래 출력을 예측할 수 없는 난수원. Python secrets.
Seed
의사난수 생성기의 초깃값. 같은 seed → 같은 수열(재현성).

O(n²) 정렬들 — 버블, 선택, 삽입

"느린 정렬"이라 불리는 O(n²) 알고리즘들이 왜 아직도 쓰이는가? 답은 세 가지 — (1) n이 작을 때 상수 인자가 가장 작고, (2) 이미 정렬된 입력에 강하며, (3) 구현이 단순해 버그가 적기 때문.

  • 삽입 정렬의 역설 — 왜 Timsort의 fallback일까?
  • 선택 정렬 — 스왑 최소 알고리즘. flash memory 친화.
  • 버블 정렬 — 교과서의 단골. 진짜 쓸모는?
  • 비교 정렬의 하한 정리 — 왜 모든 비교 정렬은 Ω(n log n)인가 (결정 트리 증명).
  • Insertion Sort가 가장 빠른 영역 — CPython은 n < 32에서 insertion sort.
