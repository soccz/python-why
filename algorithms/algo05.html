ALGO 05

O(n²) 정렬들
느린 정렬이 여전히 살아남는 이유

병합 정렬·퀵소트가 O(n log n)으로 이기는데, 왜 교과서와 실제 라이브러리(Timsort, C++ introsort)는 여전히 삽입 정렬을 쓸까요?
답은 세 가지 — 작은 n에선 상수가 작다, 거의 정렬된 입력엔 O(n)이다, 코드가 단순해 캐시에 좋다.
그리고 이번 주 마지막 질문: "비교 기반 정렬은 왜 Ω(n log n)보다 빠를 수 없는가?"

01 — 이번 주 이야기: 책상 위 4권 vs 도서관 전체

상황 1 — 책상 위 4권 정리

책상 위에 책 4권이 놓여 있습니다. 제목 순으로 정렬하는 데 어떻게 할까요? 한 권씩 집어서 알맞은 자리에 끼워넣으면 끝. 대략 3~6번 비교하고 2~3번 옮기면 됩니다. 퀵소트 구현은커녕 함수 호출 오버헤드가 더 큽니다.

상황 2 — 도서관 전체(100만 권) 정리

이제 100만 권. 한 권씩 끼워 넣으면 평균적으로 이미 정렬된 50만 권 속을 헤집어야 합니다. 100만 권 전체를 정렬하는 데 비교가 5×10¹¹번 — 현대 CPU로 몇 분이 걸립니다.

병합 정렬이면 같은 100만 권을 2×10⁷번 비교에, 약 25,000배 빠르게 끝냅니다 (Algo 03 참조).

WHY — 같은 알고리즘, 반대의 결론

책상 위 4권: 삽입 정렬이 이깁니다. 도서관 100만 권: 병합 정렬이 이깁니다.

왜? "n²과 n log n"은 n이 작을 땐 n log n이 더 클 수도 있다는 초등 수학입니다. n=4에서 n² = 16, n log₂ n ≈ 8. 비슷합니다. 하지만 실제 상수(constant factor)를 보면 — 병합 정렬은 함수 호출, 배열 복사, 포인터 관리 등으로 상수가 20~50배 더 큼. 그래서 n < 약 32~64에서는 삽입 정렬이 실제로 더 빠릅니다.

이게 Python의 list.sort(Timsort), Java의 Arrays.sort, C++의 std::sort 모두가 작은 구간에서는 삽입 정렬(또는 그 변종)을 쓰는 이유입니다. "O(n²)이니까 무조건 버린다"가 아니라, "언제 O(n²)이 이기는지"를 아는 게 중요합니다.

삽입 정렬은 카드 정렬법. 매 pass마다 다음 원소(key)를 왼쪽 정렬된 부분의 알맞은 자리에 끼워넣음. 정렬된 부분이 한 칸씩 자라남.

02 — 왜 O(n²) 정렬을 배우는가

초급

3가지 이유

  1. 작은 n에서는 빠르다 — 함수 호출·메모리 할당 같은 오버헤드가 없는 단순한 루프라 CPU가 좋아합니다.
  2. 거의 정렬된 입력을 만나면 O(n) — 삽입 정렬은 이미 정렬된 배열을 한 번만 훑고 끝냅니다. 실제 데이터는 종종 "거의 정렬된" 상태입니다 (로그 타임스탬프, 센서 데이터).
  3. "느린 정렬"은 모든 "빠른 정렬"의 부품 — Timsort, introsort 등은 작은 배열에서 삽입 정렬을 호출합니다. 삽입 정렬을 모르면 Timsort를 이해할 수 없습니다.
중급

교육적 가치 — 알고리즘 사고의 문법

O(n²) 정렬 3개는 알고리즘 설계의 세 가지 접근을 보여줍니다.

알고리즘접근불변식(대략)
삽입 정렬 "왼쪽이 정렬된 상태 유지" A[0..i-1]은 이미 정렬됨, 원래 원소들로만 구성
선택 정렬 "다음에 올 최솟값을 찾아서 자리에" A[0..i-1]은 전체의 가장 작은 i개가 정렬된 상태
버블 정렬 "큰 값을 오른쪽으로 밀어내기" 매 패스 후, 오른쪽 끝부터 k개는 최종 자리에 있음

같은 O(n²)이지만 "무엇을 매 반복마다 보장하는가"가 다릅니다 — 이게 알고리즘을 이해하는 바탕입니다.

03 — 삽입 정렬 (Insertion Sort, CLRS §2.1)

초급

카드 놀이의 정렬법

손에 카드를 한 장씩 받아서, 이미 손에 있는 정렬된 카드들 사이의 알맞은 자리에 끼워넣는 방식. 현실에서 카드 정렬할 때 사람이 자연스럽게 쓰는 방법입니다.

시작: [5, 2, 4, 7, 1, 3]
pass 1: [2, 5], 4, 7, 1, 3   // 2를 5 앞에
pass 2: [2, 4, 5], 7, 1, 3
pass 3: [2, 4, 5, 7], 1, 3
pass 4: [1, 2, 4, 5, 7], 3   // 1을 맨 앞까지 밀어넣음
pass 5: [1, 2, 3, 4, 5, 7] ✅
중급

Python 구현

insertion_sort.py
def insertion_sort(A):
    for j in range(1, len(A)):
        key = A[j]           # 지금 끼워넣을 원소
        i = j - 1
        # A[0..i]에서 key보다 큰 원소들을 오른쪽으로 한 칸씩 밀어냄
        while i >= 0 and A[i] > key:
            A[i + 1] = A[i]
            i -= 1
        A[i + 1] = key   # 빈 자리에 key 놓기
    return A


print(insertion_sort([5, 2, 4, 7, 1, 3]))
출력:
[1, 2, 3, 4, 5, 7]

정확성 증명과 루프 불변식은 Algo 02 §07에서 이미 다뤘습니다. 여기서는 복잡도에 집중.

고급

복잡도 — best / average / worst

외부 루프는 n−1번. 내부 while 루프의 반복 횟수는 입력에 따라 다름.

경우입력 예while 반복총 시간
Best 이미 정렬됨 [1,2,3,4,5] 0번 (첫 비교에 종료) Θ(n)
Average 무작위 j/2번 (평균) Θ(n²)
Worst 역순 [5,4,3,2,1] j번 (끝까지) Θ(n²)
T_worst(n) = Σj=1n-1 j = n(n-1)/2 = Θ(n²)

정확히는 T(n) = Θ(n + I) — 여기서 I는 역순쌍의 수(inversion count). 이미 정렬 → I=0 → Θ(n). 역순 → I = n(n-1)/2 → Θ(n²).

📖 원서 인용 — CLRS 4판, §2.1 "Insertion sort"

"Insertion sort, which is an efficient algorithm for sorting a small number of elements ... Humans often use it, unknowingly, when sorting a hand of playing cards."

— Cormen, Leiserson, Rivest, Stein. Introduction to Algorithms, 4th ed., §2.1

WHY — "적응적(adaptive)"의 진짜 의미

삽입 정렬은 적응적 정렬(adaptive sort)의 대표 예입니다. "적응적"이란 — 입력이 이미 어느 정도 정렬되어 있으면 더 빨라지는 성질.

  • 이미 정렬됨 → O(n). 한 번 훑고 끝.
  • k개만 자리 바뀜 (inversion k개) → O(n + k).
  • 완전 역순 → O(n²). 최악.

병합 정렬·힙 정렬은 이 적응성이 없습니다 (이미 정렬된 입력도 Θ(n log n)). 퀵소트는 부분적으로 적응적 (좋은 피벗 선택 시). Timsort가 강력한 이유도 바로 이 적응성을 최대한 활용하기 때문.

04 — 선택 정렬 (Selection Sort)

중급

"다음에 올 최솟값 찾기"

외부 루프 i를 0부터 n−2까지. 매 단계에서 A[i..n-1] 중 최솟값을 찾아 A[i]와 스왑. 결과: A[0..i]가 전체의 가장 작은 (i+1)개가 정렬된 상태.

selection_sort.py
def selection_sort(A):
    n = len(A)
    for i in range(n - 1):
        min_idx = i
        for j in range(i + 1, n):
            if A[j] < A[min_idx]:
                min_idx = j
        A[i], A[min_idx] = A[min_idx], A[i]
    return A


print(selection_sort([5, 2, 4, 7, 1, 3]))
출력:
[1, 2, 3, 4, 5, 7]
고급

복잡도 — 항상 Θ(n²) 비교, 하지만 Θ(n) 스왑

비교 횟수는 입력과 무관하게 항상 같습니다:

비교 횟수 = Σi=0n-2 (n - 1 - i) = n(n-1)/2 = Θ(n²)

하지만 스왑은 매 외부 루프마다 정확히 1번 — 총 n−1번 = Θ(n).

경우비교스왑
Best / Avg / WorstΘ(n²)Θ(n)
고급

루프 불변식

I(i): 매 외부 루프 반복 직전,
(a) A[0..i-1]은 전체 A의 가장 작은 i개 원소가 오름차순 정렬된 상태,  
(b) A[i..n-1]의 원소들은 모두 A[i-1] 이상.
  • 초기 (i=0): A[0..-1]은 빈 배열, 자명히 성립.
  • 유지: 내부 루프가 A[i..n-1]의 최솟값 인덱스를 찾고 A[i]와 스왑. 가정 (b)에 의해 최솟값은 A[i-1] 이상. 따라서 A[0..i]는 여전히 정렬됨.
  • 종료 (i = n-1): A[0..n-2]는 가장 작은 n-1개가 정렬됨. A[n-1]은 최댓값. 따라서 A 전체가 정렬됨. ∎
WHY — 스왑이 비쌀 때 빛나는 알고리즘

선택 정렬의 독특한 장점: 스왑이 Θ(n)뿐. 삽입 정렬은 원소를 한 칸씩 밀어내므로 최악 Θ(n²) 쓰기.

이게 언제 중요할까요? 쓰기가 읽기보다 비쌀 때.

  • Flash 메모리·SSD: 셀 수명이 유한. 불필요한 쓰기는 수명을 깎음.
  • EEPROM·구형 하드웨어: 쓰기 속도가 읽기의 100배 이상 느림.
  • 원소가 거대한 경우: 예컨대 100MB짜리 이미지 배열 정렬. 비교는 key만 보면 되지만, 스왑은 실제 데이터를 복사.

현대 DRAM에서는 쓰기와 읽기 비용이 비슷해 선택 정렬의 이 장점이 묻힙니다. 그래서 범용 라이브러리에서는 거의 안 씁니다.

⚠️ 선택 정렬은 기본적으로 unstable (안정 정렬 아님)

이 구현에서 A = [5ₐ, 3, 5_b, 2, 4]를 정렬하면 5ₐ와 5_b의 순서가 뒤바뀔 수 있습니다. 예컨대 i=0일 때 min_idx=3 (값 2), swap(A[0], A[3]) 하면 A = [2, 3, 5_b, 5ₐ, 4]. 원래 5ₐ가 5_b보다 앞이었는데, 뒤로 갔습니다.

안정 선택 정렬을 만들려면 스왑 대신 "최솟값을 쉬프트로 앞으로 당기기"를 해야 — 하지만 그러면 선택 정렬의 Θ(n) 스왑 장점이 사라집니다.

05 — 버블 정렬 (Bubble Sort, CLRS Problem 2-2)

중급

"거품이 물 위로 떠오르듯"

인접한 두 원소를 비교해 순서가 잘못되어 있으면 스왑. 한 번 배열을 쭉 훑으면 가장 큰 원소가 맨 뒤로 "떠오릅니다".

bubble_sort.py
def bubble_sort(A):
    n = len(A)
    for i in range(n - 1):
        # i번째 패스 후, 오른쪽 끝 (i+1)개는 제자리
        for j in range(n - 1 - i):
            if A[j] > A[j + 1]:
                A[j], A[j + 1] = A[j + 1], A[j]
    return A
고급

최적화: early exit

한 패스 동안 스왑이 한 번도 없었다면 배열은 이미 정렬된 것. 이를 감지해 루프를 종료:

bubble_sort_optimized.py
def bubble_sort_opt(A):
    n = len(A)
    for i in range(n - 1):
        swapped = False
        for j in range(n - 1 - i):
            if A[j] > A[j + 1]:
                A[j], A[j + 1] = A[j + 1], A[j]
                swapped = True
        if not swapped:
            break   # 이미 정렬 → 종료
    return A

이걸로 best case가 Θ(n)이 됩니다 (이미 정렬된 입력은 1패스만).

고급

복잡도

경우비교스왑총 시간
Best (최적화, 이미 정렬)Θ(n)0Θ(n)
AverageΘ(n²)Θ(n²)Θ(n²)
Worst (역순)Θ(n²)Θ(n²)Θ(n²)

역순 입력에서 스왑도 Θ(n²) — 삽입 정렬의 쉬프트와 같은 양. 하지만 삽입 정렬은 대개 빠릅니다 (더 적은 쓰기·더 좋은 캐시 패턴).

⚠️ 버블 정렬은 거의 모든 면에서 삽입 정렬에게 진다

Donald Knuth TAOCP Vol. 3 (§5.2.2): "In short, the bubble sort seems to have nothing to recommend it, except a catchy name and the fact that it leads to some interesting theoretical problems."

버블 정렬을 "교육용으로 훌륭하다"고 옹호하는 의견도 있지만 — 삽입 정렬도 똑같이 단순하면서 실제로 더 빠릅니다. 그래서 이번 주 핵심 권고: "정렬을 직접 구현할 일이 있으면 삽입 정렬을 써라. 버블 정렬은 공부용으로만."

WHY — 삽입 정렬이 승리하는 이유

세 알고리즘 모두 점근적으로 같은 Θ(n²)지만, 삽입 정렬이 실전에서 가장 빠릅니다. 이유:

  • 적응성: 입력이 거의 정렬되어 있으면 O(n). 선택·버블엔 이 장점 없음 (선택), 약함 (버블).
  • 내부 루프가 단순: A[i+1] = A[i]; i-- — CPU 분기 예측이 잘 맞음.
  • 캐시 지역성: 인접 원소만 접근.
  • 안정성 기본 제공: 같은 키의 원소 순서 유지.

그래서 Python(Timsort), Java(Dual-Pivot Quicksort + insertion), V8(Timsort), Rust(Timsort 기반) — 모든 주요 표준 라이브러리가 작은 구간에서 삽입 정렬을 쓰는 이유.

📖 원서 요약 — Knuth TAOCP Vol. 3, §5.2.1

Knuth는 삽입 정렬이 큰 n에서는 비효율적이지만 거의 정렬된 데이터와 작은 n에서는 탁월하다고 분석합니다. 이 두 특성 때문에 복합 정렬 알고리즘(Timsort, introsort, pdqsort 등)에서 작은 구간의 마무리 루틴으로 널리 채택됩니다.

— 요약 참조: Donald E. Knuth. The Art of Computer Programming, Vol. 3, §5.2.1 "Sorting by Insertion"

07 — 비교 기반 정렬의 하한 (CLRS §8.1)

고급

왜 비교 기반 정렬은 O(n log n)보다 빠를 수 없는가

삽입·선택·버블(이번 주) + 병합·힙·퀵(Algo 06) — 모두 "원소 두 개를 비교해 크기를 결정한다"는 공통점이 있습니다. 이런 알고리즘을 비교 기반 정렬(comparison sort)이라 부릅니다.

놀라운 정리: 모든 비교 기반 정렬은 최악의 경우 Ω(n log n)번의 비교가 필요합니다. 즉 어떤 천재적인 알고리즘이 나타나도, 입력 크기 n에 대해 비교를 n log n번보다 적게 할 수는 없습니다.

증명의 도구: 결정 트리(decision tree).

고급

결정 트리란 무엇인가

결정 트리는 알고리즘의 "비교 흐름"을 트리로 그린 것.

  • 내부 노드: "A[i] vs A[j]" 비교 한 번.
  • 왼쪽 자식: "A[i] ≤ A[j]" 분기. 오른쪽 자식: "A[i] > A[j]" 분기.
  • 리프 노드: 알고리즘 종료. 어떤 순열(permutation)을 정답으로 출력함.

입력이 주어지면 루트에서 리프까지 한 경로를 따라 내려갑니다. 경로 길이 = 비교 횟수.

예: n=3 삽입 정렬의 결정 트리 스케치

           A[1] vs A[2]
          /         \
    A[1]≤A[2]      A[1]>A[2]
    A[2] vs A[3]    A[2] vs A[3]
    /  \           /  \
  [1,2,3] ...     ... ...

// 6개(=3!) 리프가 있어야 모든 순열에 대응
전공자

정리와 증명 (CLRS §8.1 Theorem 8.1)

정리 (CLRS Thm 8.1): 비교 기반 정렬 알고리즘 A가 n개 원소를 정렬하는 데 최악의 경우 h번의 비교를 한다면, h ≥ log₂(n!) = Ω(n log n).

증명 (3단계)

  1. 리프 개수의 하한: A가 옳다면, 임의의 순열로 주어진 입력을 정렬할 수 있어야 합니다. 서로 다른 순열 n!개는 결정 트리에서 서로 다른 리프에 도달해야 합니다 (같은 리프에 가면 알고리즘이 같은 출력을 내는데, 두 입력 순열의 정답이 다르므로 모순). 따라서 리프 수 ≥ n!.
  2. 트리 높이와 리프 수의 관계: 이진 트리에서 높이 h인 트리는 최대 2h개의 리프를 가집니다. 2h ≥ n!.
  3. Stirling 근사: log₂(n!) = n log₂ n − n log₂ e + O(log n) = Θ(n log n). 양변에 log₂를 취하면 h ≥ log₂(n!) = Θ(n log n). ∎
📖 원서 인용 — CLRS 4판, §8.1 "Lower bounds for sorting"

"Theorem 8.1: Any comparison sort algorithm requires Ω(n lg n) comparisons in the worst case."

전공자

하한을 "깨는" 방법 — 비비교 정렬 (Algo 07에서)

Ω(n log n) 하한은 비교에만 의존하는 알고리즘에 대한 것. 원소의 다른 성질을 쓸 수 있다면 이 하한을 넘을 수 있습니다.

  • Counting Sort (CLRS §8.2): 원소가 작은 정수면 O(n + k), k는 값 범위.
  • Radix Sort (CLRS §8.3): d자리 수를 d번 안정 정렬. O(d(n + k)).
  • Bucket Sort (CLRS §8.4): 입력이 균등 분포면 기대 O(n).

이들은 "원소를 비교"하는 대신 "원소 자체를 배열 인덱스로" 씁니다 — 질적으로 다른 패러다임. Algo 07(Timsort) 이후에 별도로 다룹니다.

WHY — 하한 증명의 의의

"더 나은 알고리즘이 나올 수 없다"는 것을 수학적으로 보이는 것은 알고리즘 이론의 핵심입니다. 우리가 최선을 다했는지 (또는 개선의 여지가 있는지)를 객관적으로 판단하게 해줍니다.

이 증명은 또한 "왜 병합 정렬·힙 정렬이 점근적으로 최적인가"를 보여줍니다. Θ(n log n)이 바로 이 하한에 닿습니다. 더 빠른 비교 정렬은 존재하지 않습니다.

08 — 적응형 정렬과 역순쌍

고급

역순쌍(inversion)의 정밀한 정의

배열 A에서 i < j이면서 A[i] > A[j]인 쌍 (i, j)의 수를 inv(A)로 표기. 이미 Algo 03 §13에서 다뤘습니다.

  • 정렬된 배열: inv = 0
  • 역순 배열: inv = n(n-1)/2 = Θ(n²) (최댓값)
  • 무작위 배열: 평균 inv = n(n-1)/4 (절반)
전공자

삽입 정렬의 정확한 복잡도 — Θ(n + inv(A))

삽입 정렬의 내부 while 루프는 A[i] > key일 때 계속 밀어냅니다. 매 밀기 한 번 = 역순쌍 하나를 해소.

증명 스케치: key를 A[j]라 하고 자리 A[p..j-1]로 옮긴다고 합시다. 이 동안 밀어낸 횟수(= 쉬프트 수)는 j−p. 이는 정확히 "key 앞쪽에 key보다 큰 원소의 수" = 이 key와 연결된 역순쌍의 수. 한 번의 쉬프트에는 한 번의 비교가 수반되고(A[i] > key가 참), while 루프가 A[i] ≤ key로 종료될 때만 추가 비교가 1번 더 일어납니다. 외부 루프 전체를 합하면:

총 쉬프트 수 = inv(A),   총 비교 수 ≤ n − 1 + inv(A)

따라서 T(n) = Θ(n + inv(A)). 즉 삽입 정렬의 성능은 "입력이 얼마나 정렬에 가까운가"에 정확히 비례. (정확한 비교 수는 inv(A) + (앞으로 끝까지 밀리지 않은 key의 수)이며, 뒤 항은 0 ~ n−1 사이. 점근적으로는 Θ(n + inv(A))로 같음.)

WHY — "거의 정렬된 입력"이 실무에서 흔한 이유
  • 로그 파일: 시간순으로 거의 정렬됨. 가끔씩 지연된 entry 몇 개만 순서가 어긋남.
  • 버전 관리: 브랜치 병합 시 두 정렬된 커밋 리스트를 합침 — inv 작음.
  • 센서 데이터: 타임스탬프가 거의 단조 증가, 가끔 네트워크 재전송으로 순서 바뀜.
  • 점진적 업데이트: 이미 정렬된 큰 배열에 새 원소 몇 개를 추가 후 재정렬.

이런 데이터에선 삽입 정렬(또는 Timsort)이 O(n)으로 끝납니다. 퀵소트는 Θ(n log n). 현실에선 삽입 정렬이 빠른 경우가 꽤 많습니다.

09 — CPython의 binarysort: Timsort 내부의 작은 정렬

전공자

Part A — CPython 소스 뜯기

Python의 list.sort()는 Timsort입니다 (cpython/Objects/listobject.c). Timsort는 입력을 run(이미 정렬된 구간)으로 쪼개 병합하는데, run이 너무 짧으면 binarysort 함수로 확장합니다.

cpython/Objects/listobject.c — binarysort (발췌·의사코드)
// binarysort: A[lo..start-1]은 이미 정렬됨.
// A[start..hi-1]을 하나씩 끼워넣어 A[lo..hi-1] 전체를 정렬.
static int binarysort(sortslice lo, Py_ssize_t hi, Py_ssize_t start) {
    for (; start < hi; ++start) {
        pivot = lo[start];
        // lo[lo..start-1]에서 pivot이 들어갈 자리를 이진 탐색 → O(log k)
        l = 0; r = start;
        while (l < r) {
            p = (l + r) >> 1;
            if (lo[p] < pivot) l = p + 1;
            else r = p;
        }
        // lo[l..start-1]을 오른쪽으로 한 칸씩 쉬프트 → O(k)
        memmove(&lo[l+1], &lo[l], (start-l) * sizeof(PyObject*));
        lo[l] = pivot;
    }
}

이게 바로 binary insertion sort — 삽입 정렬 + 이진 탐색.

  • 삽입 위치 찾기: O(log k) (이진 탐색)
  • 쉬프트: O(k) (memmove — C 레벨 고속)
  • 전체 한 원소 삽입: O(k)
  • n개 원소 정렬: O(k log k) 비교, O(k²) 쉬프트

"비교는 O(n log n)이지만 쉬프트는 O(n²)이니 결국 Θ(n²)이잖아?" — 맞습니다. 그래서 작은 n에 대해서만 호출합니다. 작은 n에선 Θ(n²)이 Θ(n log n)보다 상수 배로 빠를 수 있기 때문.

왜 "그냥 삽입 정렬" 대신 "binary insertion"?

  • Python 객체 비교는 매우 비쌈 (__lt__ 호출, 타입 체크, 예외 처리).
  • 비교를 log n번으로 줄이면 파이썬 레벨 비용이 크게 감소.
  • 쉬프트는 C의 memmove로 매우 빠름 — 캐시 친화적 연속 메모리 복사.
  • 즉 "비교는 비싸고 메모리 이동은 싼 환경"에서 유리한 변종.
전공자

Part B — MIN_RUN과 언제 binarysort가 호출되는가

Timsort는 natural run(이미 정렬된 구간)을 찾아 stack에 쌓습니다. run 길이가 MIN_RUN(32 또는 64)보다 짧으면 binarysort로 MIN_RUN까지 확장.

MIN_RUN 결정법: n을 이진 표현했을 때 상위 6비트. n ≤ 64면 n, 아니면 그보다 조금 더 작은 수. 예: n=186 → MIN_RUN=47.

merge_compute_minrun(n) = n의 상위 6비트 + (하위 비트 중 1이 하나라도 있으면 1)

핵심은 전체 run 수가 2의 거듭제곱 또는 그에 가까운 수가 되도록 맞추는 것. 이래야 Timsort의 병합 트리가 균형 잡힘.

📖 원서 인용 — Tim Peters, listsort.txt (CPython 문서)

"If N < 64, minrun is N. If N is an exact power of 2, minrun is 32. Otherwise, minrun is chosen from the range [32, 64] such that N/minrun is an exact power of 2, or if that isn't possible, is close to, but strictly less than, an exact power of 2."

10 — 실전에서의 작은 n: 언제 O(n²)이 이기는가

중급

교차점(crossover) — 대략 n = 20~60

언어와 CPU에 따라 다르지만 대체로 n ≈ 20~60에서 삽입 정렬과 O(n log n) 정렬의 실행 시간이 같아집니다. 그 미만에서는 삽입 정렬이 빠르고, 그 이상은 O(n log n) 정렬이 빠릅니다.

라이브러리임계값 (approx)작은 구간 알고리즘
CPython list.sort (Timsort)32~64 (MIN_RUN)binarysort (binary insertion)
Java Arrays.sort(int[])44~47 (MAX_INSERTION_SORT_SIZE, JDK 버전별)insertion sort
C++ std::sort (libstdc++, introsort)16 (_S_threshold)insertion sort

각 언어마다 약간씩 다른 이유: 비교 비용(Python 객체 vs C int), 캐시 크기, 함수 호출 오버헤드 등이 모두 영향.

고급

n < 임계값일 때 왜 삽입 정렬이 이기는가 — 상수 분석

두 알고리즘의 실제 시간을 추정:

T_insertion(n) ≈ c₁ · n²
T_mergesort(n) ≈ c₂ · n log₂ n + c₃ · n   // c₃: 재귀·슬라이싱 등

실측 상수 대략: c₁ ≈ 2 ns/비교 (Python), c₂ ≈ 100 ns (함수 호출·슬라이싱 포함), c₃ ≈ 200 ns.
교차점: c₁·n² = c₂·n log n + c₃·n. n=32 풀면: 2·32² = 2048, 100·32·5 + 200·32 = 22400. 삽입 정렬이 10배 빠름.
n=64: 2·4096 = 8192, 100·64·6 + 200·64 = 51200. 여전히 삽입 정렬.
n=128: 2·16384 = 32768, 100·128·7 + 200·128 = 115200. 여전히 삽입. 교차점은 ~n=300쯤.

핵심: 교차점은 "이론적 n²=n log n"이 아니라 "실제 측정된 상수를 반영한 수식". 이래서 Timsort의 MIN_RUN이 64 정도까지 높을 수 있는 것.

⚠️ "작은 n에선 O(n²)이 빠르다"의 오해 주의

이건 같은 언어·같은 배열·같은 CPU에서 구현체를 정성스럽게 튜닝한 경우에 한합니다.

"Python에서 순수 파이썬 삽입 정렬 vs 내장 list.sort()"를 비교하면 — n=5에서도 list.sort()가 압도적으로 빠릅니다. 왜? list.sort는 C로 구현됐고, 삽입 정렬을 순수 파이썬으로 쓰면 바이트코드 오버헤드가 어마어마하기 때문. "O(n²)이 작은 n에선 좋다"는 주장은 같은 레벨의 구현을 비교할 때만 성립.

11 — AI는 정렬을 어떻게 다루는가

AI — "정렬해줘"는 세상에서 가장 쉬운 요청이다

LLM에 "이 리스트를 정렬하는 파이썬 함수"라고 하면 거의 항상 이 둘 중 하나가 나옵니다:

  1. sorted(A) 또는 A.sort() — 가장 현명한 답.
  2. 버블 정렬 또는 삽입 정렬 — "교과서적 코드"를 요청받은 것처럼 해석.

후자가 나오면 코드 자체는 대부분 정확합니다 (훈련 데이터에 수천 번 등장). 하지만 "왜 버블을 쓰지 말라고 해야 하는지", "언제 삽입이 유리한지"는 명시적으로 묻지 않으면 설명하지 않습니다.

사람의 사고
# 1) 입력 크기 n 추정 (10? 1000? 100만?)
# 2) 정렬 상태 짐작 (거의 정렬? 무작위? 역순?)
# 3) 안정성 필요 여부
# 4) n이 작고 + 거의 정렬 → insertion
# 5) 그 외 → sorted() (Timsort = 둘 다 처리)
# 6) 필요 시 key 함수·stable 고려
LLM의 흔한 흐름
# 1) "정렬" → 바로 sorted() 또는 버블 코드
# 2) n 고려 안 함
# 3) 입력 특성 안 물어봄
# 4) "교과서 버블"로 O(n²) 코드 자신있게 제공
# 5) "최적화 버전 보여줘" 요청 전엔 early-exit 안 넣음
# 6) 안정성 묻지 않으면 무시
WHY — 정렬 코드를 AI에게 맡길 때 5가지 확인
  1. "n은 얼마인가? 거의 정렬된 입력인가?"
  2. "같은 키의 원소 순서를 지켜야 하는가(stable)?"
  3. "원소 비교 비용이 큰가(긴 문자열, 객체)? → binary insertion 후보."
  4. "제자리(in-place)여야 하는가? 메모리 여유 있는가?"
  5. "왜 버블이 아닌 삽입인지 이유를 명시하라."

기본 답 sorted(A)가 90% 경우 최선인 것은 맞지만, "왜"를 이해하지 못한 채 쓰면 나머지 10%에서 크게 엇나갑니다.

12 — 실습: timeit으로 세 정렬 비교

고급

벤치마크 코드

sort_bench.py
import random, timeit, copy


def insertion_sort(A):
    for j in range(1, len(A)):
        key = A[j]; i = j - 1
        while i >= 0 and A[i] > key:
            A[i + 1] = A[i]; i -= 1
        A[i + 1] = key


def selection_sort(A):
    n = len(A)
    for i in range(n - 1):
        m = i
        for j in range(i + 1, n):
            if A[j] < A[m]: m = j
        A[i], A[m] = A[m], A[i]


def bubble_sort_opt(A):
    n = len(A)
    for i in range(n - 1):
        swapped = False
        for j in range(n - 1 - i):
            if A[j] > A[j + 1]:
                A[j], A[j + 1] = A[j + 1], A[j]
                swapped = True
        if not swapped: break


def bench(fn, data):
    start = timeit.default_timer()
    fn(list(data))
    return timeit.default_timer() - start


random.seed(42)
for n in [100, 1000, 5000]:
    random_data = [random.randint(0, 10000) for _ in range(n)]
    sorted_data = sorted(random_data)
    print(f"\nn={n}")
    for name, fn in [("insertion", insertion_sort),
                      ("selection", selection_sort),
                      ("bubble-opt", bubble_sort_opt),
                      ("sorted()", sorted)]:
        t_rand = bench(fn, random_data)
        t_sort = bench(fn, sorted_data)
        print(f"  {name:12s}: random={t_rand*1000:6.2f}ms, sorted={t_sort*1000:6.2f}ms")

PREDICT 문제 1

삽입 정렬이 이미 정렬된 배열 [1,2,3,4,5]을 처리할 때 while 루프의 총 반복 횟수는?

풀이 펼치기
답

0번. 매 외부 루프에서 A[i] > key가 즉시 거짓이 되어 while이 한 번도 실행되지 않음. 총 비교는 n−1 = 4번 (외부 루프만). 따라서 T(n) = Θ(n).

PREDICT 문제 2

선택 정렬이 [3, 1, 2]를 처리할 때 몇 번의 스왑이 일어나는가?

풀이 펼치기
풀이

i=0: min_idx=1 (값 1), swap(A[0], A[1]) → [1, 3, 2]. 스왑 1회.
i=1: min_idx=2 (값 2), swap(A[1], A[2]) → [1, 2, 3]. 스왑 1회.
총 2번. 일반적으로 n−1번의 스왑(자기 자신과 스왑할 수도 있지만 여전히 카운트).

WHY 문제 3

삽입 정렬이 적응형(adaptive)인 이유를 inversion 개념으로 한 문장으로 설명하시오.

풀이 펼치기
답

삽입 정렬의 총 비교 수 = n − 1 + inv(A)이므로, 입력에 역순쌍이 적을수록(= 정렬에 가까울수록) 실제 실행 시간이 짧아집니다. 따라서 입력의 "정렬됨 정도"에 적응합니다.

FIX 문제 4

다음 버블 정렬의 버그는? 고친 버전을 제시하시오.

buggy_bubble.py
def bubble(A):
    n = len(A)
    for i in range(n):
        for j in range(n):
            if A[j] > A[j + 1]:
                A[j], A[j + 1] = A[j + 1], A[j]
    return A
풀이 펼치기
버그 분석

내부 루프가 range(n)까지 가는데 A[j+1]에 접근하므로 j=n-1일 때 IndexError. 또한 외부 루프도 n까지면 불필요한 반복.

수정: for j in range(n - 1 - i):, 외부도 range(n - 1):.

BUILD 문제 5

선택 정렬을 안정(stable)하게 만드는 방법을 제시하시오 (힌트: 스왑 대신 다른 연산 사용).

풀이 펼치기
stable_selection.py
def stable_selection(A):
    n = len(A)
    for i in range(n - 1):
        m = i
        for j in range(i + 1, n):
            if A[j] < A[m]: m = j
        # 스왑 대신, A[m]을 A[i] 앞으로 옮기기 위해 A[i..m-1]을 한 칸씩 쉬프트
        key = A[m]
        for k in range(m, i, -1):
            A[k] = A[k - 1]
        A[i] = key
    return A
분석

이 변형은 O(n²) 쉬프트를 하게 되어 선택 정렬의 "스왑 Θ(n)" 장점이 사라집니다. 안정성을 얻는 대신 쓰기 비용이 커짐 — 근본적 트레이드오프.

PREDICT 문제 6

배열 [5, 3, 8, 1, 2]의 inversion 개수 inv(A)는?

풀이 펼치기
풀이

역순쌍: (5,3), (5,1), (5,2), (3,1), (3,2), (8,1), (8,2) — 총 7개.

따라서 삽입 정렬의 쉬프트 수 = 7, 비교 수 ≤ n−1 + 7 = 11번 (상한). 실제 트레이스: j=1 (key=3) 1회, j=2 (key=8) 1회, j=3 (key=1) 3회, j=4 (key=2) 4회 = 9번 비교. 점근적으로는 Θ(n + inv(A)).

WHY 문제 7

결정 트리 증명에서 "리프 수 ≥ n!"이라는 주장의 근거를 한두 문장으로.

풀이 펼치기
답

서로 다른 n개 원소의 순열은 n!개. 각 순열은 서로 다른 정답 배치를 요구하므로 알고리즘이 서로 다른 출력을 내야 하고, 출력은 리프와 1:1 대응이므로 리프 수 ≥ n!.

PREDICT 문제 8

n=5일 때 비교 기반 정렬이 최악의 경우 최소 몇 번의 비교를 해야 하는가? (log₂(5!) 계산)

풀이 펼치기
풀이

5! = 120. log₂ 120 ≈ 6.91. 이진 트리 높이는 정수이므로 최소 7번. 실제로 merge-insertion sort (Ford-Johnson)가 이 하한에 닿아 7번에 정렬 가능.

DESUGAR 문제 9

CPython의 binarysort가 "비교 O(log k) + 쉬프트 O(k)" 구조를 쓰는 이유를 Python 객체의 특성으로 설명하시오.

풀이 펼치기
답

Python 객체의 비교는 __lt__ 호출, 타입 체크, GIL 획득, 예외 처리 등으로 수십~수백 ns 걸립니다. 반면 포인터 쉬프트는 C memmove로 수 ns. 따라서 비교 수를 줄이는 게 훨씬 이득이므로, 선형 탐색(O(k) 비교) 대신 이진 탐색(O(log k) 비교)을 씁니다. 쉬프트는 여전히 O(k²)이지만 상수가 훨씬 작음.

BUILD 문제 10

삽입 정렬 + 이진 탐색("binary insertion sort") 버전을 Python으로 구현하시오.

풀이 펼치기
binary_insertion.py
import bisect

def binary_insertion_sort(A):
    for j in range(1, len(A)):
        key = A[j]
        # A[0..j-1]에서 key가 들어갈 위치를 O(log j)로 찾기
        pos = bisect.bisect_right(A, key, 0, j)
        # A[pos..j-1]을 한 칸씩 오른쪽으로 쉬프트
        A[pos+1:j+1] = A[pos:j]
        A[pos] = key
    return A
분석

비교 수 Θ(n log n) — 하한에 닿음. 하지만 쉬프트는 여전히 Θ(n²). 따라서 시간 복잡도는 Θ(n²)이지만, 비교 비용이 큰 경우(예: 긴 문자열 비교) 실용적으로 빠름. bisect.bisect_right를 쓰면 안정 정렬이 유지됨.

14 — 이번 주 정리

🃏

삽입 정렬의 왕좌

세 O(n²) 정렬 중 유일하게 적응형·안정·단순. 거의 모든 표준 라이브러리가 작은 구간에서 이걸 씀.

🎯

선택 정렬의 틈새

스왑 Θ(n)의 특수 장점 — 쓰기가 비싼 매체(Flash, EEPROM)에서만 의미 있음.

🫧

버블 정렬의 퇴장

모든 면에서 삽입 정렬에게 진다. 교육용으로 남았지만 실전 금지. — Knuth

📉

Ω(n log n) 하한

결정 트리 논증: 리프 n! 필요 → 높이 log₂(n!) = Θ(n log n). 비교 기반의 근본적 한계.

🔁

적응성 = inv(A) 함수

삽입 정렬 시간 = Θ(n + inv(A)). "정렬에 얼마나 가까운지"에 정확히 비례.

🐍

CPython의 binarysort

Timsort 내부의 작은 정렬. 비교 비싼 Python 객체를 위해 binary insertion 사용.

15 — 용어 사전 + 다음 주 맛보기

용어 사전 (25개)

Insertion Sort
삽입 정렬 — 왼쪽 정렬 구간에 원소를 한 개씩 끼워넣음. Θ(n + inv(A)).
Selection Sort
선택 정렬 — 남은 구간의 최솟값을 찾아 맨 앞으로. 비교 Θ(n²), 스왑 Θ(n).
Bubble Sort
버블 정렬 — 인접 쌍 교환으로 큰 값을 끝으로. 최적화해도 삽입 정렬에 밀림.
Adaptive Sort
적응형 정렬 — 입력의 정렬 정도에 따라 빨라지는 정렬. 대표 예: 삽입 정렬, Timsort.
Stable Sort
안정 정렬 — 같은 키 원소의 원래 순서를 유지. 삽입·버블은 안정, 선택은 기본 구현에서 불안정.
In-place Sort
제자리 정렬 — O(1) 또는 O(log n) 보조 공간만 사용. 세 알고리즘 모두 해당.
Comparison Sort
비교 기반 정렬 — 원소를 비교만으로 순서를 결정. 하한 Ω(n log n).
Decision Tree
결정 트리 — 알고리즘의 모든 비교 흐름을 트리로 표현. 높이 = 최악 비교 수.
Inversion
역순쌍 — i<j이고 A[i]>A[j]인 쌍. 삽입 정렬의 정확한 복잡도 = Θ(n + inv(A)).
Best / Average / Worst Case
최선/평균/최악 — 입력에 따른 실행 시간의 세 극값. 평균은 입력 분포 가정 필요.
Lower Bound (하한)
어떤 알고리즘도 이보다 빠를 수 없다는 증명. 비교 정렬의 Ω(n log n).
Permutation
순열 — n개 원소의 서로 다른 배치. 총 n!개.
Stirling's Approximation
스털링 근사 — log₂(n!) ≈ n log₂ n − n log₂ e. 결정 트리 증명의 도구.
Binary Insertion Sort
이진 삽입 정렬 — 삽입 위치를 이진 탐색으로. 비교 Θ(n log n), 쉬프트 Θ(n²).
Timsort
Tim Peters의 적응형 병합 정렬. Python list.sort. 작은 구간은 binarysort.
MIN_RUN
Timsort가 binarysort로 확장할 run의 최소 길이. 보통 32~64.
Natural Run
자연 run — 입력에 이미 존재하는 정렬된 연속 구간. Timsort의 핵심.
Crossover Point
교차점 — 두 알고리즘의 실제 실행 시간이 같아지는 n. 라이브러리마다 10~64.
Constant Factor
상수 인자 — 빅오에 숨겨진 계수. n이 작을 때 실측의 결정자.
Cache Locality
캐시 지역성 — 인접 메모리 접근이 CPU 캐시에 잘 맞는 성질. 삽입 정렬이 유리.
Branch Prediction
분기 예측 — CPU가 if 문 결과를 예측. 단순 루프일수록 잘 맞음.
Early Exit
조기 종료 — 조건 만족 시 루프를 즉시 탈출. 버블 정렬의 표준 최적화.
Counting Sort
계수 정렬 — 정수 범위 k면 O(n+k). 비비교 정렬로 Ω(n log n) 하한 우회. (CLRS §8.2)
Radix Sort
기수 정렬 — 자릿수마다 안정 정렬. O(d(n+k)). (CLRS §8.3)
Ford-Johnson / Merge-Insertion
비교 횟수 최소화에 가까운 알고리즘. n=5일 때 7번 비교로 하한에 닿음.
