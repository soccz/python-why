고급 자료구조 — Trie, Skip List, B-Tree, LSM
자동완성부터 PostgreSQL 인덱스까지

1960년 MIT의 Edward Fredkin이 retrieval의 중간 음절을 따서 Trie(트라이)를 명명. 12년 뒤 1972년 보잉 연구소의 Bayer & McCreight가 디스크 지향 균형 트리 B-Tree를 발표 — 오늘날 PostgreSQL, MySQL 인덱스의 실체. 1990년 William Pugh가 트리 없이 O(log n) 정렬 집합(sorted set)을 달성하는 Skip List를 CACM에 게재. 1996년 O'Neil et al.이 쓰기 최적 LSM-Tree를 제안 — Cassandra, RocksDB, LevelDB의 뿌리.
Algo 10-11의 기본 트리를 넘어 디스크·문자열·randomized·쓰기 최적의 세계. 현대 저장 엔진의 네 기둥.

01 — 이번 주 이야기: 60년의 자료구조 혁신

Fredkin은 "reTRIEval"의 중간 네 글자에서 이름을 따왔다고 밝혔으며, 원래는 "tree"와 같이 발음하도록 의도했다. 오늘날 트리(tree)와 구분하기 위해 "트라이(try)"로 발음하는 것이 일반적.
— Edward Fredkin, "Trie Memory", CACM 3(9):490-499, September 1960

1960 — Edward Fredkin, MIT

MIT 연구원 Fredkin이 "Trie Memory" (CACM 3(9):490-499, 1960)에서 문자열 자동완성·사전·IP 주소 조회의 기반이 되는 자료구조를 제안. "retrieval"의 중간 음절을 따서 trie라 명명. 발음은 "tree"와 같게(트리) 또는 "try"(트라이) — Fredkin 자신은 "tree"로 발음했으나, 혼동 때문에 지금은 "트라이"가 일반적.

1972 — Rudolf Bayer & Edward McCreight, Boeing Research Labs

"Organization and Maintenance of Large Ordered Indexes"(Acta Informatica 1:173-189, 1972; 1970 Boeing 내부 보고서로 먼저 유통). B-Tree가 보잉에서 태어남 — 당시 Bayer는 "B는 Boeing의 약자인가, Bayer인가, balanced인가?"라는 질문에 "B stands for balanced, Boeing, Bayer, or whatever you want" 애매하게 답함. 디스크 I/O 지향 → 데이터베이스 인덱스의 표준이 됨.

1990 — William Pugh, University of Maryland

"Skip Lists: A Probabilistic Alternative to Balanced Trees"(CACM 33(6):668-676, 1990). 트리 회전(rotation) 없이 확률적 포인터 건너뛰기로 O(log n) 삽입/조회/삭제. 구현이 단순하고 분석이 우아 — "비결정론적 자료구조"의 교과서 예.

현재 Redis Sorted Set, LevelDB MemTable, Apache Lucene의 내부에서 활용.

1996 — Patrick O'Neil, Edward Cheng, Dieter Gawlick, Elizabeth O'Neil

"The Log-Structured Merge-Tree (LSM-Tree)"(Acta Informatica 33(4):351-385, 1996). 쓰기 폭주 워크로드를 위한 설계. "최근 쓰기는 메모리, 배치로 디스크에 머지" — 하드디스크의 순차 쓰기 속도를 극대화. 배경: Rosenblum & Ousterhout의 1992년 log-structured file system(LFS, ACM TOCS 10(1))이 "순차 쓰기가 가장 빠르다"는 철학을 먼저 제시한 뒤 인덱스 구조로 진화한 것.

Google Bigtable 2006에서 재발견·대중화. 이후 Cassandra, HBase, RocksDB, LevelDB 등 NoSQL 스토리지 엔진의 기반. 2020년대 CockroachDB, TiDB까지 확장.

60년의 자료구조 계보: Fredkin 1960 Trie (문자열) → Bayer-McCreight 1972 B-Tree (디스크) → Pugh 1990 Skip List (randomized) → O'Neil 1996 LSM-Tree (쓰기 최적) → 2006 Bigtable로 NoSQL 시대 개막.

02 — Trie — Fredkin 1960

중급

아이디어

문자 하나씩 분기하는 트리. 루트에서 리프까지의 경로가 하나의 문자열. 공통 prefix는 공유.

  • 삽입: 문자 하나씩 따라가며 없는 가지 만들기. O(|word|).
  • 조회: 같은 경로 따라가기. O(|word|).
  • prefix 탐색: prefix까지 내려간 뒤 서브트리 DFS. "ca"로 시작하는 모든 단어.
  • 공간: 문자 집합 Σ 기준 O(Σ · n · L) (n=단어 수, L=평균 길이). 최악. Patricia trie로 압축 가능.
Trie 구조: 각 edge가 한 문자. 리프 또는 중간 노드에 ★(단어 끝) 표시. "ca"는 cat/car/card가 공유, "do"는 do/dog가 공유. 공통 prefix 저장 공간 절약.

03 — Trie Python 구현

고급

dict 기반 구현

trie.py
class TrieNode:
    __slots__ = ('children', 'is_end')
    def __init__(self):
        self.children = {}
        self.is_end = False


class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_end = True

    def search(self, word):
        node = self._find(word)
        return node is not None and node.is_end

    def starts_with(self, prefix):
        return self._find(prefix) is not None

    def _find(self, s):
        node = self.root
        for ch in s:
            if ch not in node.children: return None
            node = node.children[ch]
        return node

    def autocomplete(self, prefix):
        node = self._find(prefix)
        if node is None: return []
        result = []
        def dfs(n, path):
            if n.is_end: result.append(prefix + ''.join(path))
            for ch, child in sorted(n.children.items()):
                path.append(ch); dfs(child, path); path.pop()
        dfs(node, [])
        return result


# 테스트
t = Trie()
for w in ["cat", "car", "card", "do", "dog"]:
    t.insert(w)
print(t.search("cat"))          # True
print(t.search("ca"))           # False (prefix but not word)
print(t.starts_with("ca"))     # True
print(t.autocomplete("ca"))    # ['car', 'card', 'cat']
  • dict 기반: 각 노드가 dict로 children 관리. 알파벳 크기 가변 (영어 26 또는 유니코드 전체).
  • 배열 기반: 고정 알파벳(예: 소문자 26자)에 [None]*26. 빠르지만 공간 낭비.
  • __slots__: Python에서 객체 공간 절약. 큰 trie에 필수.
  • 복잡도: 삽입/조회 O(|word|). prefix search + DFS O(|prefix| + k·L) (k=매치 수).

04 — Patricia / Radix Tree

전공자

Trie의 압축

표준 trie는 공간 낭비가 큼 — 중간에 자식이 하나뿐인 체인들이 많음. Patricia Trie(Morrison 1968, "PATRICIA: Practical Algorithm To Retrieve Information Coded in Alphanumeric") 또는 Radix Tree: 단일 자식 체인을 한 edge로 압축 → 공간 O(n·L) 이하.

  • edge에 문자열 저장: "car"의 경우 하나의 edge가 "car" 전체를 담음.
  • 삽입 시 분할: 새 단어가 기존 edge와 prefix 공유 시 edge를 쪼개고 분기 생성.
  • 응용: IP 라우팅 테이블 (Longest Prefix Match), Linux 커널 route table, Rust BTreeMap(내부는 B-tree지만 radix 유사 기법도 혼합), Redis Streams 내부.

LC-Trie(Nilsson-Karlsson 1999): level-compressed trie. 자식이 꽉 찬 내부 노드는 높이를 압축. 라우터 성능을 위한 DB 인덱스 변종.

05 — Skip List — Pugh 1990

고급

"여러 레벨의 링크드 리스트"

정렬된 연결 리스트에 고속도로 같은 상위 레벨을 추가. 높은 레벨은 듬성듬성 원소만 포함 → 검색 시 빠르게 건너뜀.

  • 각 노드가 무작위 높이 (레벨). 확률 p=1/2로 "한 레벨 더 올라감" 반복.
  • 높은 레벨에서 "너무 크면 한 레벨 내려감" → 단계적으로 좁혀가며 검색.
  • 기대 O(log n) 시간 (모든 연산). 확률적 보장 — 최악은 O(n)이지만 확률 극소.

왜 이 단순한 구조가 쓰이는가:

  • 구현이 AVL/Red-Black 트리보다 훨씬 짧음 (회전 로직 없음).
  • 동시성(lock-free) 구현이 용이 — 각 노드의 next 포인터 CAS 한 번.
  • Redis Sorted Set, LevelDB MemTable, Lucene internals에 사용.
Skip list search(17): L3에서 시작 → 20>17이라 한 레벨 내려감 → L2 7에서 시작, 20>17로 내려감 → L1 12에서 시작, 20>17로 내려감 → L0에서 15→17 발견. 4 레벨 × 평균 2 스텝 = O(log n).

06 — Skip List Python 구현

고급

확률적 자료구조

skip_list.py
import random

class SkipNode:
    def __init__(self, key, level):
        self.key = key
        self.forward = [None] * (level + 1)


class SkipList:
    def __init__(self, max_level=16, p=0.5):
        self.max_level = max_level
        self.p = p
        self.head = SkipNode(-float('inf'), max_level)
        self.level = 0

    def _random_level(self):
        lvl = 0
        while random.random() < self.p and lvl < self.max_level:
            lvl += 1
        return lvl

    def insert(self, key):
        update = [None] * (self.max_level + 1)
        curr = self.head
        for i in range(self.level, -1, -1):
            while curr.forward[i] and curr.forward[i].key < key:
                curr = curr.forward[i]
            update[i] = curr

        if curr.forward[0] and curr.forward[0].key == key:
            return    # 중복 방지

        lvl = self._random_level()
        if lvl > self.level:
            for i in range(self.level + 1, lvl + 1):
                update[i] = self.head
            self.level = lvl

        new_node = SkipNode(key, lvl)
        for i in range(lvl + 1):
            new_node.forward[i] = update[i].forward[i]
            update[i].forward[i] = new_node

    def search(self, key):
        curr = self.head
        for i in range(self.level, -1, -1):
            while curr.forward[i] and curr.forward[i].key < key:
                curr = curr.forward[i]
        curr = curr.forward[0]
        return curr is not None and curr.key == key
  • 기대 레벨: p=1/2일 때 기대 높이 ≈ log2 n. 공간 기대 O(n / (1-p)) = O(2n).
  • 시간 보장: 확률 1 − n−c로 O(log n) — 매우 높은 확률.
  • Determinism: 무작위성 없이 동일 성능 보장은 어려움. 실무에선 고정 seed로 재현성 확보 가능.

07 — B-Tree — Bayer-McCreight 1972

고급

디스크 지향 트리

이진 트리의 문제: 디스크 접근이 많음. 각 노드 = 한 디스크 블록(보통 4KB-16KB). 이진 트리는 노드당 키 1개 → 1M 원소면 깊이 20. 20번 디스크 I/O는 치명적.

B-Tree 해결책: 각 노드에 많은 키(예: 수십~수백 개). 노드는 정확히 한 디스크 블록. 깊이가 logB n으로 줄어듦 (B=branching factor).

  • 각 내부 노드: k개 키 + k+1개 자식 포인터.
  • 키들은 정렬된 상태.
  • 노드당 키 수는 [t-1, 2t-1] 범위 (t는 minimum degree).
  • 모든 리프가 같은 깊이.
  • 조회: O(logt n) — 디스크 지향에선 t=100이면 n=10억도 깊이 5-6.
전공자

B의 의미

Knuth TAOCP vol.3에 따르면 Bayer는 "B가 무엇을 의미하는지 생각할수록 B-Tree를 더 잘 이해하게 된다"고 답했다고 전해짐. "balanced / Boeing / Bayer 중 어느 것이든"이라는 대중적 버전은 출처 불분명한 민담. 확실한 건: Boeing Research Labs에서 1970년 내부 보고서, 1972년 Acta Informatica 논문이라는 역사.

08 — B-Tree 삽입과 분할

전공자

분할(split) 메커니즘

  1. 리프까지 내려가며 올바른 위치 찾기.
  2. 리프가 꽉 차있지 않으면 (키 수 < 2t-1) 단순 삽입.
  3. 꽉 차있으면 분할: 중간 키를 부모로 올리고, 좌/우 절반으로 쪼갬.
  4. 부모도 꽉 차면 재귀적으로 분할.
  5. 루트가 분할되면 트리 높이 +1.

선제적 분할(proactive split): CLRS 구현은 내려가면서 "꽉 찬 노드"를 미리 분할. 재귀 없이 한 번에 내려감. 실무엔 "꽉 찬 후 분할"도 혼재.

B-Tree 삽입 22: 꽉 찬 리프를 분할 → 중간 키 올리기 → 22 삽입. B-tree는 모든 리프가 같은 깊이 보장 — 최악·평균 O(log n) 보장.

09 — B+ Tree — DB 인덱스의 실체

고급

B-Tree의 실무 변종

B+ Tree: B-Tree의 변종 (Comer의 1979 survey "The Ubiquitous B-Tree", ACM Computing Surveys 11(2)에서 정리). 오늘날 PostgreSQL, MySQL/InnoDB, SQLite, Oracle의 주 인덱스.

  • 데이터는 리프에만: 내부 노드는 라우팅만.
  • 리프 연결 리스트: 인접 리프가 수평 포인터로 연결 → 범위 스캔 효율.
  • 장점: 내부 노드에 더 많은 키를 담을 수 있어 분기 수 ↑ 깊이 ↓. 범위 질의("WHERE price BETWEEN 100 AND 500")에 매우 빠름.
  • Fill factor: 리프 70-80% 차있는 상태 유지. 삽입 여유 + 공간 효율 균형.

PostgreSQL의 CREATE INDEX는 기본 B+ Tree. psql \d+ table_name로 확인 가능. 10억 row도 깊이 4-5의 B+ Tree로 관리.

전공자

B* Tree, B-link Tree

  • B* Tree: 분할 전에 형제 노드와 키 분배 — 공간 효율 ↑. IBM VSAM.
  • B-link Tree(Lehman-Yao 1981): 리프뿐 아니라 모든 레벨이 오른쪽 링크 유지. 동시성 friendly — PostgreSQL의 기본 인덱스 구현.
  • Blink Tree 변형: 현대 DB의 lock-free 인덱스 기반.

10 — LSM-Tree — O'Neil 1996

전공자

쓰기 최적화의 혁명

B-Tree의 약점: 랜덤 쓰기. 레코드 삽입마다 해당 리프 블록을 디스크에서 읽고, 수정하고, 다시 씀. HDD의 seek 시간 때문에 초당 수백 IOPS 제한.

LSM의 아이디어: 쓰기를 순차적으로 모아서 나중에 정리.

  1. MemTable: 메모리 내 정렬 구조 (Skip list/BST). 쓰기는 여기에.
  2. Immutable MemTable: 꽉 차면 flush 대기.
  3. SSTable (Sorted String Table): 디스크의 불변 정렬 파일. flush로 생성.
  4. Compaction: 여러 SSTable을 주기적으로 머지 → 중복 제거 + 삭제 적용.
  5. Level/Tiered: 여러 레벨로 SSTable 조직. Lk는 Lk-1의 10배 크기 (LevelDB 기본).

조회: MemTable → Immutable → L0 → L1 → ... 순서대로 탐색. Bloom filter로 가속(Algo 25) — 각 SSTable에 "이 키 있을 수도" 빠른 판단.

전공자

B-Tree vs LSM: 쓰기/읽기 trade-off

측면B-TreeLSM-Tree
점 읽기O(log n) 디스크 I/OO(log n) but 여러 SSTable 확인 필요

공간Fill factor 70-80%중복 버전 존재 (compaction 전)

요약: 쓰기가 많으면 LSM, 균형/읽기 중심이면 B-Tree. 최근 하이브리드(예: MySQL X Engine, Wiredtiger)도 증가.

11 — BST vs B-Tree vs LSM 비교

고급

언제 무엇을

  • BST / AVL / Red-Black (Algo 11): 메모리 내 소~중규모. 모든 연산 O(log n). Java TreeMap, C++ std::map.
  • Skip List: BST 대안, 구현 단순, 동시성 friendly. Redis Sorted Set, LevelDB MemTable.
  • Trie: 문자열 prefix 질의. 자동완성, IP 라우팅.
  • B-Tree / B+ Tree: 디스크 지향. 대규모 정렬 인덱스, 범위 스캔. 전통 RDBMS.
  • LSM-Tree: 쓰기 폭주. NoSQL, 로그/시계열, 블록체인 상태.
  • Fractal Tree / Bε-Tree: B-Tree와 LSM의 중간. Tokutek, MongoDB 일부.

12 — 응용 — Redis, Cassandra, Postgres

고급

Redis — Skip List의 주 사용처

Redis의 Sorted Set (ZSET)은 내부적으로 두 자료구조 유지:

  • Skip List: 점수 기준 정렬. 범위 질의 O(log n).
  • Hash Table: 멤버 → 점수 O(1) 매핑.

게임 리더보드, 순위 조회, 타임라인 등 Redis의 킬러 기능. Antirez(Redis 창시자)는 skip list를 선택한 이유로 "구현 단순성 + 범위 질의 효율 + 메모리 효율"을 꼽음.

고급

Cassandra / RocksDB — LSM 기반

  • Apache Cassandra(2008, Facebook): LSM. 쓰기 폭주 환경 (시계열, 로그, IoT).
  • RocksDB(Facebook 2012): Google LevelDB fork. Ethereum, TiKV, MyRocks, CockroachDB 내부 엔진.
  • LevelDB(Google 2011): Jeff Dean & Sanjay Ghemawat. Bigtable의 단일 노드 버전.
  • Amazon DynamoDB: LSM 기반 스토리지.
고급

IP 라우팅 — Radix Tree

인터넷 라우터가 패킷을 포워딩할 때 목적지 IP에서 최장 prefix 일치 (Longest Prefix Match)를 찾음. 테이블이 수십만~수백만 개 prefix.

  • Linux 커널: FIB_TRIE (Level Compressed Trie, Nilsson-Karlsson 1999).
  • Cisco/Juniper 라우터: 하드웨어 TCAM + Patricia 변종.
  • 100 Gbps 링크에서 초당 수억 패킷 처리 — 자료구조 선택이 네트워크 인프라의 속도를 결정.
고급

자동완성 — Trie + Ranking

  • Google, Naver, YouTube 검색 자동완성: Trie로 prefix 매칭 + 인기 가중치 ranking.
  • T9 키패드(1990s 피처폰): 숫자 → 가능 단어 매핑을 trie로.
  • Elasticsearch: Completion Suggester가 Finite State Transducer(FST) — trie의 일반화.
전공자

AI가 자료구조를 잘못 고르는 패턴

  • dict로 자동완성: prefix 찾으려면 모든 키 순회 O(n). Trie는 O(|prefix|).
  • BST로 디스크 DB 인덱스: 깊이 log₂(10⁹) ≈ 30 → I/O 30번. B-Tree는 4-5번.
  • 균형 트리 구현 고집: Red-Black 회전 디버깅 지옥. Skip list나 treap이 훨씬 간단.
  • 일반 DB에 LSM: 읽기/범위 스캔이 많으면 B-Tree가 빠를 수 있음.
  • IP 라우팅에 해시: 해시는 최장 prefix 안 됨. Radix tree 필수.

교정 질문: "prefix 질의 필요?", "디스크 기반?", "쓰기 vs 읽기 비율?", "범위 스캔?"

13 — Persistent 자료구조

전공자

불변성의 힘

Persistent data structure: 업데이트 시 원본 보존 + 새 버전 반환. 함수형 언어(Haskell, Clojure)의 표준. Git도 persistent tree (commit마다 새 tree, 공유 노드 재사용).

  • Path copying: 업데이트 경로의 노드만 복사, 나머지는 공유. 트리 자료구조에 자연스러움.
  • Hash Array Mapped Trie (HAMT): Bagwell 2000. Clojure/Scala의 불변 Map/Set 기반.
  • Finger Trees: Hinze & Paterson 2006. 불변 시퀀스의 앞/뒤 O(1) 접근.
  • RRB Tree: relaxed radix balanced. Scala Vector, Clojure vector 내부.

응용: 함수형 프로그래밍, 버전 관리 시스템 (Git, Mercurial), Undo/Redo, 분산 시스템 (Event Sourcing, CRDTs).

14 — 연습문제 (12문제)

WHY 문제 1

n개 단어 총 길이 L의 trie에서 공간 상한은?

풀이 펼치기
답

최악 O(L·σ) (σ=알파벳 크기, 배열 기반). dict 기반이면 O(L). 공통 prefix 공유로 L보다 적을 수 있음. 극단: 모든 단어가 같은 prefix → 크게 줄어듦. 반대로 prefix 없는 단어들 → L 그대로.

PREDICT 문제 2

Skip list에서 p=1/4로 하면 공간과 검색 시간은?

풀이 펼치기
답

기대 노드 수 = n/(1-p) = n/0.75 ≈ 1.33n (p=1/2의 2n보다 작음 — 공간 효율 ↑). 기대 비교 수 = log_{1/p}(n)/p = 4·log₄(n) = 2·log₂(n) — p=1/2의 log₂(n)의 약 2배. 공간은 줄지만 비교 수는 늘어남. Pugh는 p=1/4 전후가 총 작업량(포인터 저장 + 비교) 관점에서 optimal이라고 분석함.

BUILD 문제 3

Trie로 "단어 경로" 문제 (Word Search II, LeetCode 212) 접근.

풀이 펼치기
답

입력 단어들로 trie 구축. 보드의 각 셀에서 DFS + trie 동시 추적. 현재 문자가 trie의 한 자식에 있으면 진행, 없으면 가지치기. O(M·4^L)이 아닌 O(M·σ·L) 같은 더 tight한 시간. 단순 단어별 DFS보다 훨씬 빠름.

DESUGAR 문제 4

B-Tree 분할의 중간 키 "올리기"가 왜 필요한가?

풀이 펼치기
답

키 분할 후 두 자식이 각각 절반씩 가짐. 부모는 "이 키 미만은 왼쪽, 이상은 오른쪽"을 안내해야 하므로 중간 키를 라우팅 지표로 가져야 함. B+ Tree는 중간 키를 "복사"해서 내부와 리프 양쪽에 둠 — 리프 스캔 위해.

WHY 문제 5

왜 PostgreSQL 기본 인덱스가 hash가 아니라 B+ Tree?

풀이 펼치기
답

Hash는 점 조회만 O(1). 범위 질의(WHERE price > 100), 정렬(ORDER BY), prefix 매칭(LIKE 'abc%')에 무력. B+ Tree는 이 모두 자연스럽게. 실제로 PostgreSQL에 hash index 있지만 기본/주력은 B+ Tree. 특수 목적에만 hash index.

BUILD 문제 6

LSM-Tree에서 삭제는 어떻게 구현?

풀이 펼치기
답

Tombstone: 삭제도 "삭제 마커"를 넣는 쓰기. 조회 시 tombstone 만나면 "없음" 반환. Compaction 때 tombstone + 해당 키의 옛 버전 모두 제거. 결과: 삭제도 순차 쓰기라 빠름. 단점: 삭제 후 공간 즉시 회수 안 됨.

PREDICT 문제 7

n=10억 row의 테이블에 B+ Tree 인덱스 깊이 예상.

풀이 펼치기
답

페이지 8KB, 키 100바이트 가정 → 노드당 ~80 키/포인터. depth = log₈₀(10⁹) ≈ 4.7 → 5레벨. 조회 디스크 I/O 5번. 상위 레벨은 캐시되어 실제 I/O 1-2번. 10억 row도 밀리초 이내 조회 가능.

DESUGAR 문제 8

Skip list가 "lock-free 친화적"인 이유.

풀이 펼치기
답

트리와 달리 회전 없음. 삽입은 각 레벨별 forward 포인터 CAS 한 번으로 끝남. 동시성 충돌 범위가 작고 로컬. Red-Black/AVL의 회전은 여러 포인터 동시 수정 필요 → lock-free 구현 극도로 복잡. Java ConcurrentSkipListMap의 설계 배경.

WHY 문제 9

Bloom filter가 LSM-Tree에 왜 필수?

풀이 펼치기
답

조회 시 최악 모든 SSTable 열어 탐색 → 디스크 I/O 폭발. Bloom filter(각 SSTable당 수 KB)로 "이 키 없음" 빠르게 확인 → 불필요한 디스크 읽기 회피. 일반적 워크로드에서 조회 시 대부분의 SSTable은 Bloom filter로 즉시 스킵. LSM 성능의 숨은 주역.

BUILD 문제 10

Patricia trie의 edge label 압축 예.

풀이 펼치기
답

{apple, application, banana}:

  • 기본 trie: 루트 → a → p → p → l → {e(★), i → c → a → t → i → o → n(★)} / b → a → n → a → n → a(★). 깊이 7+, 많은 단일 자식 체인.
  • Patricia: 루트 → "appl" → {"e"(★), "ication"(★)} / "banana"(★). 3개 내부 노드로 압축.
IP 라우팅 테이블처럼 넓고 희소한 키 공간에 대폭 압축.

AI 문제 11

AI가 "대규모 timeseries DB에 B-Tree"를 제안. 교정?

풀이 펼치기
질문
  1. "쓰기 폭주 (초당 수만~수백만 이벤트) 예상인가?"
  2. "쓰기가 많으면 B-Tree 랜덤 I/O 병목. LSM (InfluxDB, TimescaleDB continuous aggregates) 적합."
  3. "InfluxDB는 TSI(Time-Series Index) + TSM(Tree-Structured Merge)로 LSM 변형."
  4. "삭제보다 TTL 기반 retention이 대부분 — tombstone 문제 최소."
WHY 문제 12

Git의 "commit이 persistent tree"라는 말의 의미.

풀이 펼치기
답

각 commit은 그 시점의 파일 트리 전체를 표현하지만, 실제로는 변경된 파일만 새로 저장 + 나머지는 이전 commit과 해시 주소 공유. 파일 한 개 바꾸고 commit하면 그 파일의 blob + 포함 디렉토리 트리들만 새로 만듦. 수천 개 이전 commit이 공존해도 공간은 누적 변경분만. Path copying persistent tree의 실제 구현.
