왜 Python은 linked list를 주지 않는가

교과서의 첫 자료구조인 단일 연결 리스트(singly linked list)가 Python 표준 라이브러리에는 없습니다. 왜일까요? 이론적으로 완벽한 자료구조가 실전에서 외면받는 이유 — 캐시 지역성.
그리고 Python이 대신 주는 것 — collections.deque. 이건 linked list지만, 여러분이 아는 그 linked list가 아닙니다.
이번 주의 핵심 질문: "교과서와 현실 사이에서 CPython은 어떤 타협을 했는가?"

01 — 이번 주 이야기: 인라인 스케이터와 버스 승객

버스 승객 — 배열 (array)

버스의 좌석은 연속된 메모리와 같습니다. 승객(원소)이 좌석 1, 2, 3, ...에 차례로 앉아 있습니다. 5번 좌석 승객을 찾아라 하면? — 기사가 5번 좌석으로 직행. O(1).

새 승객이 타면 마지막 빈 좌석에. 좌석이 꽉 차면? — 모든 승객이 내려서 더 큰 버스로 이동. Algo 08의 동적 배열 이야기.

인라인 스케이터 그룹 — 단일 연결 리스트 (linked list)

인라인 스케이터들이 일렬로 타며 각자 앞 사람 어깨만 잡고 있습니다. 누군가 끼어들기 쉬움 — 두 사람 사이를 끊고 끼어들면 됨. O(1) 삽입.

하지만 5번째 사람을 찾으려면? — 줄 맨 앞에서 시작해 한 명씩 세며 따라가야 함. O(n) 접근.

WHY — 교과서가 찬양하는 linked list가 현실에서 밀리는 이유

교과서는 이렇게 말합니다: "linked list는 삽입·삭제가 O(1)이니 많은 경우 배열보다 낫다."

현실 하드웨어에서는 틀립니다.

  • 버스 승객은 옆자리 사람 얼굴이 바로 보임. CPU의 L1 캐시는 이웃한 메모리를 한 번에 가져옴.
  • 인라인 스케이터는 각자가 다른 도시에 흩어져 있음 (malloc이 각 node를 힙의 임의 위치에 할당). 다음 사람을 찾는 데 매번 "새 도시로 이동" — 캐시 미스.

n=10,000개 원소에 대해:

연산Array 이론Linked List 이론Array 실측Linked List 실측
순회 (traversal)O(n)O(n)0.01ms0.1~1ms (10~100배 느림)
중간 삽입O(n)O(1)*빠름노드 찾는 데 O(n), 결국 비슷

*원소 포인터를 이미 가지고 있을 때만 O(1). 노드 "찾는" 건 여전히 O(n).
이래서 Python은 단일 연결 리스트를 표준으로 주지 않습니다. 대신 list(동적 배열)와 deque(block-linked list)만.

Python의 영리한 절충: deque = block-linked list. 64개씩 묶어 캐시 친화 + 메타데이터 오버헤드 1/16. 양 끝 push/pop 모두 O(1).
큐(queue)는 deque로! list.pop(0)는 모든 원소를 옮기느라 O(n). deque.popleft()는 포인터만 한 칸 이동 → O(1). BFS·작업 큐 같은 패턴에서 절대 list 쓰면 안 되는 이유.

02 — 단일 연결 리스트 (Singly Linked List)

초급

기본 구조

각 노드(node)는 두 부분: 데이터 + 다음 노드 포인터.

head → [A|·] → [B|·] → [C|·] → [D|·] → None
             ↑            ↑           ↑           ↑
            data    data    data    data
            next    next    next    next

head 포인터로 리스트의 시작을 안다. 마지막 노드의 next는 None(또는 NULL).

중급

Python 구현 (교육용)

singly_linked.py
class Node:
    def __init__(self, data, next=None):
        self.data = data
        self.next = next


class LinkedList:
    def __init__(self):
        self.head = None
        self.size = 0

    def push_front(self, x):   # O(1)
        self.head = Node(x, self.head)
        self.size += 1

    def pop_front(self):       # O(1)
        if not self.head: raise IndexError
        x = self.head.data
        self.head = self.head.next
        self.size -= 1
        return x

    def find(self, target):    # O(n)
        cur = self.head
        while cur:
            if cur.data == target: return cur
            cur = cur.next
        return None

    def __getitem__(self, i):  # O(i) — 배열과 결정적 차이
        cur = self.head
        for _ in range(i):
            cur = cur.next
        return cur.data
고급

각 연산의 복잡도 — 양 끝의 비대칭

pop_front (앞 삭제)O(1)O(n) (pop(0))
push_back (뒤 삽입)O(n) (끝까지 가야)O(1) amortized

단일 연결 리스트가 array를 이기는 곳: 앞 삽입/삭제. 그게 전부입니다. 나머지는 array가 이기거나 비슷.

WHY — push_back을 어떻게 O(1)로?

tail 포인터를 따로 유지하면 됩니다. head와 tail 둘 다 저장.

  • push_back: tail.next = Node(x); tail = tail.next. O(1).
  • 하지만 pop_back은 여전히 O(n). 왜? tail의 prev를 모르니 끝까지 다시 순회해야.

이 문제를 풀려면 이중 연결 리스트로 가야 함 (§04).

03 — 왜 Python은 단일 연결 리스트가 없는가

중급

이유 1 — 캐시 지역성

현대 CPU의 구조:

  • L1 캐시: ~32KB, 접근 1~3 사이클 (~1ns).
  • L2 캐시: ~256KB, 접근 10~20 사이클.
  • L3 캐시: ~8MB, 접근 40~50 사이클.
  • DRAM: 무제한, 접근 100~300 사이클 (캐시의 100배 느림).

CPU가 메모리를 읽을 때 64바이트 "캐시 라인"을 한 번에 가져옴. 배열의 한 원소를 읽으면 옆 7~8개도 공짜로 캐시에 들어옴 (int 기준).

반면 linked list 순회는 매 노드마다 다른 메모리 위치로 점프. 매번 캐시 미스 가능성. 이론 O(n)이라도 실측 10~100배 느림.

고급

이유 2 — Python 객체 overhead

Python에서 linked list를 구현한다면 각 Node가 Python 객체. 오버헤드:

  • PyObject 헤더: 16바이트 (refcount + type pointer).
  • __dict__ 또는 __slots__: 속성 저장소.
  • 각 Node마다 malloc 호출: 수 μs (bulk allocation 없음).

실제 데이터 1개당 수십~수백 바이트 overhead. list는 ob_item 하나에 포인터만 압축해 저장해 효율 높음.

고급

이유 3 — list로 대부분 충분하다

linked list의 킬러 장점은 "앞/뒤 O(1)"인데, 이건 deque가 해결. 중간 삽입은 어떤 자료구조든 어렵고, 진짜 필요하면 tree나 skip list 같은 고급 자료구조.

그래서 Python 표준 라이브러리의 자료구조:

  • list: 동적 배열. 대부분 경우의 답.
  • deque: 양 끝 O(1) 필요할 때. Block-linked list 내부.
  • dict / set: O(1) 조회. Hash table (Algo 10).
  • heapq: 우선순위 큐. Binary heap (Algo 12).

단일 linked list는 — 이 표에 없습니다. 필요하면 직접 만들어 쓰세요가 Python의 입장.

📖 유명한 인용 — Bjarne Stroustrup (C++ 창시자)

"I have often been asked which is better: vector or list? Most of the time, in an experienced programmer's code, you will find something surprising. Why? It is because cache locality matters enormously..."

— Bjarne Stroustrup, GoingNative 2012 keynote. vector가 linked list보다 삽입/삭제에서도 빠를 수 있다는 유명한 벤치마크 발표.

04 — 이중 연결 리스트 (Doubly Linked List)

중급

양방향 포인터

각 노드가 next와 prev 둘 다 가지는 구조.

None ← [·|A|·] ↔ [·|B|·] ↔ [·|C|·] ↔ [·|D|·] → None
        ↑                                        ↑
       head                                 tail

head + tail 두 포인터 유지. 양 끝에서의 삽입/삭제 O(1) 보장.

고급

트레이드오프

  • 장점: 양 끝 O(1) + "노드 포인터가 있으면 그 위치에서 제거/삽입 O(1)".
  • 단점: 메모리 2배 (prev + next). 코드 복잡도 증가 (모든 삽입·삭제에 양쪽 포인터 업데이트).

실무에서 이중 연결 리스트가 빛나는 대표 사례: LRU Cache. 해시로 노드 포인터를 O(1)에 찾고, 포인터로 즉시 O(1) 이동. Algo 10 맛보기 + §11 참조.

Python 표준 라이브러리에서 숨은 이중 연결 리스트

  • collections.OrderedDict: Python 3.7 전까지 이중 연결 리스트 + dict. 3.7+에서 dict 자체가 삽입 순서 유지하면서 단순화.
  • functools.lru_cache: 현재도 순환 이중 연결 리스트 + dict. C 구현.

05 — Stack ADT — list로 충분하다

초급

Stack이란

LIFO (Last In, First Out). 마지막에 넣은 것이 가장 먼저 나옴. 접시 쌓기 비유.

  • push(x): 맨 위에 추가.
  • pop(): 맨 위 제거 + 반환.
  • peek() or top(): 맨 위 조회.
중급

Python의 stack — 그냥 list

별도 클래스 불필요. list의 append와 pop이 amortized O(1).

stack_with_list.py
stack = []
stack.append(1)       # push. O(1) amortized
stack.append(2)
stack.append(3)
top = stack[-1]       # peek. O(1)
x = stack.pop()      # pop. O(1). 반환값 3

피해야 할 것: stack.insert(0, x) + stack.pop(0) — 이건 queue 패턴이고 O(n).

Stack의 고전 활용

  • 함수 호출 스택: 재귀 호출 → 스택 프레임.
  • 괄호 균형 검사: ({[]})가 올바른지.
  • DFS (깊이 우선 탐색): 재귀 또는 명시적 스택.
  • Undo 기능: 에디터의 Ctrl-Z.
  • 수식 평가: 후위 표기법(RPN) 계산.
고급

괄호 균형 검사 예제

balance_parens.py
def is_balanced(s):
    stack = []
    pairs = {')': '(', ']': '[', '}': '{'}
    for c in s:
        if c in "([{":
            stack.append(c)
        elif c in pairs:
            if not stack or stack[-1] != pairs[c]:
                return False
            stack.pop()
    return not stack


print(is_balanced("({[]})"))    # True
print(is_balanced("([)]"))       # False
print(is_balanced("((("))         # False

O(n) 시간 + O(n) 공간. 단순하면서 강력한 스택 활용 예.

06 — Queue ADT — deque가 필수다

중급

Queue란

FIFO (First In, First Out). 먼저 들어간 것이 먼저 나옴. 줄서기 비유.

  • enqueue(x): 뒤에 추가.
  • dequeue(): 앞에서 제거 + 반환.
  • front(): 앞 조회.
중급

collections.deque 사용

queue_with_deque.py
from collections import deque

q = deque()
q.append(1)          # enqueue. O(1)
q.append(2)
q.append(3)
front = q[0]         # peek. O(1)
x = q.popleft()     # dequeue. O(1)

deque는 "double-ended queue"의 약자. 양 끝 모두 O(1)이므로 stack으로도, queue로도, 양방향 queue로도 쓸 수 있음.

고급

BFS — Queue의 고전 활용

bfs.py
from collections import deque

def bfs(graph, start):
    visited = {start}
    queue = deque([start])
    while queue:
        node = queue.popleft()       # O(1)
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append(neighbor)    # O(1)
    return visited

만약 list를 썼다면?: queue.pop(0)이 O(n). n개 노드 BFS가 O(n²). Algo 14에서 그래프를 다룰 때 이게 왜 중요한지 봅니다.

⚠️ queue.Queue 모듈과 혼동 금지

Python 표준 라이브러리에 queue라는 별도 모듈이 있습니다 (queue.Queue, queue.LifoQueue, queue.PriorityQueue).

  • 이 모듈은 스레드 간 통신용. 내부 락(lock) 포함, 훨씬 느림.
  • 단일 스레드 알고리즘에서는 절대 쓰지 말 것 — collections.deque가 10~100배 빠름.
  • producer-consumer 패턴처럼 스레드 안전 필요할 때만 queue.Queue.

07 — CPython deque: _collectionsmodule.c

전공자

Part A — 핵심 구조체

Modules/_collectionsmodule.c — 발췌
/* The block length can be any positive integer. Larger numbers reduce the
 * number of calls to the memory allocator, give faster indexing and rotation,
 * and reduce the link-to-data overhead ratio. Setting the block length to a
 * power of two speeds-up the modulo and division calculations. */
#define BLOCKLEN 64
#define CENTER ((BLOCKLEN - 1) / 2)

typedef struct BLOCK {
    struct BLOCK *leftlink;
    PyObject *data[BLOCKLEN];
    struct BLOCK *rightlink;
} block;

typedef struct {
    PyObject_VAR_HEAD
    block *leftblock;
    block *rightblock;
    Py_ssize_t leftindex;    /* 0 <= leftindex < BLOCKLEN */
    Py_ssize_t rightindex;   /* 0 <= rightindex < BLOCKLEN */
    size_t state;
    Py_ssize_t maxlen;
    size_t numfreeblocks;
    block *freeblocks[MAXFREEBLOCKS];
    PyObject *weakreflist;
} dequeobject;
  • BLOCKLEN = 64: 각 block이 64개의 PyObject 포인터 저장.
  • block: 이중 연결 (leftlink, rightlink). 즉 blocks가 doubly-linked list.
  • leftblock, rightblock: deque의 양 끝에 있는 block 포인터.
  • leftindex, rightindex: 각 끝 block 내에서의 인덱스.
  • freeblocks: 재사용 가능한 빈 block들의 pool. malloc 호출을 줄임.
전공자

Part B — 왜 "block-linked"인가

교과서 이중 연결 리스트는 각 노드마다 데이터 1개 + 포인터 2개:

전통 linked list: 노드당 (data 8바이트 + prev 8 + next 8) = 24바이트
→ data:metadata 비율 = 1:2 (67% overhead)

CPython deque는 block 하나가 64개 데이터 + 포인터 2개:

Block-linked: block당 (data 64·8 = 512 + prev 8 + next 8) = 528바이트
→ data:metadata 비율 = 512:16 = 32:1 (3% overhead)

20배 이상의 메모리 효율. 그리고 block 안의 64개 원소는 연속 메모리라 캐시 친화적.

📖 원문 — _collectionsmodule.c 주석

"Data for deque objects is stored in a doubly-linked list of fixed-length blocks. ... Another advantage is that it completely avoids use of realloc(), resulting in more predictable performance. Textbook implementations of doubly-linked lists store one datum per link, but that gives them a 200% memory overhead (a prev and next link for each datum) and it costs one malloc() call per data element. By using fixed-length blocks instead, the memory overhead is significantly reduced."

— CPython 소스 주석, Raymond D. Hettinger 유지 보수.

08 — Block-Linked List: BLOCKLEN=64의 천재성

전공자

Part A — 왜 64인가?

64 = 2⁶ 선택의 이유:

  • 캐시 라인 정렬: 대부분 CPU의 캐시 라인이 64바이트. 64개 8바이트 포인터 = 512바이트 = 8 캐시 라인. 정렬이 깔끔.
  • 2의 거듭제곱: 나눗셈과 modulo가 비트 연산으로 고속화 (i / 64 → i >> 6, i % 64 → i & 63).
  • data:metadata 3%: 충분히 낮은 overhead.
  • 실측 최적: 너무 크면 (예: 1024) 작은 deque에서 과할당. 너무 작으면 (예: 8) overhead 증가. 64가 균형점.
전공자

Part B — append 연산의 흐름

d.append(x)를 호출하면:

  1. rightindex가 BLOCKLEN − 1이 아니면 (= rightblock 내 여유 있음): rightblock->data[++rightindex] = x. O(1).
  2. rightblock이 꽉 찼다면:
    • freeblocks에 여유 block이 있으면 하나 꺼냄.
    • 없으면 malloc으로 새 block 할당.
    • rightblock의 rightlink로 연결.
    • 새 block의 data[0]에 x 저장.

핵심 관찰: list의 append와 달리 realloc으로 전체 복사 없음. 새 block만 추가. "최악도 O(1)" (amortized 아니라 단순 O(1)).

deque.append: 최악 O(1) (list는 amortized O(1))

이 차이는 실시간 시스템에서 중요: list의 "가끔 매우 느린 append"가 허용 안 되는 환경에서 deque는 예측 가능한 성능 제공.

전공자

Part C — indexing d[i]의 영리한 트릭

deque에서 d[i]의 복잡도는 원리적으로 O(min(i, n−i)) — 양 끝에서 가까운 쪽부터 순회. 양 끝은 O(1), 중앙은 O(n/2) (여전히 선형). "상수가 작을" 뿐, O(1) 보장은 없습니다.

구현 세부: BLOCKLEN=64이고 block이 doubly-linked이므로:

  • block index = i / 64 (또는 가까운 끝 기준).
  • block 내부 offset = i % 64.
  • CPython은 i < n/2이면 leftblock부터, 아니면 rightblock부터 순회 (소스의 deque_item 참조).
  • d[0], d[-1]은 block 순회 없이 즉시 접근 → O(1).
  • d[n/2]는 약 n/2 / 64 block 순회. 점근적으로 여전히 O(n)이지만 상수가 배열보다 64배 큼.

결론: random access는 list를 쓰세요. deque는 양 끝 O(1)이 목적이고, 중간 접근은 보조적. 중간 접근이 많은 패턴이면 자료구조 선택을 재고.

WHY — Raymond Hettinger의 설계 철학

Raymond Hettinger가 Python 2.4(2004)에 deque를 CPython에 추가하며 다음 원칙을 세웠습니다:

  1. 양 끝 O(1): append/appendleft/pop/popleft 모두 최악 O(1).
  2. 예측 가능한 성능: realloc 없음. 가비지 컬렉션 spike 없음.
  3. 메모리 효율: block overhead 최소화.
  4. 캐시 친화: block 내부 64개는 연속 메모리.
  5. list와 구별: indexing은 list 쓰라고 유도 (O(1)이 아님을 명시).

이 설계가 20년 간 변하지 않은 이유 — 자료구조 설계의 교과서적 성공 사례.

전공자

rotate의 마법 — 스마트한 O(k)

d.rotate(k): deque를 k칸 오른쪽으로 회전 (음수면 왼쪽). 예: deque([1,2,3,4,5]).rotate(2) → [4,5,1,2,3].

순진한 구현은 O(n). CPython은 O(min(k, n−k)) — block 단위(64원소)로 memmove하므로 상수가 매우 작지만, 원리적으로는 블록 일부를 실제로 이동합니다 (deque는 circular buffer가 아니라 block-doubly-linked list). 한 바퀴 가까운 rotate는 반대 방향이 더 쌉니다.

덕분에 rotate는 순환 이동, 큐 회전 등에 쓸 수 있는 강력한 연산. 고정 크기 deque에서 원형 버퍼 시뮬레이션(§10).

⚠️ extendleft의 함정

d.extendleft([1, 2, 3])은 각 원소를 "왼쪽에 추가"하므로 순서가 뒤바뀝니다. 결과: [3, 2, 1, ...] 순서로 앞에 붙음.

"왼쪽에 원래 순서로 추가"하려면 d.extendleft(reversed([1,2,3]))을 써야 함.

11 — LRU Cache: 이중 연결 리스트 + dict

고급

LRU (Least Recently Used) 정책

고정 크기 캐시에서 "가장 오래 사용 안 한 항목을 제거" 정책. 캐시 hit 시 그 항목을 "가장 최근 사용"으로 이동.

  • O(1) get(key): 캐시에 있으면 반환 + 최근으로 이동.
  • O(1) put(key, value): 삽입. 크기 초과면 가장 오래된 것 제거.
고급

자료구조 조합 — "hashmap + doubly linked list"

"왜 둘 다 필요한가?"

  • dict만: O(1) 조회 OK, 하지만 "가장 오래된 항목"을 O(1)에 찾을 수 없음.
  • linked list만: 순서는 유지 OK, 하지만 특정 key를 O(1)에 찾을 수 없음.
  • 둘 다: dict가 key → linked list 노드 포인터 저장. 조회는 dict로, 순서 조작은 linked list 포인터로. 둘 다 O(1).
dict: {key_A → Node_A, key_B → Node_B, ...}
                 ↓ 포인터
DLL: [Node_A] ↔ [Node_B] ↔ [Node_C]
(head = 가장 오래된, tail = 가장 최근)
전공자

Python 구현 (LeetCode #146)

lru_cache.py
class Node:
    def __init__(self, key, val):
        self.key, self.val = key, val
        self.prev = self.next = None


class LRUCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.cache = {}                 # key → Node
        self.head = Node(0, 0)        # sentinel
        self.tail = Node(0, 0)
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_back(self, node):  # tail 직전(= 최신)
        p = self.tail.prev
        p.next = node; node.prev = p
        node.next = self.tail; self.tail.prev = node

    def get(self, key):
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)              # O(1)
        self._add_to_back(node)         # O(1) — 최근으로 이동
        return node.val

    def put(self, key, val):
        if key in self.cache:
            self._remove(self.cache[key])
        node = Node(key, val)
        self.cache[key] = node
        self._add_to_back(node)
        if len(self.cache) > self.capacity:
            oldest = self.head.next     # 가장 오래된
            self._remove(oldest)
            del self.cache[oldest.key]

모든 연산 O(1). 이게 인터뷰 단골 문제이자 — functools.lru_cache의 내부 구조이기도 합니다 (C로 구현).

전공자

12 — AI가 흔히 놓치는 linked list 함정

AI — "linked list 구현해줘"의 4가지 흔한 실수
  1. sentinel 노드 생략: head가 None일 때의 특수 케이스를 매번 체크. 버그 천국. sentinel(가짜 head/tail)을 두면 모든 삽입/삭제가 uniform.
  2. tail 포인터 없이 push_back O(1): "append이니 O(1)" — tail 없으면 O(n). 쉬운 실수.
  3. free된 노드 참조: C/C++라면 use-after-free. Python은 GC 덕에 괜찮지만 로직 버그는 여전.
  4. Reverse에서 next를 덮어씀: cur.next = prev 전에 next_node = cur.next 저장 안 하면 chain 깨짐.
경험 있는 개발자
# sentinel 사용, tail 유지
class LinkedList:
    def __init__(self):
        self.head = Node(None)   # sentinel
        self.tail = self.head      # 빈 리스트는 head=tail
        self.size = 0
    def append(self, x):   # O(1)
        self.tail.next = Node(x)
        self.tail = self.tail.next
        self.size += 1
LLM의 흔한 답
# sentinel 없음, tail도 없음
class LinkedList:
    def __init__(self):
        self.head = None       # 매번 None 체크 필요
    def append(self, x):   # O(n)! 매번 끝까지 순회
        if not self.head:
            self.head = Node(x)
            return
        cur = self.head
        while cur.next:
            cur = cur.next
        cur.next = Node(x)

  1. "sentinel 노드를 썼나? 왜/왜 안 썼나?"
  2. "append의 복잡도는? tail 포인터 유지 여부?"
  3. "reverse 구현 시 next 변수 저장 순서는?"
  4. "그리고 — 진짜 linked list가 필요한가? Python list나 deque로 충분한가?"

4번이 가장 중요: Python 맥락에서 "linked list 구현"은 대부분 필요 없음. 진짜 필요한 건 LRU Cache 같은 특수 구조. 단순 리스트면 list 또는 deque.

13 — 실습: LinkedList + reverse + cycle detection

고급

실습 1 — Reverse a linked list (LeetCode #206)

reverse_list.py
def reverse(head):
    prev = None
    cur = head
    while cur:
        next_node = cur.next    # 1) 다음 저장 (핵심!)
        cur.next = prev         # 2) 현재의 next를 이전으로
        prev = cur              # 3) prev 한 칸 전진
        cur = next_node         # 4) cur 한 칸 전진
    return prev                 # 새 head

O(n) 시간, O(1) 공간. 반드시 next_node를 먼저 저장해야 chain이 안 깨짐.

고급

실습 2 — Floyd's Cycle Detection (Tortoise and Hare)

linked list에 사이클이 있는지 판단 (LeetCode #141). 토끼와 거북이 알고리즘.

cycle_detect.py
def has_cycle(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next         # 1칸 이동
        fast = fast.next.next    # 2칸 이동
        if slow is fast:         # 만나면 사이클
            return True
    return False                # fast가 None에 도달 — 사이클 없음

왜 작동하는가: 사이클이 있으면 fast가 slow를 언젠가 따라잡음 (1칸씩 거리 좁혀짐). 사이클 없으면 fast가 먼저 None.

O(n) 시간, O(1) 공간. set으로 방문 확인하는 방법은 O(n) 공간. Floyd의 알고리즘이 공간 효율성에서 압도적.

전공자

실습 3 — list vs deque vs 수제 LinkedList 벤치마크

benchmark_queues.py
from collections import deque
import timeit


class Node:
    __slots__ = ('data', 'next')
    def __init__(self, data):
        self.data = data
        self.next = None


class SLL:
    def __init__(self):
        self.head = self.tail = None
    def push_back(self, x):
        node = Node(x)
        if not self.head:
            self.head = self.tail = node
        else:
            self.tail.next = node
            self.tail = node
    def pop_front(self):
        x = self.head.data
        self.head = self.head.next
        return x
    def __bool__(self): return self.head is not None


def queue_ops(container, n, push_method, pop_method):
    for i in range(n): push_method(container, i)
    while container: pop_method(container)


n = 100_000
print(f"n = {n}")

t1 = timeit.timeit(lambda: queue_ops(deque(), n,
                   lambda c, i: c.append(i),
                   lambda c: c.popleft()), number=1)
print(f"  deque:   {t1*1000:.0f}ms")

t2 = timeit.timeit(lambda: queue_ops(SLL(), n,
                   lambda c, i: c.push_back(i),
                   lambda c: c.pop_front()), number=1)
print(f"  SLL(수제): {t2*1000:.0f}ms (deque의 {t2/t1:.1f}배)")
예상 출력 (CPython 3.12):
n = 100000
  deque:    8ms
  SLL(수제): 150ms (deque의 19배)

같은 O(1) 연산이지만 deque가 압도적으로 빠름 — C 구현 + block 캐시 지역성. 순수 파이썬 linked list는 객체 생성 overhead로 20배 느림.

교훈: Python에서 "실전 linked list"는 거의 항상 deque. 교육 목적이 아니면 직접 구현하지 말 것.

WHY 문제 1

"linked list는 삽입이 O(1)이라서 배열보다 낫다"라는 주장이 실전에서 틀리는 두 가지 이유를 서술.

풀이 펼치기
답
  1. 캐시 지역성: linked list 순회는 포인터를 따라 임의 메모리로 점프, 캐시 미스 폭증. 배열은 연속 메모리 → 캐시 히트.

답

O(k). 내부 leftindex/rightindex 조정 + block 경계 넘을 때 포인터 조정. k ≥ n이면 k mod n으로 환산하지만 여전히 O(k mod n).

BUILD 문제 6

두 linked list의 k번째 노드 중 큰 값을 찾는 함수 작성 (두 리스트 정렬 X, 사이즈 n).

풀이 펼치기
kth_max.py
def kth_of_two(head1, head2, k):
    def at(h, k):
        for _ in range(k): h = h.next
        return h.data
    return max(at(head1, k), at(head2, k))
복잡도

O(k) × 2 = O(k). 배열이라면 O(1) — linked list 인덱싱의 대표적 비용 예.

WHY 문제 7

LRU cache에 dict만 쓰거나 linked list만 쓰면 어떤 연산이 O(n)이 되는가?

풀이 펼치기
답
  • dict만: "가장 오래된 항목 찾기"가 O(n). 삽입 순서 정보가 없거나 (구 Python), 있어도 "맨 앞" 접근 O(n).
  • linked list만: 특정 key 조회가 O(n). 노드를 head부터 선형 탐색.

둘 다 써야만 O(1) get + O(1) put 달성. hashmap이 key → node 매핑으로 linked list에 O(1) 접근 제공.

DESUGAR 문제 8

@functools.lru_cache와 @functools.cache의 차이는?

풀이 펼치기
답
  • lru_cache(maxsize=128): 크기 제한 있음. 초과 시 LRU로 제거. 이중 연결 리스트 + dict.
  • cache (Python 3.9+): 무한 크기. dict만. 제거 없음. 코드 더 단순, 빠름.

메모리 걱정 없고 입력 다양성 제한적이면 @cache. 입력 공간이 크면 @lru_cache.

PREDICT 문제 9

Floyd's cycle detection에서 fast = fast.next.next인데 fast.next가 None이면? 버그 없이 처리되는가?

풀이 펼치기
답

while 조건 fast and fast.next가 먼저 검사됨. fast.next가 None이면 loop 종료 → fast.next.next 접근 안 함. Python의 단락 평가(short-circuit) 덕분.

만약 조건이 fast만이면 AttributeError: 'NoneType' object has no attribute 'next' 발생.

BUILD 문제 10

`deque(maxlen=3)`으로 최근 3개 값의 평균을 O(1)에 계산하는 클래스 작성 (스트림 입력).

풀이 펼치기
moving_avg.py
from collections import deque

class MovingAverage:
    def __init__(self, window):
        self.window = deque(maxlen=window)
        self.total = 0
    def next(self, x):
        if len(self.window) == self.window.maxlen:
            self.total -= self.window[0]
        self.window.append(x)
        self.total += x
        return self.total / len(self.window)
복잡도

각 next 호출 O(1): deque 양 끝 접근 O(1), 합 갱신 O(1). 전체 스트림 길이 n에 대해 O(n).

AI 문제 11

AI가 "stack을 구현해달라"에 Node 기반 linked list로 답했다. 이 답에 무엇이 문제인가?

풀이 펼치기
답

Python 맥락에서 불필요한 오버엔지니어링. stack은 그냥 list로 — append() / pop()이 O(1). 수제 linked list는 객체 생성 overhead로 실측 10~20배 느림.

"교육용" 또는 "C/C++ 마인드로 접근" 외에는 안 씁니다. 올바른 질문: "Python stack 관용구는 뭐지?" → list.

WHY 문제 12

CPython deque의 block 크기가 64가 아니라 4 또는 1024라면 어떤 문제가 생기는가?

풀이 펼치기
답
  • 4: block overhead 비율 = 32/32 = 50%. 교과서 linked list 수준으로 비효율. 캐시 라인(64바이트)도 채우지 못함.
  • 1024: 작은 deque (예: 10원소)도 1024 × 8 = 8KB 할당. 메모리 낭비. 또한 L1 캐시 초과(32KB 제한)로 캐시 효율 감소.
  • 64: 캐시 라인 8개 정확히 맞음. overhead 3%. 작은 deque도 낭비 적음. 이론+실측의 스위트 스팟.

Raymond Hettinger 2004 (Python 2.4). 최악 O(1) append (list는 amortized). 실시간 시스템 적합.
