01 — 이번 주 이야기: 카드 한 벌 정렬

할머니께 카드 한 벌(52장)을 섞어 드리고 정렬을 부탁드렸습니다.
할머니는 두 가지 방식을 보여주셨습니다.

방식 A — 한 장씩 자기 자리에 끼워넣기 (삽입 정렬)

카드를 한 장씩 뽑아 이미 정렬된 더미의 알맞은 위치에 끼워 넣습니다. 처음엔 빠릅니다 — 카드가 1장뿐이니 비교할 게 없죠. 하지만 더미가 50장이 되면, 51번째 카드를 끼워넣기 위해 최악의 경우 50번 비교해야 합니다.

52장 전체를 정렬하는 데 든 비교는 최악의 경우 약 1350번 (n² / 2 = 52² / 2 ≈ 1352). 카드가 1000장이 되면 50만 번. 100만 장이면 5000억 번.

방식 B — 절반씩 나눠서 정렬한 뒤 합치기 (병합 정렬)

할머니는 카드를 정확히 반으로 나누셨습니다 (26장 + 26장). 그리고 두 자녀에게 한 더미씩 주시며 "각자 정렬하고 가져와" 라고 하셨습니다. 자녀들은 자기 26장을 또 13장씩 나눠 손주들에게 보냈습니다. 손주들은 또 7장과 6장으로… 1장이 되면 그건 이미 정렬된 것이므로 그대로 돌려보냅니다.

되돌아오는 더미들은 항상 이미 정렬된 두 더미입니다. 두 더미를 합치는 일은 맨 위 카드만 비교하면 되니, 26+26장 합치는 데 약 51번 비교면 끝. 52장 전체에서 든 비교는 약 300번 (n log₂ n ≈ 52 × 5.7 ≈ 296). 100만 장이면 2000만 번. 방식 A의 25,000배 빠릅니다.

WHY — 같은 정렬, 같은 컴퓨터, 25,000배 차이

두 방식 모두 "비교하고 옮긴다"는 같은 도구를 씁니다. 차이는 "문제를 어떻게 쪼개는가"뿐입니다.

  • 방식 A: 한 번에 1칸씩 줄어드는 문제 (n → n−1 → n−2 → …)
  • 방식 B: 한 번에 절반씩 줄어드는 문제 (n → n/2 → n/4 → …)

"1씩 줄어드는 문제"는 n번 만에 끝나고, 매 단계가 O(n) 일이니 O(n²).
"절반씩 줄어드는 문제"는 log n번 만에 끝나고, 매 단계의 합이 O(n) 일이니 O(n log n).
이번 주 우리는 이 마법의 정확한 정체를 봅니다.

병합 정렬의 재귀 트리. 깊이 log₂ 8 = 3, 각 깊이에서 총 O(n) 일 → 전체 Θ(n log n).
나눠서 정복하기 (Divide & Conquer)의 힘. 큰 문제 → 작은 문제 → 더 작은 → … (1개) → 합치고 → 더 합치고 → … → 정답. 같은 작업을 반복하지만 매번 절반이라 폭발적으로 빠릅니다.

02 — 왜 분할 정복인가? 절반의 마법

초급

반으로 쪼개면 왜 빠른가?

도서관에서 책을 찾을 때 처음부터 끝까지 한 권씩 보는 사람과 가운데를 펼쳐 절반을 버리는 사람이 있다고 했죠 (Algo 01의 이야기).

분할 정복은 그 아이디어의 일반화입니다. "한 권씩 줄여나가는" 대신 "절반씩 쪼개서 동시에 풀고, 결과를 합친다."

쪼개진 두 부분은 같은 종류의 문제지만 크기가 절반입니다. 그래서 같은 함수를 자기 자신에게 다시 써먹을 수 있습니다 — 이걸 재귀(recursion)라고 부릅니다.

중급

재귀 ≠ 분할 정복 — 차이를 분명히

모든 분할 정복은 재귀를 쓰지만, 모든 재귀가 분할 정복은 아닙니다.

구분재귀분할 정복
핵심 조건 자기 자신을 호출 자기 자신을 여러 번 호출 + 결과를 합침
예 factorial(n) = n * factorial(n-1) merge_sort(A) = merge(merge_sort(L), merge_sort(R))
크기 감소 대개 1씩 (n−1) 대개 비례적 (n/2, n/3 …)
전형적 복잡도 O(n) 또는 O(n²) O(n log n) 또는 그보다 빠른 변종

factorial(n) = n * factorial(n-1) 은 자기 자신을 한 번만 부르고, "두 결과를 합치는" 단계가 의미 있게 없습니다 (그냥 곱하기 한 번). 이런 건 보통 선형 재귀(linear recursion)라고 부르고, 분할 정복으로 분류하지 않습니다.

고급

절반의 수학적 정체 — log n의 의미

n개를 절반으로 계속 쪼개면, 1이 될 때까지 몇 번 쪼개야 할까요?

n → n/2 → n/4 → … → n/2k = 1
⟺ 2k = n  ⟺  k = log₂ n

그래서 분할 정복의 깊이는 늘 log n 수준입니다. 매 깊이에서 하는 일이 O(n)이라면 전체는 O(n log n).

이게 단순한 직관이지만, 정확한 증명을 위해서는 점화식(recurrence relation)이 필요합니다. 병합 정렬의 점화식은

T(n) = 2·T(n/2) + Θ(n),   T(1) = Θ(1)

이고, 마스터 정리(CLRS §4.5)로 풀면 정확히 Θ(n log n)입니다. 이번 주 §06에서 두 가지 방법으로 직접 풀어봅니다.

📖 원서 인용 — CLRS 4판, §2.3 도입부

"Many useful algorithms are recursive in structure: to solve a given problem, they call themselves recursively one or more times to deal with closely related subproblems. These algorithms typically follow a divide-and-conquer approach: they break the problem into several subproblems that are similar to the original problem but smaller in size, solve the subproblems recursively, and then combine these solutions to create a solution to the original problem."

— Cormen, Leiserson, Rivest, Stein. Introduction to Algorithms, 4th ed., MIT Press 2022, §2.3 "Designing Algorithms"

03 — 분할 정복의 3단계: Divide / Conquer / Combine

중급

3단계 템플릿

CLRS는 분할 정복 알고리즘을 다음 세 단계로 정의합니다.

  1. Divide (분할) — 문제를 같은 종류의 더 작은 부분문제 여러 개로 쪼갠다. 쪼갤 수 없을 만큼 작은 경우(base case)는 직접 푼다.
  2. Conquer (정복) — 부분문제를 재귀적으로 해결한다.
  3. Combine (결합) — 부분문제의 해를 합쳐 원래 문제의 해를 만든다.
function DC(P):
  if size(P) ≤ threshold:
    return solve_directly(P)   // base case
  (P₁, P₂, …, Pₖ) = divide(P)   // Divide
  S₁ = DC(P₁);   S₂ = DC(P₂); …   // Conquer (재귀)
  return combine(S₁, S₂, …, Sₖ)   // Combine
고급

왜 base case가 반드시 필요한가

base case가 없으면 재귀가 영원히 자기 자신을 부릅니다 — Python에서는 RecursionError가 나거나 (기본 한계 1000), C에서는 stack overflow로 크래시합니다.

더 본질적으로, base case는 귀납법의 기저(base of induction)입니다. Algo 02에서 본 강귀납법(strong induction)으로 정확성을 증명할 때, base case가 P(n₀)에 해당합니다. base case를 잘못 작성하면 — 예를 들어 빈 배열을 빈 배열 두 개로 쪼개면 — 무한 루프에 빠집니다.

⚠️ 분할 정복의 흔한 함정
  • Divide가 진짜로 작아지지 않음: divide([x])가 ([x], [])를 반환하면 영원히 같은 크기 호출.
  • base case 누락: 빈 배열, 한 원소 배열에서 종료 조건 안 잡으면 stack overflow.
  • Combine이 비싸면 무용지물: 합치는 데 O(n²) 들면 전체가 O(n²)이 됩니다.
  • 중복 부분문제: 같은 부분문제를 여러 번 풀면 (피보나치 재귀처럼) 지수 시간. → 동적 계획법(Algo 20).
WHY — Combine이 알고리즘의 진짜 심장

"쪼개고 재귀로 풀자"는 누구나 떠올립니다. 분할 정복의 진짜 어려움은 Combine입니다.

  • 병합 정렬의 Combine = "정렬된 두 배열을 합치기" → O(n).
  • 퀵소트의 Combine = "원래 자리에서 합쳐짐" → O(1). (대신 Divide가 O(n))
  • 최대 부분배열의 Combine = "왼쪽, 오른쪽, 가운데를 가로지르는 것 중 최댓값" → 이걸 O(n)에 푸는 게 핵심.
  • Strassen의 Combine = "7개의 부분 곱을 더해 4개의 부분 행렬로" → 영리한 대수.

"어떻게 쪼갤지"보다 "어떻게 합칠지"가 알고리즘의 깊이를 결정합니다.

04 — 사례 1: 병합 정렬 (Merge Sort, CLRS §2.3.1)

초급

한눈에 보는 병합 정렬

8개 원소 배열 [5, 2, 4, 7, 1, 3, 2, 6]을 정렬한다고 합시다.

[5,2,4,7,1,3,2,6]
  ↓ 반으로
[5,2,4,7]      [1,3,2,6]
  ↓ 또 반으로
[5,2] [4,7]   [1,3] [2,6]
  ↓ 또 반으로 (1개씩 = 이미 정렬됨)
[5][2] [4][7]   [1][3] [2][6]
  ↓ 합치기 (merge)
[2,5] [4,7]   [1,3] [2,6]
  ↓ 또 합치기
[2,4,5,7]      [1,2,3,6]
  ↓ 마지막 합치기
[1,2,2,3,4,5,6,7] ✅

쪼개는 데 log₂ 8 = 3단계, 매 단계에서 8개 원소를 한 번씩 비교 — 총 약 3 × 8 = 24번 비교. 삽입 정렬은 같은 입력에서 약 8² / 2 = 32번. 작은 n에선 차이가 미미하지만, n이 커질수록 격차가 폭발적으로 벌어집니다.

중급

Python 구현

merge_sort.py
def merge_sort(A):
    # Base case — 0개나 1개는 이미 정렬됨
    if len(A) <= 1:
        return A[:]  # 복사본 반환 (원본 보호)

    # Divide — 정확히 반으로
    mid = len(A) // 2
    L = merge_sort(A[:mid])   # Conquer (왼쪽 재귀)
    R = merge_sort(A[mid:])   # Conquer (오른쪽 재귀)

    # Combine — 정렬된 두 배열을 합치기
    return merge(L, R)


def merge(L, R):
    result = []
    i = j = 0
    while i < len(L) and j < len(R):
        if L[i] <= R[j]:
            result.append(L[i]); i += 1
        else:
            result.append(R[j]); j += 1
    # 한쪽이 끝났다면, 남은 쪽을 그대로 붙임
    result.extend(L[i:])
    result.extend(R[j:])
    return result


print(merge_sort([5, 2, 4, 7, 1, 3, 2, 6]))
출력:
[1, 2, 2, 3, 4, 5, 6, 7]

주의: L <= R이 아니라 L[i] <= R[j]를 비교합니다. 그리고 <= (등호 포함) 으로 써야 안정 정렬(stable sort)이 됩니다 — 같은 키의 원소들이 원래 순서를 유지합니다.

전공자

CLRS의 의사코드 (Part A)

CLRS는 in-place에 가까운 형태로 의사코드를 씁니다 (보조 배열만 임시 사용). A[p..r]은 인덱스 p부터 r까지의 부분배열을 뜻합니다.

CLRS §2.3.1 의사코드 (Python으로 옮김)
def merge_sort_inplace(A, p, r):
    if p < r:
        q = (p + r) // 2            # Divide
        merge_sort_inplace(A, p, q)
        merge_sort_inplace(A, q + 1, r)
        merge_inplace(A, p, q, r)     # Combine


def merge_inplace(A, p, q, r):
    # A[p..q]와 A[q+1..r]은 이미 정렬됨
    L = A[p:q+1]
    R = A[q+1:r+1]
    # Sentinel(보초): CLRS 원서는 무한대 sentinel을 두지만, Python에선 인덱스 검사로 충분
    i = j = 0
    k = p
    while i < len(L) and j < len(R):
        if L[i] <= R[j]:
            A[k] = L[i]; i += 1
        else:
            A[k] = R[j]; j += 1
        k += 1
    while i < len(L):
        A[k] = L[i]; i += 1; k += 1
    while j < len(R):
        A[k] = R[j]; j += 1; k += 1
📖 원서 인용 — CLRS 4판, §2.3.1 "The divide-and-conquer method"

"The merge-sort algorithm closely follows the divide-and-conquer paradigm. Intuitively, it operates as follows. Divide the subarray A[p:r] to be sorted into two adjacent subarrays, each of half the size. Conquer by sorting each of the two subarrays A[p:q] and A[q+1:r] recursively using merge sort. Combine by merging the two sorted subarrays A[p:q] and A[q+1:r] back into A[p:r], producing the sorted answer."

Part B — Python 표준라이브러리는 왜 병합 정렬을 안 쓰는가?

list.sort()는 Timsort를 씁니다 (Tim Peters, 2002). Timsort는 병합 정렬을 베이스로 하지만:

  • 이미 정렬된 부분(natural run) 감지: 입력에 정렬된 구간이 있으면 그걸 그대로 활용해 best case가 O(n).
  • 작은 run은 binary insertion sort로 정렬 (32~64개 미만).
  • Galloping mode: 한쪽에서 연속해 이기면 이진 탐색으로 점프해 비교 횟수 감소.

소스: cpython/Objects/listobject.c의 list_sort_impl. Algo 07에서 정밀하게 다룹니다.

05 — 병합의 정확성: 루프 불변식

고급

왜 merge가 옳은가? (Algo 02의 도구로 증명)

병합 정렬의 정확성은 두 부분으로 나뉩니다.

  1. merge 함수가 옳다 — 정렬된 L, R을 받아 정렬된 합본을 반환한다.
  2. 재귀가 옳다 — base case가 옳고, 귀납 단계가 옳다.

merge 함수의 사전조건과 사후조건

{ L과 R은 비내림차순 정렬됨 }
   result = merge(L, R)
{ result는 비내림차순 정렬, 그리고 multiset(result) = multiset(L) ∪ multiset(R) }

루프 불변식 (Loop Invariant)

I(i, j, k): 매 while 반복 직전에
(a) result[0..k-1]은 비내림차순 정렬됨,  
(b) multiset(result[0..k-1]) = multiset(L[0..i-1]) ∪ multiset(R[0..j-1]),  
(c) result의 마지막 원소(있다면)는 L[i], R[j] 어느 것보다도 작거나 같음.

3속성 증명

  • 초기 (Initialization): i = j = k = 0. result[0..-1]은 빈 시퀀스이므로 (a)(b)(c) 모두 자명히 성립.
  • 유지 (Maintenance): 가정 — 반복 직전 I 성립. L[i] <= R[j]인 경우 L[i]를 result에 append. (c)에 의해 직전 result의 마지막 원소 ≤ L[i]이므로 (a) 유지. (b)는 L의 인덱스가 1 늘었고 result도 같은 원소가 추가됐으므로 유지. (c)는 새로 추가된 원소가 L[i]이고, 다음 후보는 L[i+1]과 R[j]인데 L 정렬성으로 L[i] ≤ L[i+1], 비교 조건으로 L[i] ≤ R[j]이므로 유지.
  • 종료 (Termination): 루프 종료 조건은 i >= len(L) 또는 j >= len(R). 남은 원소들을 extend 하면, 남은 쪽은 이미 정렬되어 있고 (c)에 의해 result의 마지막 원소보다 모두 크거나 같으므로 정렬성이 유지. 그리고 (b)에 의해 모든 원소가 정확히 한 번씩 들어감 → multiset(result) = multiset(L) ∪ multiset(R). ∎
전공자

재귀의 정확성 — 강귀납법

주장 P(n): 길이 n인 임의의 배열 A에 대해 merge_sort(A)는 A를 비내림차순 정렬한 결과를 반환한다.

Base: P(0), P(1). len(A) <= 1일 때 함수는 A[:]를 반환. 빈 배열과 한 원소 배열은 정의상 정렬됨. ✓

귀납 (강귀납법): 모든 k < n에 대해 P(k)를 가정. 임의의 길이 n ≥ 2 배열 A를 생각.

  • mid = n // 2이므로 1 ≤ mid < n, 1 ≤ n − mid < n.
  • 귀납 가정에 의해 L = merge_sort(A[:mid])는 정렬됨, R = merge_sort(A[mid:])는 정렬됨.
  • 위에서 증명한 merge의 정확성에 의해 merge(L, R)는 정렬됨, multiset도 보존됨.
  • multiset(L) ∪ multiset(R) = multiset(A[:mid]) ∪ multiset(A[mid:]) = multiset(A). 따라서 P(n) 성립. ∎
WHY — 왜 multiset을 따로 챙겨야 하는가?

"정렬되었다"는 사실 두 가지를 동시에 요구합니다: (1) 비내림차순이고 (2) 원소 구성이 원래와 같다. 둘 중 하나만 만족하면 잘못된 답이 됩니다.

  • (1)만 만족: return [1, 2, 3] — 어떤 입력이든 정렬된 답을 주지만 multiset이 다름. 명백한 오답.
  • (2)만 만족: return A.copy() — 원소는 같지만 정렬 안 됨. 역시 오답.

"merge가 옳다"를 증명하려면 두 가지 모두가 불변식에 들어가야 합니다. Algo 02에서 본 "적은 불변식은 약한 보장만 준다"의 좋은 예입니다.

06 — 병합 정렬의 복잡도: 점화식과 재귀 트리

고급

점화식 세우기

merge_sort(A)가 길이 n에 대해 하는 일을 그대로 식으로 옮기면:

  • 크기가 1 이하면 상수 시간: T(1) = Θ(1).
  • 그 외엔: 절반 크기 두 번 재귀 + 길이 n의 병합 + 슬라이싱(O(n)).
T(n) = 2·T(n/2) + Θ(n),   T(1) = Θ(1)

이걸 푸는 방법은 두 가지(다음 §10에서 자세히): 마스터 정리로 직접 풀거나, 재귀 트리로 시각화.

고급

재귀 트리로 풀기

각 깊이 ℓ에서 호출 개수와 호출당 일을 표로 정리하면:

깊이 ℓ호출 개수호출당 크기호출당 일 (병합)층 합계
01ncncn
12n/2c·n/2cn
24n/4c·n/4cn
…………cn
log₂nn1ccn

층마다 합계가 항상 cn이고, 층 수는 log₂ n + 1. 따라서 총합 = cn(log₂ n + 1) = Θ(n log n).

T(n) = Θ(n log n)
전공자

마스터 정리로 검증

마스터 정리(CLRS §4.5)의 형태: T(n) = a·T(n/b) + f(n).
병합 정렬: a = 2, b = 2, f(n) = Θ(n). 그러면 nlogba = nlog₂ 2 = n.

  • f(n) = Θ(nlogba) = Θ(n) → Case 2가 적용.
  • Case 2 결론: T(n) = Θ(nlogba · log n) = Θ(n log n). ✓

두 방법이 같은 답을 줍니다 — 점화식의 해는 한 가지뿐이므로.

공간 복잡도

Python 구현은 O(n) 추가 공간을 씁니다 (보조 배열). 재귀 호출 스택의 깊이가 log n이니, 보조 공간 총합은 O(n) + O(log n) = O(n).

"in-place merge sort"라는 변종이 있지만, O(1) 보조 공간으로 merge 하는 알고리즘은 매우 복잡하고 (Kronrod 1969) 실용에서는 거의 안 씁니다 — quicksort가 in-place에선 더 깔끔하기 때문.

⚠️ Python 슬라이싱의 숨은 비용

A[:mid]와 A[mid:]는 새 list 객체를 만듭니다 — O(n) 시간. 이론적 점화식에는 이미 +Θ(n)이 들어 있어 결론은 같지만, 실제 측정에서는 슬라이싱이 큰 상수 인자입니다. in-place 버전(merge_sort_inplace)이 약 2~3배 빠른 이유.

07 — 사례 2: 최대 부분배열 (Maximum Subarray)

중급

문제 정의

배열 A[0..n-1]이 주어졌을 때, 연속된 부분배열 A[i..j]의 합이 최대가 되는 (i, j)를 찾으시오.

예: A = [-2, 1, -3, 4, -1, 2, 1, -5, 4] → 답은 A[3..6] = [4, -1, 2, 1], 합 = 6.

주의: 모두 음수면 답은 가장 큰 음수 한 개. 빈 부분배열을 허용하면 답은 0 — 이번 글은 비공집합만 고려합니다.

중급

단순 해법 — O(n²)

brute_force.py
def max_subarray_brute(A):
    n = len(A)
    best = A[0]
    for i in range(n):
        s = 0
        for j in range(i, n):
            s += A[j]
            if s > best:
                best = s
    return best

이중 루프로 모든 (i, j) 쌍을 — O(n²). n = 10⁶이면 10¹² 연산, 한참 걸립니다.

고급

분할 정복 해법 — O(n log n)

배열을 가운데에서 둘로 쪼갭니다. 최대 부분배열은 셋 중 하나에 있습니다.

  1. 왼쪽에 완전히 포함됨 → 왼쪽에 재귀.
  2. 오른쪽에 완전히 포함됨 → 오른쪽에 재귀.
  3. 중앙을 가로지름 → 중앙에서 양쪽으로 펼치며 최댓값 찾기 (O(n)).
divide_and_conquer.py
def max_subarray_dc(A, lo, hi):
    if lo == hi:
        return A[lo]                 # Base: 한 원소
    mid = (lo + hi) // 2

    left = max_subarray_dc(A, lo, mid)        # Conquer 왼쪽
    right = max_subarray_dc(A, mid + 1, hi)   # Conquer 오른쪽

    # Combine: 중앙을 가로지르는 최대 — O(hi - lo)
    left_sum, s = float('-inf'), 0
    for i in range(mid, lo - 1, -1):
        s += A[i]
        if s > left_sum: left_sum = s
    right_sum, s = float('-inf'), 0
    for j in range(mid + 1, hi + 1):
        s += A[j]
        if s > right_sum: right_sum = s
    cross = left_sum + right_sum

    return max(left, right, cross)

점화식: T(n) = 2T(n/2) + Θ(n) → 마스터 정리 Case 2 → Θ(n log n). 병합 정렬과 똑같은 점화식이라는 점에 주목. "Combine이 O(n)인 분할 정복"의 전형.

전공자

Kadane의 알고리즘 — O(n)

재미있게도 이 문제는 O(n)에 풀 수 있습니다. 분할 정복이 최선이 아닙니다. (Algo 20에서 다룰 동적 계획법의 가장 단순한 예시이기도 합니다.)

kadane.py — Bentley 1984
def max_subarray_kadane(A):
    best = cur = A[0]
    for x in A[1:]:
        cur = max(x, cur + x)   # 여기서 끝나는 최대합
        best = max(best, cur)
    return best

불변식: 매 반복 직전, cur = "x 직전 인덱스에서 끝나는 최대 부분배열의 합", best = "지금까지 본 최대". 증명은 강귀납법 (Algo 02 §10).

WHY — 분할 정복이 항상 최선은 아니다

분할 정복은 강력한 도구지만 만능이 아닙니다. 최대 부분배열처럼 "왼쪽에서 만든 부분 답을 그대로 오른쪽으로 확장 가능"한 문제는 선형 스캔이 더 빠릅니다. 분할 정복이 진가를 발휘하는 곳은 "부분 답을 합치는 것이 본질적으로 문제의 일부인" 경우입니다 — 정렬, 행렬 곱셈, FFT 등.

교훈: "분할 정복이 가능하다 ≠ 분할 정복이 최선이다." 항상 더 단순한 풀이가 없는지 먼저 따져봅니다.

📖 역사 — 어디서 왔나

최대 부분배열 문제는 1977년 Ulf Grenander가 통계 패턴 인식에서 제기. 분할 정복 O(n log n) 해는 Shamos가 발견, Kadane의 O(n) 알고리즘은 Jay Kadane이 같은 1977년에 (논문 발표는 Bentley 1984 Programming Pearls). CLRS 3판 §4.1에서 분할 정복 해를 자세히 다뤘으나 4판에서는 행렬 곱셈으로 교체됨 — 그래도 교육적으로 중요한 클래식.

08 — 사례 3: 카라츠바 곱셈 (전공자)

전공자

큰 정수 곱셈은 O(n²)인가? — Karatsuba 1962

두 n-자릿수 정수 X, Y의 곱셈을 학교에서 배운 방식대로 하면, 자릿수마다 자릿수를 곱하니 O(n²). 1960년 안드레이 콜모고로프는 "곱셈은 본질적으로 Ω(n²)"이라고 추측했습니다. 그런데 23세의 대학원생 아나톨리 카라츠바가 콜모고로프 세미나에서 단 한 주만에 이 추측을 깨버립니다 (1960년 발견, 1962년 Doklady Akademii Nauk SSSR에 발표).

핵심 아이디어

X = 10m·X₁ + X₀,   Y = 10m·Y₁ + Y₀  (반으로 나누기).

X·Y = 102m·X₁Y₁ + 10m·(X₁Y₀ + X₀Y₁) + X₀Y₀

이 식은 곱셈을 4번 합니다 (X₁Y₁, X₁Y₀, X₀Y₁, X₀Y₀). 그러면 T(n) = 4T(n/2) + O(n) = O(n²) — 본전.

카라츠바의 통찰: 가운데 항 (X₁Y₀ + X₀Y₁)은 다음 식으로 한 번의 곱셈으로 얻을 수 있다.

Z = (X₁ + X₀)(Y₁ + Y₀)  ⟹  X₁Y₀ + X₀Y₁ = Z − X₁Y₁ − X₀Y₀

이미 X₁Y₁와 X₀Y₀를 계산했으니 추가 곱셈은 Z 한 번뿐 — 총 3번의 절반 크기 곱셈.

T(n) = 3·T(n/2) + O(n)  ⟹  T(n) = O(nlog₂ 3) ≈ O(n1.585)

Python 구현

karatsuba.py
def karatsuba(x, y):
    if x < 10 or y < 10:
        return x * y          # Base: 작은 수는 직접

    n = max(len(str(x)), len(str(y)))
    m = n // 2

    high1, low1 = divmod(x, 10**m)
    high2, low2 = divmod(y, 10**m)

    z0 = karatsuba(low1, low2)
    z2 = karatsuba(high1, high2)
    z1 = karatsuba(low1 + high1, low2 + high2) - z2 - z0

    return z2 * 10**(2*m) + z1 * 10**m + z0


print(karatsuba(1234, 5678))
출력:
7006652
CPython 내부 — Python int는 카라츠바를 쓴다

CPython의 정수 곱셈 (longobject.c의 k_mul, KARATSUBA_CUTOFF = 70)은 n이 70 internal digits 이상일 때 카라츠바를 씁니다 (64비트 빌드 기준 digit = 30비트 → 약 2100비트, 십진수로 약 630자리). 그 미만은 학교 곱셈(schoolbook)이 더 빠릅니다 — 카라츠바는 큰 상수 인자가 있어서, 작은 n에서는 오히려 손해.

실험: (10**1000)*(10**1000)는 카라츠바로 거의 즉시. (10**100000)*(10**100000)는 더 발전된 Toom-Cook 3(O(nlog₃ 5) ≈ O(n1.465))로 처리. 가장 큰 곱셈은 SSA(Schönhage-Strassen)가 O(n log n log log n). 2019년 Harvey와 van der Hoeven이 O(n log n) 알고리즘을 발표했지만 아직 실용적이진 않음.

— Donald E. Knuth. The Art of Computer Programming, Vol. 2: Seminumerical Algorithms, 3rd ed., §4.3.3 "How Fast Can We Multiply?"

09 — 사례 4: Strassen 행렬 곱셈 (전공자)

전공자

n×n 행렬 곱셈은 정말로 O(n³)인가? — Strassen 1969

두 n×n 행렬 A, B의 곱 C = A·B. 정의에 따르면 C[i,j] = Σ A[i,k]·B[k,j], 곱셈 횟수 n³ — 명백히 O(n³). 1969년까지 사람들은 이게 최선이라고 생각했습니다.

분할 정복 시도 — 단순한 시도는 본전

n×n 행렬을 4개의 (n/2)×(n/2) 부분행렬로 나누면:

A = [[A₁₁, A₁₂], [A₂₁, A₂₂]]   B = [[B₁₁, B₁₂], [B₂₁, B₂₂]]
C₁₁ = A₁₁B₁₁ + A₁₂B₂₁
C₁₂ = A₁₁B₁₂ + A₁₂B₂₂
C₂₁ = A₂₁B₁₁ + A₂₂B₂₁
C₂₂ = A₂₁B₁₂ + A₂₂B₂₂

8번의 (n/2)×(n/2) 곱셈 + Θ(n²)의 덧셈. 점화식: T(n) = 8T(n/2) + Θ(n²) — 마스터 정리 Case 1 → Θ(n³). 본전.

Strassen의 통찰 — 7번의 곱셈으로

Strassen은 영리한 대수로 곱셈을 7번으로 줄였습니다. 7개의 보조 곱(M₁ ~ M₇)을 정의:

M₁ = (A₁₁ + A₂₂)(B₁₁ + B₂₂)
M₂ = (A₂₁ + A₂₂)·B₁₁
M₃ = A₁₁·(B₁₂ − B₂₂)
M₄ = A₂₂·(B₂₁ − B₁₁)
M₅ = (A₁₁ + A₁₂)·B₂₂
M₆ = (A₂₁ − A₁₁)(B₁₁ + B₁₂)
M₇ = (A₁₂ − A₂₂)(B₂₁ + B₂₂)

그러면:

C₁₁ = M₁ + M₄ − M₅ + M₇
C₁₂ = M₃ + M₅
C₂₁ = M₂ + M₄
C₂₂ = M₁ − M₂ + M₃ + M₆
T(n) = 7·T(n/2) + Θ(n²)  ⟹  T(n) = Θ(nlog₂ 7) ≈ Θ(n2.807)

왜 실용에선 잘 안 쓰는가

  • 큰 상수: 부분 행렬을 18번 더하고 빼야 — n이 작으면 표준 알고리즘이 더 빠름. 보통 n > 100~1000에서 이득.
  • 수치 불안정성: 부동소수 연산에서 부분 행렬의 차(M₃, M₄ 등)는 cancellation 오차를 키움. 표준보다 정확도가 떨어짐.
  • 캐시 친화도: 표준 곱셈은 BLAS로 거의 메모리 대역폭 한계까지 최적화됨. Strassen은 메모리 패턴이 복잡해 캐시에 불리.

NumPy의 @ 연산자는 BLAS 라이브러리의 ?gemm(general matrix multiply)을 호출 — O(n³)이지만 벡터화/SIMD/캐시 최적화로 실용에선 압도적으로 빠릅니다. Strassen은 거의 안 씁니다.

📖 원서 요약 — CLRS 4판, §4.2 "Strassen's algorithm for matrix multiplication"

Strassen의 재귀 알고리즘은 곱 C = A·B를 Θ(nlg 7) 시간에 계산하므로, 충분히 큰 n에서는 단순 Θ(n³) 행렬 곱셈보다 빠르다 — 이것이 §4.2의 핵심 결과. 이후 50년의 후속 연구로 행렬 곱셈의 점근 상한은 계속 낮아져 왔으며, 최근 결과로는 Williams, Xu, Xu, Zhou (2024)의 약 O(n2.371552)가 알려져 있음(원서 외 후속 인용).

WHY — Strassen이 "이론적으로" 중요한 이유

Strassen의 진짜 의의는 빠른 코드가 아니라 "O(n³)이 행렬 곱셈의 한계가 아님"을 보인 것입니다. 이후 50년간 연구자들이 지수를 깎아왔습니다 (현재 약 ω < 2.371552). 이 ω가 n×n 가우스 소거, 역행렬, 행렬식 등 선형대수 거의 모든 것의 복잡도를 결정합니다 (CLRS §28).

10 — 점화식을 푸는 두 가지 방법 (CLRS §4.3, §4.4)

고급

방법 1 — 대입법 (Substitution Method, CLRS §4.3)

"답을 추측하고, 귀납법으로 증명한다."

예: T(n) = 2T(n/2) + n, T(1) = 1을 푼다.

  1. 추측: 재귀 트리 등 직관으로 T(n) = O(n log n)을 추측. 정확히는 T(n) ≤ c·n log n (충분히 큰 n과 적절한 c > 0에 대해)을 보이려 함.
  2. 귀납: n/2에 대해 T(n/2) ≤ c·(n/2) log(n/2)가 성립한다고 가정.
    T(n) = 2T(n/2) + n  ≤  2 · c·(n/2) log(n/2) + n
         = c·n·(log n − 1) + n
         = c·n log n − c·n + n
         ≤ c·n log n   (c ≥ 1이면)
  3. Base 확인: T(1) = 1 ≤ c·1·log 1 = 0 — 실패. 그래서 T(2) 같은 다른 base를 쓰면: T(2) = 2T(1) + 2 = 4, c·2 log 2 = 2c. c ≥ 2면 OK.

결론: T(n) = O(n log n). 하한도 비슷하게 보이면 T(n) = Θ(n log n).

⚠️ 대입법의 흔한 함정 — "정확한" 형태로 추측해야 한다

T(n) ≤ c·n log n을 직접 증명하려다 막히는 경우, 종종 더 강한 가설로 시도해야 합니다. 예: T(n) ≤ c·n log n − d·n (낮은 차수 항 빼기). 그러면 귀납이 깔끔하게 통과합니다. 직관적으로: "낮은 차수의 노이즈"를 미리 빼놓아야 귀납이 본 항을 잡을 수 있습니다. CLRS §4.3에 자세한 예. 처음 보면 헷갈리는 부분.

§06에서 병합 정렬을 풀었던 방식. 트리를 그려 깊이별 일을 모두 더한다.

예: T(n) = 3T(n/4) + cn² (Strassen스러운 형태).

11 — 마스터 정리를 넘어서: Akra-Bazzi (전공자)

전공자

마스터 정리가 못 푸는 점화식

마스터 정리는 같은 크기로 균등 분할되는 경우(T(n/b))만 다룹니다. 다음 같은 식은 막힙니다.

T(n) = T(n/3) + T(2n/3) + Θ(n)   // 불균등 분할 (median-of-3 quicksort 등)
T(n) = 2T(n/2) + n / log n   // f(n)이 polylog
T(n) = T(n/2) + T(n/4) + T(n/8) + n   // 부분문제가 셋 다 다름

Akra-Bazzi 정리 (Akra & Bazzi 1998)

T(n) = Σᵢ aᵢ·T(n/bᵢ) + g(n) 형태에서 (단, aᵢ > 0, bᵢ > 1):

Σᵢ aᵢ·bi−p = 1을 만족하는 p (유일하게 존재) 를 찾는다. 그러면:

T(n) = Θ(np · (1 + ∫₁n g(u)/up+1 du))

예 1: T(n) = T(n/3) + T(2n/3) + n. (1/3)p + (2/3)p = 1을 풀면 p = 1. 적분 ∫₁n u/u² du = ln n. 따라서 T(n) = Θ(n · (1 + ln n)) = Θ(n log n).

예 2: T(n) = 2T(n/2) + n / log n. p = 1, 적분 ∫₁n 1/(u log u) du = log log n — 매우 느리게 증가. 결과는 Θ(n log log n).

📖 원서 인용 — CLRS 4판, §4.7 "Akra-Bazzi recurrences"

"The Akra-Bazzi method handles a class of divide-and-conquer recurrences that arise from algorithms whose subproblems do not all have the same size, including unbalanced quicksort and certain convex-hull algorithms."

WHY — 실무에서는 거의 직접 안 풀어도 된다

Akra-Bazzi가 마스터 정리의 진짜 일반화이지만, 실무에서는 거의 균등 분할이라 마스터 정리만으로 충분. Akra-Bazzi가 진가를 발휘하는 곳은 알고리즘 분석 논문에서 정확한 점근식이 필요할 때. 시험에 나오면 "어떤 정리를 쓰는지" 정도만 알아두면 OK.

12 — AI는 분할 정복을 어떻게 인식하는가

AI — "쪼개라"는 너무 일반적이다

LLM에 "이 문제 분할 정복으로 풀어줘"라고 하면 코드가 즉시 나옵니다. 하지만 잘못된 분할이 자주 나옵니다.

  • 병합 정렬을 짜라 → 거의 정확히 나옵니다 (훈련 데이터에 무수히 많음).

  • 점화식의 정확한 형태는 묻지 않으면 잘 안 씁니다. 점화식 없이 "효율적이다"라고만 말하는 답은 신뢰하기 어렵습니다.

WHY — AI에게 시킬 때의 5가지 검증 질문

  2. "마스터 정리의 어느 케이스에 해당하는지 보여줘."
  3. "이 문제를 분할 정복 말고 더 빠르게 풀 수 있는가? (예: 동적 계획법, 그리디)"
  4. "base case가 빈 입력, 한 원소, 두 원소에 대해 모두 옳은가?"
  5. "입력이 다음 경계 케이스일 때도 동작? (전부 같은 값, 이미 정렬됨, 역순)"

13 — 실습: Inversion Counting (역순쌍 세기)

고급

문제 — 배열의 "흐트러진 정도"를 재기

배열 A에서 i < j이면서 A[i] > A[j]인 (i, j) 쌍을 역순쌍(inversion)이라 합니다. 역순쌍의 수는 "이 배열이 정렬에서 얼마나 멀리 있는가"의 정량적 척도. 정렬되어 있으면 0, 완전히 역순이면 n(n−1)/2.

예: A = [2, 4, 1, 3, 5] → 역순쌍: (2,1), (4,1), (4,3) → 3개.

고급

단순 풀이 — O(n²)

brute_inversions.py
def count_inversions_brute(A):
    n = len(A)
    cnt = 0
    for i in range(n):
        for j in range(i + 1, n):
            if A[i] > A[j]:
                cnt += 1
    return cnt
전공자

병합 정렬을 변형해 O(n log n)에 풀기

핵심 통찰: 병합 단계에서 "오른쪽 원소를 result에 넣을 때, 왼쪽에 남은 원소들 모두가 더 큼" — 이게 곧 새로운 역순쌍.

merge_inversions.py
def count_inversions(A):
    sorted_A, cnt = _merge_count(A)
    return cnt


def _merge_count(A):
    if len(A) <= 1:
        return A[:], 0
    mid = len(A) // 2
    L, c1 = _merge_count(A[:mid])
    R, c2 = _merge_count(A[mid:])
    merged, c3 = _merge_with_count(L, R)
    return merged, c1 + c2 + c3


def _merge_with_count(L, R):
    result, i, j, cnt = [], 0, 0, 0
    while i < len(L) and j < len(R):
        if L[i] <= R[j]:
            result.append(L[i]); i += 1
        else:
            result.append(R[j]); j += 1
            cnt += len(L) - i   # L[i..]가 모두 R[j]보다 큼 → 역순쌍 추가
    result.extend(L[i:])
    result.extend(R[j:])
    return result, cnt


print(count_inversions([2, 4, 1, 3, 5]))  # 3
출력:
3

병합 정렬과 동일한 점화식 T(n) = 2T(n/2) + O(n) → Θ(n log n). cnt 누적 한 줄로 알고리즘이 풀리는 마법은 — "분할 정복은 같은 구조를 재활용한다"의 좋은 예.

WHY — 정렬 알고리즘이 정렬 외 문제도 푼다

역순쌍 카운팅은 정렬과 무관해 보이는 문제인데 병합 정렬을 살짝 고쳐서 풉니다. 이런 패턴은 흔합니다.

  • 가장 가까운 두 점 (CLRS §33): 분할 정복 + 정렬 = O(n log n).
  • K번째 작은 원소: 퀵소트 변형(Quickselect) = O(n) 평균.
  • 다항식 곱셈: FFT(빠른 푸리에 변환)는 분할 정복 = O(n log n).

"분할 정복"은 정렬에 한정되지 않는 — 알고리즘 사고의 한 패러다임입니다.

LLM에게 "리스트의 최댓값을 분할 정복으로"라고 시켰더니 O(n) 코드가 나왔다. 이 답은 옳은가? "왜 문제 5의 답과 같은 복잡도인지"를 설명하시오.

풀이 펼치기
답

옳습니다. 최댓값 찾기는 모든 원소를 한 번씩은 봐야 하므로 하한이 Ω(n). 분할 정복은 그 하한을 깨지 못합니다 — 그저 같은 일을 다른 순서로 할 뿐. 마스터 정리 결과(Θ(n))가 이를 정확히 보여줍니다. 분할 정복이 모든 문제에 대해 더 빠른 게 아님을 보여주는 좋은 예입니다.

15 — 이번 주 정리 + 용어 사전 + 다음 주 맛보기

핵심 정리

✂️

3단계 패러다임

Divide(쪼개기) — Conquer(재귀로 풀기) — Combine(결과 합치기). 진짜 어려움은 Combine.

🪄

절반의 마법

n → n/2 → n/4 → … → 1은 log n 단계. 매 단계 O(n) 일이면 총 Θ(n log n).

📐

점화식 풀이 3종

대입법(자유), 재귀 트리(직관적), 마스터 정리(즉답). 일반화는 Akra-Bazzi.

⚖️

분할 정복 ≠ 만능

최대 부분배열은 Kadane(O(n))이 더 빠름. 분할 정복이 가능 ≠ 최선.

🧠

대수적 영리함

Karatsuba(4→3 곱), Strassen(8→7 곱) — "곱셈을 덧셈으로 환원"하는 대수가 복잡도를 깎는다.

🐍

Python 안의 분할 정복

list.sort(Timsort), int 곱셈(Karatsuba/Toom-Cook), numpy(BLAS는 안 씀).
