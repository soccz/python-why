문자열 알고리즘 — KMP, Boyer-Moore, Rabin-Karp
"건초 더미에서 바늘 찾기"의 세 가지 수학

1970년 Stanford, Vaughan Pratt과 James Morris가 기술보고서로 아이디어를 정리. 몇 주 뒤 Donald Knuth가 오토마타 이론에서 같은 결과에 독립 도달. 1977년 세 명이 공동 저자로 SIAM J. Comput.에 발표 — KMP 알고리즘. 같은 해 Texas Austin의 Boyer & Moore가 CACM에 "A Fast String Searching Algorithm"을 발표. 10년 뒤 1987년 Rabin & Karp가 rolling hash 기법을 완성.
1977 = 문자열 매칭의 해. Unix grep(1973), 텍스트 에디터 검색, 유전체학, 검색 엔진, 표절 검사 — 모두 이 세 알고리즘의 변형.
핵심 질문: "왜 KMP는 '실패'에서 배우는가? Boyer-Moore가 왜 패턴을 뒤에서부터 읽는가? Rabin-Karp의 hash는 어떻게 O(n+m)을 달성하는가?"

01 — 이번 주 이야기: 1977, 세 천재의 해

"This was the first linear-time algorithm for string matching."
— Knuth, Morris, Pratt, Fast Pattern Matching in Strings, SIAM J. Comput. 6(2):323-350, 1977

1970 — Pratt & Morris의 기술보고서

UC Berkeley의 James Morris와 Stanford의 Vaughan Pratt가 선형 시간 문자열 매칭 아이디어를 Berkeley 컴파일러 프로젝트의 기술보고서로 정리. 이후 몇 주 안에 Stanford 교수 Donald Knuth가 오토마타 이론으로부터 독립적으로 같은 결과에 도달. 세 사람이 합쳐 발표하기로 결정.

1977 — 문자열 매칭의 해

  • KMP: Knuth, Morris, Pratt. "Fast Pattern Matching in Strings"(SIAM J. Comput. 6(2):323-350). O(n+m) 선형 시간. Failure function으로 "실패에서 배우는" 패턴 매칭.
  • Boyer-Moore: Robert S. Boyer & J Strother Moore(Texas Austin). "A Fast String Searching Algorithm"(CACM 20(10):762-772). 패턴을 뒤에서부터 읽는 반직관적 아이디어로 평균 sublinear — 실측 가장 빠름.

우연일까? 70년대는 텍스트 에디터(vi 1976, emacs 1976)와 Unix 유틸(grep 1973)이 대중화되던 시기. 문자열 검색이 핵심 작업이 됨. 그에 호응하여 수학적 정석이 등장.

1987 — Rabin-Karp의 Rolling Hash

Richard Karp와 Michael Rabin(Turing Award 수상자)이 "Efficient randomized pattern-matching algorithms"(IBM J. Res. Dev. 31:249-260, 1987)에서 해시 기반 접근. 단일 패턴에선 KMP/BM보다 느리지만 다중 패턴 동시 검색에 독보적.

문자열 알고리즘이 만든 세상

  • grep / sed / awk: Unix 텍스트 처리의 3대 도구. 내부적으로 BM·KMP·Boyer-Moore-Horspool.
  • 텍스트 에디터: vim, VS Code 검색 모두 BM 변종.
  • 바이오인포매틱스: BLAST(1990)는 KMP 확장, BWA(Burrows-Wheeler Aligner 2009)는 suffix 구조.
  • 검색 엔진: inverted index + suffix array.
  • 표절 검사: Rabin-Karp 기반 fingerprinting (Winnowing 2003).
  • 네트워크 침입 탐지(IDS): Snort, Suricata의 시그니처 매칭 — Aho-Corasick + BM.
1977이라는 황금의 해: KMP와 Boyer-Moore가 같은 해에 독립 발표. 1970년대 텍스트 에디터·Unix 유틸 붐과 시기적으로 일치. 1987 Rabin-Karp, 1990-1995 suffix 구조로 이어짐.

02 — 순진한 매칭 — O(nm)의 함정

초급

"모든 위치에서 직접 비교"

텍스트 T(길이 n), 패턴 P(길이 m). 가장 쉬운 방법: T의 각 위치 i에 대해 P와 하나씩 비교.

naive_match.py
def naive_search(T, P):
    n, m = len(T), len(P)
    result = []
    for i in range(n - m + 1):
        if T[i:i+m] == P:
            result.append(i)
    return result
  • 시간: 최악 O(nm). 예: T = "AAAAAAAAA...AB", P = "AAAAB" — 매 위치에서 m-1자 비교 후 실패.
  • Python in 연산자: CPython이 최적화된 구현을 씀 (≥3.10은 Two-Way 알고리즘/Crochemore-Perrin 변형, 이전엔 BMH 변종). 실제로는 빠름 — "손으로 구현"하면 이 함정.
중급

왜 이게 낭비인가

예: T = "ABCABCABD", P = "ABCABD".

  • i=0: ABCAB ↔ ABCAB 일치, C↔D 실패.
  • i=1: BCABC ↔ ABCAB 첫 글자부터 불일치.
  • i=2: ...

문제: i=0에서 ABCAB를 이미 봤는데 이 정보를 버림. KMP는 이 정보를 failure function으로 재활용.

03 — KMP 아이디어 — "실패에서 배우기" (CLRS §32.4)

중급

핵심 통찰

KMP의 깨달음: "패턴 P[0..j-1]이 일치하고 P[j]에서 실패했다면, T는 되돌아갈 필요 없이 P의 접두사 = 접미사를 이용해 이동할 수 있다."

예: P = "ABCDABD". 인덱스 0..5까지 일치, 6에서 실패했다고 하자. 현재 매치된 부분: "ABCDAB". 이 문자열의 진부분 접두사 = 진부분 접미사 중 가장 긴 것은 "AB"(길이 2). 따라서 T의 현재 위치는 그대로 두고 P를 4칸 밀어도 "AB"는 이미 맞는 상태 → j를 2로 리셋.

failure function π[j] = "P[0..j]의 진부분 접두사이자 진부분 접미사인 가장 긴 문자열의 길이"
고급

예: P = "ABABAC"의 failure

j:   0 1 2 3 4 5
P:   A B A B A C
π:   0 0 1 2 3 0
  • π[0]=0: 단일 문자, 진부분 접두·접미사 없음.
  • π[1]=0: "AB"의 진부분: "A" vs "B" → 매치 없음.
  • π[2]=1: "ABA"의 "A"="A" (길이 1).
  • π[3]=2: "ABAB"의 "AB"="AB" (길이 2).
  • π[4]=3: "ABABA"의 "ABA"="ABA" (길이 3).
  • π[5]=0: "ABABAC"에서 "C" 등장으로 공통 접두·접미사 없음.

04 — Failure Function 구축

고급

O(m) 구축

π를 자기 자신과 비교해 구함. P를 P 자신에 대해 KMP 매칭하는 것과 같음. O(m) 시간.

build_failure.py
def compute_failure(P):
    m = len(P)
    pi = [0] * m
    k = 0    # 현재 일치한 prefix 길이
    for q in range(1, m):
        while k > 0 and P[k] != P[q]:
            k = pi[k - 1]    # 이전 failure로 점프
        if P[k] == P[q]:
            k += 1
        pi[q] = k
    return pi


print(compute_failure("ABABAC"))    # [0, 0, 1, 2, 3, 0]
print(compute_failure("ABCDABD"))    # [0, 0, 0, 0, 1, 2, 0]
print(compute_failure("AABAACAABAA"))    # [0,1,0,1,2,0,1,2,3,4,5]

점화 원리: P[0..q-1]까지의 pi가 계산됐다고 가정. P[q]를 처리:

  1. k = pi[q-1]. 즉 P[0..q-1]의 최장 접두·접미사 길이.
  2. P[k] == P[q]이면 pi[q] = k+1.
  3. 아니면 k를 pi[k-1]로 점프, 반복.
  4. k=0이면 pi[q] = 0 (또는 P[0]==P[q]면 1).

05 — KMP Python 구현

고급

전체 매칭 함수

kmp.py
def kmp_search(T, P):
    """T에서 P의 모든 등장 위치."""
    n, m = len(T), len(P)
    if m == 0: return [0]

    pi = compute_failure(P)
    result = []
    q = 0    # 현재까지 일치한 P 길이

    for i in range(n):
        while q > 0 and P[q] != T[i]:
            q = pi[q - 1]    # failure 점프
        if P[q] == T[i]:
            q += 1
        if q == m:
            result.append(i - m + 1)
            q = pi[q - 1]    # 겹침 허용 매칭
    return result


print(kmp_search("ABABDABACDABABCABAB", "ABABCABAB"))    # [10]
print(kmp_search("AAAAA", "AA"))    # [0, 1, 2, 3] — 겹침
print(kmp_search("ABC", "XYZ"))    # []
  • 시간: O(n + m). T를 한 번만 순회(i는 뒤로 가지 않음), q의 총 감소량이 총 증가량 이하 → amortized O(n).
  • 공간: O(m) (failure 배열).
  • return 시 q = pi[q-1]: 매칭 성공 후에도 다음 가능한 매칭의 prefix만큼 유지. 겹치는 매칭 모두 발견.
뒤로 가지 않음 → O(n+m)
4단계 KMP 트레이스: T="ABCABCABD", P="ABCABD". q=5에서 실패 → π[4]=2로 점프(AB는 이미 맞음 활용) → C vs C 매치 → 완전 매치 위치 3. T의 i는 절대 뒤로 가지 않아 O(n+m).

06 — Boyer-Moore — 뒤에서부터 읽기

중급

반직관적이지만 강력

Boyer-Moore는 패턴 P를 T에 맞추되, 패턴의 오른쪽 끝부터 거꾸로 비교. 불일치가 발생하면 두 규칙 (bad-character / good-suffix) 중 더 큰 값만큼 패턴을 오른쪽으로 이동시킵니다.

  • Bad Character Rule: 불일치한 T의 문자를 P 안에서 찾음. P 내 그 문자의 가장 오른쪽 위치로 정렬. 없으면 전체 밀기.
  • Good Suffix Rule: 불일치 전까지 일치했던 접미사가 P 내 다른 곳에도 나타나는가? 나타나면 거기에 정렬.

핵심 이익: T의 많은 문자를 아예 보지 않고 skip. 긴 알파벳(영어 알파벳 26자)에서 평균 sublinear O(n/m). 짧은 알파벳(DNA의 4자)에선 이득이 줄어듦.

07 — Bad Character Rule

고급

구체 예제

T = "ABAAABCDACBAAD", P = "ABCDACB" (길이 7).

Bad character 테이블 for P = "ABCDACB": A→4, B→6, C→5, D→3 (각 문자의 P 내 최우측 인덱스).

  1. P를 T의 0-6에 정렬. 뒤에서부터: P[6]=B vs T[6]=C. 불일치. Bad char: T[6]='C', bad['C']=5. Shift = 6−5 = 1. i=1.
  2. P가 1-7에 정렬. P[6]=B vs T[7]=D. 불일치. bad['D']=3. Shift = 6−3 = 3. i=4.
  3. P가 4-10에 정렬. 뒤에서부터 모두 일치: P=T[4:11]="ABCDACB" ✓ 매치 발견 at i=4.
  4. i=5부터 계속 탐색, 더 이상 매치 없음.

전처리: bad_char[c] = P 내 문자 c의 가장 오른쪽 위치. 없으면 -1. O(m + σ) 공간 (σ = 알파벳 크기).

boyer_moore.py
def boyer_moore_badchar(T, P):
    """BM의 bad-character rule만 사용한 단순 버전 (good suffix 생략).
       Horspool 1980과 유사하나 엄밀히 같진 않음 — Horspool은 항상
       T[i+m-1] 문자로 shift 계산. 이 코드는 실제 mismatch 위치 사용."""
    n, m = len(T), len(P)
    if m == 0: return [0]

    # bad character table: 각 문자의 P 내 최우측 위치
    bad = {c: i for i, c in enumerate(P)}

    result = []
    i = 0
    while i <= n - m:
        j = m - 1
        while j >= 0 and P[j] == T[i + j]:
            j -= 1
        if j < 0:
            result.append(i)
            i += 1    # 겹침 매칭 고려
        else:
            shift = j - bad.get(T[i + j], -1)
            i += max(1, shift)
    return result


print(boyer_moore_badchar("ABAAABCDACBAAD", "ABCDACB"))    # [4]
print(boyer_moore_badchar("ABABDABACDABABCABAB", "ABABCABAB"))    # [10]
  • 위 구현은 엄밀히는 Boyer-Moore-Horspool 1980 — 원래 BM의 간이 버전. good suffix는 생략.
  • 평균 O(n/m), 최악 O(nm). 완전 BM은 최악 O(n+m) (good suffix + Galil 1979 개선).
  • Python in / str.find()는 CPython 3.10+에서 Two-Way 알고리즘(Crochemore-Perrin 1991)을 사용. 이전 버전은 BMH 변종.

08 — Good Suffix Rule

전공자

"이미 맞은 부분"을 활용

Bad character만 쓰면 최악 O(nm). Good suffix가 이를 O(n+m)으로 수정.

불일치 시점의 P의 매치된 접미사 S를 생각. 두 가지 경우:

  1. P 내 다른 곳에 S가 나타남: 그곳과 정렬되도록 이동.
  2. S의 접미사(suffix of suffix)가 P의 prefix와 일치: 해당 prefix 시작으로 이동.
  3. 둘 다 아니면: m칸 이동.

전처리는 KMP의 failure와 유사한 O(m) 알고리즘. 구현은 복잡하지만 효과는 큼 — 특히 반복 패턴에서 bad character만으론 부족한 경우.

정확 복잡도: BM (bad + good) 최악 비교 수 ≤ 3n (Cole 1994). Galil 1979의 개선으로 최악 선형 O(n+m).

09 — Rabin-Karp — Rolling Hash (CLRS §32.2)

고급

해시 기반 매칭

아이디어: T의 각 길이 m 부분문자열의 해시를 계산, P의 해시와 비교. 해시가 다르면 반드시 불일치. 해시가 같으면 실제 비교로 확인.

문제: 각 위치마다 해시를 O(m)에 다시 계산하면 총 O(nm) — 개선 없음. Rolling hash: 한 칸 이동 시 해시를 O(1)에 업데이트.

h(T[i..i+m-1]) = T[i]·dm-1 + T[i+1]·dm-2 + ... + T[i+m-1]    (mod q)

h(T[i+1..i+m]) = (h(T[i..i+m-1]) − T[i]·dm-1)·d + T[i+m]    (mod q)
  • d = 알파벳 크기 (보통 256).
  • q = 소수 (충돌 줄이기).
  • 한 칸 이동 = 앞 문자 제거 + 뒤 문자 추가. O(1).
전공자

Python 구현

rabin_karp.py
def rabin_karp(T, P, d=256, q=10**9 + 7):
    """T에서 P의 모든 등장 위치. d=알파벳 크기, q=큰 소수."""
    n, m = len(T), len(P)
    if m > n: return []

    # d^(m-1) mod q — rolling 때 빼낼 값의 가중치
    h_high = pow(d, m - 1, q)

    # 초기 해시 (길이 m 부분)
    hash_P = 0; hash_T = 0
    for i in range(m):
        hash_P = (hash_P * d + ord(P[i])) % q
        hash_T = (hash_T * d + ord(T[i])) % q

    result = []
    for i in range(n - m + 1):
        if hash_P == hash_T:
            # 해시 일치 — 실제 비교로 확인 (충돌 방지)
            if T[i:i + m] == P:
                result.append(i)

        # 다음 윈도우로 rolling
        if i < n - m:
            hash_T = ((hash_T - ord(T[i]) * h_high) * d + ord(T[i + m])) % q
            if hash_T < 0: hash_T += q

    return result


print(rabin_karp("ABABDABACDABABCABAB", "ABABCABAB"))    # [10]
print(rabin_karp("AAAA", "AA"))    # [0, 1, 2]
  • 기대 시간: O(n + m) with 좋은 해시. 최악 O(nm) (모든 해시가 충돌).
  • 다중 패턴: k개 패턴을 모두 해시 집합에 넣고 한 번 순회 → O(n + km). BM·KMP로는 O(nk).
  • 응용: Winnowing 2003 (표절 탐지), Karp-Rabin fingerprinting.
💡 암호학적 해시 vs Rolling hash

Rabin-Karp는 비암호학적 rolling hash를 씀. 빠르지만 충돌 발생 가능. 그래서 실제 비교(T[i:i+m] == P)로 확인. 악의적 입력엔 취약 — 그래서 "adversarial setting"에선 해시 시드를 랜덤화하거나 암호 해시 사용.

10 — Z-알고리즘 — 간결한 대안

전공자

KMP보다 간단한 구조

Z-array: 문자열 S에 대해 Z[i] = "S의 i번째 위치부터 시작하는 부분문자열이 S의 prefix와 일치하는 최대 길이".

패턴 매칭: S = P + '$' + T 로 만들고 Z-array 계산. Z[i] ≥ |P|인 위치가 매칭 시작점.

z_algorithm.py
def z_function(S):
    """Z[i] = S[i:]과 S의 최장 공통 prefix 길이. O(n)."""
    n = len(S)
    Z = [0] * n
    Z[0] = n
    l, r = 0, 0    # 현재 가장 오른쪽 Z-box [l, r]
    for i in range(1, n):
        if i < r:
            Z[i] = min(r - i, Z[i - l])
        while i + Z[i] < n and S[Z[i]] == S[i + Z[i]]:
            Z[i] += 1
        if i + Z[i] > r:
            l, r = i, i + Z[i]
    return Z


def z_search(T, P):
    S = P + "$" + T
    Z = z_function(S)
    m = len(P)
    return [i - m - 1 for i in range(m + 1, len(S)) if Z[i] >= m]


print(z_search("ABABDABACDABABCABAB", "ABABCABAB"))    # [10]
  • 시간: O(n + m). KMP와 동등.
  • 장점: 구현이 KMP보다 짧고 이해하기 쉬움 — 경쟁 프로그래밍에서 선호.
  • Z-box 유지로 반복 비교 방지.

12 — Suffix Tree / Suffix Array 소개

전공자

"같은 T에 많은 패턴" 시나리오

KMP/BM/RK는 모두 T를 매번 처음부터 순회. 하지만 고정된 T에 여러 패턴을 검색한다면 (예: 검색 엔진) T를 한 번 인덱싱해두는 게 유리.

  • Suffix Tree(Weiner 1973, McCreight 1976, Ukkonen 1995 online O(n)): T의 모든 suffix를 트라이로. 패턴 검색 O(m). 공간 O(n).
  • Suffix Array(Manber-Myers 1990): suffix를 정렬한 배열 + LCP 배열. 공간 효율 O(n). 패턴 검색 O(m log n) 이진 탐색.
  • FM-index(Ferragina-Manzini 2000): BWT 기반 압축 suffix array. BWA, Bowtie 같은 유전체 정렬기의 핵심.

예: BLAST, BWA (유전체 검색), Google 검색 엔진의 inverted index 일부. 구축은 느리지만 수백만 번의 질의를 빠르게 처리.

13 — 응용 — grep, DNA, 검색 엔진

중급

grep의 진화

  • grep 1973(Ken Thompson): 원래 Thompson의 정규표현식 NFA 기반. GNU grep (Mike Haertel, ~1988)이 BM 계열로 전환하며 유명해짐.
  • GNU grep: Boyer-Moore + Aho-Corasick(다중 패턴) 혼합. "grep is fast"로 유명.
  • ripgrep(2016, Andrew Gallant): Rust로 구현, SIMD·메모리 맵 활용. grep보다 2-10배 빠름.
  • ag, ack: Perl/Go 기반 빠른 grep 대안.
고급

바이오인포매틱스

  • BLAST(Altschul et al. 1990): seed-extend. k-mer 해시(Rabin-Karp 사촌) + 지역 alignment.
  • BWA / Bowtie: FM-index(압축 suffix array) 기반 short read alignment. Human genome(3G bp)을 수 GB로 압축, 수억 개 read를 빠르게.
  • MEGA BLAST, minimap2: 현대 long-read alignment.
고급

표절 검사 / 중복 탐지

  • Winnowing 2003(Schleimer et al.): Rabin-Karp fingerprints + local minima 선택. MOSS(Stanford), Turnitin 기반.
  • Bloom filter + Rabin-Karp: rsync의 delta 계산.
  • git의 delta 압축: Rabin-Karp 기반 chunking.
전공자

AI가 문자열 알고리즘을 잘못 쓰는 패턴

  • Python in으로 만족: 내부 BM이 빠르지만 여러 패턴을 각각 찾으면 O(nk). Aho-Corasick 1번 순회가 더 빠름.
  • DNA에 Boyer-Moore 맹신: 알파벳 4자(ACGT)라 bad-char 이동이 작음. 실제로는 KMP와 비슷하거나 더 느림. FM-index가 적절.
  • Rabin-Karp 단일 패턴: BM/KMP가 훨씬 빠름. RK는 다중 패턴·2D 매칭용.
  • Adversarial input: 정적 해시 함수 + Rabin-Karp는 공격자가 충돌 유도 가능 → DoS. 랜덤 시드 필수.
  • 전처리 시간 무시: 패턴이 자주 바뀌면 전처리(KMP의 π, BM 테이블)가 매번 필요. 짧은 T + 자주 바뀌는 P면 그냥 naive가 나을 수도.

교정 질문: "알파벳 크기는?", "동일 T에 반복 질의인가?", "다중 패턴인가?", "최악 보장 필요한가?"

14 — 연습문제 (12문제)

WHY 문제 1

KMP의 T 인덱스 i가 절대 뒤로 가지 않는다는 것을 어떻게 증명?

풀이 펼치기
답

코드 구조: for i in range(n), while 내부에서 q만 감소. i는 for loop에서만 증가. 따라서 i는 단조 증가. 총 반복 횟수 ≤ n.
q의 총 감소량 ≤ 총 증가량(q++ 횟수) ≤ n. 따라서 amortized O(n). 실패 점프(while)가 많아 보여도 전체적으론 선형.

PREDICT 문제 2

P = "AABAACAABAA"의 failure function.

풀이 펼치기
답

[0, 1, 0, 1, 2, 0, 1, 2, 3, 4, 5]. 검증: "AABAACAABAA"의 진부분 접두=접미 중 최장은 "AABAA"(길이 5).

BUILD 문제 3

Boyer-Moore bad-character 테이블 — P = "TOOTH"일 때.

풀이 펼치기
답

bad = {'T': 4, 'O': 2, 'H': 3} (각 글자의 P 내 최우측 인덱스). 0-indexed. 불일치 시 shift = j - bad[T[i+j]]. 음수면 최소 1로 clamping.

DESUGAR 문제 4

Rabin-Karp에서 modulo q가 너무 작으면 어떤 문제?

풀이 펼치기
답

해시 충돌이 자주 발생 → 매번 실제 비교 O(m) → 실질적으로 naive와 동등(O(nm)). 좋은 q: 큰 소수(2³¹-1 또는 10⁹+7). 또는 이중 해시(두 개 다른 (d, q) 쌍)로 충돌률을 사실상 0으로.

WHY 문제 5

DNA 정렬에 BM이 KMP보다 못한 이유.

풀이 펼치기
답

알파벳 크기 |Σ|=4 (ACGT). BM의 bad-character 이동은 "P 내에 없는 문자"일 때 m칸 전체 이동으로 큰 이득. 하지만 DNA는 4자 모두 P에 자주 등장 → 이동량이 작음. 긴 알파벳(영어 26자, 유니코드 수천자)에선 BM의 진가. DNA엔 FM-index(BWA) 또는 k-mer 해시가 표준.

BUILD 문제 6

다중 패턴 {he, she, his, hers} 동시 매칭 알고리즘.

풀이 펼치기
답

Aho-Corasick 1975. 모든 패턴으로 trie 구축 + KMP의 failure를 trie로 확장한 failure link. 텍스트를 한 번만 순회. 시간 O(n + m + z) (z = 매칭 총 수). 네트워크 IDS, 콘텐츠 필터링 표준.

PREDICT 문제 7

Z-array vs KMP failure — 어떤 관계?

풀이 펼치기
답

Z[i] = S[i:]과 S의 최장 공통 prefix. failure π[i] = S[0..i]의 최장 접두=접미사. 둘은 상호 변환 가능 (O(n)). Z-array는 "위치 i에서 시작하는" 관점, failure는 "위치 i에서 끝나는" 관점. 같은 문제의 두 방향 표현.

DESUGAR 문제 8

Rabin-Karp rolling 공식의 유도.

풀이 펼치기
답

h(i) = T[i]·dm-1 + T[i+1]·dm-2 + ... + T[i+m-1].
h(i+1) = T[i+1]·dm-1 + ... + T[i+m].
h(i+1) = (h(i) − T[i]·dm-1) · d + T[i+m].
(기존 값에서 T[i] 기여분 빼고, 전체를 d로 밀고, 새 글자 더함.) mod q로 크기 관리.

WHY 문제 9

Suffix Tree가 Suffix Array보다 덜 쓰이는 이유.

풀이 펼치기
답

Suffix Tree 공간은 O(n)이지만 상수가 큼 (20n 바이트 실제). Suffix Array는 4n 바이트. 같은 쿼리 속도 (접미사 배열 + LCP ≈ suffix tree). 메모리 효율이 중요한 바이오·검색엔진에서 suffix array가 선호. 단 suffix tree가 표현력은 더 풍부.

BUILD 문제 10

"T에서 P가 겹치며 몇 번 등장?" 문제 — KMP로.

풀이 펼치기
답

KMP 매칭 성공 시 q = pi[q-1]로 reset (q=0 대신). 이렇게 하면 겹치는 매칭도 모두 발견. 예: T="AAAA", P="AA" → [0,1,2]. 3번.

AI 문제 11

AI가 "100만 단어 사전에서 검색 문자열이 있는 모든 단어 찾기"에 "각 단어에 KMP" 제안 — 교정?

풀이 펼치기
질문
  1. "단어 수 N=10⁶, 각 KMP O(|word| + |query|). 총 O(N·|word|)."
  2. "더 나은 방법: 사전을 Aho-Corasick trie로 전처리 (한 번), 검색 시 O(|query| + matches)."
  3. "단어 사전이 고정이면 inverted index(n-gram → 단어 리스트) 훨씬 빠름."
  4. "실무 fuzzy 검색: BK-tree 또는 Levenshtein automaton."
WHY 문제 12

1977에 KMP와 BM이 "같은 해" 나온 것의 의미.

풀이 펼치기
답

두 그룹이 독립적으로 선형 시간 문자열 매칭을 발견. KMP는 실패 정보 활용(왼→오), BM은 뒤에서부터 skip(오→왼). 완전히 다른 수학적 관점으로 같은 복잡도 한계 도달 — "1977은 문자열 매칭이 성숙한 분야가 된 해"의 증거. 70년대 텍스트 컴퓨팅 폭발이 이런 동시 발견을 자극.
