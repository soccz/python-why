이번 주 이야기

웹 크롤러를 만들었습니다. URL 100개에서 데이터를 가져와야 합니다. 순서대로 하나씩 요청하면 얼마나 걸릴까요?

순차 실행 — URL 5개 예시 
import time
import urllib.request

urls = [
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
]

start = time.time()
for url in urls:
    # 각 요청이 ~1초씩 기다림
    with urllib.request.urlopen(url, timeout=5) as resp:
        data = resp.read()
    print(f"완료: {url[-8:]}")
end = time.time()
print(f"\n총 소요 시간: {end - start:.1f}초")
# 출력: 총 소요 시간: ~5.0초
실행 결과 (예상):
완료: /delay/1 완료: /delay/1 완료: /delay/1 완료: /delay/1 완료: /delay/1 총 소요 시간: ~5.0초

URL이 100개라면 100초. 실제 서비스에서는 용납할 수 없는 시간입니다. 동시에 여러 요청을 보낼 수는 없을까요?

여기서 동시성(Concurrency)이 등장합니다. 그리고 곧바로 만나는 벽이 GIL입니다. 오늘은 GIL이 왜 있는지부터, threading/multiprocessing/asyncio의 차이, 그리고 Event Loop가 단일 스레드로 어떻게 그 많은 요청을 처리하는지까지 CPython 내부를 분해합니다.

동시성 vs 병렬성 — 개념 정리

초급

비유: 식당 주방

식당 주방 비유:

순차(Sequential) — 요리사 1명이 메뉴 하나를 완성한 뒤 다음 메뉴를 시작합니다.
피자 다 만들고 → 파스타 시작 → 샐러드 시작. 총 30분.

동시성(Concurrency) — 요리사 1명이지만 "오븐에 피자 넣고 기다리는 동안" 파스타를 만듭니다. 오븐이 돌아가는 동안 다른 일을 합니다. 한 번에 한 가지만 하지만, 기다리는 시간을 활용합니다. 총 15분.

병렬성(Parallelism) — 요리사 여러 명이 각자 다른 메뉴를 동시에 만듭니다. 진짜로 동시에 여러 일이 진행됩니다. 총 10분.

Python의 asyncio는 동시성 — 요리사 1명이지만 기다리는 시간을 낭비하지 않습니다.
Python의 multiprocessing은 병렬성 — 진짜 여러 요리사입니다.

동시성과 병렬성을 가르는 핵심 질문은 "무엇을 기다리는가?"입니다.

Preemptive vs Cooperative — 누가 언제 양보하는가?

동시성을 구현하는 방법에는 두 가지 스케줄링 전략이 있습니다.

  • Preemptive(선점형) — OS가 강제로 컨텍스트 스위치합니다. 스레드는 원하지 않아도 CPU를 빼앗길 수 있습니다. threading이 이 방식입니다. OS 스케줄러가 결정합니다.
  • Cooperative(협력형) — 작업이 스스로 "나 잠깐 쉴게"라고 선언합니다. await가 바로 그 신호입니다. asyncio가 이 방식입니다. 프로그래머가 제어합니다.

협력형 스케줄링의 단점: await를 안 걸면 다른 태스크가 영원히 실행되지 않습니다. 한 코루틴이 CPU를 독점하면 전체가 멈춥니다.

전공자

Part A — OS 레벨: 프로세스, 스레드, 코루틴

OS 관점에서 세 추상화의 계층을 정리합니다.

  • Process: OS가 관리하는 독립 실행 단위. 고유한 메모리 공간, 독립적인 파일 디스크립터 테이블, PCB(Process Control Block). fork() 시스템 콜로 생성. IPC가 없으면 메모리를 공유하지 않음.
  • Thread: 같은 프로세스 내의 실행 단위. 힙, 코드, 전역 데이터 공유. 고유한 스택과 레지스터 세트. Linux에서는 clone(CLONE_VM | CLONE_FS | ...)로 생성 — 프로세스와 구현 레벨 차이는 작음.
  • Coroutine: OS 커널이 모르는 사용자 공간(User-space) 스케줄링. 단일 스레드 내에서 실행. 스택 스위치 비용 없음. Green Thread, Fiber라고도 부름.

컨텍스트 스위치 비용 (대략): Process > Thread > Coroutine.
Python coroutine 전환은 단순히 Python 프레임 포인터를 교체하는 것입니다.

GIL — 왜 존재하는가?

초급

비유: 신호등

신호등 비유:
좁은 교차로에 신호등이 있습니다. 한 번에 한 방향의 차만 지나갈 수 있습니다. 다른 방향의 차들은 대기합니다.

GIL(Global Interpreter Lock) = 파이썬의 신호등
한 번에 하나의 스레드만 파이썬 코드를 실행할 수 있습니다. 다른 스레드들은 신호를 기다립니다.

왜 신호등이 필요할까요? 교차로가 너무 좁아서 — 즉, 파이썬 내부 데이터 구조가 동시에 여러 차(스레드)가 접근하면 충돌하기 때문입니다.
중급

GIL이 보호하는 것: Reference Counting

CPython은 메모리를 관리하기 위해 Reference Counting(참조 카운팅)을 사용합니다. 모든 Python 객체는 자신을 가리키는 참조의 수를 내부적으로 추적합니다.

참조 카운팅 확인 
import sys

x = [1, 2, 3]          # 리스트 객체 생성, refcount = 1
y = x                  # 같은 객체를 y도 가리킴, refcount = 2
print(sys.getrefcount(x))   # 3 (getrefcount 호출 자체도 임시 참조)

del y                  # refcount = 2 → 1
print(sys.getrefcount(x))   # 2

# refcount가 0이 되면 메모리 자동 해제
z = x
del x
del z
# 여기서 [1, 2, 3] 객체가 메모리에서 해제됨
실행 결과:
3 2

문제는 refcount 증감 연산이 Atomic하지 않다는 것입니다. 두 스레드가 동시에 같은 객체의 refcount를 변경하면 데이터 경쟁(Race Condition)이 발생합니다. GIL은 이를 막기 위해 존재합니다.

Python 공식 문서에서는...
"The GIL is a mutex that protects access to Python objects, preventing multiple threads from executing Python bytecodes at once." — Python FAQ
고급

GIL의 실제 동작: sys.getswitchinterval()

GIL 스위치 인터벌 
import sys

# GIL이 스레드를 강제로 전환하는 시간 간격 (초)
print(sys.getswitchinterval())   # 0.005 (5ms)

# 변경도 가능 (실험용)
sys.setswitchinterval(0.001)     # 1ms
print(sys.getswitchinterval())   # 0.001
실행 결과:
0.005 0.001

Python 3.2+ 이전에는 100 bytecode instruction마다 GIL을 전환했습니다(check interval). Python 3.2+부터는 시간 기반(기본 5ms)으로 변경되었습니다. 이는 Python 2의 GIL보다 더 공정한 스케줄링을 제공하지만, CPU-bound 스레드들이 경합할 때 오버헤드가 발생합니다.

I/O 작업에서는 GIL이 자동으로 해제됩니다. read(), write(), sleep() 같은 blocking syscall을 호출하면 CPython은 GIL을 놓아주고 다른 스레드가 실행될 수 있도록 합니다. 이것이 threading이 I/O-bound에서는 효과적인 이유입니다.

전공자

Part A — CPython GIL 구현: ceval_gil.h

CPython 소스에서 GIL은 Python/ceval_gil.h와 Python/ceval.c에 구현됩니다. 핵심 구조체는 다음과 같습니다 (CPython 3.12 기준, 개념적 표현):

ceval_gil.h 핵심 구조 (개념적 표현) 
# CPython의 GIL은 C로 구현되어 있습니다.
# 아래는 Python에서 동등한 개념을 표현한 것입니다.

# 실제 C 구조 (ceval_gil.h):
# struct _gil_runtime_state {
#     unsigned long interval;          // switch interval (microseconds)
#     _Py_atomic_address last_holder;  // 마지막 GIL 보유자
#     _Py_atomic_int locked;           // 0=해제, 1=획득
#     unsigned long switch_number;     // 전환 횟수
#     PyCOND_T cond;                   // condition variable
#     PyMUTEX_T mutex;                 // mutex
#     _Py_atomic_int eval_breaker;     // 빠른 경로 플래그
# };

# GIL 획득 흐름 (개념):
# 1. eval_breaker 플래그를 원자적으로 확인
# 2. GIL이 잠겨 있으면 cond.wait()로 대기
# 3. interval 지나면 다른 스레드가 gil_drop_request 설정
# 4. 현재 스레드가 eval_breaker를 감지하고 GIL 해제
# 5. 대기 중인 스레드가 GIL 획득

import sys
print(f"Switch interval: {sys.getswitchinterval() * 1000:.1f}ms")

# GIL 전환이 실제로 일어나는지 확인
import threading
counter = 0

def increment():
    global counter
    for _ in range(1_000_000):
        counter += 1   # 이 연산은 Atomic하지 않음!
                       # LOAD_GLOBAL + BINARY_ADD + STORE_GLOBAL = 3 bytecodes

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)
t1.start(); t2.start()
t1.join(); t2.join()

# Race condition 발생! 2000000이 아닐 수 있음
print(f"counter = {counter}")  # 예: 1843721 (매번 다름)

Part B — GIL의 역사와 대안 모델

GIL은 Python 창시자 Guido van Rossum이 1990년대 초 도입했습니다. 당시 fine-grained locking(세밀한 잠금)은 구현 복잡도와 버그 위험이 너무 높았기 때문입니다.

GIL 없이 스레드 안전성을 보장하는 언어들의 접근:

  • Go (CSP — Communicating Sequential Processes): goroutine끼리 channel로 통신. "메모리 공유 대신 통신으로 공유하라(Don't communicate by sharing memory; share memory by communicating)." 런타임이 goroutine을 OS 스레드에 M:N으로 매핑.
  • Erlang (Actor Model): 프로세스(경량)들이 서로 메시지를 주고받음. 공유 상태 없음. 결함 격리(fault isolation) 탁월. OTP 슈퍼바이저 트리.
  • Java: synchronized, volatile, java.util.concurrent — 세밀한 잠금. happens-before 관계로 메모리 가시성 보장(JMM, Java Memory Model).
  • Rust: Ownership + Send/Sync 트레이트. 컴파일 타임에 데이터 경쟁을 원천 차단. GIL도 락도 필요 없음.

Python의 asyncio는 사실상 CSP와 Actor의 중간에 가깝습니다. 코루틴은 Actor처럼 독립적으로 실행되고, 데이터는 asyncio.Queue 같은 채널로 교환됩니다.

threading — I/O bound에서만 효과적인 이유

초급

비유: 점원 여러 명

점원 비유:
카페에 점원이 여러 명 있습니다. 각자 주문을 받고 커피를 만듭니다. 하지만 에스프레소 머신이 하나라서, 한 번에 한 명만 쓸 수 있습니다(GIL).

주문을 받거나, 컵을 준비하거나, 고객과 대화하는 동안(I/O)은 각자 동시에 할 수 있습니다. 에스프레소 추출(CPU 계산) 단계에서만 줄을 서야 합니다.
threading 기본 사용법 
import threading
import time

def download(url_num):
    """네트워크 요청 시뮬레이션 (I/O bound)"""
    print(f"[스레드] URL {url_num} 다운로드 시작")
    time.sleep(1)   # 실제로는 네트워크 대기
    print(f"[스레드] URL {url_num} 완료")

# 순차 실행: ~3초
# start = time.time()
# for i in range(3): download(i)
# print(f"순차: {time.time()-start:.1f}초")

# 동시 실행: ~1초
start = time.time()
threads = []
for i in range(3):
    t = threading.Thread(target=download, args=(i,))
    threads.append(t)
    t.start()

for t in threads:
    t.join()   # 모든 스레드가 끝날 때까지 기다림

print(f"\n총 소요 시간: {time.time()-start:.1f}초")
실행 결과:
[스레드] URL 0 다운로드 시작 [스레드] URL 1 다운로드 시작 [스레드] URL 2 다운로드 시작 [스레드] URL 0 완료 [스레드] URL 1 완료 [스레드] URL 2 완료 총 소요 시간: 1.0초
중급

ThreadPoolExecutor — 더 편리한 방법

concurrent.futures.ThreadPoolExecutor 
from concurrent.futures import ThreadPoolExecutor, as_completed
import time

def fetch(url_num):
    time.sleep(1)   # I/O 시뮬레이션
    return f"URL {url_num}: 200 OK"

urls = list(range(5))
start = time.time()

with ThreadPoolExecutor(max_workers=5) as executor:
    # submit: 각 작업을 Future로 반환
    futures = {executor.submit(fetch, url): url for url in urls}

    for future in as_completed(futures):
        result = future.result()
        print(result)

print(f"\n총 소요 시간: {time.time()-start:.1f}초")
실행 결과:
URL 2: 200 OK URL 0: 200 OK URL 3: 200 OK URL 1: 200 OK URL 4: 200 OK 총 소요 시간: 1.0초
경고: threading으로 CPU-bound 작업 → 오히려 느려집니다
GIL 때문에 CPU 계산은 어차피 한 번에 하나씩 실행됩니다. 거기에 스레드 생성 비용 + 컨텍스트 스위치 오버헤드 + GIL 경합이 추가되어 순차 실행보다 느려집니다.
CPU-bound에서 threading은 역효과 
import threading
import time

def cpu_heavy():
    """CPU를 많이 쓰는 계산"""
    total = 0
    for i in range(5_000_000):
        total += i * i
    return total

# 순차 실행
start = time.time()
cpu_heavy()
cpu_heavy()
seq_time = time.time() - start
print(f"순차: {seq_time:.2f}초")

# threading 시도 (GIL 때문에 병렬 안 됨)
start = time.time()
t1 = threading.Thread(target=cpu_heavy)
t2 = threading.Thread(target=cpu_heavy)
t1.start(); t2.start()
t1.join(); t2.join()
thread_time = time.time() - start
print(f"threading: {thread_time:.2f}초")
print(f"→ threading이 {thread_time/seq_time:.1f}배 느림 (오버헤드)")
실행 결과 (예상):
순차: 1.82초 threading: 2.14초 → threading이 1.2배 느림 (오버헤드)

Lock — 공유 데이터 보호

threading.Lock으로 Race Condition 해결 
import threading

# Race condition 있는 버전
counter_unsafe = 0
def unsafe_increment():

from concurrent.futures import ProcessPoolExecutor
import math

def factorize(n):
    """소인수분해 (CPU-bound)"""
    factors = []
    d = 2
    while d * d <= n:
        while n % d == 0:
            factors.append(d)
            n //= d
        d += 1
    if n > 1:
        factors.append(n)
    return factors

    with ProcessPoolExecutor(max_workers=4) as executor:
        results = list(executor.map(factorize, numbers))

    for n, factors in zip(numbers, results):
        print(f"{n} = {' × '.join(map(str, factors))}")

Event Loop는 끝없이 도는 루프입니다. 각 반복(iteration)을 tick이라고 합니다. 각 tick에서 Loop는:

async def count_slowly(name, n):
    """await 없이 긴 루프 — 다른 코루틴을 굶김"""
    for i in range(n):
        # await asyncio.sleep(0)  # 이 줄이 없으면 루프 독점
        pass
    print(f"{name} 완료")

async def quick_task(name):
    print(f"{name} 시작")
    await asyncio.sleep(0)   # Event Loop에 제어권 넘김
    print(f"{name} 끝")

async def main_blocking():
    """asyncio.sleep(0) 없이 — 긴 루프가 Event Loop 독점"""
    await asyncio.gather(
        count_slowly("계산", 5_000_000),
        quick_task("빠른 태스크"),
    )

async def main_cooperative():
    """asyncio.sleep(0)으로 주기적으로 양보"""
    async def count_with_yield(name, n):
        for i in range(n):
            if i % 100_000 == 0:
                await asyncio.sleep(0)   # 주기적으로 양보
        print(f"{name} 완료")

    await asyncio.gather(
        count_with_yield("계산", 5_000_000),
        quick_task("빠른 태스크"),

asyncio는 내부적으로 두 가지 Event Loop 아키텍처를 지원합니다.

  • Reactor 패턴 (SelectorEventLoop): "이 파일 디스크립터가 읽기 준비되면 나한테 알려줘" — epoll/kqueue 기반. Linux/macOS. 콜백이 I/O 완료 후 호출됨. Node.js, nginx, libuv가 이 패턴.
  • Proactor 패턴 (ProactorEventLoop): "이 작업을 비동기로 시작해줘, 끝나면 알려줘" — Windows IOCP(I/O Completion Ports) 기반. 커널이 I/O를 직접 처리하고 완료 큐에 결과를 올림.

Go의 netpoller, Rust의 tokio, Java의 NIO2(AIO)도 이 두 패턴 중 하나를 사용합니다. asyncio의 설계는 본질적으로 단일 스레드 Reactor입니다.

        # Python 3.11+ ExceptionGroup

Part A — GIL 제거의 기술적 도전

GIL 제거는 단순히 뮤텍스 하나를 빼는 것이 아닙니다. 주요 과제:

  • Reference Counting → Biased Reference Counting: No-GIL 빌드는 Biased Reference Counting을 사용합니다. 객체를 생성한 스레드("소유자")는 atomic 없이 refcount를 변경. 다른 스레드는 atomic 연산. 오너십 전환 시 "deferred" refcount로 처리. 이는 Sam Gross(PEP 703 저자)가 설계.
  • 내부 데이터 구조 Thread Safety: dict, list, set 등의 내부 구현을 thread-safe하게 재작성. Python 레벨에서는 여전히 lock 없이는 atomic하지 않음 — GIL 제거 ≠ 자동 Thread Safety.
  • 성능: 초기 No-GIL 빌드는 단일 스레드에서 ~40% 느렸음. 3.14에서 ~10~15%로 감소. 멀티스레드 CPU-bound에서는 선형적 속도 향상 달성.
중요: GIL이 없어도 Python 객체는 Thread-safe하지 않습니다.
list.append()는 원자적이지 않습니다. No-GIL 환경에서 여러 스레드가 같은 객체를 동시에 수정하면 여전히 threading.Lock()이 필요합니다.

def amdahl_speedup(parallel_fraction: float, num_processors: int) -> float:
    """
    Amdahl의 법칙: S = 1 / ((1-P) + P/N)
    P: 병렬화 가능한 비율 (0~1)
    N: 프로세서 수
    """
    serial_fraction = 1 - parallel_fraction
    return 1 / (serial_fraction + parallel_fraction / num_processors)

# 90% 병렬화 가능한 작업
P = 0.90
print(f"병렬화 비율: {P*100:.0f}%")
print("-" * 35)
for n in [1, 2, 4, 8, 16, 32, 64, 128]:
    s = amdahl_speedup(P, n)
    print(f"  {n:3d}개 코어: {s:.2f}배 (이론 최대: {1/(1-P):.0f}배)")

# 결론: 프로세서를 무한히 늘려도 속도는 1/(1-P)배가 한계

ChatGPT, Claude 같은 LLM API는 I/O-bound 작업입니다. 응답이 오기까지 수 초가 걸리는 네트워크 대기입니다. asyncio가 가장 적합한 도구입니다.

LLM API 병렬 호출 패턴 (개념) 
import asyncio
import time

# 실제 LLM API 호출을 시뮬레이션
async def call_llm(prompt_id: int, response_time: float = 2.0) -> str:
    """LLM API 호출 (네트워크 I/O 시뮬레이션)"""
    # 실제로는: async with aiohttp.ClientSession() as session: ...
    await asyncio.sleep(response_time)
    return f"LLM 응답 #{prompt_id}: 생성 완료"

async def process_batch_sequential(prompts: list) -> list:
    """순차 처리 — 하나씩 기다림"""
    results = []
    for i, _ in enumerate(prompts):
        result = await call_llm(i)
        results.append(result)
    return results

async def process_batch_concurrent(prompts: list) -> list:
    """동시 처리 — asyncio.gather"""
    tasks = [call_llm(i) for i in range(len(prompts))]
    return await asyncio.gather(*tasks)

async def main():
    prompts = ["질문1", "질문2", "질문3", "질문4", "질문5"]

    start = time.time()
    results = await process_batch_sequential(prompts)
    print(f"순차: {time.time()-start:.1f}초 ({len(results)}개 결과)")

    start = time.time()
    results = await process_batch_concurrent(prompts)
    print(f"동시: {time.time()-start:.1f}초 ({len(results)}개 결과)")

    # 실제 프로덕션: Rate Limiting도 고려
    # asyncio.Semaphore로 동시 요청 수 제한

asyncio.run(main())
실행 결과:
순차: 10.0초 (5개 결과) 동시: 2.0초 (5개 결과)

실제 LLM 서버(예: vLLM, TGI)는 GPU에서 여러 요청을 Continuous Batching으로 처리합니다. 다수의 요청이 동시에 들어오면 배치로 묶어 하나의 forward pass에서 처리하여 GPU 활용도를 최대화합니다. 서버 측은 GPU 병렬성(CPU bound), 클라이언트 측은 asyncio(I/O bound)의 조합입니다.

import asyncio

async def greedy(n):
    """양보하지 않는 코루틴 — Event Loop 독점"""
    total = 0
    for i in range(n):
        total += i
    print(f"greedy 완료: {total}")

async def polite(n):
    """주기적으로 양보하는 코루틴"""
    total = 0
    for i in range(n):
        total += i
        if i % 100_000 == 0:
            await asyncio.sleep(0)
    print(f"polite 완료: {total}")

async def quick():
    print("quick: 시작")
    await asyncio.sleep(0)
    print("quick: 끝")

async def main():
    print("=== greedy 버전 (quick이 마지막에 실행) ===")
    await asyncio.gather(greedy(500_000), quick())

    print("\n=== polite 버전 (quick이 중간에 실행) ===")
    await asyncio.gather(polite(500_000), quick())

asyncio.run(main())

import asyncio
import random

async def producer(queue, name, items):
    for item in items:
        await asyncio.sleep(random.uniform(0.05, 0.15))
        await queue.put(item)
        print(f"[{name}] 생산: {item}")
    await queue.put(None)   # 종료 신호

async def consumer(queue, name, num_producers):
    sentinel_count = 0
    while True:
        item = await queue.get()
        if item is None:
            sentinel_count += 1
            if sentinel_count == num_producers:
                break
            queue.task_done()
            continue
        await asyncio.sleep(0.1)
        print(f"  [{name}] 소비: {item}")
        queue.task_done()

async def main():
    queue = asyncio.Queue(maxsize=5)
    NUM_PRODUCERS = 2

    producers = [
        asyncio.create_task(producer(queue, f"P{i}", range(i*3, i*3+3)))
        for i in range(NUM_PRODUCERS)
    ]
    consumers = [
        asyncio.create_task(consumer(queue, f"C{i}", NUM_PRODUCERS))
        for i in range(3)
    ]

    await asyncio.gather(*producers)
    await asyncio.gather(*consumers)

asyncio.run(main())

def compute_partial_sum(args):

문제 12. Amdahl의 법칙을 활용하여, 자신이 만든 프로그램의 병렬화 가능 비율(P)을 실험으로 측정하는 코드를 작성하세요. 1코어와 4코어 실행 시간을 측정하여 역으로 P를 계산하세요.

힌트 보기

Amdahl: S = 1 / ((1-P) + P/N). 이를 P에 대해 풀면: P = (1/S - 1) / (1/N - 1)

from multiprocessing import Pool
import time

def workload(n):
    """측정할 작업"""
    total = 0
    for i in range(n):
        total += i * i
    return total

if __name__ == "__main__":
    TASKS = [2_000_000] * 4

    # 1코어 순차 실행
    start = time.time()
    for t in TASKS: workload(t)
    t1 = time.time() - start

    # 4코어 병렬 실행
    start = time.time()
    with Pool(4) as pool:
        pool.map(workload, TASKS)
    t4 = time.time() - start

    S = t1 / t4   # 실제 속도 향상
    N = 4
    # Amdahl 역산: P = (1/S - 1) / (1/N - 1)
    P = (1/S - 1) / (1/N - 1)

    print(f"순차: {t1:.2f}초, 4코어: {t4:.2f}초")
    print(f"속도 향상: {S:.2f}배")
    print(f"추정 병렬화 비율: {P*100:.1f}%")
    print(f"이론 최대 속도 향상(∞코어): {1/(1-P):.1f}배")

  • multiprocessing: GIL이 없는 독립 프로세스. 진짜 병렬이지만 IPC 비용 존재
  • asyncio: 단일 스레드 Cooperative 스케줄링. 수천 개 I/O를 메모리 효율적으로 처리
  • async def / await: 코루틴 프로토콜 위에 구현된 문법 설탕. 내부는 generator + yield
  • PEP 703: Python 3.13+에서 No-GIL 빌드가 실험적으로 제공. GIL 없어도 Lock은 여전히 필요

def choose_concurrency_tool(task_type: str, scale: str) -> str:
    """
    task_type: "io_bound" or "cpu_bound"
    scale: "small" (< 100) or "large" (>= 100)
    """
    if task_type == "io_bound":
        if scale == "large":
            return "asyncio — 수천 개 동시 I/O, 메모리 효율적"
        else:
            return "threading 또는 asyncio — 모두 적합"
    elif task_type == "cpu_bound":
        return "multiprocessing — GIL 우회, 진짜 병렬"
    else:
        return "순차 실행 — 가장 단순하고 버그 없음"

용어 사전 (25개)

GIL (Global Interpreter Lock)
CPython에서 한 번에 하나의 스레드만 Python 바이트코드를 실행할 수 있도록 하는 전역 뮤텍스. Reference Counting을 Thread-safe하게 보호하기 위해 존재한다.
Reference Counting (참조 카운팅)
CPython의 기본 메모리 관리 방식. 모든 객체는 자신을 가리키는 참조의 수를 추적하며, 이 수가 0이 되면 즉시 메모리를 해제한다.
Concurrency (동시성)
여러 작업이 겹쳐서 진행되는 구조. 실제로 동시에 실행되지 않아도 된다. 기다리는 시간을 다른 작업에 활용한다. asyncio가 대표적.
Parallelism (병렬성)
여러 작업이 물리적으로 동시에 실행되는 것. 멀티코어 프로세서를 실제로 활용한다. multiprocessing이 대표적.
I/O bound
프로그램의 병목이 CPU 계산이 아닌 네트워크, 디스크, DB 등의 I/O 대기에 있는 경우. threading 또는 asyncio로 해결한다.
CPU bound
프로그램의 병목이 CPU 계산에 있는 경우. 수치 계산, 암호화, 이미지 처리 등. multiprocessing 또는 Cython/NumPy로 해결한다.
Thread (스레드)
프로세스 내의 실행 단위. 힙, 코드, 전역 데이터를 공유하고 독립적인 스택과 레지스터를 가진다. OS 커널이 관리한다.
Process (프로세스)
OS가 관리하는 독립 실행 단위. 고유한 메모리 공간을 가지며 다른 프로세스와 메모리를 공유하지 않는다.
Coroutine (코루틴)
실행을 일시 중단하고 재개할 수 있는 함수. async def로 정의하며 호출 시 coroutine 객체를 반환한다. await로 실행하고 제어권을 Event Loop에 돌려준다.
Event Loop (이벤트 루프)
asyncio의 핵심 스케줄러. 코루틴과 콜백을 관리하고 I/O 완료를 감시하여 준비된 작업을 순서대로 실행한다. asyncio.run()이 생성하고 관리한다.
await
코루틴 내에서 다른 awaitable 객체를 기다리는 키워드. 내부적으로 __await__() 메서드를 호출하며, yield를 통해 Event Loop에 제어권을 양보할 수 있다.
async def
코루틴 함수를 정의하는 키워드. 호출 시 실행되지 않고 coroutine 객체를 반환한다. await로 실행해야 한다.
asyncio.gather()
여러 코루틴이나 태스크를 동시에 실행하고 모든 결과를 입력 순서대로 반환하는 함수. return_exceptions=True로 예외를 결과로 받을 수 있다.
asyncio.create_task()
코루틴을 Task 객체로 감싸 Event Loop에 즉시 예약하는 함수. Task 객체를 반환하여 취소, 완료 확인, 콜백 추가가 가능하다.
asyncio.sleep(0)
0초 대기가 아닌, Event Loop에 한 번 제어권을 양보하고 다음 tick에 재개되도록 예약하는 패턴. 긴 루프에서 다른 코루틴이 실행될 기회를 주기 위해 사용한다.
asyncio.Semaphore
동시에 실행될 수 있는 코루틴의 수를 제한하는 동기화 기법. asyncio.Semaphore(n)으로 생성하며 async with로 사용한다.
asyncio.Queue
코루틴 사이에서 데이터를 안전하게 전달하는 비동기 큐. Producer-Consumer 패턴 구현에 사용한다. Thread-safe하지 않으므로 asyncio 내에서만 사용.
ThreadPoolExecutor
concurrent.futures의 스레드 풀 실행기. executor.submit()으로 작업을 제출하고 Future를 반환받는다. asyncio와 연동 시 loop.run_in_executor()를 사용.
ProcessPoolExecutor
concurrent.futures의 프로세스 풀 실행기. ThreadPoolExecutor와 동일한 API로 multiprocessing을 사용한다. CPU-bound 작업을 asyncio에서 사용할 때 loop.run_in_executor()와 조합한다.
Race Condition (경쟁 조건)
여러 스레드가 공유 데이터를 동시에 읽고 쓸 때 실행 순서에 따라 결과가 달라지는 버그. threading.Lock으로 해결한다.
Deadlock (교착 상태)
두 스레드가 서로 상대방이 가진 Lock을 기다리며 영원히 대기하는 상태. Lock 획득 순서를 일관되게 유지하면 예방할 수 있다.
Cooperative Scheduling (협력적 스케줄링)
작업이 스스로 "나 잠깐 쉴게"라고 선언하는 스케줄링 방식. asyncio의 await가 이 역할을 한다. 한 코루틴이 await 없이 긴 작업을 하면 다른 코루틴이 실행되지 못한다.
Preemptive Scheduling (선점형 스케줄링)
OS가 강제로 실행 중인 작업을 중단하고 다른 작업에 CPU를 넘기는 방식. threading이 이 방식으로 OS 스케줄러가 관리한다.
PEP 703
2023년 승인된 "Making the Global Interpreter Lock Optional in CPython" 제안. Python 3.13부터 No-GIL 빌드(python3.13t)가 실험적으로 제공된다.
Amdahl의 법칙
병렬 처리의 이론적 속도 향상 한계를 설명하는 법칙. S = 1 / ((1-P) + P/N). P가 병렬화 가능한 비율, N이 프로세서 수. 10%의 순차 코드가 있으면 최대 10배 향상이 한계.
