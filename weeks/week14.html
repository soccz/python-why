이번 주 이야기

지난 주에 데코레이터와 메타클래스로 클래스의 동작을 런타임에 바꾸는 법을 배웠습니다. 그런데 프로젝트가 커질수록 함수의 인자가 무엇을 받는지, 무엇을 반환하는지 잊어버립니다. 

# 이 함수가 뭘 받고 뭘 반환하는지 알 수 있나요?
def process(data, config, mode):
    ...
    return result

세 달 후 이 코드를 다시 보면: data가 리스트인가, 딕셔너리인가? config는 어떤 키를 가져야 하는가? result는 무슨 타입인가?

타입 힌트가 없던 시절: 함수 시그니처만 보고는 알 수 없었습니다. docstring을 열심히 쓰거나, 소스를 다 읽거나, 직접 실행해서 확인해야 했습니다. 
# 타입 힌트를 붙이면 — 코드 자체가 문서가 됩니다
def process(
    data: list[dict[str, int]],
    config: dict[str, str],
    mode: str
) -> list[int]:
    ...
    return result

함수 시그니처만으로 이제 의도가 보입니다. 하지만 Python은 이것을 강제하지 않습니다 — 힌트(hint)입니다. 오늘은 이 힌트가 어떻게 저장되고, 어떻게 도구들이 활용하며, 구조적 서브타이핑이라는 강력한 개념까지 완전히 분해합니다.

타입 힌트는 왜 "힌트"인가?

초급

비유: 요리 레시피의 "추천 재료"

설명서 비유:
요리책에 "달걀 2개 권장"이라고 적혀 있습니다. 실제로 메추리알 4개를 쓰거나 달걀을 빼도 요리책이 막지는 않습니다.

Python 타입 힌트는 이 "권장 재료 설명서"입니다 — 어기는 것을 막지 않고, 다른 사람(혹은 mypy 같은 도구)에게 의도를 알려줍니다.

타입 힌트를 쓴다고 해서 Python이 타입을 검사하지는 않습니다. 다음 코드는 경고 없이 실행됩니다: 

def greet(name: str) -> str:
    return "안녕, " + name

# str을 기대하지만 int를 넘겨도 런타임에서는 오류가 나지 않습니다
# (하지만 실행하면 int + str 이라 TypeError가 발생합니다 — 타입 힌트 때문이 아니라
#  실제 연산 때문에)
result = greet("파이썬")    # "안녕, 파이썬" — 정상
print(result)
실행 결과:
안녕, 파이썬
가장 흔한 오해: 타입 힌트를 쓰면 Python이 타입을 검사할 것이라고 생각합니다. 아닙니다. 타입 힌트는 런타임에 완전히 무시됩니다. 검사는 mypy, pyright 같은 별도 도구가 합니다. 
# 타입 힌트는 런타임에 무시됩니다 — 증명
def add(a: int, b: int) -> int:
    return a + b

# int를 기대하지만 str을 넘겨도 Python은 아무 말 안 합니다
# (str + str은 합법적인 연산이므로 실제로 실행됩니다)
print(add("안녕", "세상"))   # "안녕세상" — 오류 없음!
print(add(1, 2))             # 3
실행 결과:
안녕세상 3

타입 힌트는 코드를 읽는 사람과 정적 분석 도구를 위한 것입니다. 런타임 Python 인터프리터는 신경 쓰지 않습니다.

중급

타입 힌트의 역사 — PEP 484에서 시작

Python은 원래 완전히 동적 타입 언어였습니다. 타입 힌트는 Python 3.5 (PEP 484)에서 도입되었습니다. Guido van Rossum이 직접 주도했고, 선택적(optional)으로 설계했습니다. 

# Python 3.5+ — 기본 타입 힌트
def greet(name: str) -> str:
    return f"안녕, {name}!"

# 변수에도 타입 힌트를 붙일 수 있습니다 (Python 3.6+, PEP 526)
age: int = 25
scores: list[float] = [98.5, 87.0, 93.3]

# 반환값이 없을 때
def log(message: str) -> None:
    print(f"[LOG] {message}")

타입 힌트가 유용한 이유:

  • IDE 자동완성: VSCode, PyCharm이 타입을 알면 더 정확하게 제안
  • 정적 분석: mypy가 실행 전에 타입 오류를 잡아냄
  • 문서화: docstring 없이도 함수 시그니처로 의도 전달
  • 리팩토링: 타입이 있으면 안전하게 코드를 바꿀 수 있음
# 실전 예시 — 타입 힌트가 버그를 방지하는 방법
from typing import Optional

def find_user(user_id: int) -> Optional[dict]:
    """user_id로 사용자를 찾아 반환. 없으면 None."""
    users = {1: {"name": "Alice"}, 2: {"name": "Bob"}}
    return users.get(user_id)

user = find_user(1)
# mypy는 user가 None일 수 있다는 것을 알고, None 체크 없이 접근하면 경고합니다
if user is not None:
    print(user["name"])   # "Alice"
실행 결과:
Alice

타입 힌트가 저장되는 방식 — __annotations__

타입 힌트는 실제로 __annotations__ 딕셔너리에 저장됩니다. Python이 클래스나 함수를 정의할 때 이 딕셔너리를 자동으로 만듭니다. 

def add(a: int, b: int) -> int:
    return a + b

# 타입 힌트가 어디 저장되는지 확인
print(add.__annotations__)
# {'a': <class 'int'>, 'b': <class 'int'>, 'return': <class 'int'>}
실행 결과:
{'a': <class 'int'>, 'b': <class 'int'>, 'return': <class 'int'>} 
class Point:
    x: float
    y: float

    def __init__(self, x: float, y: float) -> None:
        self.x = x
        self.y = y

# 클래스 레벨의 어노테이션
print(Point.__annotations__)   # {'x': <class 'float'>, 'y': <class 'float'>}

# 인스턴스 메서드의 어노테이션
print(Point.__init__.__annotations__)
# {'x': <class 'float'>, 'y': <class 'float'>, 'return': <class 'NoneType'>}
실행 결과:
{'x': <class 'float'>, 'y': <class 'float'>} {'x': <class 'float'>, 'y': <class 'float'>, 'return': <class 'NoneType'>}
Python 공식 문서에서는: Python 3.10부터는 inspect.get_annotations()로 접근을 권장합니다. Python 3.14부터는 annotationlib.get_annotations()가 이를 대체합니다. 클래스 상속 시 __annotations__를 직접 접근하면 부모 클래스의 것이 반환될 수 있기 때문입니다. 
import inspect

def process(data: list[int], name: str = "default") -> dict:
    return {}

# 권장 방법 (Python 3.10+)
annotations = inspect.get_annotations(process)
print(annotations)
# {'data': list[int], 'name': <class 'str'>, 'return': <class 'dict'>}
실행 결과:
{'data': list[int], 'name': <class 'str'>, 'return': <class 'dict'>}

Part A — CPython __annotations__ 내부 구현

CPython 컴파일러는 함수/클래스 정의를 컴파일할 때 어노테이션 표현식을 별도로 처리합니다. 

import dis

def add(a: int, b: int) -> int:
    return a + b

dis.dis(add)
# 어노테이션은 함수 본체의 바이트코드에 없습니다 —
# 함수 정의 시점의 MAKE_FUNCTION 단계에서 처리됩니다
# 모듈 레벨 어노테이션의 컴파일 방식 확인
import dis

code = compile("x: int = 5", "<string>", "exec")
dis.dis(code)
# RESUME          0
# LOAD_CONST      5 (5)
# STORE_NAME      0 (x)
# SETUP_ANNOTATIONS   <-- 이 바이트코드가 __annotations__ 딕셔너리를 초기화
# LOAD_NAME       1 (int)
# LOAD_NAME       2 (__annotations__)
# LOAD_CONST      3 ('x')
# STORE_SUBSCR
SETUP_ANNOTATIONS 바이트코드: 모듈이나 클래스 본체에 어노테이션이 있으면 Python 컴파일러는 SETUP_ANNOTATIONS 명령어를 삽입합니다. 이 명령어는 __annotations__ 딕셔너리가 없으면 빈 딕셔너리로 초기화합니다. 함수는 다르게 동작합니다 — 함수 어노테이션은 함수 객체의 func_annotations 슬롯에 저장되고, lazy-create됩니다.

Part B — Hindley-Milner 타입 시스템과 Python의 선택

대부분의 정적 타입 언어(Haskell, ML, Rust)는 Hindley-Milner(HM) 타입 추론을 사용합니다. HM은 타입 주석이 없어도 모든 표현식의 타입을 완전히 추론합니다.

Python이 HM을 선택하지 않은 이유:

  • 동적 타이핑과의 공존: HM은 타입을 모르는 표현식이 없어야 합니다. Python은 타입 무관 코드가 존재합니다.
  • 점진적 타이핑(Gradual Typing): PEP 484는 Gradual Type System을 채택했습니다. 타입 힌트가 없는 코드는 Any 타입으로 취급하고, Any는 모든 타입과 호환됩니다.
  • 런타임 동적성: setattr, __getattr__, 메타클래스로 런타임에 속성이 바뀔 수 있어 컴파일 타임 전체 추론이 불가능합니다.
from typing import Any

# Any는 모든 타입과 양방향 호환
def legacy_function(data: Any) -> Any:
    return data

# 타입이 없는 함수는 묵시적으로 Any → Any
def no_hints(data):
    return data

# mypy 관점에서 이 둘은 동일하게 취급됩니다

mypy 같은 도구는 PEP 484의 Gradual Typing 명세를 구현합니다. Liskov Substitution Principle(LSP): 서브타입은 슈퍼타입이 사용되는 모든 곳에 사용 가능해야 합니다. mypy는 LSP를 검사합니다.

Optional, Union, PEP 604 파이프 문법

Optional[X]는 X | None과 정확히 동일합니다. Optional[str]은 str | None과 같습니다. Python 3.10부터는 | 문법(PEP 604)을 권장합니다. 두 가지를 혼용하지 마세요 — 프로젝트 내에서 하나를 선택해 일관되게 쓰세요. 
from typing import Optional, Union

# Optional — 값이 없을 수도 있음
def find_name(user_id: int) -> Optional[str]:   # str | None과 동일
    db = {1: "Alice", 2: "Bob"}
    return db.get(user_id)

# Python 3.10+ 권장 방식 (PEP 604)
def find_name_modern(user_id: int) -> str | None:
    db = {1: "Alice", 2: "Bob"}
    return db.get(user_id)

print(find_name(1))       # "Alice"
print(find_name(99))      # None

# Union — 여러 타입 중 하나
def stringify(value: Union[int, float, str]) -> str:
    return str(value)

# PEP 604 방식 (Python 3.10+)
def stringify_modern(value: int | float | str) -> str:
    return str(value)

print(stringify(42))        # "42"
print(stringify(3.14))      # "3.14"
print(stringify("hi"))      # "hi"
실행 결과:
Alice None 42 3.14 hi

list[int] vs List[int] — PEP 585 이전/이후

# Python 3.8 이하 — typing 모듈에서 가져와야 했음
from typing import List, Dict, Tuple, Set

def old_style(items: List[int], mapping: Dict[str, List[str]]) -> Tuple[int, ...]:
    return tuple(items)

# Python 3.9+ (PEP 585) — 내장 타입을 직접 사용
# typing에서 가져올 필요 없음!
def new_style(items: list[int], mapping: dict[str, list[str]]) -> tuple[int, ...]:
    return tuple(items)

print(new_style([1, 2, 3], {"a": ["x", "y"]}))   # (1, 2, 3)
실행 결과:
(1, 2, 3)
Python 공식 문서에서는: PEP 585 (Python 3.9+)부터 list[int], dict[str, int] 등 내장 컬렉션을 제네릭으로 직접 사용할 수 있습니다. typing.List, typing.Dict는 deprecated 상태입니다. 새 코드는 내장 타입을 쓰세요. 
# 실전 패턴들
from typing import Callable, Any

# Callable — 호출 가능한 것의 타입
def apply(func: Callable[[int], int], value: int) -> int:
    return func(value)

print(apply(lambda x: x * 2, 5))   # 10

# 복잡한 중첩 타입
UserDB = dict[int, dict[str, str | int]]

def create_user_db() -> UserDB:
    return {
        1: {"name": "Alice", "age": 30},
        2: {"name": "Bob", "age": 25},
    }

db = create_user_db()
print(db[1])   # {'name': 'Alice', 'age': 30}
실행 결과:
10 {'name': 'Alice', 'age': 30}

Part A — 타입 표현식의 런타임 평가 지연 (PEP 563 / 649)

# from __future__ import annotations 를 쓰면
# 모든 어노테이션이 문자열로 저장됩니다 (lazy evaluation)
from __future__ import annotations

def add(a: int, b: int) -> int:
    return a + b

print(add.__annotations__)
# {'a': 'int', 'b': 'int', 'return': 'int'}
# — 실제 int 클래스가 아니라 문자열 'int'
실행 결과:
{'a': 'int', 'b': 'int', 'return': 'int'}

# __future__ 없이도 어노테이션이 lazy하게 평가됩니다
# annotationlib.get_annotations()로 평가 시점을 제어할 수 있습니다

# Python 3.14+ 예시:
# import annotationlib

# annotations = annotationlib.get_annotations(

#     format=annotationlib.Format.VALUE       # 실제 타입 객체

PEP 563 vs 649 논쟁: PEP 563 (from __future__ import annotations)은 어노테이션을 항상 문자열로 저장합니다. Pydantic 같은 런타임에 어노테이션을 활용하는 라이브러리들이 심각한 호환성 문제를 겪었습니다. PEP 649는 "지연 평가(lazy evaluation)"로 타협 — 필요할 때만 평가합니다. Python 3.14에서 PEP 649가 채택되었고, PEP 563은 최종적으로 취소되었습니다.

Part B — 공변성·반공변성 이론

공변성(Covariance): Dog이 Animal의 서브타입이면, Producer[Dog]도 Producer[Animal]의 서브타입입니다. "생산자"는 공변 — 더 구체적인 것을 생산해도 더 일반적인 것을 기대하는 곳에 쓸 수 있습니다.

Part B — 구조적 타이핑 vs 명목적 타이핑 이론

명목적 타이핑(Nominal Typing): Java, C# 방식. 타입 이름으로 호환성을 결정합니다. class Dog extends Animal처럼 명시적 선언이 필요합니다. "네가 누구인지"가 중요합니다.

ABC vs Protocol 선택 기준

언제 ABC를 쓰는가:
  • 공통 구현(concrete method)을 기반 클래스에서 제공하고 싶을 때
  • 서브클래스가 반드시 추상 메서드를 구현하도록 강제해야 할 때 (인스턴스 생성 시점에 검사)
  • 내가 만드는 클래스 계층구조가 명확할 때
언제 Protocol을 쓰는가:
  • 서드파티 클래스나 내가 수정할 수 없는 클래스를 타입으로 표현할 때
  • "오리 타이핑"을 타입으로 문서화하고 싶을 때
  • 상속 계층 없이 유연하게 타입을 정의하고 싶을 때

def _create_init(fields_list):
    params = ", ".join(f"{f.name}: {f.type.__name__}" for f in fields_list)
    body_lines = [f"    self.{f.name} = {f.name}" for f in fields_list]
    body = "\n".join(body_lines)
    source = f"def __init__(self, {params}):\n{body}"
    return source

from typing import TypeVar, Generic

T = TypeVar("T")

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        if not self._items:
            raise IndexError("Stack이 비어있습니다")
        return self._items.pop()

    def peek(self) -> T | None:
        return self._items[-1] if self._items else None

    def is_empty(self) -> bool:
        return len(self._items) == 0

    def __len__(self) -> int:
        return len(self._items)

class BankAccount:
    owner: str
    balance: float
    account_number: str
    transactions: list[float] = field(default_factory=list)

        if self.balance < 0:
            raise ValueError("초기 잔액은 0 이상이어야 합니다")
        if not (self.account_number.isdigit() and len(self.account_number) == 10):
            raise ValueError("계좌번호는 10자리 숫자여야 합니다")

        if amount <= 0:
            raise ValueError("입금액은 양수여야 합니다")
        self.balance += amount
        self.transactions.append(amount)

    def withdraw(self, amount: float) -> None:
        if amount <= 0:
            raise ValueError("출금액은 양수여야 합니다")
        if amount > self.balance:
            raise ValueError("잔액 부족")
        self.balance -= amount
        self.transactions.append(-amount)

    def to_dict(self) -> dict:
        return {
            "owner": self.owner,
            "balance": self.balance,
            "account_number": self.account_number,
            "transactions": self.transactions,
        }

    def to_json(self) -> str:
        return json.dumps(self.to_dict(), ensure_ascii=False)

import functools
import inspect

def enforce_types(func):
    ann = func.__annotations__
    sig = inspect.signature(func)
    param_types = {
        name: ann[name]
        for name in sig.parameters
        if name in ann
    }
    return_type = ann.get("return")

    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        bound = sig.bind(*args, **kwargs)
        bound.apply_defaults()
        for name, value in bound.arguments.items():
            if name in param_types:
                expected = param_types[name]
                if not isinstance(value, expected):
                    raise TypeError(
                        f"{func.__name__}()의 {name}는 {expected.__name__}여야 합니다, "
                        f"{type(value).__name__}이 전달되었습니다"
                    )
        result = func(*args, **kwargs)
        if return_type is not None and not isinstance(result, return_type):
            raise TypeError(
                f"{func.__name__}()의 반환값은 {return_type.__name__}여야 합니다"
            )
        return result
    return wrapper

class ProtocolMeta(type):
    def __instancecheck__(cls, instance):
        attrs = getattr(cls, "__protocol_attrs__", frozenset())
        return all(hasattr(instance, attr) for attr in attrs)

    def __subclasscheck__(cls, subclass):
        attrs = getattr(cls, "__protocol_attrs__", frozenset())
        return all(
            any(attr in base.__dict__ for base in subclass.__mro__)
            for attr in attrs
        )

이번 주 정리

핵심 3가지:
  1. 타입 힌트는 런타임에 무시됩니다. __annotations__ 딕셔너리에 저장되지만, Python 인터프리터는 타입 검사를 하지 않습니다. mypy, pyright 같은 도구의 몫입니다.
  2. Protocol은 구조적 서브타이핑. 상속 선언 없이 "어떤 메서드를 가졌는가"로 호환성을 판단합니다. 외부 라이브러리 클래스를 타입으로 표현할 때 핵심입니다.
  3. dataclass는 코드 생성. __annotations__를 읽어 exec()로 __init__을 동적 생성합니다. mutable 기본값은 반드시 field(default_factory=...)를 쓰세요.

용어 사전 (22개)

타입 힌트 (Type Hint)

Python 3.5+에서 도입된 선택적 타입 표기. 런타임에 무시되며, __annotations__ 딕셔너리에 저장됨. 정적 분석 도구와 IDE가 활용.

__annotations__

함수·클래스·모듈이 자동으로 가지는 딕셔너리. 타입 힌트가 저장되는 곳. Python 3.14+에서는 annotationlib.get_annotations()로 접근 권장.

PEP 484

Python 타입 힌트 시스템의 출발점 (Python 3.5, 2015). Gradual Typing을 채택 — 타입 힌트가 없는 코드는 Any 타입으로 취급.

PEP 585

Python 3.9+에서 내장 컬렉션(list, dict, tuple)을 제네릭으로 직접 사용 가능하게 함. List[int] → list[int].

PEP 604

Python 3.10+에서 X | Y 유니온 문법 도입. Union[X, Y]의 간결한 대체. X | None이 Optional[X]와 동일.

Optional[X]

Union[X, None]의 축약. 값이 있거나 None일 수 있음. Python 3.10+에서는 X | None으로 대체 가능.

Union[X, Y]

X 또는 Y 중 하나의 타입임을 표현. Python 3.10+에서는 X | Y가 권장.

TypeVar

타입 변수. 제네릭 함수·클래스에서 타입 매개변수를 표현. T = TypeVar("T"). bound로 상한 타입 지정 가능.

Generic[T]

타입 매개변수를 가지는 클래스의 기반. class Stack(Generic[T]):. Stack[int]처럼 구체 타입으로 인스턴스화 가능.

Protocol (PEP 544)

Python 3.8+. 구조적 서브타이핑 지원. 상속 선언 없이 특정 메서드를 가진 타입이면 호환. "오리 타이핑"의 공식화.

runtime_checkable

Protocol에 붙이는 데코레이터. isinstance()로 런타임 검사 가능하게 함. 메서드 이름 존재 여부만 검사, 시그니처는 검사 안 함.

ABC (Abstract Base Class)

명목적 서브타이핑. abc.ABC를 상속하고 @abstractmethod로 추상 메서드 선언. 구현 강제, 공통 구현 제공 가능.

@abstractmethod

ABC에서 서브클래스가 반드시 구현해야 하는 메서드 표시. 구현 없이 인스턴스를 만들면 TypeError 발생.

구조적 타이핑 (Structural Typing)

이름이 아닌 구조(메서드, 속성)로 호환성 결정. Protocol, TypeScript, Go 방식. "무엇을 할 수 있는가"가 중요.

명목적 타이핑 (Nominal Typing)

이름(상속 관계)으로 호환성 결정. Java, C#, ABC 방식. "누구에서 왔는가"가 중요.

@dataclass

Python 3.7+. 클래스 어노테이션을 읽어 __init__, __repr__, __eq__를 자동 생성하는 데코레이터.

field(default_factory=...)

dataclass에서 mutable 기본값(list, dict 등)을 안전하게 지정하는 방법. 인스턴스마다 독립된 객체를 생성.

frozen=True

dataclass 옵션. 인스턴스를 불변으로 만들고 __hash__를 자동 생성. 딕셔너리 키나 set 원소로 사용 가능.

Gradual Typing

타입 힌트가 있는 코드와 없는 코드가 공존하는 타입 시스템. 타입 없는 부분은 Any로 취급. PEP 484의 핵심 설계 원칙.

mypy

Python 타입 힌트를 검사하는 정적 분석 도구. 실행 없이 타입 오류를 찾아냄. Gradual Typing 기반 — 힌트 없는 코드는 경고 억제 가능.

공변성 (Covariance)

타입 매개변수의 상속 관계가 Generic 클래스에도 그대로 유지됨. covariant=True TypeVar. "생산자"에 적합.

import time

# 동기 방식 — 순서대로, 기다리면서
def fetch_data(url: str) -> dict:
    time.sleep(1)   # 네트워크 대기
    return {"data": "..."}

# 3개의 API를 순서대로 호출하면 — 3초 소요
start = time.time()
result1 = fetch_data("https://api1.example.com")
result2 = fetch_data("https://api2.example.com")
result3 = fetch_data("https://api3.example.com")
print(f"소요 시간: {time.time() - start:.1f}초")   # 3.0초

소요 시간: 3.0초 
import asyncio

# 비동기 방식 — 기다리는 동안 다른 일을 함
async def fetch_data_async(url: str) -> dict:
    await asyncio.sleep(1)   # 기다리는 동안 다른 코루틴 실행
    return {"data": f"from {url}"}

# 3개를 동시에! — 1초 소요
async def main():
    start = asyncio.get_event_loop().time()
    results = await asyncio.gather(
        fetch_data_async("https://api1.example.com"),
        fetch_data_async("https://api2.example.com"),
        fetch_data_async("https://api3.example.com"),
    )
    print(f"소요 시간: {asyncio.get_event_loop().time() - start:.1f}초")
    return results

asyncio.run(main())   # 1.0초!

  • asyncio.gather(), asyncio.create_task()
  • Awaitable, Coroutine, Future, Task — 타입 계층
  • async for, async with — 비동기 이터레이터와 컨텍스트 매니저
