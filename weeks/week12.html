이번 주 이야기

지난 주 리스트, 딕셔너리, 함수로 여러 데이터를 관리했습니다. 그런데 "강아지"를 표현하려면 어떻게 할까요?

Python 
# 함수 + 딕셔너리로 강아지를 표현하면
dog1 = {"name": "바둑이", "breed": "진돗개", "age": 3}
dog2 = {"name": "초코", "breed": "포메라니안", "age": 1}

def bark(dog):
    print(f"{dog['name']}이 짖습니다: 왈왈!")

def get_older(dog):
    dog["age"] += 1

bark(dog1)       # 바둑이이 짖습니다: 왈왈!
get_older(dog1)
print(dog1["age"])  # 4
실행 결과:
바둑이이 짖습니다: 왈왈! 4

이 방식의 문제는 상태(데이터)와 행동(함수)이 분리되어 있다는 점입니다. bark(dog1)처럼 데이터를 함수에 넣어줘야 합니다. 강아지가 100마리라면? bark_special(dog1), bark_loud(dog2)... 함수와 딕셔너리 사이의 관계는 아무도 보장하지 않습니다.

클래스의 핵심 아이디어:
상태(데이터)와 행동(메서드)을 하나의 객체 안에 묶는다.
dog1.bark() — 강아지 스스로 짖습니다.

왜 클래스가 필요한가? — 상태와 행동을 묶는 이유

초급

비유: 강아지 설계도

클래스 = 설계도, 인스턴스 = 실제 강아지
"강아지"라는 개념(설계도)은 하나지만, 실제 강아지(인스턴스)는 여러 마리일 수 있습니다.
설계도에는 "이름이 있다", "짖을 수 있다"고 적혀 있고, 각 강아지마다 이름이 달라집니다.
Python 
class Dog:
    # __init__: 강아지가 태어날 때 이름과 품종을 기억시킴
    def __init__(self, name, breed):
        self.name = name
        self.breed = breed
        self.age = 0

    def bark(self):
        print(f"{self.name}: 왈왈!")

    def get_older(self):
        self.age += 1
        print(f"{self.name} 나이: {self.age}살")

# 설계도(Dog)로 실제 강아지 만들기
dog1 = Dog("바둑이", "진돗개")
dog2 = Dog("초코", "포메라니안")

dog1.bark()           # 바둑이: 왈왈!
dog2.bark()           # 초코: 왈왈!
dog1.get_older()      # 바둑이 나이: 1살
print(dog1.breed)     # 진돗개
실행 결과:
바둑이: 왈왈! 초코: 왈왈! 바둑이 나이: 1살 진돗개

클래스가 없으면 생기는 문제

딕셔너리 + 함수 방식에는 세 가지 치명적 문제가 있습니다:

  1. 무결성 없음 — dog1["age"] = -5를 막을 수 없습니다.
  2. 연결 보장 없음 — bark(dog1)에서 dog1이 딕셔너리인지 아무도 확인하지 않습니다.
  3. 확장이 어려움 — "특수한 경찰견"처럼 Dog를 확장하려면 함수를 모두 복사해야 합니다.
Python 
class Dog:
    def __init__(self, name, breed, age=0):
        self.name = name
        self.breed = breed
        if age < 0:
            raise ValueError("나이는 0 이상이어야 합니다.")
        self.age = age

    def bark(self):
        print(f"{self.name}: 왈왈!")

# 상속으로 경찰견 확장
class PoliceDog(Dog):
    def __init__(self, name, badge_number):
        super().__init__(name, breed="독일 셰퍼드")
        self.badge_number = badge_number

    def arrest(self, suspect):
        print(f"경찰견 {self.name}(배지 #{self.badge_number}): {suspect} 체포!")

rex = PoliceDog("렉스", 007)
rex.bark()           # 렉스: 왈왈!  (Dog.bark 상속)
rex.arrest("범인")   # 경찰견 렉스(배지 #7): 범인 체포!

print(isinstance(rex, PoliceDog))  # True
print(isinstance(rex, Dog))        # True — 상속 관계 인식!
실행 결과:
렉스: 왈왈! 경찰견 렉스(배지 #7): 범인 체포! True True
isinstance() vs type()
isinstance(rex, Dog)는 상속 계층 전체를 확인합니다.
type(rex) is Dog는 정확히 Dog인지만 확인합니다 — False.
실무에서는 거의 항상 isinstance()를 사용합니다.

Python 공식 문서에서는: 클래스는 네임스페이스다

Python 원서(tutorial/classes)에서 클래스를 이렇게 정의합니다: "클래스는 데이터와 기능을 함께 묶는 방법을 제공합니다. 새 클래스를 만드는 것은 객체의 새 형(type)을 만들어서, 그 형의 새 인스턴스를 만들 수 있도록 합니다."

클래스 정의가 실행될 때 내부적으로 일어나는 일:

Python 
class Dog:
    species = "Canis lupus familiaris"  # 클래스 변수 — Dog.__dict__에 저장

    def __init__(self, name):
        self.name = name  # 인스턴스 변수 — instance.__dict__에 저장

d = Dog("바둑이")

# 클래스 네임스페이스
print(Dog.__dict__.keys())
# dict_keys(['__module__', 'species', '__init__', '__dict__', '__weakref__', '__doc__'])

# 인스턴스 네임스페이스
print(d.__dict__)
# {'name': '바둑이'}

# 속성 탐색 순서: instance.__dict__ → class.__dict__ → 상위 클래스 순
print(d.species)       # 'Canis lupus familiaris' (클래스 변수 상속)
print(d.name)          # '바둑이' (인스턴스 변수)

# 인스턴스 변수가 클래스 변수를 가림(Shadow)
d.species = "개"       # instance.__dict__에 추가
print(d.species)       # '개' (인스턴스 우선)
print(Dog.species)     # 'Canis lupus familiaris' (클래스 변수 그대로)
실행 결과:
dict_keys(['__module__', 'species', '__init__', '__dict__', '__weakref__', '__doc__']) {'name': '바둑이'} Canis lupus familiaris 바둑이 개 Canis lupus familiaris

Part B: Prototype 기반 vs Class 기반 OOP

OOP에는 두 가지 주요 패러다임이 있습니다:

  • Class 기반 (Python, Java, C++): 클래스가 인스턴스의 청사진. 인스턴스는 클래스로부터 메서드를 상속.
  • Prototype 기반 (JavaScript ES5 이전, Self 언어): 클래스 없음. 객체가 다른 객체를 직접 프로토타입으로 참조. 메서드는 프로토타입 체인을 따라 탐색.
JavaScript (Prototype 기반 비교) 
# JavaScript (참고 — 실행 불가)
# const dog = { name: "바둑이", bark: function() { console.log(this.name + ": 왈왈!"); } };
# const policeDog = Object.create(dog);  // dog를 프로토타입으로 설정
# policeDog.badge = 7;
# policeDog.bark();  // "바둑이: 왈왈!" — 프로토타입 체인에서 bark 탐색

# Python은 class 기반이지만 내부적으로 MRO(프로토타입 체인과 유사한 개념)를 사용
# Self 언어(1986): 모든 것이 객체, 클래스 없음, 프로토타입만 존재
# Python의 type과 object 관계는 이 경계를 흐리게 만듦:
print(isinstance(type, object))   # True — type도 object의 인스턴스
print(isinstance(object, type))   # True — object도 type의 인스턴스 (순환!)
실행 결과:
True True

Python의 type과 object의 순환 관계는 CPython 내부에서 C 레벨에서 수동으로 부트스트랩됩니다. 이는 Self 언어의 "모든 것은 객체" 철학과 Smalltalk의 Metaclass 개념의 영향을 받았습니다.

초급

비유: 공장과 인테리어

__new__ = 빈 공간 만들기, __init__ = 인테리어
집을 지을 때 먼저 건물 골조(공간)를 세우고(__new__),
그 다음 벽지·가구 등을 채웁니다(__init__).

우리가 보통 쓰는 __init__은 인테리어 담당입니다. 공간을 만드는 것은 __new__가 알아서 합니다.

# 2단계: Dog.__init__(obj, "바둑이") — 초기화

class Weird:
    def __new__(cls, value):
        print("__new__ 호출됨")
        # Dog 대신 str을 반환하면?
        return "이건 Weird 인스턴스가 아님"

print(type(w))    # <class 'str'>

언제 __new__를 오버라이드하는가?

대부분은 __init__만으로 충분합니다. __new__가 필요한 경우:

  • 불변 타입 서브클래싱 — int, str, tuple은 __init__으로 값을 바꿀 수 없습니다.
  • 싱글톤 패턴 — 인스턴스가 항상 하나임을 보장할 때.
  • 메타클래스 — 클래스를 만드는 클래스.
Python 
# 불변 타입(int)을 서브클래싱: __new__에서 값을 설정해야 함
class PositiveInt(int):
    def __new__(cls, value):
        if value <= 0:
            raise ValueError(f"{value}는 양수여야 합니다.")
        return super().__new__(cls, value)  # int의 값을 여기서 설정

n = PositiveInt(5)
print(n, type(n))   # 5 <class 'PositiveInt'>
print(n + 3)        # 8 — int처럼 덧셈 가능

try:
    bad = PositiveInt(-1)
except ValueError as e:
    print(e)         # -1는 양수여야 합니다.
실행 결과:
5 <class '__main__.PositiveInt'> 8 -1는 양수여야 합니다.

싱글톤 패턴과 __new__

Python 
class Singleton:
    _instance = None  # 클래스 변수 — 유일한 인스턴스를 저장

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            print("첫 번째 인스턴스 생성")
            cls._instance = super().__new__(cls)
        else:
            print("기존 인스턴스 반환")
        return cls._instance

    def __init__(self, value):
        self.value = value

a = Singleton(1)   # 첫 번째 인스턴스 생성
b = Singleton(2)   # 기존 인스턴스 반환

print(a is b)      # True — 같은 객체!
print(a.value)     # 2 — __init__이 두 번 호출되어 value가 덮어씌워짐
print(b.value)     # 2
실행 결과:
첫 번째 인스턴스 생성 기존 인스턴스 반환 True 2 2

Part A: CPython의 tp_new / tp_init

CPython에서 Dog("바둑이") 호출 시 내부 흐름:

Python (CPython 내부 흐름 추적) 
# CPython Objects/typeobject.c type_call() 함수 (단순화):
#
# static PyObject *
# type_call(PyTypeObject *type, PyObject *args, PyObject *kwds)
# {
#     PyObject *obj;
#     obj = type->tp_new(type, args, kwds);  /* __new__ 호출 */
#     if (obj != NULL) {
#         if (Py_TYPE(obj) — type의 서브타입이면) {
#             type->tp_init(obj, args, kwds); /* __init__ 호출 */
#         }
#     }
#     return obj;
# }

# Python 레벨에서 동일한 흐름을 직접 구현:
class Dog:
    def __init__(self, name):
        self.name = name

# 수동으로 type_call 흐름 재현
obj = Dog.__new__(Dog, "바둑이")   # tp_new
print(type(obj), hasattr(obj, 'name'))  # <class 'Dog'> False (아직 초기화 전)
Dog.__init__(obj, "바둑이")             # tp_init
print(obj.name)                         # 바둑이

해시 규칙과 비교 연산자 전체

Python Data Model의 해시 규칙:

  1. a == b이면 반드시 hash(a) == hash(b)여야 합니다.
  2. 역은 성립하지 않습니다 — 해시 충돌이 가능합니다.
  3. 가변 객체(Mutable)는 해시를 구현하면 안 됩니다 — 값이 바뀌면 해시도 바뀌어야 하는데, 이미 딕셔너리에 들어가 있으면 찾을 수 없게 됩니다.
Python 
from functools import total_ordering

@total_ordering  # __eq__와 __lt__ 하나만 정의하면 나머지 자동 생성
class Version:
    def __init__(self, major, minor, patch):
        self.major = major
        self.minor = minor
        self.patch = patch

    def __eq__(self, other):
        return (self.major, self.minor, self.patch) == \
               (other.major, other.minor, other.patch)

    def __lt__(self, other):
        return (self.major, self.minor, self.patch) < \
               (other.major, other.minor, other.patch)

    def __hash__(self):
        return hash((self.major, self.minor, self.patch))

    def __repr__(self):
        return f"Version({self.major}.{self.minor}.{self.patch})"

v1 = Version(1, 0, 0)
v2 = Version(2, 0, 0)
v3 = Version(1, 0, 0)

print(v1 < v2)     # True
print(v1 > v2)     # False  — total_ordering이 자동 생성
print(v1 == v3)    # True
print(v1 <= v2)    # True   — 자동 생성
print(sorted([v2, v1, v3]))  # [Version(1.0.0), Version(1.0.0), Version(2.0.0)]

merge에서 각 리스트의 첫 번째 원소 중 다른 리스트의 끝에 나타나지 않는 것을 선택합니다.

# L(B) = [B] + merge([A, object], [A]) = [B, A, object]
# L(C) = [C] + merge([A, object], [A]) = [C, A, object]
# L(D) = [D] + merge([B, A, object], [C, A, object], [B, C])

Part A: CPython의 MRO 구현

Python (C3 알고리즘 직접 구현) 
# CPython Objects/typeobject.c mro_implementation()은 C3을 구현
# 여기서는 동일한 알고리즘을 Python으로 재현

def c3_linearize(cls_name, bases_mro):
    """C3 선형화. bases_mro: 각 베이스의 MRO + 베이스 목록"""
    if not any(bases_mro):
        return []

    for candidate in bases_mro[0]:  # 각 리스트의 첫 번째 원소 탐색
        # 다른 리스트의 꼬리 부분에 candidate가 없으면 선택
        if all(candidate not in tail[1:] for tail in bases_mro if tail):
            return [candidate] + c3_linearize(
                cls_name,
                [lst[1:] if lst and lst[0] == candidate else lst
                 for lst in bases_mro]
            )
    raise TypeError(f"Cannot create a consistent MRO for {cls_name}")

# 실제 Python MRO와 비교

import math

class Vector:
    """2D 벡터 — 던더 메서드 완전 구현 예제"""

    def __init__(self, x, y):
        self.x = float(x)
        self.y = float(y)

    # ── 표현 ──────────────────────────────────────
    def __repr__(self):
        return f"Vector({self.x!r}, {self.y!r})"

    def __str__(self):
        return f"({self.x}, {self.y})"

    # ── 비교 ──────────────────────────────────────
    def __eq__(self, other):
        if not isinstance(other, Vector):
            return NotImplemented
        return self.x == other.x and self.y == other.y

    def __hash__(self):
        return hash((self.x, self.y))

    # ── 산술 ──────────────────────────────────────
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __sub__(self, other):
        return Vector(self.x - other.x, self.y - other.y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)

    def __rmul__(self, scalar):  # scalar * vector
        return self.__mul__(scalar)

    def __neg__(self):
        return Vector(-self.x, -self.y)

    def __abs__(self):   # abs(v) = 크기(magnitude)
        return math.sqrt(self.x**2 + self.y**2)

    # ── 논리/컨테이너 ──────────────────────────────
    def __bool__(self):   # bool(v) — 영벡터면 False
        return bool(abs(self))

    def __len__(self):    # len(v) — 차원 수
        return 2

    def __getitem__(self, index):  # v[0], v[1]
        return (self.x, self.y)[index]

    def __iter__(self):   # for component in v
        yield self.x
        yield self.y

    def dot(self, other):
        return self.x * other.x + self.y * other.y

        mag = abs(self)
        if mag == 0:
            raise ValueError("영벡터는 정규화할 수 없습니다")
        return Vector(self.x / mag, self.y / mag)

class FrozenPoint:
    _cache = {}

    def __new__(cls, x, y):
        key = (x, y)
        if key not in cls._cache:
            instance = super().__new__(cls)
            cls._cache[key] = instance
        return cls._cache[key]

    def __init__(self, x, y):
        if hasattr(self, '_initialized'):
            return   # 이미 초기화됨 — skip
        # object.__setattr__을 직접 호출 (우리의 __setattr__ bypass)
        object.__setattr__(self, 'x', x)
        object.__setattr__(self, 'y', y)
        object.__setattr__(self, '_initialized', True)

    def __setattr__(self, name, value):
        if hasattr(self, '_initialized'):
            raise AttributeError(f"FrozenPoint는 불변입니다: {name}")
        object.__setattr__(self, name, value)

    def __repr__(self):
        return f"FrozenPoint({self.x}, {self.y})"

    def __hash__(self):
        return hash((self.x, self.y))

    def __eq__(self, other):
        return isinstance(other, FrozenPoint) and \
               self.x == other.x and self.y == other.y

def merge(sequences):
    result = []
    while True:
        sequences = [s for s in sequences if s]  # 빈 리스트 제거
        if not sequences:
            return result
        for seq in sequences:
            candidate = seq[0]
            # candidate가 다른 리스트의 꼬리에 없으면 선택
            if not any(candidate in s[1:] for s in sequences):

                for s in sequences:
                    if s and s[0] == candidate:
                        del s[0]
                break
        else:
            raise TypeError("MRO를 계산할 수 없습니다 (순환 상속)")

def mro(cls):
    if cls is object:
        return [object]
    return [cls] + merge(
        [mro(b) for b in cls.__bases__] + [list(cls.__bases__)]
    )

class SingletonMeta(type):
    _instances = {}

    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            # type.__call__ = __new__ + __init__
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

용어 사전 (25개)

클래스 (Class)

데이터(속성)와 기능(메서드)을 묶는 객체 설계도. Python에서 클래스 자체도 type의 인스턴스인 객체.

인스턴스 (Instance)

클래스로부터 생성된 실제 객체. Dog("바둑이")는 Dog 클래스의 인스턴스.

던더 메서드 (Dunder Method)

Double UNDERscore — __init__, __repr__ 등 앞뒤 이중 밑줄 메서드. Special Method, Magic Method라고도 함. Python Data Model의 핵심.

__new__

클래스의 인스턴스 메모리 공간을 할당하고 반환하는 클래스 메서드. CPython의 tp_new 슬롯. 불변 타입 서브클래싱이나 싱글톤 패턴에 필요.

__init__

이미 만들어진 인스턴스를 초기화하는 메서드. 반환값은 항상 None. "생성자"가 아닌 "초기화자". CPython의 tp_init 슬롯.

__repr__

개발자용 공식 표현. repr(obj), 컨테이너([obj]) 출력, REPL에서 호출. 가능하면 eval(repr(obj)) == obj가 성립하도록 설계.

__str__

사람이 읽기 좋은 표현. str(obj), print(obj), f-string f"{obj}"에서 호출. 없으면 __repr__로 폴백.

__eq__

== 연산자 오버로드. NotImplemented를 반환하면 Python이 반대편 객체에게 위임. 오버라이드 시 __hash__도 반드시 오버라이드.

__hash__

hash() 함수 오버로드. dict/set에서 빠른 검색을 위해 필요. __eq__ 오버라이드 시 자동으로 None이 됨 — unhashable!

MRO (Method Resolution Order)

메서드를 탐색하는 순서. Python 2.3부터 C3 선형화 알고리즘 사용. Class.__mro__로 확인.

C3 선형화

다중 상속에서 MRO를 계산하는 알고리즘. Dylan 언어에서 기원. 단조성(Monotonicity)과 지역 우선순위 순서(Local Precedence Ordering)를 보장.

super()

MRO에서 현재 클래스의 다음을 가리키는 프록시 객체. "부모 클래스"가 아님. Python 3에서 __class__ 셀 변수로 자동 바인딩.

다중 상속 (Multiple Inheritance)

class D(B, C):처럼 여러 베이스 클래스에서 상속. MRO가 메서드 탐색 순서를 결정. 다이아몬드 문제를 C3+super()로 해결.

Context Manager

__enter__와 __exit__를 구현한 객체. with문과 함께 사용. 리소스 획득/해제를 자동화.

__enter__

with EXPR as VAR:에서 VAR에 바인딩될 값을 반환. 리소스 획득 담당.

__exit__

with 블록 종료 시(정상/예외) 호출. (exc_type, exc_val, exc_tb)를 받음. True 반환 시 예외 억제, False 반환 시 예외 전파.

__slots__

클래스 변수로 허용된 속성 이름을 고정. __dict__ 대신 C 수준 배열 사용 → 메모리 절약 + 접근 속도 향상. 고정된 속성 이외의 추가 불가.

PyTypeObject

CPython에서 클래스(타입)를 표현하는 C 구조체. tp_new, tp_init, tp_repr, tp_hash 등 함수 포인터 슬롯을 포함.

tp_new / tp_init

CPython C 레벨에서 __new__와 __init__에 해당하는 함수 포인터. type_call()에서 순서대로 호출됨.

Descriptor Protocol

__get__, __set__, __delete__를 구현한 객체. 클래스 속성으로 사용 시 속성 접근을 가로챔. @property, classmethod, staticmethod의 구현 기반.

Metaclass

클래스를 만드는 클래스. type이 기본 메타클래스. class Foo(metaclass=Meta):로 지정. 클래스 생성 과정을 커스터마이즈.

Exception Chaining

예외에서 다른 예외를 raise할 때 원인 예외를 연결. raise X from Y(명시적, __cause__), raise Xin except(암시적, __context__).

total_ordering

functools의 데코레이터. __eq__와 __lt__ 하나만 정의하면 나머지 비교 연산자(__gt__, __le__, __ge__)를 자동 생성.

NotImplemented

이진 연산 메서드(__eq__, __add__ 등)가 반환하는 특수 싱글톤. "이 타입과의 연산을 처리할 수 없음"을 알리면 Python이 반대편 객체에게 위임.

Zero-cost Exception Handling

Python 3.11+에서 도입. try 블록이 정상 실행되면 오버헤드 0. 예외 발생 시에만 예외 테이블을 조회하는 방식으로 구현.

다음 주 맛보기 — Week 13: 이터레이터 & 제너레이터

이번 주 클래스와 던더 메서드로 Python Data Model의 핵심을 배웠습니다. 특히 __iter__와 __next__를 Vector 클래스에서 살짝 보았습니다. 다음 주에는 이 Iterator Protocol을 완전히 분해합니다.

Python 
# for 루프를 완전히 분해하면:
data = [1, 2, 3]

# 이게 실제로 일어나는 일:
_iter = iter(data)       # data.__iter__() 호출
while True:
    try:
        item = next(_iter)  # _iter.__next__() 호출
        print(item)
    except StopIteration:
        break

# 제너레이터 — 함수가 값을 yield하는 방식
def countdown(n):
    while n > 0:
        yield n         # 여기서 일시정지하고 값을 반환
        n -= 1

for i in countdown(3):
    print(i)
# 3
# 2
# 1

# 무한 수열도 가능 — 메모리에 전부 올리지 않음!
def infinite_counter(start=0):
    n = start
    while True:
        yield n
        n += 1

gen = infinite_counter(10)
print(next(gen))  # 10
print(next(gen))  # 11
print(next(gen))  # 12
실행 결과:
1 2 3 3 2 1 10 11 12
